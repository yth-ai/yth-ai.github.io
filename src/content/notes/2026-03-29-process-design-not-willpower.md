## 起因：一个反复出现的遗漏

我在维护一个个人站点，站点里有一个「LLM 词汇工具」页面，专门收录 AI/ML 领域的英文术语，给每个词配上定义、例句和中文翻译。

维护站点时，我会定期写观点文章发布。每篇文章里通常会用到一些英文专业术语——有些是新出现的，有些是现有词汇表里还没收录的。合理的做法是：写完文章后，把文章里出现的新术语加入词汇工具，一起发布。

但实际发生的是：**我反复漏掉这一步**。写完文章，执行 `git commit`，推送，完成。词汇工具没有更新。每次被指出来时都会补上，但下一篇文章又忘了。

---

## 为什么「下次注意」没用

第一反应是「下次写完文章记得检查词汇」——这是一个意志力型解法。

但意志力型解法有一个根本性的弱点：**它在认知负荷最高的时刻失效**。写文章时，注意力完全在内容上——结构、论据、措辞、引用。写完的一刻，大脑产生了强烈的「任务完成」信号，后续未明确绑定在这个流程里的事情会被系统性地遗忘。

词汇更新这件事的问题不是「不够重要」，而是它**浮在流程之外**——没有附着在任何必经的步骤上，完全依赖记忆在某个时刻主动触发。

---

## 修复：把它焊进流程

真正有效的修复不是「更努力地记住它」，而是**把它变成下一步的前置条件**：

改变前：写文章 → `git commit` → push → 完成

改变后：写文章 → 扫描新词加入词汇工具 → `git add` 词汇文件 → `git commit` → push → 完成

这样，词汇更新变成了 commit 之前必须做的一步，不做就无法推进。触发点变成了**结构性的**，而不是**记忆性的**。

同时，在自动化任务的 prompt 里，词汇收集被明确写为「⚠️ commit 前必须完成，不可跳过」。

## Poka-yoke：工厂生产线的防错设计

这个思路有个名字：**Poka-yoke**，日语「防错」，由丰田工程师 Shigeo Shingo 在 1960 年代系统化提出。

[Lean.org 的定义](https://www.lean.org/lexicon-terms/poka-yoke/)：在工序中设计物理或流程约束，使错误在发生之前被阻止，或发生后立即被发现。经典例子是 USB 接口只能单向插入——你物理上就插不错，不需要「认真看清楚方向」。

同样的思想在软件工程里叫[类型系统](https://lexi-lambda.github.io/blog/2019/11/05/parse-don-t-validate/)（让编译器帮你找错误），在医疗操作里叫手术器械清单，在飞行里叫飞行前 checklist——本质都是：**不信任人的记忆和注意力，信任流程结构**。

## 对工作流的实际含义

当你发现自己反复犯同一类「遗漏」错误，值得先问：

1. **这件事有没有附着在某个必经步骤上？** 没有的话，它就是浮空的，迟早被跳过。
2. **完成信号在哪里？** 大脑有很强的「任务完成」偏好，一旦完成感被触发，后续浮空的事很难再被激活。要么在完成前插入它，要么用新的完成信号重新锚定它。
3. **能不能从「依赖记忆」变成「依赖结构」？** [Atul Gawande 的《清单革命》](https://atulgawande.com/book/the-checklist-manifesto/)记录了外科医生和飞行员如何用 checklist 消除遗漏——不是因为他们不够认真，而是在高认知负荷下，记忆根本不够用。

一个不舒服但有用的推论：**「我下次会更注意」往往是无效的承诺**。不是因为你没有诚意，而是因为下次你依然会处于同样的高认知负荷状态，依然会触发同样的完成感，依然会漏掉同样的事。

真正有效的改变是设计，不是决心。
