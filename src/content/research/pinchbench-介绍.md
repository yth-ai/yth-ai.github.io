# PinchBench 系统介绍：专为 AI Agent 设计的智能体基准测试

---

## 1. 概览

PinchBench 是由 Kilo AI 团队于 2026 年 3 月推出的开源 AI Agent 基准测试系统，专门评估大语言模型（LLM）作为 OpenClaw 编码代理"大脑"的任务执行能力。与传统知识问答或数学推理评测不同，PinchBench 聚焦于真实工作流中的任务完成能力，从成功率（Success Rate）、速度（Speed）和成本（Cost）三个维度对全球主流大模型进行横向对比。

发布后，OpenClaw 创始人 Peter Steinberger 亲自推荐该榜单为开发者选择适配模型的权威参考。截至 2026 年 3 月中旬，PinchBench 已评测超过 40 款主流大模型，所有任务定义与评分逻辑完全开源。

> **核心定位**: 不测"知道什么"，测"能干什么"——衡量 LLM 在 Agent 场景下的真实执行力。

| 项目 | 详情 |

## 2. 背景：为什么需要 PinchBench

### 2.1 OpenClaw 的兴起与模型选择难题

OpenClaw 是 2026 年初爆火的开源 AI Agent 框架（昵称"龙虾"），它本身不具备"做事"能力，真正执行任务的是底层接入的大语言模型。用户在 OpenClaw 上的花费本质上是 LLM 的 API 调用费用。随着 OpenClaw 普及，开发者面临核心难题：面对数十款大模型，哪一个最适合驱动 OpenClaw？

### 2.2 传统评测的局限性

传统大模型评测（如 MMLU、HumanEval、MATH）主要衡量知识储备、推理能力或代码生成质量，无法直接反映 Agent 场景下的实际表现。Agent 场景的独特需求包括：

- **工具使用（Tool Use）**：模型能否选择正确的工具并填入正确参数
- **多步推理（Multi-step Reasoning）**：能否将多个动作串联为完整工作流
- **现实世界的混乱性**：能否处理模糊指令和不完整信息
- **实际结果产出**：是否真正创建了文件、发送了邮件或安排了会议

PinchBench 正是为填补这一评测空白而诞生。

## 3. 技术架构

### 3.1 系统组成

PinchBench 的完整系统由三个开源仓库组成：

| 仓库 | 功能 | 技术栈 |
|------|------|--------|
| `pinchbench/skill` | 基准运行器、任务定义和评分逻辑 | Python |
| `pinchbench/leaderboard` | 排行榜网站前端 | Next.js, React, Tailwind |
| `pinchbench/api` | 排行榜数据后端 API | Cloudflare Workers |

### 3.2 运行方式

```bash
# 克隆仓库
git clone https://github.com/pinchbench/skill.git
cd skill

# 运行全部任务
./scripts/run.sh --model openrouter/anthropic/claude-sonnet-4

# 运行指定任务
./scripts/run.sh --model openrouter/openai/gpt-4o --suite task_01_calendar,task_02_stock
```

评测环境通过 `Dockerfile.benchmark` 标准化，确保所有模型在相同条件下运行。

### 3.3 版本控制机制

- 每个基准版本由运行时仓库的 **Git commit hash** 唯一标识
- 系统区分"实质性变更"（修改任务提示或评分逻辑）和"非实质性变更"（文档、CI 等）
- 实质性变更开启新一代基准，不同代际的结果分开保存
- 旧版本及其结果永不删除，可通过版本选择器查看历史排行榜

## 4. 任务设计

### 4.1 任务定义格式

每个测试任务以 **Markdown + YAML 前置元数据** 存储在 `tasks/` 目录中，包含五个核心要素：

| 要素 | 说明 |
|------|------|
| **Prompt** | 发送给 Agent 的完整指令，模拟真实用户请求 |
| **Expected Behavior** | 描述成功完成任务的方法路径和关键决策 |
| **Grading Criteria** | 原子化、可验证的成功标准检查清单 |
| **Automated Checks** | 基于工作区文件和交互记录评分的 Python 函数 |
| **LLM Judge Rubric** | 供 Claude Opus 对定性标准评分的详细细则 |

YAML 元数据字段包括：`id`（唯一标识，`task_XX_name` 格式）、`name`（显示名称）、`category`（所属类别）、`grading_type`（`automated` / `llm_judge` / `hybrid`）、`timeout_seconds`（超时时间）、`workspace_files`（预置文件）。

### 4.2 23 项标准化任务（7 大类别）

| 类别 | 典型任务 | 测试能力 |
|------|----------|----------|
| **生产力** | 日历事件创建、每日摘要 | 时间解析、日程安排 |
| **研究** | 股票价格查询、会议研究、市场分析 | 网络搜索、数据提取、综合分析 |
| **写作** | 博客文章、邮件撰写、拟人化写作 | 内容生成、语气调整、格式化 |
| **编程** | 天气脚本、文件结构创建 | 代码生成、文件操作 |
| **分析** | 电子表格、PDF 解读、文档摘要 | 数据处理、信息提取 |
| **电子邮件** | 邮件分类、邮件搜索 | 收件箱管理、邮件过滤 |
| **记忆与技能** | 上下文检索、ClawHub 集成 | 长期记忆、生态系统集成 |

### 4.3 典型任务深度解析

**任务 No.21：OpenClaw 报告理解（自动化评分）**

要求 Agent 阅读 `openclaw_report.pdf` 并回答 8 个具体问题，如"过滤前社区内的技能总共有多少？"（答案：5705）。评分极其严格——答对 7 道题但最后一题数字错一位，得分为 0。考察结构化与非结构化信息提取能力。

**任务 No.16：竞争市场研究（混合评分）**

要求生成企业级 APM 市场竞争分析报告，涉及识别头部厂商、分析差异化定位、梳理定价模式。自动化部分检查"是否涵盖 5+ 竞品"、"是否包含对比表格"；主观部分（研究质量、洞察力）由 Claude Opus 打分。

### 4.4 优质任务的四项标准

- **真实性（Realistic）**：真实用户会要求 Agent 做的事
- **可衡量性（Measurable）**：有明确的成功标准
- **可重现性（Reproducible）**：同一任务产生一致的评分结果
- **挑战性（Challenging）**：测试 Agent 能力，非仅考察知识库

## 5. 三级架构评分机制

### 5.1 自动化验证（AUTOMATED）

- **执行方式**：Python 函数，签名为 `def grade(transcript: list, workspace_path: str) -> dict:`
- **输入**：`transcript`（JSONL 事件列表）+ `workspace_path`（隔离工作区路径）
- **输出**：评分标准名称 → 0.0-1.0 分数的字典
- **适用场景**：文件存在性、关键词匹配、数字精准匹配、日期格式验证
- **约束**：仅允许标准库 + pathlib，支持部分得分

### 5.2 LLM 评判（LLM JUDGE）

- **执行方式**：调用 Claude Opus 作为"评委"
- **评分等级**：1.0（完美）→ 0.75（良好）→ 0.5（可接受）→ 0.25（差）→ 0.0（失败）
- **适用场景**：内容质量、分析深度、风格一致性、商业趋势判断
- **权重机制**：各评判标准权重总和 = 100%

### 5.3 混合评分（HYBRID）

将两种方式结合：客观部分由脚本检查，主观部分由 LLM 打分。例如市场研究任务中，自动检查竞品数量和表格结构，LLM 评估研究质量和洞察深度。

| 评分类型 | 适用任务示例 |
|----------|-------------|
| 自动化 | 日历创建、股票查询、天气脚本、文件结构 |
| LLM 评判 | 博文写作、文档摘要、会议研究、邮件草拟 |
| 混合 | 多步 API 工作流、图像生成、市场研究、Excel 分析 |

> 两次排名差异源于版本迭代（新增模型、更新任务/评分标准），体现了 PinchBench 实时更新的特点。

## 7. 关键发现与洞察

### 7.1 轻量模型的逆袭

PinchBench 最令人意外的发现之一是：针对 Agent 优化的轻量模型往往超越传统旗舰。Gemini 3 Flash（轻量版）以 95.1% 成功率力压自家 Pro 版本和 Claude、GPT 系列。这表明在 Agent 场景中，参数规模不是决定性因素——快速响应、精准工具调用、高效的多步推理才是关键。

### 7.2 国产模型的强势表现

MiniMax M2.1（成功率 93.6%，速度冠军）和 Kimi K2.5（93.4%）在成功率排行中紧随 Gemini 3 Flash，跻身全球前三。在综合考量成功率与价格的推荐榜单中，8 个入围模型有 4 个来自中国。这说明国产大模型在 Agent 垂直领域已具备国际一线竞争力。

### 7.3 成本控制的重要性

Agent 场景的 token 消耗远超普通对话。一次复杂任务可能涉及成百上千次 API 调用，模糊指令会导致更多重试，交互次数的线性增长带来 token 消耗的指数级上升。PinchBench 数据显示，排名第十的模型 token 成本可能是排名第一的 12 倍以上。有重度用户月花费达 1000-2000 美元，因此成本维度对实际选型至关重要。

### 7.4 没有"万能模型"

PinchBench 揭示了一个重要事实：没有单一模型能在所有维度称霸。成功率冠军（Gemini 3 Flash）既非速度最快也非成本最低；速度冠军（MiniMax M2.5）在成功率上不敌前三；成本最低的 GPT-5 Nano 成功率仅 85.8%。开发者必须根据自身场景在三个维度间做出权衡。

## 8. 场景化选型建议

基于 PinchBench 数据，不同场景的推荐策略如下：

| 场景 | 推荐模型 | 理由 |
|------|----------|------|
| **生产环境（追求成功率）** | Gemini 3 Flash | 95.1% 成功率 + $0.72 成本，综合最优 |
| **快速迭代（追求速度）** | MiniMax M2.5 / Gemini 2.0 Flash | 均 ~106 秒完成，节省开发时间 |
| **个人/小团队（追求性价比）** | Gemini 2.5 Flash Lite | $0.05 成本，83.2% 成功率，入门首选 |
| **国产模型偏好** | MiniMax M2.1 / Kimi K2.5 | 成功率 93%+，MiniMax 兼具速度和性价比 |
| **预算极度有限** | GPT-5 Nano | $0.03 成本，85.8% 成功率 |

## 9. 与传统评测的对比

| 对比维度 | 传统评测（MMLU/HumanEval 等） | PinchBench |
|----------|-------------------------------|------------|
| 测试对象 | LLM 的知识/推理/代码能力 | LLM 驱动 Agent 的任务执行能力 |
| 任务类型 | 选择题、数学题、代码补全 | 日历安排、邮件处理、市场研究等真实任务 |
| 评估维度 | 准确率 | 成功率 + 速度 + 成本 |
| 评分方式 | 标准答案匹配 | 自动化 + LLM 评判 + 混合三级架构 |
| 工具使用 | 不涉及 | 核心考察点 |
| 多步推理 | 有限 | 核心考察点 |
| 实际产出验证 | 不涉及 | 检查文件创建、API 调用等实际结果 |

## 10. 局限性与未来展望

### 10.1 当前局限

- **任务覆盖有限**：23 个任务虽涵盖多个类别，但尚无法完整代表所有 Agent 使用场景
- **OpenClaw 绑定**：专为 OpenClaw 设计，其他 Agent 框架的适用性有待验证
- **LLM 评判偏差**：使用 Claude Opus 作为评委可能引入对特定风格的偏好
- **排行榜波动**：实时更新机制导致不同时间点的数据差异较大

### 10.2 未来方向

- 持续扩充任务集，覆盖更多垂直场景
- 社区驱动的任务贡献机制已建立，任何人可通过 PR 提交新任务
- 更多 Agent 框架的适配与评测
- 更精细的评分维度（如安全性、鲁棒性）
