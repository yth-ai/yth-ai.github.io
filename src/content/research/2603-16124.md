## 这篇论文在解决什么问题

Agentic Coding 正在快速发展——Claude Code、Cursor、Copilot 等工具让模型在真实代码仓库中导航、搜索、推理。但评估这类能力时，我们遇到了一个尴尬的局面：**现有基准不够用**。

问题出在两个层面：

**第一，多样性缺失。** 现有的仓库级基准（如 SWE-bench）集中在少数热门开源项目上。这些仓库——Django、Flask、scikit-learn——模型在预训练阶段大概率已经见过，甚至记住了大量实现细节。当一个模型能靠"记忆"回答问题时，我们无法区分它是否真的具备仓库探索能力。

**第二，难度标定不足。** 很多基准中的问题可以通过直接回答（不访问仓库）来解决。如果一个问题不需要工具使用就能答对，那它根本无法测出 Agent 的核心价值。

SWE-QA-Pro 试图同时解决这两个问题。它不仅是一个基准，还附带了一套训练配方——论文的核心论点是：**评估和训练应该是一个闭环**。

## 基准构建：四步流程

### 1. 数据源与分类体系

起点是 SWE-Rebench 中的 168 万条 GitHub Issues。作者用 Qwen3-8B-Embedding 模型计算语义向量，然后做两层层次化 K-Means 聚类（第一层 10 个大类，第二层 50 个子类），再用 GPT-4.1 提取语义标签，人工验证后精简为 **48 个任务子类**。

这一步的目的很明确：建立一个完整的"代码理解任务分类学"，确保基准不会偏向某几类问题（比如只有 bug 修复和 API 用法查询）。48 个子类覆盖了配置管理、数据管道、错误诊断、性能分析等实际工程中的长尾场景。

### 2. 数据合成与采样

关键的设计决策是**不用热门仓库**。训练集从 1,484 个仓库中均匀采样，测试集选择覆盖所有类别的 26 个长尾仓库。所谓"长尾"，指的是 star 数较少、模型预训练语料中出现概率较低的仓库。

数据合成用 Claude Code 在沙盒环境中完成——给它一个仓库和对应的 Issue，让它生成 QA 对。这里有个隐含的假设：Claude Code 的仓库探索能力足够强，可以产出可靠的参考答案。后文会讨论这个假设的局限性。

### 3. 难度校准——最有价值的设计

这是整篇论文最值得关注的技术贡献。

做法如下：让 GPT-4o、Claude Sonnet 4.5、Gemini 2.5 Pro 等模型**不访问仓库**直接回答问题，然后将直接回答的得分与仓库级参考答案做对比。计算跨模型共识的 z-score，把那些"直接回答也能答对"的低难度问题过滤掉。

过滤后，Claude Sonnet 4.5 的直接回答与 Agent 工作流之间出现了约 **13 分的差距**。这个差距就是"必须通过仓库探索才能获取的信息增量"。

这个方法论的价值超越了 SWE-QA-Pro 本身。它提供了一个可复制的范式：**任何 Agentic 基准都应该有"去工具化基线"来标定难度**。如果一个 Agent 基准中大量问题不需要工具就能解决，那这个基准本质上是在测语言模型能力而非 Agent 能力。

### 4. 最终数据集

260 个问题，来自 26 个长尾仓库，每个问题都有可执行环境。数据量不大，但论文论证了质量和代表性比规模更重要。

## Agent 工作流：轻量但有效

SWE-QA-Pro 的 Agent 工作流是标准的 ReAct 循环，但做了两个值得注意的选择：

**抛弃 RAG 预索引。** 不预先构建代码索引或嵌入数据库，而是让模型直接用 Search、View、CommandLine 三个工具实时探索仓库。理由是：预索引会引入额外的工程复杂度，且对长尾仓库的适配成本高。

**长度受限的工具输出。** 每个工具调用返回的内容有长度限制，迫使模型学会精准的查询策略而不是"把整个文件读进来"。

这种设计的哲学是：**让 Agent 的能力来自推理策略，而非信息堆砌**。

## 训练配方：SFT → RLAIF

这是论文的另一个核心贡献。

### 阶段一：SFT

用 Claude Sonnet 4.5 生成 1,000 条高质量多轮对话轨迹（包含工具调用），在 Qwen3-8B 上做监督微调。这一步的目标是**教会模型工具调用的语法和基本策略**。

1,000 条数据量很小，但论文实验表明增加到 1,464 条时提升有限。SFT 的天花板很快到来——它只能教模型"怎么用工具"，但无法教模型"什么时候该停止探索、如何权衡精度和效率"。

### 阶段二：RLAIF

在 SFT 基础上用 GRPO 算法做强化学习。奖励信号来自 LLM-as-a-Judge，从五个维度评分：正确性、完整性、相关性、清晰度、推理质量。

关键发现：**RL 带来的提升和 SFT 是正交的**。仅增加 SFT 数据从 1,000 到 1,464 条只有微小提升，但在 1,000 条 SFT 基础上加 464 条 RL 训练数据则产生了显著的性能跳跃，尤其在正确性和完整性两个维度上。

这个结论对 Agentic Coding 训练有直接的实践意义：**不要无脑堆 SFT 数据，及早引入 RL**。SFT 负责教语法和基础行为模式，RL 负责优化决策策略。

## 实验结果：几个关键数字

| 模型 | 工作流 | 表现概要 |
|------|--------|---------|
| Claude Sonnet 4.5 | Agent | 最高分，工具调用次数最多 |
| Gemini 2.5 Pro | Agent | 调用次数少但仍有竞争力——内部推理补偿了探索不足 |
| GPT-4o | Agent | 被训练后的 8B 模型超越 |
| **SWE-QA-Pro-8B (SFT+RL)** | Agent | **超越 GPT-4o 2.3 分**，接近部分更大的开源模型 |

工具使用分析揭示了一个有趣的分化：

- **Claude Sonnet 4.5** 采取的是"广泛探索"策略，大量调用工具，然后从丰富的上下文中提炼答案。
- **Gemini 2.5 Pro** 更像"精准狙击"，调用次数少但每次命中率高，推测是利用了更强的内部推理能力来减少不必要的探索。
- **8B 模型经过 RL 训练后**，不是简单地增加调用频率，而是学到了更高效的执行策略——用更少的步骤获取关键信息。

## 我的评论

### 这篇论文做对了什么

**难度校准的方法论**是最大的贡献。在 Agentic 系统的评估中，"去工具化基线"应该成为标准配置。SWE-bench 长期被批评的一个点就是模型可以靠记忆作弊，SWE-QA-Pro 用跨模型共识的方式量化了这个问题并做了过滤。

**SFT+RL 的两阶段配方**验证了一个重要假说：对于工具使用和复杂推理行为，SFT 的天花板来得很快，RL 能提供互补的优化信号。8B 模型超越 GPT-4o 是一个有说服力的实验结果。

**长尾仓库的选择**是正确的方向。在评估 Agent 能力时，我们需要的是模型没有预训练记忆的场景，长尾仓库天然满足这个条件。

### 局限性与需要注意的地方

**数据合成的循环依赖。** 参考答案由 Claude Code 生成，评估也用 LLM-as-a-Judge。这意味着整个系统在某种程度上是"LLM 评价 LLM"，缺乏外部的 ground truth 验证。虽然论文提到了人工验证，但验证覆盖率和深度不明确。

**260 个测试样本的统计显著性。** 样本量较小，模型之间 2-3 分的差异在统计上是否显著？论文没有报告置信区间或显著性检验。

**RL 的可复现性。** RLAIF 用 LLM 打分作为奖励信号，不同评判模型的偏好可能导致不同的训练结果。换一个 Judge 模型，结论是否依然成立？

**工具集的局限。** 只有 Search、View、CommandLine 三个工具。实际的 Agentic Coding 场景中，模型可能需要运行测试、执行代码、查看 git history 等更丰富的操作。在限制性工具集上训练出的策略是否能迁移到更复杂的工具环境？

### 对实际工作的启示

对于在做 Agentic Coding 训练数据的团队，这篇论文提供了几个直接可用的思路：

1. **基准设计时引入难度校准**——用多个强模型的"无工具基线"过滤掉不需要 Agent 能力的问题
2. **SFT 数据不需要很多，但质量要高**——1,000 条高质量轨迹就够了
3. **SFT 之后尽早加 RL**——RL 和 SFT 提供正交的学习信号
4. **评估工具使用效率，不只是最终答案质量**——好的 Agent 不是调用工具最多的，而是策略最优的

## 总结

SWE-QA-Pro 在"如何评估 Agentic 代码理解能力"这个问题上给出了一个比较完整的方案：长尾仓库避免记忆作弊，跨模型共识标定难度，SFT+RL 闭环训练。260 个样本的规模和 LLM-as-a-Judge 的评估方式存在固有局限，但核心方法论——特别是难度校准——值得在更广泛的 Agentic 基准设计中借鉴。

对于关注"如何训练出好的 Code Agent"的读者，1,000 SFT + 464 RL 就让 8B 模型超越 GPT-4o 这个结果，是一个信号：在 Agent 训练中，**策略优化（RL）的边际收益高于行为克隆（SFT）的数据堆叠**。这可能是当前这个阶段最值得投入的方向。
