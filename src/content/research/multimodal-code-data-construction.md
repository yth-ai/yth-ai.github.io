## 核心问题：多模态代码数据和纯文本代码数据有什么不同？

**纯文本模型的代码数据**：代码文本 + 注释 + 文档，输入输出都是字符串。来源是 GitHub 爬取、Stack Overflow、LeetCode 题解，本质是 token 序列，训练目标是「给文字描述，生成代码」或「给代码，预测下一行」。

**多模态模型的代码数据**，在此基础上增加了视觉维度：

| 新增维度 | 描述 | 典型示例 |
|---------|------|---------|
| **代码 → 渲染结果** | 代码执行后的视觉输出 | HTML/CSS → 网页截图；matplotlib → 图表图片 |
| **截图 → 代码** | 逆向：给界面截图，写出实现代码 | 给 UI 设计稿截图，生成对应 React 组件 |
| **图表 → 代码** | 给数据可视化图，反推生成代码 | 给折线图，写出 pandas + matplotlib 代码 |
| **代码 + 错误截图** | 运行时报错截图（非文字），定位 bug | 给 Python 报错的终端截图，修复代码 |
| **文档图片 → 代码** | PDF/扫描技术文档中的伪代码/算法框图 → 可运行代码 | 论文 Algorithm 1 截图，实现为 Python |

关键结构差异：
- 纯文本：`(描述, 代码)` 二元对
- 多模态：`(图像, 描述, 代码)` 三元对——**多模态对齐的核心难点在这里**

三元组天然稀缺，「截图 + 对应代码 + 语义描述」同时出现的地方非常少，这决定了多模态代码数据的构建路径和成本都与纯文本有本质区别。

## 一、Code-Vision 配对数据：代码 ↔ 渲染结果

最主流的构建方向，核心思想是「执行代码得到视觉输出，形成配对」。

### WebSight / Design2Code

**论文**：[arxiv.org/abs/2403.09029](https://arxiv.org/abs/2403.09029)（HuggingFace，2024）

**构建方案**：
- 用 LLM 合成生成 HTML/CSS 代码
- Headless Browser（Playwright）渲染截图
- 构成「截图 ↔ 代码」配对数据集

**规模**：约 **82 万对**，已开源到 HuggingFace（`HuggingFaceM4/WebSight`）

**反向任务 Screenshot2Code**：给网页截图，要求模型写出 HTML/CSS，已成为多模态代码能力的标准评估维度。

**局限**：合成代码结构较简单，缺乏真实网页的复杂交互（动画、JS 事件、响应式布局）。

---

### Plot2Code

**论文**：[arxiv.org/abs/2405.07990](https://arxiv.org/abs/2405.07990)（2024）

**构建方案**：
- 收集 GitHub 上的 matplotlib/seaborn 代码
- Python 沙箱执行渲染 → 保存图表图片
- 构成图表图片 + 代码配对

**规模**：约 **132K 对**

**任务形式**：给图表图片，写出能精确复现该图的 Python 代码

**关键发现**：GPT-4V 在此任务上准确率仅约 **56%**，暴露了当时顶尖多模态模型在代码-视觉对齐上的明显短板。

---

### ChartMimic

**论文**：[arxiv.org/abs/2406.09152](https://arxiv.org/abs/2406.09152)（2024）

**特点**：专注科学论文中的图表（figure）→ 代码复现
- 覆盖 bar / line / scatter / pie / heatmap / 组合图等多种类型
- 难度高于 Plot2Code：论文图表通常有多子图、误差棒、复杂图例
- 任务形式：给截图 + 部分数据，写出完整可运行的绘图代码

## 二、UI2Code：界面截图 → 前端代码

### Pix2Code（2017，奠基工作）

最早的 UI2Code 工作，给 App 截图 / UI 设计稿 → 生成 DSL / HTML 代码。后续大量工作在此基础上扩展到 React/Vue/Flutter 等现代框架。

---

### Design2Code Benchmark

**论文**：[arxiv.org/abs/2403.03163](https://arxiv.org/abs/2403.03163)（斯坦福，2024）

**数据**：484 个真实网页截图 → 生成 HTML

**评估维度**：
- 视觉相似度（像素级和结构级）
- UI 元素匹配（文字、按钮、布局是否对应）
- 文字内容准确性

**关键发现**：GPT-4V 生成的 HTML 视觉相似度约 **0.8**，但细节还原仍有明显差距（字体、间距、颜色精度）。

---

### 业界产品映射

这个方向已有成熟的商业落地，背后都依赖大量此类配对训练数据：

| 产品 | 公司 | 核心能力 |
|------|------|---------|
| [Vercel v0](https://v0.dev/) | Vercel | UI 截图/描述 → React + Tailwind 组件 |
| [Locofy.ai](https://locofy.ai/) | Locofy | Figma 设计稿 → React/Vue/Flutter 代码 |
| Claude Artifacts | Anthropic | 描述/截图 → 可运行 Web 应用 |
| GPT-4 Canvas | OpenAI | 代码生成 + 实时预览 |

这些产品级能力的背后，都需要大量高质量的「设计稿/截图 ↔ 代码」配对数据持续训练。

## 三、代码执行轨迹 + 视觉反馈（最难，也最有价值）

这是最接近「真实编程过程」的多模态数据形态，也是目前**最大的数据空白**。

### 核心数据形态

```
代码 → 执行 → [输出截图 / 图表 / 报错截图]
→ 模型看到视觉输出
→ 判断是否符合预期
→ 修改代码
→ 再执行
→ 迭代直到正确
```

这个多轮闭环在纯语言模型里不存在——纯文本模型的「执行反馈」只有文字报错，没有视觉输出。但对多模态模型来说，这个闭环同时需要：
- **视觉理解**：看懂输出截图（图表是否正确、UI 是否符合要求、报错页面显示什么）
- **代码生成**：基于视觉反馈修改代码
- **多轮推理**：维护任务目标状态，判断当前进展

### 业界现状

**OSWorld / SWE-bench Multimodal** 里已有此类场景的评估：
- 运行 matplotlib 代码，截图结果不符合要求，模型需要看图修改
- 浏览器中 JS 代码运行后的 DOM 渲染结果，对照截图调试
- 网页爬虫脚本运行后的结果截图，修复提取逻辑

但**大规模训练数据几乎没有**：构建这类数据需要真实的沙箱执行环境（能运行代码、截图、传回给模型），每条数据的生产成本是普通代码数据的 10x 以上。

这是目前多模态代码数据领域**最明显的空白**，也是能力提升最依赖的数据类型。

## 四、技术文档与论文中的代码理解

### DocCoder

**论文**：[arxiv.org/abs/2309.02702](https://arxiv.org/abs/2309.02702)（2023）

- 给 API 文档截图（PDF/HTML 格式），根据文档写代码调用
- 解决「文档是图片而非文字」的场景——大量 SDK 文档、技术手册是 PDF 格式，纯文本模型直接读不了

### ArXiv 论文伪代码 → 实现

论文中的 Algorithm 框图是图片格式，纯文本模型完全无法处理，多模态模型需要：
1. 读懂算法框图（理解伪代码 + 数学符号）
2. 实现为可运行的 Python/PyTorch 代码
3. 验证实现是否符合原始算法描述

目前没有专门的公开大规模数据集。[SciCode](https://arxiv.org/abs/2407.13168) 在推动科学代码方向，但对视觉输入的覆盖还很有限。**这是一个高价值的空白方向**，特别是对 Research Agent 的训练有直接价值。

## 五、构建管线：业界主流方案

```
① 种子代码采集
   ├── GitHub 爬取（HTML/CSS/Python viz/JS/Flutter）
   ├── LLM 合成生成（GPT-4 / Claude 生成多样化代码）
   └── 现有数据集增强（CodeSearchNet、The Stack 等）

② 执行渲染
   ├── Headless Browser（Puppeteer/Playwright）→ HTML/CSS/JS 截图
   ├── Python 沙箱（Docker 隔离）→ matplotlib/seaborn/plotly 图表
   ├── Android Emulator → Flutter/原生 App 截图
   └── Node.js 沙箱 → React/Vue 组件渲染

③ 质量过滤
   ├── 渲染成功率：代码是否无报错执行
   ├── 视觉质量：截图非空白、非纯色、有实际内容
   ├── 代码-截图一致性：用另一个模型交叉验证（caption 是否匹配）
   └── 去重：感知哈希去除视觉上高度相似的截图

④ 配对构建
   ├── 正向：代码 → 截图（「给代码，预测渲染结果」训练）
   └── 反向：截图 → 代码（「给截图，生成实现代码」训练）

⑤ 指令化包装
   "以下是一个网页截图，请写出对应的 HTML/CSS 代码，使渲染效果与截图一致。"
   "请根据此图表写出能精确复现它的 Python 代码（使用 matplotlib）。"
   "以下代码运行后输出了截图中的界面，请修改代码使其满足以下要求：..."
```

## 六、关键差距与机会点

| 方向 | 现状 | 数据规模估计 | 核心差距 |
|------|------|------------|---------|
| HTML/CSS ↔ 截图 | 相对充分 | ~100万对（WebSight + 内部数据）| 复杂交互、动画、响应式布局覆盖不足 |
| 图表 ↔ 代码 | 有基础 | ~20万对 | 科学图表（误差棒、多子图）、交互式图表覆盖少 |
| UI 设计稿 → 代码 | 商业化成熟，数据封闭 | 未知（各家内部） | 高保真还原、复杂组件状态 |
| 代码执行视觉反馈 | **几乎空白** | <1万对（公开） | 需要沙箱基础设施，每条数据成本高 |
| 论文伪代码 → 实现 | **几乎空白** | 极少 | 高价值，无专门数据集 |
| 移动端 App ↔ 截图 | 早期 | <10万对 | Flutter/SwiftUI 代码量大但配对数据少 |
| 多轮调试轨迹 | **完全空白** | 0（公开） | 需要完整执行环境 + 多轮交互录制 |

### 最值得关注的机会

**代码执行轨迹 + 视觉反馈**是目前最大的差距，也是能力提升空间最大的方向：

1. **难以被合成替代**：不像静态截图-代码配对可以大量合成，执行反馈轨迹需要真实运行环境
2. **对多轮 Agent 能力有决定性影响**：Coding Agent 的核心不是「一次生成正确代码」，而是「看到结果 → 判断 → 修改 → 迭代」
3. **训练信号最强**：相比单轮「截图→代码」，多轮轨迹包含了推理过程和错误修复，DPO/RL 信号密度高

## 数据规模的不对称问题

这是理解多模态代码能力瓶颈的核心：

| 数据类型 | 估计规模 | 差距倍数 |
|---------|---------|---------|
| GitHub 纯代码 | 千亿 token | 基准 |
| 有配对截图的代码 | 百万量级 | 差 ~3-4 个数量级 |
| 完整三元组（截图+代码+描述） | 十万量级 | 差 ~4-5 个数量级 |
| 多轮执行反馈轨迹 | 千量级（公开） | 差 ~6-7 个数量级 |

**这个不对称直接解释了为什么**：GPT-4o 在纯文本代码上已经很强，但在「截图转代码」上还有明显差距——能力差距几乎完全反映了训练数据密度差距。

填补这个数据密度差距，是多模态编程能力提升最直接的路径。
