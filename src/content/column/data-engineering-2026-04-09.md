## 本期主题：当标注员变成大模型

前 12 期我们系统梳理了预训练数据工程的各个环节——从质量评估、去重、选择、配比、合成数据，到持续预训练的全生命周期方法论。但有一个横切所有环节的核心能力被刻意留到了现在：**数据标注**。

这不是一个新话题。从 ImageNet 时代的众包标注，到 GPT-3 时代的 RLHF 人工偏好标注，数据标注始终是 AI 系统中最昂贵、最难以规模化的瓶颈。全球数据标注市场规模在 2026 年已突破 **30 亿美元**（Grand View Research 估计 2024 年为 14.5 亿美元，预计 2033 年将达 131 亿美元，CAGR ~27%），但这个数字背后隐藏着一场静默的革命：

**标注员正在被大模型替代。**

准确地说，不是简单的替代，而是一种方法论的代际跃迁：

```
数据标注的三个时代：

时代 1（2012-2020）：纯人工标注
  众包平台（AMT/Scale AI）+ 质量控制（多人冗余+金标准检查）
  成本：$0.10-$5.00/样本  |  速度：~1000 样本/人/天
  问题：成本线性增长，无法跟上数据需求

时代 2（2020-2024）：人机协作标注
  LLM 预标注 + 人工审核/纠正
  成本：降低 50-70%  |  速度：提升 2-5×
  问题：人类审核者倾向于接受 LLM 建议，引入系统性偏差

时代 3（2024-2026）：LLM 原生标注
  LLM 直接标注 + 自动化质量验证 + 人类仅处理边界案例
  成本：降低 90%+  |  速度：提升 100×+
  问题：偏差检测、可靠性校准、主观性任务的质量保证
```

本期围绕四个核心问题深入解读 2024-2026 年的关键研究：

1. **平替验证**：LLM 标注能否真正替代人工标注？在什么条件下可以、什么条件下不行？
2. **偏差暗礁**：LLM-as-Judge 系统存在哪些系统性偏差？如何自动发现和缓解？
3. **范式创新**：从"强制单标签"到"候选标注蒸馏"，标注方法论本身在经历什么变革？
4. **偏好数据工厂**：RLHF/RLAIF 的偏好数据如何自动化生产？质量保证的红线在哪里？

## 一、RLAIF 的里程碑实验：AI 反馈追平人类反馈

### "[RLAIF vs. RLHF: Scaling Reinforcement Learning from Human Feedback with AI Feedback](https://arxiv.org/abs/2309.00267)" (Google / ICML 2024)

**来自 Harrison Lee, Samrat Phatale, Hassan Mansoor 等 11 人**

这可能是"LLM 标注能否替代人工"这个问题迄今为止最有说服力的实证回答。

**实验设计**：在 PaLM 2 XS 模型上，系统对比了三种对齐方式：

| 方法 | 偏好数据来源 | 流程 |
|------|------------|------|
| RLHF | 人类标注者两两比较 | 人标注 → 训练 RM → RL 训练策略 |
| RLAIF | LLM 标注者两两比较 | AI 标注 → 训练 RM → RL 训练策略 |
| d-RLAIF | LLM 直接打分 | AI 直接给奖励 → RL 训练策略（跳过 RM） |

**三个任务上的核心结果**：

```
摘要任务（Summarization）：
  RLHF ≈ RLAIF（人类评估无显著差异）

有帮助的对话（Helpful Dialogue）：
  RLAIF ≈ RLHF

无害对话（Harmless Dialogue）：
  RLAIF ≈ RLHF
```

更令人振奋的发现：

1. **d-RLAIF > RLAIF > SFT 基线**。跳过奖励模型训练、直接用 LLM 打分做 RL，效果反而更好——流程更简、效果更优
2. **自我改进有效**：用与策略模型同等规模的 LLM 做标注器，甚至用策略模型自身的早期 checkpoint 做标注器，效果仍然优于纯 SFT
3. **扩展友好**：AI 标注的成本是人工标注的 ~1/100，且可以无限扩展

> **对实践的启示**：RLAIF 的核心价值不是"AI 比人类标得好"——在简单任务上两者持平，在复杂主观任务上人类仍有优势。RLAIF 的核心价值是 **scalability**：当你需要 100 万条偏好数据而非 1 万条时，人工标注在成本上不可行，RLAIF 是唯一的路。d-RLAIF 更进一步简化了流程——连奖励模型都省了。

## 二、"只要把人放进循环就行了？"——人机协作标注的隐性陷阱

### "[Just Put a Human in the Loop? Investigating LLM-Assisted Annotation for Subjective Tasks](https://aclanthology.org/2025.findings-acl.1323/)" (MIT / ACL 2025 Findings)

**来自 Hope Schroeder, Deb Roy, Jad Kabbara**

当很多团队采用"LLM 预标注 + 人工审核"的人机协作流程时，MIT 的这项预注册实验给出了一个反直觉的警告。

**实验规模**：350 名独立标注员 × 2 个模型 × 2 个数据集 × 4 种条件 = **7,000 个标注**。

**四种实验条件**：

```
条件 A：纯人工标注（无 LLM 辅助）   ← 基线
条件 B：展示 LLM 标注建议后人工判断  ← "人在循环中"
条件 C：模型 1 的建议
条件 D：模型 2 的建议
```

**四个关键发现**：

| 发现 | 预期 | 实际 |
|------|------|------|
| 效率提升 | ✅ 人工更快 | ❌ 没有更快 |
| 标注信心 | — | ⬆️ 标注者更自信（但不更准确） |
| 标签分布偏移 | 不应偏移 | ❌ 标注者大量采纳 LLM 建议，分布显著偏移 |
| 模型评估虚高 | 不应虚高 | ❌ 用"人类审核"的标签评估 LLM，性能虚高 |

**第四个发现尤其危险**：当你用 LLM 辅助人工标注创建"金标准"评估集，然后用它评估 LLM 性能时——你其实在让 LLM 间接评估自己。人类审核者引入的 LLM 偏差会"污染"评估集，导致模型性能被系统性高估。

```
LLM 辅助标注的反馈回路陷阱：

LLM 生成建议 → 人类倾向于接受
                ↓
        标注数据偏向 LLM 的分布
                ↓
        用该数据评估 LLM
                ↓
        LLM 性能被高估 → 更信任 LLM 建议
                ↓
              ╭──────╮
              │ 正反馈 │
              │ 循环   │
              ╰──────╯
```

> **对实践的启示**：
> 1. **"人在循环中"不是万能药。** 特别是主观性任务（如偏好判断、有害性评估），人类审核者很容易被 LLM 建议锚定
> 2. **评估集和训练集必须使用不同的标注流程。** 如果训练数据用 LLM 辅助标注可以接受，评估集则应使用无 LLM 干预的纯人工标注
> 3. **监控标注分布漂移。** 部署 LLM 辅助标注后，持续比较有/无 LLM 辅助条件下的标签分布——如果差异过大，说明人类审核形同虚设

## 三、LLM 标注的承诺与陷阱：48K 合成标注的大规模实验

### "[The Promises and Pitfalls of LLM Annotations in Dataset Labeling](https://arxiv.org/abs/2411.11081)" (NAACL 2025 Findings)

**来自 Tomas Horych, Christoph Mandl, Terry Ruas 等 7 人**

如果说 MIT 的实验揭示了"人机协作"的陷阱，这篇 NAACL 2025 论文则从另一个角度验证了"纯 LLM 标注"的潜力。

**实验设计**：在媒体偏见检测这一复杂分类任务上，使用 LLM 生成了超过 **48,000 条**合成标注数据（命名为 annolexical 数据集），然后用这些数据训练下游分类器。

**核心发现**：

```
三层蒸馏效果：

第一层：LLM 直接做分类
  → 基线准确率

第二层：用 LLM 标注数据训练 SLM 分类器
  → MCC 提升 5-9 个百分点（超越标注用的 LLM 本身）

第三层：对比人工标注数据
  → 在 BABE 和 BASIL 基准上：
     LLM 标注训练的模型 ≈ 人工标注训练的模型
     部分场景下甚至 > 人工标注
```

**为什么蒸馏后的小模型能超越标注用的大模型？**

这个看似悖论的结果其实是合理的。LLM 直接推理时受限于 prompt 格式和上下文窗口；而用 LLM 标注的大量数据训练专用分类器时，小模型在特定任务上的参数利用效率更高——这本质上是知识蒸馏的标准机制。

**行为压力测试揭示的局限性**：

尽管总体结果积极，后续的压力测试发现：
- LLM 标注在**边界案例**上的一致性低于人工标注
- 某些特定的偏见模式（如隐性偏见）LLM 检测率显著低于人类专家
- 标注分布的长尾部分存在系统性缺失

> **对实践的启示**：对于明确定义的分类任务（如文本分类、意图识别、情感分析），LLM 标注 + 蒸馏训练是一条高性价比的路径——成本降低一到两个数量级，质量接近或持平人工标注。但对于需要深层语义理解的边界案例，仍需人工介入。

## 四、LLM-as-Judge 的系统性偏差：当裁判也有偏见

标注质量的评估本身是一个递归问题——如果我们用 LLM 来评估标注质量（LLM-as-Judge），那谁来评估 LLM 评估的质量？2024-2026 年的三项关键工作正在系统地回答这个问题。

### "[A Survey on LLM-as-a-Judge](https://arxiv.org/abs/2411.15594)" (arXiv 2024, 持续更新至 2025.10)

**来自 Jiawei Gu, Xuhui Jiang, Zhichao Shi 等 16 人**

这篇综述首次系统化了 LLM-as-Judge 的方法论框架，核心贡献是提出了"What × How × Where to Judge"的三维分类法：

| 维度 | 内容 | 典型方法 |
|------|------|---------|
| What to Judge | 评估什么——文本质量/安全性/事实性/推理/代码… | 不同维度需要不同的提示策略 |
| How to Judge | 如何评估——打分/排序/两两比较/分类 | 两两比较最稳定但成本高 |
| Where to Judge | 在哪评估——训练数据筛选/模型评估/RLHF 奖励信号 | 每个场景的可靠性阈值不同 |

**综述识别的六类主要偏差**：

```
LLM-as-Judge 的六大偏差：

1. 位置偏差（Position Bias）
   两两比较时，偏向第一个/最后一个选项
   → 影响幅度：可达 10-15% 准确率差异

2. 长度偏差（Length Bias）
   偏好更长的回答，即使长度不代表质量
   → 典型表现：verbose ≠ correct

3. 自我增强偏差（Self-Enhancement Bias）
   偏好与自身风格相似的输出
   → GPT-4 倾向于高评 GPT-4 的输出

4. 权威偏差（Authority Bias）
   当提示中包含"专家观点"等锚定信息时，评分偏移
   → 与 Vol.11 CQF 的"分布偏移"本质相同

5. 慈悲偏差（Leniency Bias）
   LLM 倾向于给高分，避免低分
   → 评分分布整体偏右

6. 一致性不足（Inconsistency）
   同一输入多次评估结果不同
   → 温度、prompt 微调都会影响
```

---

### "[BiasScope: Towards Automated Detection of Bias in LLM-as-a-Judge Evaluation](https://arxiv.org/abs/2602.09383)" (ICLR 2026)

**来自 Peng Lai, Zhihao Ou, Yong Wang, Longyue Wang, Jian Yang, Yun Chen, Guanhua Chen**

如果说上述综述梳理了已知偏差，BiasScope 则试图发现**未知偏差**。

**核心方法**：BiasScope 是一个完全由 LLM 驱动的迭代式框架，将偏差发现从"人工枚举已知偏差"转变为"自动化探索未知偏差"：

```
BiasScope 的工作流程：

传统方法（人工驱动）：
  研究者假设偏差类型 → 设计实验 → 验证是否存在
  → 只能发现已知的偏差类型
  → 覆盖面有限

BiasScope（LLM 驱动）：
  LLM Agent 自动生成候选偏差假设
  → 自动构造测试用例
  → 自动验证偏差是否存在
  → 迭代发现新偏差
  → 可以跨模型系列和规模发现潜在偏差
```

**JudgeBench-Pro 基准的震撼结果**：

基于 BiasScope 发现的偏差构建的 JudgeBench-Pro 基准中，**即使是最强大的 LLM 评委，错误率也超过 50%**。

这意味着：在精心设计的对抗性场景下，LLM-as-Judge 几乎和随机猜测一样不可靠。

---

### "[Evaluating Scoring Bias in LLM-as-a-Judge](https://arxiv.org/abs/2506.22316)" (DASFAA 2026)

**来自 Qingquan Li, Shaoyu Dou, Kailai Shao, Chao Chen, Haixiang Hu**

这篇论文关注了一个更微妙的偏差来源——**评分提示词本身**。

**三种新发现的偏差类型**：

| 偏差类型 | 描述 | 实践风险 |
|---------|------|---------|
| 评分标准顺序偏差 | 提示词中评分标准的排列顺序影响打分 | 同一份 rubric 换个顺序，分数就变了 |
| 分数 ID 偏差 | 用数字(1-5)、字母(A-E)还是文字作为分数选项会影响结果 | 评分量表的设计本身引入偏差 |
| 参考答案分数偏差 | 提示词中的参考答案样例分数会锚定后续评分 | few-shot 示例的分数分布会"传染" |

> **对实践的启示——三篇论文的综合判断**：
>
> | 使用场景 | 可靠性 | 建议 |
> |---------|--------|------|
> | 大规模数据粗筛（如预训练数据质量过滤） | 🟢 足够 | LLM-as-Judge 的成本优势远超偏差影响 |
> | RLHF 偏好数据标注 | 🟡 需校准 | 使用多 Judge 投票 + 位置交换去偏 |
> | 模型能力评估（benchmark） | 🟠 风险高 | 必须配合人类评估做交叉验证 |
> | 安全性/有害性评估 | 🔴 不可靠 | 高风险场景不能完全依赖 LLM Judge |
>
> **黄金法则**：LLM-as-Judge 的可靠性与任务的客观性正相关。事实性判断 > 文本质量 > 偏好排序 > 主观性评估 > 安全性判断。越主观的任务，越需要人类参与。

## 五、CanDist：从"强制单标签"到"候选标注蒸馏"

### "[Prompt Candidates, then Distill: A Teacher-Student Framework for LLM-driven Data Annotation](https://arxiv.org/abs/2506.03857)" (ACL 2025 Main)

**来自 Mingxuan Xia, Haobo Wang, Yixuan Li 等 7 人**

这篇 ACL 2025 主会论文对 LLM 标注的方法论本身做了一个精妙的改进。

**问题诊断**：

```
传统 LLM 标注（强制单标签）：

输入："这篇文章的情感倾向是什么？"
LLM（不确定时）→ 被迫输出一个标签 → "积极"（实际可能是中性偏积极）
                                     → 错误标签

根因：LLM 存在内在不确定性，但传统范式强迫它给出确定性答案。
      在困难样本上，这种"强制确定"会系统性地生成错误标签。
```

**CanDist 的解决方案**：

```
候选标注范式（Candidate Annotation）：

输入："这篇文章的情感倾向是什么？列出所有可能的标签。"
LLM（教师）→ 输出候选集：{"积极": 0.6, "中性": 0.35, "消极": 0.05}

SLM（学生）→ 从候选集中蒸馏出最终标签
            → 利用多个样本的候选分布做联合推断
            → 最终标签比单标签更准确
```

**关键创新**：

1. **模糊厌恶对抗**：受行为经济学中"模糊厌恶"启发——当面对不确定性时，人类倾向于逃避而非做出错误选择。CanDist 允许 LLM "坦诚"自己的不确定性
2. **理论保证**：论文严格证明了候选标注蒸馏在理论上优于单标签直接使用——因为候选集保留了标签间的概率关系，信息量严格大于单标签
3. **教师-学生架构**：大模型（教师）生成候选集，小模型（学生）做蒸馏——实现了成本和质量的最优平衡

**六个文本分类任务上的实验**：

在全部六个任务上，CanDist 均优于直接使用 LLM 单标签标注。在困难样本上（LLM 不确定性高的样本），优势更为显著。

> **对实践的启示**：如果你在用 LLM 做大规模标注，考虑把 prompt 从"给出一个标签"改为"给出所有可能的标签及其置信度"。然后用一个小模型做蒸馏。这个改动几乎零成本（只需要修改 prompt 和加一个轻量蒸馏步骤），但在困难样本上的标注质量提升是显著的。

## 六、Anyprefer：Agent 化的偏好数据合成工厂

### "[Anyprefer: An Agentic Framework for Preference Data Synthesis](https://arxiv.org/abs/2504.19276)" (ICLR 2025)

**来自 Yiyang Zhou, Zhaoyang Wang, Tianle Wang 等 16 人**

前面讨论的 RLAIF 验证了"AI 反馈可以替代人类反馈"，但具体怎么生产高质量的偏好数据？Anyprefer 给出了一个系统化的工程方案。

**核心架构——协作双人博弈**：

```
Anyprefer 的协作框架：

┌─────────────────────────────────────────┐
│            Anyprefer Framework           │
│                                         │
│  ┌───────────┐       ┌───────────┐     │
│  │ 目标模型   │←──→  │ 判官模型   │     │
│  │ (Player 1) │       │ (Player 2) │     │
│  │ 生成回答   │       │ 评判质量   │     │
│  └─────┬─────┘       └─────┬─────┘     │
│        │                   │            │
│        │   ┌───────────┐   │            │
│        └──→│ 外部工具   │←──┘            │
│            │ (去偏校准)  │               │
│            └─────┬─────┘               │
│                  │                      │
│            ┌─────▼─────┐               │
│            │ 反馈优化   │               │
│            │ (提示词迭代)│               │
│            └───────────┘               │
└─────────────────────────────────────────┘
```

**与自奖励方法的关键区别**：

| 维度 | 自奖励方法 | Anyprefer |
|------|----------|-----------|
| 评判者 | 目标模型自己 | 独立的判官模型 |
| 偏差来源 | 共享权重 → 偏差放大 | 独立模型 + 外部工具 → 去偏 |
| 工具使用 | 无 | 外部验证工具校准评判标准 |
| 优化方式 | 静态 | 动态反馈优化提示词 |

**跨领域实验结果**：

| 应用领域 | 数据集数 | 平均提升 |
|---------|---------|---------|
| 自然语言生成 | 5 | **+18.55%** |
| 视觉-语言理解 | 9 | **+3.66%** |
| 医学图像分析 | 3 | **+30.05%** |
| 视觉-运动控制 | 4 | **+16.00%** |

**Anyprefer-V1 数据集**：包含 58K 高质量偏好对，已开源。

> **对实践的启示**：Anyprefer 的核心启示是——偏好数据的生产不应该是"让模型自己标注自己"，而应该是一个**带外部验证的协作系统**。目标模型和判官模型分离、引入外部工具做去偏校准、动态优化提示词——这三层设计有效打断了自奖励方法中的偏差放大循环。这与 Vol.11 讨论的模型崩溃逃逸机制（外部验证器）在思路上高度一致。

## 七、细粒度奖励标注：从粗糙偏好到精准信号

### "[Data-adaptive Safety Rules for Training Reward Models](https://arxiv.org/abs/2501.15453)" (arXiv 2025)

**来自 Xiaomin Li, Mingye Gao, Zhiwei Zhang, Jingxuan Fan, Weiyu Li**

传统的偏好标注是粗粒度的——给两个回答，标注哪个更好。但"更好"是一个模糊的概念——更安全？更有帮助？更准确？更流畅？

**从二元偏好到细粒度规则**：

```
传统偏好标注（粗粒度）：
  回答 A vs 回答 B → "A 更好" / "B 更好"
  → 奖励模型只知道谁赢了，不知道为什么赢

细粒度规则标注：
  回答 A vs 回答 B →
    安全性：A > B（B 包含不当内容）
    有帮助度：B > A（B 更直接回答了问题）
    准确性：A = B
    → 奖励模型学到多维度的偏好信号
```

**本文的关键创新——数据自适应规则选择**：

不是对所有样本使用固定的规则集，而是为每个响应对**动态选择最重要的规则**。方法基于"最大差异原则"——选择两个回答差异最大的维度进行标注。

**理论保证**：论文证明该方法能最大化规则标注与真实偏好之间的互信息。

**实战成果**：用这种自适应标注的偏好数据训练的 8B 奖励模型，在 RewardBench 排行榜上取得了 **安全性维度最高分**（截至 2025.01.25），超越了多个更大参数量的模型。

> **对实践的启示**：如果你在构建 RLHF 的偏好数据标注流程，考虑从"谁更好"升级为"在哪些维度上谁更好"。细粒度标注的额外成本不高（特别是用 LLM 自动化后），但训练出的奖励模型能力显著更强——因为它学到的不是模糊的"好坏"，而是可分解的多维偏好信号。

## 八、Curriculum-RLAIF：当课程学习遇见 AI 反馈

### "[Curriculum-RLAIF: Curriculum Alignment with Reinforcement Learning from AI Feedback](https://arxiv.org/abs/2505.20075)" (arXiv 2025)

**来自 Mengdi Li, Jiaye Lin, Xufeng Zhao 等 7 人**

这篇论文直接呼应了 Vol.12 讨论的"课程学习 × LR 衰减"冲突——但这次是在 RLAIF 的语境下。

**问题**：RLAIF 使用 AI 生成的偏好数据训练奖励模型，但这些数据存在三个交织的困难：
- **分布偏移**：AI 生成的偏好分布 ≠ 真实人类偏好分布
- **标签噪声**：AI 标注不可避免地包含错误
- **难度不匹配**：某些样本对当前奖励模型来说太难，强制学习反而引入噪声

**Curriculum-RLAIF 的方案**：

```
传统 RLAIF（随机训练）：
  所有偏好对 → 打乱 → 训练奖励模型
  → 困难样本的噪声标签污染早期训练

Curriculum-RLAIF（课程训练）：
  所有偏好对 → 按难度排序 → 从易到难训练
  → 奖励模型先建立稳定基础，再处理困难案例
  → 噪声标签的负面影响被有效缓解
```

**核心结果**：在不增加任何推理成本的前提下，Curriculum-RLAIF 训练的奖励模型泛化能力和策略模型的对齐性能均显著优于随机训练基线。

> **对实践的启示**：这与 Vol.12 的发现形成了跨领域呼应——**数据的排列顺序比数据量更重要**。无论是预训练的课程学习，还是 RLAIF 的奖励模型训练，"从易到难"的数据组织都比"随机打乱"更有效。如果你在做 RLAIF，花一小时实现数据难度排序带来的收益，可能超过花十小时收集更多数据。

## 本期思考

> 回顾本期讨论的十余项工作，数据标注领域正在经历一场深刻的方法论重塑。这场重塑的核心不是"用 AI 替代人类"——这个叙事过于简化——而是**标注工程从手工作坊走向工业化生产**。
>
> **三个核心认知需要更新**：
>
> **第一，标注的瓶颈已经从"产能"转移到"质量验证"。** RLAIF 证明了 AI 标注在产能上可以无限扩展，成本降低两个数量级。但 BiasScope 同时证明了 LLM-as-Judge 存在大量系统性偏差——在对抗性测试中错误率超过 50%。这意味着**未来数据标注工程的核心竞争力不是"谁标得快"，而是"谁能可靠地验证标注质量"**。从 CQF 的质量幻觉（Vol.11）到 LLM-as-Judge 的偏差，"看起来高质量"和"真正高质量"之间的鸿沟是数据工程面临的最深层挑战。
>
> **第二，"人在循环中"需要重新设计，而非简单部署。** MIT 的 7,000 标注实验揭示了一个令人不安的事实：把人类放进 LLM 标注流程后，人类不仅没有起到纠错作用，反而被 LLM 锚定了判断。这不是说人类审核没用——而是说**审核流程的设计**决定了审核是否有效。有效的人机协作需要：盲审（不展示 LLM 建议）、对照组（保留无 LLM 辅助的基线）、分布监控（持续检查标签偏移）。
>
> **第三，标注方法论正在从"确定性"走向"概率性"。** CanDist 的候选标注范式代表了一个重要的范式转换——承认不确定性、保留概率信息、通过蒸馏获得最终标签。这比传统的"强制单标签"在理论和实践上都更合理。与此同时，细粒度标注（多维度评估）和数据自适应标注（动态选择评估维度）正在取代粗粒度的二元偏好标注。数据标注正在从"给个答案"变成"给出带置信度的多维判断"。
>
> **对数据工程团队的建议**：
> 1. **立即可做**：审视你的标注 prompt——把"给出一个标签"改为"给出所有可能的标签及置信度"，零成本提升困难样本的标注质量（CanDist 启示）
> 2. **短期投入**：建立标注质量的自动化验证管线——不依赖单一 LLM Judge，而是多 Judge 投票 + 位置交换 + 分布监控
> 3. **中期建设**：如果在做 RLHF，评估从二元偏好升级到细粒度多维标注的 ROI——8B 模型 + 细粒度标注可以超越更大模型 + 粗粒度标注
> 4. **认知红线**：安全性和有害性评估不能完全交给 LLM。高风险决策的标注，人类必须保留最终决定权
