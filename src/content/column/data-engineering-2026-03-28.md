## 本期主题：AI 生成的训练数据，能训练出更好的 AI 吗？

2024 年中，一个词开始在大模型圈弥漫开来——**数据墙（Data Wall）**。当预训练语料库从 2T 膨胀到 15T 再向 100T tokens 迈进时，一个尴尬的现实浮出水面：**互联网上可用的高质量自然文本正在枯竭**。

FineWeb-Edu 和 DCLM 通过激进的模型过滤把 Common Crawl 的精华榨了出来，代价是丢弃了 90% 以上的原始数据。剩下的 token 不够用，怎么办？答案似乎显而易见：**让 AI 自己造数据**。

从微软的 Phi 系列首次证明"教科书级合成数据可以让小模型超越预期"，到 HuggingFace 的 Cosmopedia、NVIDIA 的 Nemotron-CC、DatologyAI 的 BeyondWeb，合成数据正以惊人的速度渗透到预训练管线的核心。2025 年的多个技术报告显示，主流大模型的预训练数据中已有 **15%-30%** 是合成生成的。

但这里有一个根本性问题被低估了：**我们怎么知道合成数据是"好"的？**

当训练数据由 AI 生成，传统的质量评估方法——perplexity、分类器过滤、人工抽检——还够用吗？更深层的矛盾是：合成数据看起来总是很"干净"、很"流畅"、很"结构化"，但这些表面特征能等同于"对模型有用"吗？Nature 2024 的模型崩溃警告还适用吗？

本期深度拆解。

## 一、理论基线：模型崩溃——最坏情况有多坏？

### "[AI Models Collapse When Trained on Recursively Generated Data](https://www.nature.com/articles/s41586-024-07566-y)" (Nature, 2024)

**来自牛津 / 剑桥 / 多伦多大学（Ilia Shumailov, Zakhar Shumaylov, Yiren Gal 等）**

这篇发表在 Nature 正刊上的论文，为合成数据的讨论设定了理论上限——如果处理不当，最坏情况到底有多坏。

**核心发现**：当模型 A 生成数据 → 模型 B 在此数据上训练 → 模型 B 生成数据 → 模型 C 在此数据上训练……这个递归过程会导致**不可逆的分布坍缩**。

```
模型崩溃的机制：

第 0 代：原始人类数据分布 P₀
  │  尾部覆盖丰富 ████████████████▓▓▓▒▒░░
  ↓
第 1 代：模型 A 训练在 P₀ 上，生成 P₁
  │  尾部开始缺失 ████████████████▓▓░
  ↓
第 n 代：模型 N 训练在 P_{n-1} 上，生成 P_n
  │  分布坍缩     ████████
  ↓
极限：所有生成内容趋向均值，多样性消失

三个误差源的累积：
┌─────────────────────────────────────────────┐
│  ① 统计近似误差：有限样本导致低概率事件丢失   │
│  ② 功能表达性误差：模型能力有限，无法完美拟合  │
│  ③ 学习过程误差：SGD 等优化器的结构偏差       │
│                                             │
│  单代误差可能很小，但误差会跨代累积！          │
└─────────────────────────────────────────────┘
```

**最关键的实验结论**：

| 条件 | 结果 |
|------|------|
| 100% 合成数据递归训练 | 数代后 perplexity 显著上升，生成文本出现大量重复和幻觉 |
| 保留 **10%** 原始人类数据 | 崩溃被大幅缓解 |
| 单轮合成（非递归）| 影响远小于递归情况 |

**对实践的启示**：

> 模型崩溃论文的核心信息不是"不能用合成数据"，而是给出了三个硬约束：
> 1. **绝对避免递归链条**：不要用模型 A 生成的数据训练模型 B，再用 B 生成的数据训练 C——每一代都应该从原始数据出发
> 2. **必须保留原始人类数据**：合成数据只能是补充，不能是替代。10% 的原始数据可能就是"救命稻草"
> 3. **单轮合成的风险远低于递归**：当前主流的"用强模型一次性改写/生成"范式，并不在模型崩溃的最严格警告范围内

## 二、千模实验：Meta 的合成数据系统性拆解

### "[Demystifying Synthetic Data in LLM Pre-training](https://arxiv.org/abs/2510.01631)" (Meta FAIR / EMNLP 2025)

**来自 Meta FAIR（Feiyang Kang, Newsha Ardalani, Michael Kuchnik 等）**

如果说模型崩溃论文设定了理论上的恐惧，Meta 的这项工作则用 **1000+ 个 LLM、10 万+ GPU 小时**的大规模实证实验，给出了具体到可直接指导实践的答案。

**实验设计的关键优势**：使用统一的实验协议和 Scaling Laws 框架，系统比较了自然网络数据、改写合成数据（rephrased synthetic）、教科书式合成数据（textbook-style）以及各种混合比例。这种规模的对照实验，在合成数据领域前所未有。

**六大核心发现**：

```
发现 1：纯改写合成数据 ≠ 更快的训练
┌────────────────────────────────────────────┐
│  预期：合成数据更干净 → 学习效率更高          │
│  现实：纯改写数据的训练效率并不优于自然文本     │
│                                            │
│  但！混合使用时效果截然不同 ↓                 │
└────────────────────────────────────────────┘

发现 2：1/3 合成 + 2/3 自然 = 最佳配比
┌────────────────────────────────────────────┐
│  在大数据预算下（>100B tokens）：             │
│  混合 ~30% 改写合成数据 + ~70% 自然数据       │
│  → 达到相同验证损失的速度提升 5-10×           │
│                                            │
│  这个 ~30% 的比例在不同模型规模下              │
│  经验性地收敛到类似值                         │
└────────────────────────────────────────────┘

发现 3：教科书式合成数据有严重局限
┌────────────────────────────────────────────┐
│  纯教科书风格数据训练：                       │
│  → 在许多下游领域的 loss 显著偏高              │
│  → 尤其在小数据预算下问题严重                  │
│  → 表现出模型崩溃论文预测的模式               │
│                                            │
│  ↔ 改写型数据没有这个问题                     │
└────────────────────────────────────────────┘

发现 4：生成器不是越大越好
┌────────────────────────────────────────────┐
│  使用 70B 模型生成的合成数据                  │
│  不一定优于 8B 模型生成的                     │
│                                            │
│  → 在 ~8B 后，生成器规模的收益开始饱和        │
│  → 这对成本控制是个极好的消息                 │
└────────────────────────────────────────────┘

发现 5：对模型崩溃的混合证据
┌────────────────────────────────────────────┐
│  改写型合成数据：大规模单轮训练无崩溃迹象      │
│  教科书型合成数据：出现模型崩溃预测的模式       │
│                                            │
│  关键区别：改写保留了原始文本的信息结构         │
│  教科书式生成创造了全新的、更窄的分布           │
└────────────────────────────────────────────┘

发现 6：合成数据的 Scaling Law 存在但不同
┌────────────────────────────────────────────┐
│  最优合成配比取决于模型规模和数据预算          │
│  → 不存在"万能配比"                         │
│  → 需要根据具体场景调参                      │
│  → 但 ~30% 是一个稳健的经验起点              │
└────────────────────────────────────────────┘
```

**对实践的启示**：

> Meta 这篇论文是目前合成预训练数据领域最重要的实证参考：
> 1. **改写 > 生成**：如果你要在预训练中引入合成数据，优先选择"改写"范式（基于真实文本重写），而非"从头生成"范式（如教科书式）
> 2. **30% 是黄金比例的起点**：在没有条件做大规模搜索时，30% 合成 + 70% 自然是最安全的起始配比
> 3. **用 8B 模型生成就够了**：不需要用最大最贵的模型来生成合成数据，8B 参数级别已经足够
> 4. **质量的定义需要改变**：合成数据的"质量"不能只看表面的流畅度和结构性——教科书式数据看起来最"高质量"，但对预训练的贡献反而有限

## 三、工业级实践：谁在造合成数据？怎么造的？

### 1. Nemotron-CC：NVIDIA 的 6.3T 增强管线

**"[Nemotron-CC: Transforming Common Crawl into a Refined Long-Horizon Pretraining Dataset](https://arxiv.org/abs/2412.02595)" (NVIDIA / ACL 2025)**

Nemotron-CC 是目前规模最大的**融合了合成增强的开源预训练数据集**。它的核心问题意识是：FineWeb-Edu 和 DCLM 为了质量丢掉了 90% 的数据，对于 15T token 规模的长程训练来说根本不够用。

**三步解法**：

```
传统方案的困境：
  质量高 → 数据少（FineWeb-Edu：~1T HQ tokens）
  数据多 → 质量低（原始 CC：~240T tokens）

Nemotron-CC 的解法：
┌────────────────────────────────────────────┐
│  ① 分类器集成（Classifier Ensembling）        │
│     用多个质量分类器投票而非单一模型           │
│     → 保留更多"在边界上"的高质量文档          │
│                                            │
│  ② 合成改写（Synthetic Rephrasing）           │
│     对中等质量文档进行改写提升                 │
│     → 不是丢弃，而是"救活"                   │
│                                            │
│  ③ 减少启发式过滤                            │
│     用模型判断替代硬编码规则                   │
│     → 避免规则对非典型内容的误杀               │
└────────────────────────────────────────────┘

结果：
  总量 6.3T tokens（含 1.9T 合成增强）
  vs DCLM：相同 MMLU 水平下，独特真实 token 多 4×
  vs Llama 3.1 8B（15T训练）：MMLU +5.0, ARC +3.1
```

**Nemotron-CC v2**（2025.12 发布）进一步更新：
- 新增 8 个 Common Crawl 快照（2024-2025 年数据）
- 改用 **Qwen3-30B-A3B** 作为改写器
- 使用 **5 种不同提示词**生成合成变体，增加多样性
- 合成 Diverse QA 翻译到 15 种语言
- 英文合成部分达 **1.26T tokens**

### 2. BeyondWeb：DatologyAI 的合成数据新前沿

**"[BeyondWeb: Lessons from Scaling Synthetic Data for Trillion-scale Pretraining](https://arxiv.org/abs/2508.10975)" (DatologyAI, 2025)**

BeyondWeb 是 2025 年合成预训练数据领域最令人印象深刻的工作之一。它不仅在性能上建立了新的帕累托前沿，更重要的是**总结了 7 条关于合成数据质量的可复现经验**。

**核心数据**：

| 对比 | BeyondWeb | Cosmopedia | Nemotron-Synth | 原始 Web (RPJ) |
|------|-----------|------------|----------------|---------------|
| 8B 模型准确率 (180B tokens) | **63.7%** | 58.6% | 61.1% | 56.6% |
| vs 原始 Web 提升 | **+7.1pp** | +2.0pp | +4.5pp | — |
| 达到 RPJ 基线所需 tokens | **23.2B** (7.7× 加速) | 较多 | 66.2B (2.7×) | 180B |
| 3B 超越 8B？ | ✅ 3B BeyondWeb > 8B Cosmopedia | — | — | — |

**BeyondWeb 的 7 条质量法则**（对实践极具指导价值）：

```
法则 1：合成数据 ≠ 知识蒸馏
  → 简单摘要几乎匹配 Cosmopedia
  → 但精心设计的改写远超摘要
  → 关键增益来自"提高每 token 信息密度"

法则 2：朴素合成无法突破数据墙
  → 简单用 LLM 扩展 Web 文档 ≈ 重复 Web 数据
  → 只有填补分布空白的合成数据才有突破性价值

法则 3：高质量输入 ≠ 高质量输出
  → DCLM 精选数据作为种子 > 低质量种子
  → 但仅此不够，还需要策略优化

法则 4：风格匹配有用但不够
  → Web 中对话式内容 < 5%，但 LLM 主要用于对话
  → 上采样对话数据有帮助，但收益递减

法则 5：多样性是可扩展学习的关键 ⭐
  → 单策略合成数据很快饱和
  → 多策略（格式转换 + 风格修改 + 内容重构）
    在整个训练过程中持续提供学习信号

法则 6：大多数改写器模型都可以
  → 收益对改写器家族不敏感
  → 模型基准分数不预示合成质量

法则 7：小模型 (3B-8B) 足以做改写器
  → 1B→3B 有提升，3B→8B 饱和
  → 改写任务的简单性使大模型不必要
  → 这意味着成本可以大幅降低
```

**对实践的启示**：

> BeyondWeb 的最大贡献不是某个具体数字，而是它对**合成数据质量的系统性定义**：
> - 不是"看起来更干净"（那只是表面质量）
> - 不是"更像高质量参考集"（那是 [Vol.5](/column/data-engineering-2026-03-26) 中 Apple 批判的分布相似性幻觉）
> - 而是**"能否填补自然数据的分布空白"** + **"每 token 的信息密度"** + **"策略多样性"**
> 
> 这三个维度构成了目前最实用的合成数据质量评估框架。

### 3. SynthLLM：微软的修正 Scaling Law

**"[SynthLLM: Breaking the AI Data Wall with Scalable Synthetic Data](https://www.microsoft.com/en-us/research/articles/synthllm-breaking-the-ai-data-wall-with-scalable-synthetic-data/)" (微软研究院亚洲, 2025)**

微软的贡献不在于又造了一个合成数据集，而在于首次验证了**合成数据也遵循 Scaling Laws——但需要修正**。

**修正 Scaling Law 的三个关键发现**：

```
发现 1：性能可预测缩放
┌────────────────────────────────────────────┐
│  合成数据量 ↑ → 性能 ↑（可预测的幂律关系）   │
│  这验证了合成数据作为"可再生资源"的可行性      │
└────────────────────────────────────────────┘

发现 2：300B tokens 平台期
┌────────────────────────────────────────────┐
│  当合成数据量达到 ~300B tokens 时             │
│  性能提升趋于平缓                            │
│  → 继续堆量只有微小收益                       │
│  → 这是 Scaling Law 的"修正"部分             │
│                                            │
│  自然数据的平台期远高于此                      │
│  → 合成数据的有效信息天花板更低                │
└────────────────────────────────────────────┘

发现 3：大模型对合成数据的需求更小
┌────────────────────────────────────────────┐
│  8B 模型：~1T tokens 即接近最优               │
│  3B 模型：需要 ~4T tokens                    │
│                                            │
│  → 与自然数据的趋势相反！                     │
│  → 大模型更擅长从有限合成数据中提取信息         │
└────────────────────────────────────────────┘
```

**SynthLLM 的技术创新**：用图算法识别和重组来自多文档的高级概念，而非简单的回译或问题提取。这种方法在有限的参考文档下生成了更高多样性的数据。

## 四、多样性：合成数据质量的隐藏维度

### "[On the Diversity of Synthetic Data and its Impact on Training Large Language Models](https://arxiv.org/abs/2410.15226)" (ICLR 2025)

**来自佐治亚理工 / KAIST 等（Hao Chen, Abdul Waheed 等）**

前面的研究大多关注"信息密度"和"配比"，但这篇论文揭示了一个被严重低估的维度：**合成数据的多样性**。

**新指标：LLM Cluster-Agent**

传统的多样性度量（如 n-gram 多样性、Self-BLEU）无法捕捉语义层面的多样性。作者提出了一种基于 LLM 的聚类方法来度量数据的**概念多样性**——不是词汇层面的不重复，而是知识和推理模式的覆盖广度。

**核心实验结论**：

| 发现 | 具体结论 |
|------|---------|
| 多样性正相关 | LLM cluster-agent 多样性分数与预训练 + SFT 性能正相关 |
| SFT 比预训练更敏感 | 合成数据多样性对微调的影响 **显著大于** 对预训练的影响 |
| 小模型更依赖多样性 | 小模型从高多样性数据中受益更大 |
| 多样性 > 重复 | 高多样性 + 较少重复 >> 低多样性 + 大量数据 |

**与 BeyondWeb 的呼应**：

BeyondWeb 的"法则 5"（多样性是可扩展学习的关键）和这篇论文的定量分析完美互证——**合成数据的质量不能只看单条样本的"好坏"，还要看整体集合的分布覆盖**。一个由 100 条"满分"样本组成的数据集，可能不如由 100 条"80 分"但覆盖不同概念的样本组成的数据集。

## 合成数据质量评估框架

综合本期覆盖的所有研究，这里提出一个可操作的合成数据质量评估框架：

```
                合成数据质量的五维评估模型

维度 1：信息密度 (Information Density)
┌──────────────────────────────────────────────┐
│  度量：每 token 的"新知识"含量                  │
│  方法：与原始文本的信息论比较（如压缩率差异）    │
│  基线：改写后的文本应包含等量或更多信息          │
│                                              │
│  ❌ 反例：教科书式生成往往"稀释"了信息          │
│  ✅ 正例：BeyondWeb 式重构提高了密度            │
└──────────────────────────────────────────────┘

维度 2：分布互补性 (Distribution Complementarity)
┌──────────────────────────────────────────────┐
│  度量：合成数据是否填补了自然数据的空白          │
│  方法：嵌入空间覆盖率分析                       │
│  基线：合成数据应拓展，而非复制自然分布          │
│                                              │
│  ❌ 反例：朴素扩展 ≈ 重复自然数据               │
│  ✅ 正例：格式转换、风格转化、概念重组            │
└──────────────────────────────────────────────┘

维度 3：概念多样性 (Concept Diversity)
┌──────────────────────────────────────────────┐
│  度量：数据集覆盖的知识和推理模式的广度          │
│  方法：LLM Cluster-Agent 或语义聚类             │
│  基线：多策略生成 > 单策略生成                   │
│                                              │
│  ❌ 反例：单一提示词生成的数据快速饱和           │
│  ✅ 正例：Nemotron-CC v2 的 5 种提示词策略      │
└──────────────────────────────────────────────┘

维度 4：源信号保真度 (Source Signal Fidelity)
┌──────────────────────────────────────────────┐
│  度量：合成数据是否保留了原始文本的核心事实      │
│  方法：事实一致性检查、幻觉检测                  │
│  基线：改写 > 生成（改写天然保留更多源信号）     │
│                                              │
│  ❌ 反例：从头生成容易引入幻觉                   │
│  ✅ 正例：基于源文档的改写保留了事实锚点          │
└──────────────────────────────────────────────┘

维度 5：递归安全性 (Recursive Safety)
┌──────────────────────────────────────────────┐
│  度量：合成数据链条是否引入累积偏差              │
│  方法：监控多代训练后的分布漂移                   │
│  基线：始终从原始人类数据出发生成               │
│                                              │
│  ❌ 反例：用合成数据训练的模型再生成数据          │
│  ✅ 正例：单轮生成 + 保留原始数据锚              │
└──────────────────────────────────────────────┘
```

**实操检查清单**：

| 检查项 | 方法 | 通过标准 |
|--------|------|---------|
| 混合配比 | 消融实验 | 合成占比 20%-30% 时验证损失最优 |
| 改写 vs 生成 | 对比实验 | 改写型在多领域 loss 上无退化 |
| 多样性充足 | LLM 聚类/嵌入覆盖率 | 覆盖率 ≥ 自然数据的 80% |
| 无递归链 | 溯源审计 | 每条合成数据可追溯到人类源文档 |
| 尾部保留 | 分布分析 | 低频概念仍有合理覆盖 |
| 事实一致 | 抽样检查 + NLI | 幻觉率 < 5% |

## 本期思考

> **合成数据正在从大模型训练的"可选调料"变成"核心原料"。** 当 NVIDIA 在 Nemotron-CC v2 中放入 1.26T 合成 tokens、当 BeyondWeb 证明 3B 模型可以靠合成数据超越 8B 模型、当微软验证合成数据也遵循（修正的）Scaling Laws——合成数据已经不可逆地成为预训练管线的标配。
>
> 但本期覆盖的研究也清晰地画出了**合成数据的能力边界**：
>
> **第一，合成数据有信息天花板。** SynthLLM 发现 300B tokens 后收益趋平，BeyondWeb 发现朴素合成无法突破数据墙。这说明合成数据并不能无限替代真实数据——它能"增强"和"补充"，但不能"替代"。当你的训练预算超过 1T tokens，必须有大量真实数据作为基底。
>
> **第二，"好"的定义已经改变。** 传统的数据质量评估——perplexity 低、结构清晰、语法正确——对合成数据几乎毫无区分度。所有合成数据都"看起来很好"。真正区分合成数据质量的是**信息密度、分布互补性、概念多样性**这三个更深层的维度。BeyondWeb 的 7 条法则和本期提出的五维评估模型，是目前最接近实用的评估框架。
>
> **第三，改写范式的胜出不是偶然。** Meta 的千模实验、NVIDIA 的 Nemotron-CC、DatologyAI 的 BeyondWeb，三个独立的大规模实验共同验证了同一个结论：**改写型合成数据 >> 生成型合成数据**。改写保留了源文档的事实基础和信息结构，同时提升了密度和格式多样性。这与模型崩溃论文的理论分析完全一致——改写本质上是对原始分布的"增强"而非"替代"，因此不会导致分布坍缩。
>
> 对于数据工程团队，**此刻最该做的三件事**：
> 1. **在预训练管线中引入合成改写**（如果还没有的话）。以 30% 改写 + 70% 自然作为起点，用 8B 模型做改写器，多种提示词策略确保多样性
> 2. **建立合成数据的质量监控指标**。不要只看 perplexity——用嵌入空间覆盖率监控多样性，用事实一致性检查监控幻觉，用消融实验验证每种合成策略的边际贡献
> 3. **投资数据溯源**。每条合成数据都应该能追溯到它的人类源文档。这不仅是 [Vol.6](/column/data-engineering-2026-03-27) 中讨论的合规要求，也是防止递归崩溃的技术保障
