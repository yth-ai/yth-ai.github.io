💡 核心理念

AI 编码代理不应该"直接写代码"，而应该遵循完整的软件工程流程。
传统的 Vibe Coding（用户给需求 → AI 吐代码）会产生大量技术债务。Superpowers 强制插入了设计、计划、测试、审查等关键环节。
🧪 TDD First
永远先写测试，再写代码。红-绿-重构循环是不可跳过的硬性约束。
🔧 System over Ad-hoc
用流程代替即兴发挥。每次决策都有结构化的步骤支撑。
📊 Evidence over Claims
验证通过才算完成。不接受"看起来对了"。
✂️ Reduce Complexity
简单性是首要目标。YAGNI — 你不会需要它。

🏗️ 技术架构

整个系统的核心是 SKILL.md — 用结构化 Markdown 定义的技能文件。每个技能描述了触发条件、执行流程、红旗警告和与其他技能的集成关系。

项目结构： skills/ ├── brainstorming/ # 苏格拉底式需求探索 ├── writing-plans/ # 细粒度实施计划生成 ├── executing-plans/ # 当前会话内批量执行 ├── subagent-driven-development/ # ⭐ 子代理驱动开发（核心） ├── dispatching-parallel-agents/ # 并发子代理工作流 ├── test-driven-development/ # 红-绿-重构 TDD 循环 ├── systematic-debugging/ # 4 阶段根因分析 ├── verification-before-completion/ # 完成前验证 ├── requesting-code-review/ # 请求代码审查 ├── receiving-code-review/ # 接收审查反馈 ├── using-git-worktrees/ # Git Worktree 并行开发 ├── finishing-a-development-branch/ # 合并/PR 决策 ├── using-superpowers/ # 框架使用入门 └── writing-skills/ # 创建新技能（元技能） 配置目录： ├── .claude-plugin/ # Claude Code 插件配置 ├── .cursor-plugin/ # Cursor 插件配置 ├── .codex/ # Codex 配置 ├── .opencode/ # OpenCode 配置 ├── agents/ # 智能体定义 ├── commands/ # 命令注册 ├── hooks/ # 钩子脚本 └── tests/ # 测试文件

协作与流程类（8 个）

brainstorming
苏格拉底式设计细化。在展示设计并获得批准前严禁写任何代码。一次只问一个问题。

假设执行者"无项目上下文且厌恶测试"，因此计划必须极细粒度（2-5 分钟/步），严禁 TBD。

在当前会话中批量执行计划，包含检查点。适合紧密耦合的任务序列。

并行
requesting-code-review

审查
receiving-code-review
接收审查反馈时的处理规范。要求技术严谨性，拒绝表演性同意或盲目实施。
审查
finishing-a-development-branch
分支完成后的决策工作流：合并、创建 PR、或清理。
收尾

⚡ 关键设计点：自动触发

  • 技能之间通过触发条件自动衔接，无需用户手动调用
  • brainstorming 完成后自动过渡到 writing-plans
  • 计划编写完成后提供 subagent-driven 和 inline 两种执行路径选择
  • 每个技能完成时自动触发下一个技能的"should I use this skill?"检查

🤖 子代理驱动开发（核心创新）

这是 Superpowers 最重要的技能创新。它将软件开发任务分解为可独立执行的单元，通过子代理隔离执行，确保上下文不会污染。

主控代理（Orchestrator） │ ├── 📖 读取实施计划 │ ├── 对每个任务重复： │ │ │ ├── 📤 分派 → 实施者子代理（Implementer） │ │ ├── 隔离上下文（不继承主控历史） │ │ ├── TDD 实现（红→绿→重构） │ │ ├── 自审 + 提交 │ │ └── 状态报告: DONE | DONE_WITH_CONCERNS | NEEDS_CONTEXT | BLOCKED │ │ │ ├── 📤 分派 → 规格审查子代理（Spec Reviewer） │ │ └── 检查是否符合规格，不合规则回到实施者修复 │ │ │ └── 📤 分派 → 代码质量审查子代理（Quality Reviewer） │ └── 检查代码质量，有问题则修复并重审 │ └── ✅ 全部通过 → 最终代码审查 → 分支收尾

模型分级策略

任务类型推荐模型场景示例
机械实施 快速、便宜的模型 隔离函数、规格清晰、1-2 个文件的改动
集成判断 标准模型 多文件协调、模式匹配、调试定位
架构审查 最强模型 架构设计、代码审查、设计方案评估

🚩 红旗警告（绝对禁止）

  • 在未经明确同意的情况下在主分支上开始实施
  • 跳过审查（规格合规性或代码质量）
  • 带着未修复的问题继续下一个任务
  • 并行分派多个实施子代理（会导致代码冲突）
  • 让子代理读取计划文件（应直接提供全文）
  • 接受"足够好"的规格合规性（必须完全修复）

📊 对数据工作的价值

直接可借鉴的方法论

① 数据管道的 TDD 思路

Superpowers 的 TDD 强制模式对数据管道开发非常有参考价值：先写数据质量断言（schema 验证、空值检查、范围约束），再写 ETL 逻辑。

类比映射

  • 单元测试 → dbt test / Great Expectations / Soda 数据质量测试
  • 红阶段 → 写断言，确认数据不满足（因为还没写变换）
  • 绿阶段 → 写最小 ETL 使断言通过
  • 重构 → 优化 SQL 性能、增量策略

② 数据项目的"头脑风暴"流程

数据工作中最常见的问题是"需求不明确就开始建模"。Superpowers 的 brainstorming 流程可以直接套用到数据建模讨论中。

流程迁移

  • 先明确业务问题而不是直接写 SQL
  • 探索数据源可用性和质量
  • 设计数据模型方案（2-3 种）并权衡
  • 输出数据设计文档再开始实施

③ 子代理驱动的数据工程

对于复杂的数据工程任务（如数据迁移、多源整合），可以借鉴子代理驱动开发模式：

角色数据工程中的对应
主控 Agent理解全局数据架构，制定迁移计划
实施子 Agent处理单个表/数据源的迁移
规格审查 Agent验证迁移结果的数据一致性
质量审查 Agent检查 SQL 性能和代码质量

对 Agentic Coding 训练数据的价值

🎯 与 Agentic Coding 训练数据构建方案的交叉点

  • 高质量 Agent 轨迹数据源 — 94K 用户意味着海量规范化轨迹（brainstorm→plan→implement→review→commit），自带审查标签
  • SKILL.md = Prompt Engineering 教材 — 用心理学原则（权威、承诺一致性）增强指令遵循的实战范例
  • 子代理协调模式参考 — 4 种状态反馈机制可直接用于元认知训练数据生成
  • 模型分级策略 — 为不同任务匹配不同能力等级的模型，是构建高效 Agent 系统的重要设计模式

🛠️ 我们编写的数据工程 Skills

基于 Superpowers 的技能框架，我们为数据工程场景编写了 5 个自定义 Skills，已部署到项目的 .workbuddy/skills/ 目录：

🧪
data-pipeline-tdd
数据管道测试驱动开发。先写数据质量断言（P0/P1/P2 分级），再写 ETL 变换逻辑，遵循红-绿-重构循环。
减少数据质量事故
💭
data-modeling-brainstorm
数据建模头脑风暴。强制在写 SQL 前明确业务场景、评估数据源、设计 2-3 种建模方案并权衡。
避免需求遗漏
🔍
pipeline-debugging
数据管道系统化调试。4 阶段根因分析（症状收集→假设生成→假设验证→修复验证），附调试文档模板。
缩短故障定位时间
📋
migration-plan
数据迁移计划编写。5-15 分钟粒度步骤，每步含验证点 + 回滚方案，严禁 TBD/TODO。
降低迁移风险
📐
schema-review
Schema 两阶段审查。先业务合规（命名一致性、语义正确性），再技术质量（性能、兼容性、存储成本）。
提升 Schema 质量
