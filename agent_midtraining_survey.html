Section 0

定义与背景：为什么需要 Mid-training for Agent？

大模型训练三阶段定位，以及 Agent 中训练的核心必要性

Stage 1
Pre-training
Foundational Knowledge
通用知识与语言能力。在万亿 token 网络文本上学习语言模式、世界知识、基础推理。数据规模大，但几乎不含 Agent 执行轨迹。
Stage 2 ← 本文聚焦
Mid-training
Specialized Skills
定向强化 Agent 所需的规划/工具/执行能力。通过大规模 agentic data（轨迹、工具调用、推理链）填补 distribution gap，是 SFT 的前置增强。
Stage 3
Post-training
Alignment
SFT + RL 对齐指令跟随与偏好。数据量相对小，依赖 mid-training 已奠定的能力基础，进行最终行为对齐。

Agent Mid-training 的三重必要性

1

分布鸿沟（Distribution Gap）

自然爬取文本 vs. Agent 执行轨迹之间存在巨大 distribution gap。SFT 阶段数据量有限（万级），难以弥补预训练阶段造成的能力空洞。Mid-training 用百亿级 agentic token 填补这一空白。

2

隐式知识不足

工具使用规范、API 调用格式、错误处理范式在通用预训练数据中极为稀疏。模型不知道"如何调用工具"不是智力问题，而是分布问题——需要专项数据注入。

3

规划能力是"学出来"的

多步规划、自我反思、工具串联等核心 Agent 能力，需要大规模轨迹数据才能激活。研究表明这些能力在 mid-training 后呈现显著的 emergent gain，而非靠 SFT 微调可达。

Section 1

数据来源维度：四大类范式

从数据获取角度，Agent mid-training 数据可分为四大类型，覆盖真实挖掘到合成生成

🌐
1A · Real Trajectory Mining
真实轨迹挖掘
Web Agent 轨迹

从网页交互日志（Click/Type/Scroll/Navigate 序列）中提取 GUI 操作轨迹，天然包含用户意图→操作序列映射。

AgentTrek (arXiv:2412.09605) · 数万条高质量 GUI 轨迹
代码执行轨迹

GitHub PR/Issue/Commit 序列，天然包含"问题→分析→修复→验证"的 agent 范式，是最丰富的免费 agentic data 来源。

KAT-Coder (2025) · 20B GitHub tokens
工具调用日志

从 API 使用文档、Stack Overflow 答案、Jupyter Notebook 中提取工具调用序列，覆盖真实使用场景。

ToolBench / APIBench
🏗️
1B · Synthetic Trajectory Generation
合成轨迹生成
基于 LLM 的轨迹合成

用强模型（GPT-4/Claude）在模拟环境中执行任务生成完整轨迹。优势：可规模化、质量可控；挑战：hallucinated tool calls、impossible actions。

基于环境的执行验证

在真实/模拟环境中执行生成的轨迹，保留成功轨迹作为训练数据——执行验证是关键质量门控。

ScaleEnv (arXiv:2602.06820, 2026) · 从零构建合成环境
PAO 循环（Plan-Action-Observation）

结构化合成"Plan → Action → Observation → Reflect"四元组序列，是当前工业界最广泛采用的 agent 数据格式。

KAT-Coder / GLM4.5 均采用此范式
🔧
1C · Tool & API Knowledge Mining
工具/文档知识挖掘
工具文档合成 QA

从 API 文档生成"如何使用此工具解决 X 问题"的 QA 对。ToolDoc → ToolQA：覆盖工具选择/参数理解/错误处理多视角。

工具 Schema 理解训练

JSON schema、OpenAPI spec 格式理解训练；工具描述→调用格式对齐，让模型学会"读懂"工具定义并正确调用。

跨工具组合训练

从单工具调用渐进到多工具串联，进一步构建工具依赖图（哪些工具需要组合使用），提升工具编排能力。

🧠
1D · Reasoning & Reflection Data
推理链/反思数据
自我反思轨迹

模型执行失败后的 retry + correction 序列。Reflexion (Shinn et al., 2023)：failure → verbal reflection → retry，提供负例驱动的修正数据。

多步推理 CoT（工具×推理混合链）

在 agent 上下文中的 chain-of-thought，包含工具使用中间步骤——不是纯数学 CoT，而是"工具调用嵌入推理链"的混合格式。

Agent 自进化轨迹

通过 Reasoning Memory 机制让 agent 积累推理经验，用于生成更高质量的后续训练数据，实现闭环自我提升。

ReasoningBank (Ouyang et al., 2025)

核心方法论文详情

7 篇代表性论文/系统，覆盖 Web 轨迹、环境合成、工业实践、自进化等方向

1. AgentTrek 轨迹合成 Web GUI arXiv:2412.09605
Xu et al., 2024 · Web GUI 轨迹自动合成

核心思路：从互联网 web tutorial 文章（"如何操作 XXX 网站"类教程）自动解析出操作步骤，用 VLM 在真实浏览器环境执行并验证，生成高质量 instruction-trajectory 对，无需人工标注。

  • aTutorial Crawling → Step Extraction：NLP 解析教程文章，提取结构化操作步骤序列（"点击→输入→提交"）
  • bVLM-driven Execution：在真实浏览器中执行提取的步骤，VLM 负责截图理解与操作验证
  • c自动质量过滤：过滤执行失败、步骤不完整或环境异常的轨迹，仅保留端到端成功的 trajectory
意义：证明"用教程文档自动生成训练轨迹"技术路线可行。开辟了无需人工标注的大规模 web trajectory 获取新路径，GUI agent 能力显著提升。
2. ScaleEnv 工具/环境 合成环境 arXiv:2602.06820
2026 · 环境合成 × 通用工具使用 Agent 中训练

核心思路：不依赖预设 benchmark 环境，从零自动生成多样化交互环境，并在其中合成训练轨迹，训练通用 interactive tool-use agent。解决高质量 agentic 训练环境稀缺的根本问题。

  • a自动环境生成：自动设计任务空间 + 工具空间 + 状态转移规则，生成多样化交互环境（不依赖 WebArena/OSWorld 等现有环境）
  • b环境内轨迹采集：在生成的环境中运行 LLM agent 收集执行轨迹，覆盖成功/失败/部分成功等多种状态
  • c成功率过滤 + 多样性保证：按成功率阈值过滤，同时保证环境和任务的多样性覆盖
意义：在多个 agent benchmark 上超越仅用真实数据训练的基线。环境合成是轨迹合成的前提，ScaleEnv 从根本上解决了"哪里来足够多、足够多样的训练环境"这一开放问题。
3. Qwen Agent Continual Pre-training 工业实践 多源混合
Qwen Team, 2025 · 工业规模中训练 · Scaling Agents via CPT

核心思路：在 Qwen base 模型上通过 continual pre-training 系统性提升 agent 能力，核心是构建工业规模的 agentic data pipeline，多源数据融合训练。

  • a多源 Agentic Data：Web 轨迹 + 代码执行序列 + 工具调用日志 + PAO 循环，构成多维度 agent 能力覆盖
  • b数据配比搜索：系统性搜索 reasoning 数据与 agent 轨迹数据的最优混合比例，避免某类能力退化
  • c专项能力评估：在 agent benchmark 上进行专项测试，量化 mid-training 前后的 delta gain
意义：工业级验证 mid-training 对 agent 能力的增益可测量且显著。证明 continual pre-training 是比 SFT-only 更有效的 agent 能力提升路径。
4. KAT-Coder 轨迹合成 代码 Agent
2025 · 代码 Agent 专项中训练

核心思路：针对 coding agent 能力，构建四类专项 mid-training 数据，覆盖真实软件工程轨迹、推理链、agentic 循环与指令遵循，系统性提升代码修复与执行能力。

  • aReal-world SWE 数据：20B token GitHub PR / Issue / Commit 序列，天然包含"问题发现→根因分析→代码修复→验证"的完整 agent 轨迹
  • bReasoning CoT：code / STEM / logic 推理链，强化模型在代码场景下的多步推理能力
  • cAgentic Data：Plan-Action-Observation 循环，结构化编程 agent 执行范式
  • dComplex Instruction Following：多约束指令遵循数据，提升模型处理复杂编程规范和约束的能力
效果：在 SWE-bench 类代码修复任务上大幅提升，证明针对特定 agent 场景的专项数据构建策略的有效性。
5. GLM4.5 Mid-training 实践 工业实践 多能力
Zhipu AI, 2025 · 推理 × 长文档 × Agent 三维 mid-training

核心思路：系统性 mid-training，同时覆盖推理能力、长文档处理、agent 执行三个维度，采用精细化数据工程策略保证长轨迹数据完整性。

  • aRepo-level Data：仓库原始文件 + PR + Issue + Commit，提供代码 agent 所需的仓库级上下文理解
  • bReasoning Data：math / science / coding competitions 数据，合成 reasoning CoT，覆盖 STEM 推理场景
  • cLong Context & Agent Data：上采样长文档 + agentic trajectory；采用 best-fit packing 策略，避免长轨迹在 sequence packing 时被截断
工程洞察：Best-fit packing 是 mid-training 阶段处理长轨迹数据的关键工程细节——传统 random packing 会截断长轨迹，导致训练信号不完整，对 agent 能力影响显著。
6. Large-Scale Diverse Synthesis for Mid-Training 跨域/通用 系统性研究
2025 · 学术系统性综述与实验

核心思路：专门为 mid-training 设计大规模多样合成数据 pipeline，通过系统性实验评估各类合成策略对不同下游能力的增益，建立 mid-training 数据合成的方法论框架。

  • a系统性框架建立：对 mid-training 数据合成的类型、质量标准、评估体系进行系统梳理，建立可复用的 pipeline 设计规范
  • b能力增益对比：对比不同合成策略（指令多样性、难度分布、领域覆盖）对各下游 agent 能力的 delta gain
  • c多样性度量：提出数据多样性对 mid-training 效果的重要性度量，量化覆盖度与增益的关系
贡献：为工业界和学术界提供了 mid-training 数据工程的参考标准，明确了数据多样性是影响 mid-training 效果的核心变量之一。
7. ReasoningBank 推理/规划 自进化
Ouyang et al., 2025 · Agent Self-Evolving with Reasoning Memory

核心思路：通过 Reasoning Memory 机制让 agent 在任务执行中积累历史推理路径，用于生成更高质量的自进化轨迹数据——模型用自己的经验训练自己，无需外部 oracle。

  • aReasoning Memory：存储历史推理路径（成功/失败路径及其结果），作为未来执行时的上下文参考，避免重复失败模式
  • bSelf-evolving Data Generation：模型利用积累的推理记忆生成更高质量的新训练轨迹，数据质量随迭代轮次提升
  • c闭环自我提升：无需外部 teacher model 或人工反馈，形成自主改进循环
意义：指向未来"模型自己合成自己的训练数据"这一重要范式。ReasoningBank 证明了 reasoning memory 是驱动 agent 自进化的有效机制，但同时也引出了自进化数据质量控制的开放问题。

三条关键洞察

1

🔵 合成优于真实（规模角度）

真实轨迹稀缺且噪声大；合成轨迹可控、可验证。Qwen / GLM 工业实践表明，最终训练数据中合成比例可安全达到 70%+，且效果不逊于全真实数据。

2

🟢 执行验证是关键质量门控

在真实/模拟环境中执行验证，可将合成轨迹错误率从 ~40% 降低到 ~10%。无验证的合成轨迹会引入大量 hallucinated tool calls，损害训练效果。

3

🟡 PAO 循环是通用范式

Plan → Action → Observation 三元组是当前工业界最广泛采用的 agent 数据格式。KAT-Coder、GLM4.5、Qwen CPT 均采用此结构，可作为 agent mid-training 数据的标准化格式。

Section 5

前沿方向与开放问题

当前 Agent Mid-training 领域三个重要的未解决挑战

🌍
Open Problem 1

环境合成的 Scaling 与真实感

ScaleEnv 开辟了"自动生成训练环境"的路径，但如何自动生成足够多样且具备"真实感"的交互环境仍是难题。当前合成环境与真实 Web/OS 环境之间存在 sim-to-real gap——模型在合成环境中训练后，未必能在真实环境中泛化。如何缩小这一 gap，同时保持环境生成的可规模化，是关键开放挑战。

🔄
Open Problem 2

自进化数据的质量控制与错误防扩散

ReasoningBank 等自进化框架面临"错误自我强化"风险：如果 agent 积累了错误的推理记忆，会生成有缺陷的训练数据，进而强化错误行为，形成恶性循环。目前缺乏有效的自进化轨迹质量检测机制——如何在无外部 oracle 的情况下，识别并过滤低质量或错误的自进化轨迹，是保证自进化可靠性的关键问题。

⚖️
Open Problem 3

Mid-training vs RL 的能力边界划分

哪些 Agent 能力更适合通过 mid-training（大规模数据注入）习得？哪些更适合通过 RL（环境交互探索）获得？目前缺乏系统性的比较研究。初步观察表明：工具格式理解、API 调用规范等"知识型"能力适合 mid-training；而长程规划、动态策略调整等"探索型"能力可能更适合 RL。厘清这一边界将指导未来 agent 训练资源的最优分配。
