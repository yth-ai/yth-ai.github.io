从概念起源到工程实践的完整解析

Mid-Training（中间训练）是 2024-2025 年大模型研发中最重要的范式革新之一。 它位于大规模预训练与指令微调之间，通过精心设计的数据配比与学习率策略， 系统性地激活模型在数学、代码、长文本、领域知识等方面的能力。 本报告面向预训练/后训练团队，提供从理论到实操的深度技术指南。

⚡ 执行摘要

战略定位

Mid-Training 是预训练与后训练之间的"桥接器"——用百亿至千亿 token 的精选数据，完成通用基座向垂直能力的低摩擦过渡，填补 SFT 无法注入深度知识的结构性缺口。

与前后阶段的关系

预训练奠定语言理解基础（万亿 token），Mid-Training 激活/注入特定能力（百亿 token 量级），后训练（SFT/RLHF）完成行为对齐（百万条样本）。三者分工明确、缺一不可。

行业现状

Meta（Llama 3）、阿里（Qwen2.5）、深度求索（DeepSeek）、OpenAI（专设 Mid-Training 团队）均已将其纳入标准训练流水线。2025 年已成主流实践而非实验探索。

核心发现与建议

  • Mid-Training 对数学/代码/长文本能力提升效果显著，MATH benchmark 可提升 20-50%，代码 pass@1 可提升 15-30%
  • 数据配比是最关键超参数：通用数据占比建议 5-20%，过少导致灾难性遗忘，过多稀释目标能力
  • 学习率应比预训练低 5-10 倍，需要配套 warmup 重启以稳定收敛
  • 小规模 Proxy 实验（1B 模型 + 10B token）可以较准确预测大规模效果，节省 90%+ 算力
  • Annealing 阶段（最终 1-5% token）引入高质量数据的边际收益极高，是性价比最高的 Mid-Training 形式
  • 2025 年趋势：Mid-Training 正与强化学习融合，出现"Mid-RL"形态；OpenAI 研究员 Noam Brown 提出"Mid-Training is the new Pre-Training"
CH 01

概念由来与定义

1.1 从训练流水线说起

大语言模型的训练范式经历了从两阶段到三阶段的演进。经典的两阶段框架是：大规模预训练（在万亿 token 的通用语料上学习语言表征）→ 指令微调/RLHF（将模型行为与人类意图对齐）。这一框架催生了 GPT-3、早期 ChatGPT 等里程碑产品。

然而，随着应用需求深化，两阶段框架的裂缝逐渐显现：纯粹的预训练语料（以 Common Crawl 为主体）缺乏足够的数学、代码、专业领域内容；而 SFT/RLHF 虽能塑造对话风格与安全行为，却难以在有限样本中注入大量系统性知识。这一空白，催生了 Mid-Training 的出现。

📘 Research Insight

2025 年 10 月，arXiv 上同期出现了两篇 Mid-Training 综述（arXiv:2510.06826 和 arXiv:2510.23081），标志着该方向已从零散实践上升为系统研究领域。两篇综述均认为，mid-training 区别于 continued pretraining 的核心在于：它不仅是数据分布的延续，更是有意识的能力塑造行为。

1.2 为什么需要 Mid-Training

纯预训练的局限：以 Common Crawl 为骨干的预训练数据中，数学内容约占 0.5-1%，代码约占 5-10%，医疗/法律等专业内容占比更低。这种数据分布导致通用模型在专业领域表现薄弱。即便扩大预训练规模，若不改变数据配比，专业能力提升也会面临收益递减。

后训练的局限：SFT（监督微调）的本质是行为克隆——模型学的是"如何表达"，而非"知道什么"。大量研究表明，SFT 无法有效注入大量新知识：若目标知识不在预训练语料中，模型往往产生幻觉而非学会该知识。RLHF 的情况类似——它调整的是输出偏好，而非底层知识表征。

Mid-Training 的独特价值，可从四个维度理解：

  • 知识注入（Knowledge Injection）：通过大规模领域数据训练，将专业知识深度编码进模型参数，而非依赖检索或提示。典型如 CodeLlama 通过 500B token 代码数据使模型真正"理解"代码语义。
  • 能力激活（Capability Activation）：模型在预训练后已具备某些潜在能力（如长文本关联、复杂推理链），但缺乏充分的训练信号。Mid-Training 通过定向数据激活这些潜能，典型如长文本 Mid-Training（引入 > 32K 的文档后，原本 8K context 模型可处理 128K）。
  • 格式适应（Format Adaptation）：某些任务需要特定的输入输出格式（如代码的函数填充 Infilling、数学的逐步推理格式），预训练数据中此类格式稀缺，需通过 Mid-Training 专门强化。
  • 数据配比修正（Data Mix Correction）：在预训练最后阶段（Annealing）通过大幅上调高质量数据比例，修正早期训练中的数据分布偏差。Llama 3 的 Annealing 阶段仅用 40M token 高质量数据，就在 MATH 上提升了约 24 个百分点。
💡 Tip：Mid-Training vs RAG

参数化知识注入（Mid-Training）与检索增强（RAG）并非互斥。前者适合频繁调用的核心领域知识，具有无延迟优势；后者适合长尾知识和需要实时更新的内容。两者互补，成熟系统通常结合使用。

CH 02

Mid-Training 的主要类型

2.1 领域适应型（Domain-Adaptive）

领域适应型 Mid-Training 的目标是将通用语言模型迁移为某一特定领域的专家模型。其核心假设是：通用预训练虽然赋予了模型强大的语言理解能力，但领域专用知识仍需通过大规模领域语料的持续训练来深度内化。

法律 SaulLM-7B, LegalBERT Mistral-7B 30B-54B token 法律条文、判决书、法学期刊
金融 BloombergGPT, FinLLaMA BLOOM / LLaMA 363B-500B token Bloomberg News、SEC 文件、财报

关键设计原则：领域适应型 Mid-Training 必须解决能力保持与能力获取的张力。纯粹的领域数据训练会导致通用能力（如常识推理、对话能力）严重退化。实践中，成功的做法是数据回放：在领域数据中混入 5-20% 的通用语料，以"锚定"模型的通用表征。

⚠️ Warning：领域数据质量陷阱

领域数据的质量比数量更重要。某医疗 LLM 研究发现，使用 10B token 高质量医学教材数据（教科书、权威指南）的效果，显著优于使用 100B token 低质量医疗论坛数据。包含错误医疗信息的训练数据不仅无益，还会污染模型的已有知识。数据质量过滤是领域适应 Mid-Training 的核心工程挑战。

2.2 能力增强型（Capability Enhancement）

🔢 长文本能力激活

长文本是 2024-2025 年 Mid-Training 最活跃的方向之一。大多数模型的预训练上下文窗口为 4K-8K token，但实际应用（代码库分析、长文档处理、多轮对话）往往需要 32K-128K+。长文本 Mid-Training 通常分两步：

  1. 位置编码外推：修改 RoPE 参数（如 YaRN、LongRoPE、ABF），使位置编码能支持更长序列
  2. 长文本数据课程：逐步引入更长的文档，避免训练初期因 attention 矩阵过大导致的不稳定

Llama 3 的长文本 Mid-Training 将 context window 从 8K 扩展到 128K，使用的是逐步扩展策略：先在 32K 上训练，再在 128K 上继续。关键数据配比：合成长文档（书籍、代码库文件聚合）占 65%，自然长文本（学术论文、GitHub issue 长链）占 35%。

🧮 数学推理增强

数学是 Mid-Training 效果最显著的领域之一。以 DeepSeekMath 为例：

  • 基于 DeepSeek-Coder-v1.5 7B，进行 500B token 的数学专项 Mid-Training
  • 数据来源：web 数学内容 120B + GitHub 数学相关代码 60B + 合成推理链数据 320B
  • 结果：在 MATH benchmark 上从 46.8% 提升到 64.1%（+17.3%），逼近当时的 GPT-4 表现
📘 Research Insight：代码辅助数学推理

多项研究发现，代码数据对数学推理有正迁移效应：代码训练强化了结构化推理、变量追踪和逐步分解的能力，这些能力同样有助于数学解题。Qwen2.5-Math 和 DeepSeekMath 均在数学数据中混入约 20% 的代码数据，并报告了显著的协同增益。

💻 代码能力增强

代码 Mid-Training 是最成熟的子领域，已有多个从同一基座出发的专项模型：

  • CodeLlama：基于 Llama 2，500B token 代码 Mid-Training。关键创新：专门的"代码填充"（Infilling）格式训练，支持 FIM（Fill-in-the-Middle）
  • Qwen2.5-Coder：基于 Qwen2.5，5.5T token 代码数据继续预训练。多个规模（0.5B-72B）均公开，在 HumanEval 上 72B 版本超越 GPT-4o
  • DeepSeek-Coder-V2：6T token 代码+数学数据，MoE 架构，236B 总参数

🧠 推理能力增强

2024-2025 年最新趋势：使用合成推理链数据（CoT traces）进行 Mid-Training，使模型在预训练阶段就内化"思考步骤"的模式。代表工作包括 OpenAI 的 o1 系列前置训练和各种基于 Distillation 的推理链 Mid-Training 方案。

2.3 退火阶段（Annealing / Final Stage）

退火阶段（Annealing Stage）是 Mid-Training 中最精巧、性价比最高的形式。其核心思想是：在预训练接近尾声时，同时执行两个操作——降低学习率（Cosine 衰减至接近零）并引入高质量精选数据集。

为什么退火阶段特别有效？理论角度：学习率降低使模型更细粒度地"记忆"数据，而低噪声的高质量数据此时能以极高效率被吸收。信息瓶颈理论也支持这一解释：退火期模型处于从"记忆"向"压缩"过渡的临界状态，此时高质量数据的学习效率最高。

Llama 3 退火阶段的关键数据

  • 总预训练：约 15.6 万亿 token
  • 退火阶段：仅 40M token（占总量约 0.00026%）
  • 数据组成：高质量数学数据（MATH train set 直接引入）、高质量代码（精选 GitHub）、多样化教材
  • 效果：GSM8K 提升约 +20 分，MATH 提升约 +24 分，HumanEval 提升约 +15 分
📘 Research Insight：Phi 系列的启示

微软的 Phi-1/Phi-2/Phi-3 系列验证了"教科书即是一切"（Textbooks Are All You Need）假说：用 GPT-4 生成的高质量合成教材数据（Phi-1 约 7B token），在编程能力上超越了数倍大小的模型。这一发现强化了退火阶段的设计哲学：数据质量对最终性能的影响，远超数据数量。

2.4 知识更新型（Knowledge Update）

预训练的"知识截止"（Knowledge Cutoff）问题日益突出。2024 年之前训练的模型对 2024 年后的事件一无所知，这在需要最新信息的应用场景中是硬伤。知识更新型 Mid-Training 尝试解决这一问题。

主要技术路线：

  1. 定期增量 Mid-Training：每隔几个月，用最新的网络数据（经过质量过滤）进行一轮 Mid-Training。这是最直接的方案，但成本较高。
  2. 时间戳感知训练：在数据预处理时加入时间戳，训练模型理解知识的时效性，避免将过时信息与最新信息混淆。
  3. 知识冲突处理：当新数据与旧知识矛盾时（如某公司的 CEO 换人了），模型如何正确"覆盖"旧知识而非两者共存产生幻觉，是当前研究的难点。
⚠️ Warning：知识更新的"双重遗忘"风险

知识更新型 Mid-Training 面临双重挑战：一方面要避免遗忘旧的通用能力（经典的灾难性遗忘问题），另一方面要确保新知识真正覆盖了旧的错误知识（而非两者并存）。目前尚无完美解法，实践中建议：(1) 新知识数据与通用数据以 3:1 比例混合，(2) 在知识截止日期之后的问题上做专项评估。

CH 03

核心技术挑战

3.1 灾难性遗忘（Catastrophic Forgetting）

灾难性遗忘是 Mid-Training 最核心的技术挑战。当模型在新的领域数据上持续训练时，其参数会被新任务的梯度更新逐渐覆盖，导致通用能力显著退化。这不是偶发现象，而是神经网络的结构性特点：BP 算法本质上是局部优化，没有机制保护"历史知识"。

💡 Tip：实践中的数据回放比例

工业界普遍接受的数据回放比例是 5-15% 通用数据。比例过低（< 2%）基本无效；比例过高（> 30%）会稀释目标能力的提升速度，延长训练时间。推荐起点：10% 通用数据回放，根据遗忘程度动态调整。

3.2 数据配比问题

数据配比（Data Mixture Ratio）是 Mid-Training 中影响最终性能的核心超参数，但也是最难优化的一个，因为其影响是多维的且具有非线性交互效应。

三类典型配比设计

  1. 重领域轻通用（领域数据 85-95%，通用 5-15%）：快速获得领域能力，但遗忘风险较高。适用于：目标明确的垂直模型（如专用代码助手）。
  2. 均衡配比（领域数据 50-70%，通用 30-50%）：能力提升与遗忘抑制平衡。适用于：通用+专项的混合场景。
  3. 轻量注入（领域数据 20-40%，通用 60-80%）：通用能力保持最好，但领域能力提升有限。适用于：以提升整体能力为主、顺带强化某方向的场景（如 Qwen2.5 的多语言和长文本强化）。
📘 Research Insight：arXiv:2510.14865 的关键发现

2025 年的研究"Midtraining Bridges Pretraining and Posttraining Distributions"发现：引入专业数据的时机（timing）与数据比例（mixture weight）存在强交互效应。早期引入专业数据（预训练前期）时，可承受更高比例（40-50%）；晚期引入（退火阶段）则必须用低比例（5-20%）。这意味着存在一个"可塑性窗口"（Plasticity Window），错过后无法通过提高比例补偿，这对训练计划制定有重要指导意义。

3.3 学习率策略

Mid-Training 的学习率策略需要在两个方向上做出权衡：足够大以使模型有效学习新数据，足够小以避免破坏预训练阶段建立的良好表征。

最优起始学习率选择

经验规律：Mid-Training 的起始学习率通常设为预训练峰值学习率的 1/5 到 1/10。以 Llama 3 70B 为例，预训练峰值 LR 为 1.5e-4，则 Mid-Training 起始 LR 通常在 1.5e-5 到 3e-5 之间。

优化器状态处理

是否需要重置 Adam 的动量（m1, m2）？这是一个有争议的问题：

  • 不重置：利用预训练积累的动量信息，训练更稳定，但可能导致初始阶段学习偏向旧数据分布
  • 完全重置：相当于从一个好的初始化点重新开始，更干净但可能导致初始损失突刺
  • 部分重置（推荐）：重置 m1（梯度动量），保留 m2（二阶矩），在稳定性和适应性之间取得平衡

Warmup 策略

Mid-Training 通常需要一个短暂的 warmup 阶段（约 1-5% 的总 Mid-Training 步数），即使从相对较小的学习率开始也建议保留。原因：预训练结束后的参数状态与 Mid-Training 新数据分布之间存在分布漂移，warmup 有助于平滑这一过渡，减少初始损失突刺。

# 典型 Mid-Training 学习率调度示例
def mid_training_lr_schedule(step, total_steps, base_lr=2e-5, warmup_ratio=0.03):
    warmup_steps = int(total_steps * warmup_ratio)
    
    if step < warmup_steps:
        # 线性 warmup
        return base_lr * (step / warmup_steps)
    
    # Cosine decay 到 base_lr 的 10%
    progress = (step - warmup_steps) / (total_steps - warmup_steps)
    return base_lr * (0.1 + 0.9 * 0.5 * (1 + math.cos(math.pi * progress)))

3.4 训练稳定性

损失突刺（Loss Spike）

Loss Spike 是 Mid-Training 中的常见问题，通常在以下情况下出现：

  • 学习率从预训练结束值直接跳跃到 Mid-Training 起始值
  • 数据分布发生突变（如批次中突然出现大量极长序列）
  • 梯度 norm 因异常样本而突然爆炸

处理方案：(1) 启用梯度裁剪（max_grad_norm = 1.0），(2) 设置序列长度过滤（丢弃过于极端的长序列），(3) 使用 Z-loss（对 logit 的 softmax 指数做 L2 正则，稳定大批量训练），(4) 损失突刺检测自动回滚到上一个健康 checkpoint。

💡 Tip：Checkpoint 策略

Mid-Training 期间建议每隔 0.5B-2B token 保存一个 checkpoint，并记录每个 checkpoint 在代理评估集（Proxy Eval Set）上的性能。原因：(1) 出现损失突刺时可以回滚；(2) 不同下游任务可能在不同 checkpoint 处性能最优（"过训练"现象）；(3) 支持后续的 checkpoint averaging 操作（有时能提升 1-2 分）。

CH 05

数据工程详解

5.1 Mid-Training 数据的特点

Mid-Training 数据与预训练数据在质量要求上存在根本差异。预训练数据强调规模与多样性，容忍一定的噪声（因为大量重复见到各种文本有助于稳健表征学习）；而 Mid-Training 数据强调质量与相关性，低质量数据的危害比预训练期更大，因为此时模型已具备区分"好内容"和"坏内容"的基本能力，但坏内容仍会污染其特定领域的知识。

关键差异对比：

维度 预训练数据 Mid-Training 数据
规模 T 量级（宽松过滤） B-T 量级（严格过滤）
质量要求 中等（过滤明显垃圾） 高（LLM judge 评分）
领域分布 尽量均匀覆盖 刻意偏向目标领域
合成数据比例 低（< 5%） 高（可达 30-50%）
重复数据处理 全局去重 领域内严格去重（更重要）

5.2 数据构建方法

领域数据采集

  • 垂直网站定向爬取：使用领域关键词/URL 白名单精确爬取专业网站。医疗领域：PubMed Central、ClinicalTrials.gov、医学教材扫描版；法律领域：law.cornell.edu、各国法律数据库、判决书数据集；金融领域：SEC EDGAR、财经媒体、研究报告。
  • 书籍与教材数字化：高质量教材是 Mid-Training 数据的"黄金"。Project Gutenberg（古典著作）、O'Reilly（技术书籍）、学术出版社开放获取内容。关键处理步骤：OCR 质量检验、格式标准化（公式 → LaTeX，图表描述化）。
  • 学术论文全文：arXiv（支持 bulk download）、Semantic Scholar Open Corpus、PubMed Open Access。注意：需过滤掉参考文献、致谢等非正文部分。
  • 专业问答社区：StackExchange 的特定子版块（StackOverflow for 代码，Math.StackExchange for 数学，MedSci 等）。QA 格式数据特别有价值：包含问题、回答、评分，天然具有难度标签。

合成数据生成

合成数据在现代 Mid-Training 中占据越来越重要的地位，在某些方向（如数学推理）中甚至成为主导数据源。

  • 基于真实文档生成 QA 对：使用 GPT-4/Qwen-max 等强模型，对真实文档生成多种类型的问题（事实性问答、推理问答、摘要问答），并提供带推理链的参考答案。质量过滤：保留 LLM self-consistency 分数 > 0.8 的样本。
  • 难度分级的练习题生成：特别适合数学和代码领域。使用 WizardMath/WizardCoder 类方法（Evol-Instruct）：从简单问题开始，通过"增加约束、增加推理步骤、组合多个概念"等方式逐步生成更难的变体。
  • 错误示例与纠正对（Error-Correction Pairs）：生成包含常见错误的解题过程，并提供正确的修正版本。这种数据对训练模型的自我纠错能力尤为有效。

数据清洗与过滤流水线

# Mid-Training 数据过滤流水线示例
class MidTrainingDataPipeline:
    def process(self, documents):
        # 1. 基础过滤
        docs = self.filter_basic(documents)
          # - 长度过滤 (50-100K chars)
          # - 语言检测（确保目标语言）
          # - 格式检测（移除乱码/编码错误）
        
        # 2. 领域相关性打分
        docs = self.domain_score(docs)
          # - 使用领域关键词 TF-IDF 打分
          # - 用小型分类器预测领域概率
          # - 保留 top-70% 按相关性排序
        
        # 3. 质量打分
        docs = self.quality_score(docs)
          # - 困惑度过滤（太低=重复，太高=乱码）
          # - LLM judge 打分（0-5分，保留>=3分）
          # - 事实性检验（领域事实一致性）
        
        # 4. 去重
        docs = self.dedup(docs)
          # - MinHash LSH 全局去重
          # - 同来源文档近重复去重（更严格）
        
        # 5. 有害内容过滤（领域特定）
        docs = self.safety_filter(docs)
          # - 医疗：过滤错误用药建议
          # - 法律：过滤明显错误的法律建议
        
        return docs

5.3 数据配比实验方法

直接在全量模型上搜索最优数据配比成本极高（搜索一次配比可能需要数十万 GPU 小时）。工业界采用分层实验策略：

1

构建代理任务（Proxy Tasks）

选择 5-10 个能代表最终目标能力的简短 benchmark（如 MATH-100 子集、HumanEval-20 子集）。代理任务需与最终目标高度相关，且可在小模型上快速计算。

2

1B 模型快速 Ablation

用 1B 参数模型 + 10B token 数据，在代理任务上测试多种配比方案。每个配比实验约 4-8 GPU 小时，可并行测试 10-20 种配比。

3

7B 验证实验

将 1B 实验中排名前 3 的配比，在 7B 模型 + 50B token 上验证。验证小模型结论是否在更大规模上保持（规模迁移性检验）。

4

全量 Mid-Training

使用验证后的最优配比进行正式 Mid-Training。同时设置 checkpoint 评估，监控训练过程中能力的演进曲线，根据需要微调配比。

5.4 工具链与最佳实践

数据格式统一

Mid-Training 数据格式应与预训练格式完全对齐。推荐使用 jsonl 格式，每行一个 JSON 对象，字段包含 text（正文）、source（来源）、quality_score（质量分）、domain（领域标签）。Tokenization 过程中不应引入特殊 token（除非是专门的格式训练如 FIM），以避免破坏预训练学到的 token 分布。

序列长度分布设计

序列长度分布对训练效率影响显著。推荐：(1) 将同领域、长度相近的文档 pack 在一起（document packing），避免过多 padding；(2) 设置合理的长度上限（如 4096 token），超长文档切分时保留语义完整的段落；(3) 在数据 shuffle 时保持 bucket-based shuffle（相同长度的样本优先相邻），减少 padding 浪费。

💡 Tip：Tokenizer 兼容性检查

如果 Mid-Training 目标语言与预训练语言不同（如在英文预训练基础上做中文专项训练），需特别检查 Tokenizer 对目标语言的覆盖率。若目标语言词汇碎片化严重（fertility > 3 token/字），建议考虑词表扩充（Vocabulary Extension），并对新增 embedding 进行热启动初始化（使用语义相近的已有 embedding 的均值）。

TruthfulQA 事实准确性/幻觉 817 ≤ 2%
WinoGrande 常识/指代消解 1,267 ≤ 2%
GSM8K 基础数学推理 1,319 +（通常提升）

遗忘率（Forgetting Rate）计算：

forgetting_rate = (score_after_midtraining - score_before) / score_before × 100%

# 判断标准：
# FR > 5%: 严重遗忘，需要调整配比（增加通用数据回放比例）
# 2% < FR ≤ 5%: 中度遗忘，可接受但需关注
# FR ≤ 2%: 轻微遗忘，正常范围
# FR < 0: 能力提升（理想情况，通常在数学等与预训练数据重叠高的任务上出现）

6.3 评估流水线设计

Checkpoint 评估频率

Mid-Training 过程中的评估频率需要平衡成本与信息密度：

  • 轻量代理评估（Fast Eval）：每 1B token 一次，仅用 5-10 个代理 benchmark（约 100-200 个测试样本）。目的：早期发现训练问题（Loss Spike、严重遗忘）并触发告警。
  • 完整评估（Full Eval）：每 10-20B token 一次，完整通用+领域 benchmark 套件。目的：精确追踪能力演进曲线，支持配比调整决策。
  • 最终评估：Mid-Training 完成后，全面对比基线模型和 Mid-Training 后模型在所有 benchmark 上的表现，并进行人工评估补充。

Early Stopping 策略

Mid-Training 不像有监督训练那样有明确的 validation loss 作为早停信号。推荐策略：(1) 以目标能力的改善斜率作为主要信号——如果连续 3 次评估改善 < 0.5%，说明已接近收益天花板；(2) 以通用能力的遗忘率作为约束——若任一指标遗忘超过 5%，强制触发早停；(3) 结合训练效率：若每增加 10B token 的改善小于预设的 ROI 阈值，则停止。

⚠️ Warning：避免"过训练"（Over-fitting to Domain）

Mid-Training 的"过训练"与监督学习的 overfitting 不同，它表现为：目标领域性能持续上升，但通用能力持续退化，且两者的曲线出现"交叉"。一旦出现交叉信号，应立即停止并回滚到交叉点之前的 checkpoint。这个最优 checkpoint 通常不是训练最长时间的 checkpoint。

前沿研究与趋势

7.1 持续学习（Continual Learning）理论视角

从持续学习理论的视角看，Mid-Training 是解决稳定性-可塑性困境（Stability-Plasticity Dilemma）的工程化实践。稳定性（Stability）指模型保持已学知识的能力；可塑性（Plasticity）指模型获取新知识的能力。这两者在共享参数空间中天然存在矛盾。

持续学习理论将解决方案分为三类：

  • 基于正则化的方法（EWC, SI, MAS）：通过约束参数更新保护旧知识。理论完善，但在亿级参数模型上计算 Fisher 矩阵代价极高，实用性受限。
  • 基于回放的方法（Experience Replay, Generative Replay）：存储或生成旧任务样本进行复习。工程上简单有效，是当前 LLM Mid-Training 的主流做法。
  • 基于架构的方法（Progressive Neural Networks, PackNet）：为新任务分配专属参数空间，不影响旧参数。LoRA/Adapter 是这类方法在 LLM 中的轻量实现。
📘 Research Insight：ADEPT 框架（arXiv:2510.10071）

2025 年 10 月，ADEPT（Adaptive Expansion for Continual Pretraining）提出了一种结合参数扩展和选择性重用的 Mid-Training 方案：通过动态添加轻量模块（类似于 LoRA 专家）处理新领域，同时保持原有参数冻结。在医疗和法律领域测试显示，ADEPT 比标准 CPT 减少约 60% 遗忘，同时保持 85% 以上的领域能力获取效率。

7.2 动态数据配比

静态数据配比（训练前预先确定，训练中不变）是当前主流做法，但存在明显局限：不同能力的学习速度不同，最优配比会随训练进展变化。动态数据配比是 2024-2025 年的研究热点。

主要方向：

  • 基于损失反馈的在线配比调整：监控每类数据的 perplexity 下降速度，对下降慢的领域增加采样权重（类似 DoReMi 方法）。已有多项研究证明这比静态配比平均提升 1-3%。
  • 基于模型能力估计的自适应配比：每隔一定 step，在代理 benchmark 上评估各维度能力，根据与目标能力的 gap 动态调整配比。开销较大但针对性强。
  • Online Mid-Training：将 Mid-Training 与部署融合，根据用户实际查询的领域分布动态更新模型。技术上具有挑战性，需要解决训练-推理并发的工程问题。

7.3 Mid-Training 与 PEFT 的结合

全参数 Mid-Training 的计算成本使许多中小团队望而却步。PEFT（Parameter-Efficient Fine-Tuning）方法与 Mid-Training 的结合是重要的效率优化方向。

方案对比

方案 训练参数量 遗忘程度 能力上限 典型场景
全参数 Mid-Training 100% 中（需回放） 最高 大型团队，追求极致性能
LoRA Mid-Training（r=64-128） 1-3% 最低 中等 领域适应，快速迭代
混合方案（底层全参+顶层 LoRA） 40-60% 低 较高 资源有限的能力增强
Prefix Tuning / Prompt Tuning < 0.1% 极低（参数冻结） 较低 格式适应

实践建议：数学/代码等需要深度知识改变的能力增强，全参数训练明显优于 LoRA（在 70B token 数据量时，全参数 MATH 得分比 LoRA 高约 5-8 分）。但格式适应类任务（如特定输出格式、特定语言风格），LoRA 性价比更高。

7.4 多模态 Mid-Training

随着多模态模型（LLM + Vision）成为主流，多模态 Mid-Training 成为新的热点方向：

  • 视觉-语言对齐 Mid-Training：在语言预训练基础上，注入大量视觉-文本对（图文对、图文交错文档、视觉 QA 数据），激活视觉理解能力。LLaVA、Qwen-VL 等均采用分阶段的视觉 Mid-Training 策略。
  • 代码执行能力注入：通过包含代码执行轨迹（输入代码 → 运行结果）的数据进行 Mid-Training，使模型内化代码语义。这比纯代码文本训练更能提升代码调试和错误理解能力。
  • Tool-use 能力的 Mid-Training：注入工具调用轨迹数据（如 API 调用记录、搜索轨迹、计算器使用过程），使模型在底层就理解工具的使用模式，而非仅依赖 SFT 阶段的格式学习。

7.5 未来趋势预测（2025-2026）

🔭 趋势一：Mid-Training 即新 Pre-Training

OpenAI 研究员 Noam Brown 在 2025 年 7 月的播客中提出"Mid-Training is the New Pre-Training"观点：随着数据质量优化、合成数据崛起，传统意义上的大规模网络爬取预训练可能逐渐收缩，而以高质量数据为核心的 Mid-Training 将承担更多"教育模型"的职责。这一趋势已体现在 Phi 系列的极简预训练路径上。

🔭 趋势二：与强化学习的融合

Mid-Training + RL 的融合形态（"Mid-RL"）正在兴起。典型范式：先用 Mid-Training 注入领域知识（让模型"知道"该领域的解法框架），再用 RL 强化正确解题行为（让模型"练会"高难度问题）。DeepSeek-R1 的成功部分归功于这一流水线：在基础 Mid-Training 后，GRPO/REINFORCE 强化学习能更高效地激发推理能力。

🔭 趋势三：自动化 Mid-Training 配方搜索

类似 AutoML 的思路，使用小规模代理实验自动搜索最优的数据配比、学习率调度和训练时长。基于梯度的配方优化（如通过元学习估计每类数据对最终能力的贡献梯度）是当前研究热点，有望将配比搜索成本从"数十次实验"压缩到"数次实验"。

🔭 趋势四：快速垂直化模式

Mid-Training 正在成为大模型"快速垂直化"的标准路径：开源通用基座（如 Llama 3.1、Qwen2.5）→ 领域专项 Mid-Training → SFT → 垂直行业模型。这一模式的成本已从数百万美元降至数万美元量级，使中小企业也能打造领域专家模型。2025-2026 年预期将有大量行业专属模型通过此路径诞生。

8.1 何时做 Mid-Training

不是所有能力需求都需要 Mid-Training。在决策之前，需要进行系统的能力 Gap 分析：

场景描述 推荐方案 原因
模型"知道"答案但回答格式不对 SFT 即可 格式问题通过行为克隆解决
模型在简单问题上表现好，但复杂问题差 SFT + 困难样本 推理链数据的 SFT 可能足够
模型对专业领域知识完全不熟悉（大量幻觉） 必须 Mid-Training 知识缺口无法靠 SFT 填补
模型无法处理长文本（> 4 倍训练长度） 长文本 Mid-Training 位置编码问题需要在预训练层级解决
代码能力差但语言理解好 代码 Mid-Training 代码是新的语言，需要足够的代码 token 训练
轻微能力不足，有充足的高质量 SFT 数据 SFT 先试 避免 Mid-Training 的过高成本

资源评估

Mid-Training 的计算成本估算（以 7B 模型为例）：

  • 50B token Mid-Training ≈ 50,000 GPU 小时（A100 80G）≈ 约 50 万元人民币（公有云价格）
  • 100B token Mid-Training ≈ 100,000 GPU 小时 ≈ 约 100 万元
  • 使用 Proxy 实验（1B 模型 + 10B token）预实验成本：约 2,000 GPU 小时 ≈ 2 万元
💡 Tip：算力预算不足时的替代方案

如果算力预算不足以支撑全参数 Mid-Training，考虑：(1) LoRA Mid-Training：成本降至约 1/10，能力提升约 60-70%；(2) 更小的基础模型：7B 模型的 Mid-Training 成本约为 70B 的 1/10；(3) 仅做 Annealing：在预训练最后 1% token 阶段引入高质量数据，成本极低，但要求能控制或影响预训练进程。

8.2 标准 Mid-Training 流程

1

确定目标能力与评估指标

明确目标领域/能力，选定 3-5 个核心 benchmark 作为成功标准（如"在 MATH 上从 40% → 65%"）。同时确定通用能力保持的下限（如"MMLU 不低于基线 3%"）。

2

数据采集与构建

按 5.2 节方法构建目标领域数据集。同时准备通用数据回放池（从预训练数据中随机抽取约 5-10% 比例）。建议数据集规模：10-50B token（用于后续代理实验）。

3

小规模 Proxy 实验

用 1B 模型测试 5-10 种数据配比方案（覆盖通用数据比例 5%/10%/20%/30%）。每个实验训练 10B token，在代理 benchmark 上评估。选择能力提升最大且遗忘率 < 5% 的配比。

4

全量 Mid-Training

使用验证后的最优配比，设置合适的学习率（预训练峰值 LR 的 1/5-1/10），启用梯度裁剪（max_grad_norm=1.0），每 5B token 保存一个 checkpoint，每 10B token 进行一次完整评估。

5

评估：双维度系统评估

目标能力提升 + 通用能力保持，选择两个维度综合最优的 checkpoint（非训练最长的 checkpoint）。进行人工评估补充，测试模型在真实场景下的表现。

6

迭代优化 → 后训练

根据评估结果决定是否需要第二轮 Mid-Training（通常不需要超过 2 轮）。确认 Mid-Training 效果后，进行 SFT 和 RLHF 阶段，将预训练期注入的知识与指令遵循能力融合。

8.3 常见坑与避坑

⚠️ 坑 1：忘记加回放数据

最常见的错误。纯领域数据训练会导致通用能力快速退化，MMLU 可能在 50B token 训练后下降 30%+。避坑：强制要求数据配方中至少包含 5% 通用数据，并将其写入训练配置文件的必填项中，设置 lint 检查。

⚠️ 坑 2：学习率设置过大

从预训练峰值 LR（如 1e-4）直接开始 Mid-Training 会导致严重的 Loss Spike 甚至训练崩溃。避坑：起始 LR 设为预训练峰值的 1/5-1/10，并设置短暂 warmup（3% 步数）。如果出现 Loss Spike，立即回滚到上一个稳定 checkpoint，将 LR 降低 2 倍后重启。

⚠️ 坑 3：只看目标能力不看通用能力

兴奋于目标能力提升，忽视通用能力退化，导致交付一个"偏科"模型。避坑：在训练 Dashboard 中将通用能力监控 benchmark（MMLU, HellaSwag）与目标 benchmark 放在同一个看板上，设置通用能力下降 5% 的自动告警。

⚠️ 坑 4：领域数据质量不足时适得其反

有时低质量领域数据（如充满错误的医疗论坛帖子）会使模型在领域任务上的表现比基线更差，因为它"学到"了错误的知识。避坑：在大规模 Mid-Training 前，先用 1B token 小实验验证数据质量——如果在这个规模上领域能力没有提升，不要期望更多数据会神奇地改善情况。先解决数据质量问题再扩大规模。

⚠️ 坑 5：使用被污染的 Benchmark 评估

如果训练数据中包含了 benchmark 的测试题（即使是无意的），评估结果会虚高。尤其是 MATH 和 HumanEval 这类数量有限的 benchmark。避坑：在数据清洗阶段，用 N-gram 匹配过滤掉与常见 benchmark 测试集高度重合的样本。使用 LiveCodeBench（持续更新）替代 HumanEval 评估代码能力。

8.4 快速验证方案（低成本 Proxy 实验）

Mid-Training 团队面临的核心工程问题是：如何以最低成本验证数据配比假设？以下是一套经过验证的 Proxy 实验方法论：

Scale-Down 原则

核心假设：在小规模实验中，数据配比对性能的相对排序具有规模迁移性（即 1B 模型上最好的配比，在 70B 上通常也最好），但绝对值会不同（大模型的提升通常更大）。

# Proxy 实验设计示例
proxy_experiments = {
    "baseline": {"domain": 0.0, "general": 1.0},  # 无 Mid-Training
    "heavy_domain": {"domain": 0.9, "general": 0.1},
    "balanced": {"domain": .7, "general": 0.3},
    "mild": {"domain": 0.5, "general": 0.5},
    "light": {"domain": 0.3, "general": 0.7},
}

# 每个实验：1B 模型，10B token，约 4 GPU 小时
# 并行 5 个实验：总计约 20 GPU 小时 ≈ 2000 元

规模推断规则（Scale-Up Rules）

  • 能力提升幅度：7B/70B 模型的绝对提升通常是 1B 模型的 1.5-2.5 倍（对数线性关系）
  • 遗忘幅度：大模型通常比小模型遗忘更少（更大的参数空间有更好的表征分离）
  • 最优配比迁移性：1B 实验中排名第一的配比，在 7B/70B 上维持第一的概率约 70-80%
  • 警惕反常现象：若 1B 实验中不同配比的差异极小（< 1%），可能需要更长的训练来分辨，不要据此下结论
