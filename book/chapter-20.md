## 20.1 数据处理框架

### 主流框架对比

| 框架 | 开发方 | 特点 | 适用规模 | 许可 |
|------|-------|------|---------|------|
| **datatrove** | HuggingFace | FineWeb 使用的框架，模块化设计 | PB 级 | Apache 2.0 |
| **NeMo-Curator** | NVIDIA | 企业级，GPU 加速去重和过滤 | PB 级 | Apache 2.0 |
| **RedPajama** | Together AI | 完整的数据管线参考实现 | TB 级 | Apache 2.0 |
| **dolma** | AI2 | OLMo 的数据管线 | PB 级 | Apache 2.0 |
| **自建方案** | - | 完全可控 | 取决于工程 | - |

### 框架选择指南

```
你应该选择 datatrove 如果：
  - 你在复现或改进 FineWeb 风格的管线
  - 你需要灵活的模块化组合
  - 你的团队熟悉 Python 和 HuggingFace 生态

你应该选择 NeMo-Curator 如果：
  - 你有大量 GPU 资源
  - 你需要 GPU 加速的去重和过滤
  - 你在 NVIDIA 生态中

你应该自建如果：
  - 你有非标准的数据源或格式
  - 你需要与内部系统深度集成
  - 你的数据量极大（>10PB），需要极致优化
```

## 20.2 大规模数据处理的工程挑战

### PB 级数据的处理

```
处理 10PB 的预训练数据需要考虑：

存储：
  - 原始数据 ~10PB
  - 中间结果 ~30PB（各阶段的临时文件）
  - 最终数据 ~3PB（经过清洗去重后）
  → 总计需要 ~50PB 存储空间

计算：
  - 文本提取：~1000 CPU·天
  - 语言识别：~200 CPU·天
  - 去重（MinHash）：~500 CPU·天 + 2TB 内存
  - 质量过滤：~300 CPU·天（或 50 GPU·天）
  → 总计 ~2000 CPU·天（使用 1000 核集群约 2 天）

成本估算（云）：
  - 存储：50PB × $0.023/GB/月 ≈ $1.15M/月
  - 计算：~$50K-200K（取决于实例类型）
  → 一次完整的数据管线运行成本 ~$100K-300K
```

### 分布式去重的实现

去重是数据管线中计算量最大的步骤。在万亿 token 级别，单机去重不现实。

```
分布式 MinHash 去重方案：

Step 1: 分片计算 MinHash 签名
  - 每台机器处理一个数据分片
  - 输出：(document_id, minhash_signature)
  - 可高度并行

Step 2: LSH 索引构建
  - 将签名分成 b 个 band
  - 相同 band hash 的文档对是候选重复对
  - 使用分布式 hash table（如 Redis cluster）

Step 3: 候选对验证
  - 对候选对计算精确 Jaccard 相似度
  - 超过阈值的标记为重复

Step 4: 连通分量（Union-Find）
  - 将重复对构建成连通图
  - 每个连通分量保留一个代表文档
  - 使用分布式 Union-Find 算法
```

## 20.3 数据管理与版本控制

### 工具选择

```
小规模（< 1TB）：
  DVC + Git
  → 简单易用，与 Git 无缝集成

中规模（1-100TB）：
  lakeFS
  → Git-like 分支模型
  → 支持 S3/GCS 对象存储

大规模（> 100TB）：
  Delta Lake / Apache Iceberg
  → 与 Spark 生态集成
  → 支持 ACID 事务
  → PB 级数据的高效版本管理
```

### 数据血缘追踪

```
数据血缘（Data Lineage）记录每条数据的完整生命周期：

  原始来源 → 提取 → 清洗 → 去重 → 过滤 → 配比 → 训练

每一步都需要记录：
  - 输入是什么
  - 使用了什么参数
  - 输出是什么
  - 有多少数据被过滤掉
  - 时间戳
```

## 20.4 自动化数据管线

### CI/CD for Data

```
数据管线的 CI/CD 流程：

  数据更新（新的 CC 快照、新的代码数据...）
      ↓
  触发数据管线（自动）
      ↓
  阶段性质量检查（自动）
      ├── 通过 → 继续
      └── 失败 → 告警，暂停
      ↓
  数据统计和分布检查（自动）
      ├── 正常 → 继续
      └── 异常 → 告警，人工审核
      ↓
  与历史版本对比（自动）
      ↓
  生成数据报告
      ↓
  人工审批后发布
```

## 本章要点回顾

1. **datatrove 和 NeMo-Curator 是当前最主流的开源框架**
2. **PB 级数据处理的成本**：存储是大头，一次完整管线运行约 $100K-300K
3. **分布式去重是工程最难的环节**——需要分布式 hash table + Union-Find
4. **数据版本管理不是奢侈品**——至少用 manifest 文件记录每个版本
5. **CI/CD for Data**：数据管线应该像代码一样自动化测试和部署
