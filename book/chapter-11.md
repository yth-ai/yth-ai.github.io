# 第 11 章：SFT 数据的本质——表面对齐假说

> *"Almost all knowledge in large language models is learned during pretraining, and only limited instruction tuning data is necessary to teach models to produce high quality output."*
> *——LIMA (Zhou et al., 2023)*
>
> 这句话深刻地改变了我们对 SFT 的认知。

---

## 11.1 LIMA 的启示：Less is More

### 表面对齐假说的深层含义

2023 年，Meta 的研究者用 LIMA 模型向世界展示了一个惊人的结果：**仅用 1000 条精心挑选的指令-回复对，在 65B 参数的 LLaMA 上做 SFT，就能得到接近 GPT-4 水平的对话质量。**

这个结果背后的理论解释被称为"表面对齐假说"（Superficial Alignment Hypothesis）：

> **模型的知识和能力几乎全部在预训练阶段学习。SFT 的作用只是教模型以正确的格式和风格来"展示"这些已有的知识。**

用一个类比：
- 预训练 = 一个人读了十年书，知识渊博但不善表达
- SFT = 教这个人"如何在面试中展示自己"——回答要结构清晰、有条理、有礼貌
- 关键洞察：**SFT 不是在教新知识，而是在教"包装"已有的知识**

### 1000 条精选数据 vs 100 万条噪声数据

LIMA 的实验对比了多种方案：

| 方案 | SFT 数据量 | 数据质量 | 效果 |
|------|-----------|---------|------|
| LIMA | 1,000 条 | 极高（人工精选） | 接近 GPT-4 |
| Alpaca | 52,000 条 | 中等（GPT-3.5 合成） | 明显不如 LIMA |
| 无 SFT | 0 条 | - | 基座模型，无法正常对话 |

这个对比揭示了一个核心原则：**在 SFT 阶段，数据的质量比数量重要得多。** 一条高质量的样本胜过一千条平庸的样本。

### 认知变化链

> **SFT 数据的认知演变**
> - 2022：InstructGPT 时代——"人工标注是金标准，13K 条人工数据 + RLHF"
> - 2023.3：Self-Instruct / Alpaca——"原来可以用 GPT 来合成 SFT 数据！52K 条就行"
> - 2023.5：LIMA——"等等，1000 条精选数据就够了？质量>>数量"
> - 2023-2024：Phi 系列——"精心构造的合成数据可以非常强大"
> - 2024：Llama 3 / Qwen 2.5——"工业实践中用数万到数十万条 SFT 数据"
> - 2025-2026：多阶段 SFT 成为主流——"不同阶段需要不同类型和难度的数据"

## 11.2 SFT 数据的角色定位

### SFT 不是教知识，是教格式和风格

这是理解 SFT 的最关键认知。

```
基座模型（预训练后）的行为：
  User: "什么是光合作用？"
  Model: "光合作用是植物利用光能将二氧化碳和水"
         "转化为有机物和氧气的过程光合作用的化学"
         "方程式为6CO2+6H2O→C6H12O6+6O2光合作用"
         "在叶绿体中进行主要分为光反应和暗反应..."

  → 模型知道答案，但输出是一团糟——没有分段、没有标点、
    没有结构化的解释，像是在"背诵"而不是"回答"。

SFT 后的行为：
  User: "什么是光合作用？"
  Model: "光合作用是植物利用光能将无机物转化为有机物的过程。
         
         ## 基本原理
         植物通过叶绿体中的叶绿素吸收光能...
         
         ## 化学方程式
         6CO₂ + 6H₂O → C₆H₁₂O₆ + 6O₂
         
         ## 两个阶段
         1. **光反应**：在类囊体膜上进行...
         2. **暗反应（Calvin 循环）**：在基质中进行..."

  → 同样的知识，但以清晰、结构化、用户友好的方式呈现。
```

### SFT 激活预训练知识的机制

SFT 的作用可以理解为在模型的"知识空间"和"输出空间"之间建立了一个映射：

1. **格式映射**：学习"用户提问 → 结构化回答"的模式
2. **风格校准**：学习"用什么语气、什么详细度来回答"
3. **安全边界**：学习"哪些问题应该拒绝、如何拒绝"
4. **角色设定**：学习"我是一个助手，我应该如何行为"

### SFT 数据对模型行为的影响边界

SFT 有其能力边界——有些事情它能做到，有些做不到：

| SFT 能做到 | SFT 做不到 |
|-----------|-----------|
| 改变回答的格式（markdown、代码块） | 教模型全新的知识领域 |
| 改变回答的风格（正式/幽默/简洁） | 提升基础语言理解能力 |
| 建立对话的轮次意识 | 大幅提升数学推理能力 |
| 设定安全边界和拒答规则 | 修复预训练阶段的知识错误 |
| 教模型使用工具的格式 | 从根本上消除幻觉 |

> **💡 反直觉发现**
>
> 很多团队在模型"不知道某个知识"时，会尝试通过在 SFT 数据中加入相关知识来"补救"。这通常效果很差——因为几百条 SFT 样本无法与万亿 token 的预训练竞争。**如果模型缺知识，应该回到中训练或预训练去补。SFT 只负责"展示"。**

## 11.3 SFT 数据的分类学

### 按任务类型

```
SFT 数据的任务分类

├── 对话与问答
│   ├── 开放域问答（世界知识）
│   ├── 闲聊对话
│   └── 多轮对话
│
├── 创作与生成
│   ├── 文章写作（报告、邮件、文案）
│   ├── 创意写作（故事、诗歌）
│   ├── 翻译
│   └── 摘要
│
├── 推理与分析
│   ├── 数学推理
│   ├── 逻辑推理
│   ├── 代码生成与调试
│   └── 数据分析
│
├── 指令遵循
│   ├── 格式约束（"用 JSON 格式回答"）
│   ├── 长度约束（"用三句话总结"）
│   ├── 角色扮演
│   └── 多步骤任务
│
└── 工具使用
    ├── 函数调用
    ├── API 交互
    └── 搜索增强
```

### 按难度分层

```
难度级别设计

Level 1: 简单指令遵循
  - "把这段话翻译成英文"
  - "给这篇文章写一个标题"
  - "解释什么是 HTTP 协议"

Level 2: 中等复杂度
  - "对比 REST 和 GraphQL 的优缺点，给出建议"
  - "帮我重构这段 Python 代码，提高可读性"
  - "分析这份销售数据的趋势"

Level 3: 高复杂度
  - "设计一个分布式缓存系统，需要考虑一致性和可用性的权衡"
  - "阅读这三篇论文的摘要，找出它们的共同点和分歧"
  - "帮我写一个完整的 REST API，包括错误处理和测试"

Level 4: 专家级
  - "证明 P≠NP 的一个充分条件（如果存在的话）"
  - "分析这个分布式系统中的死锁条件，给出避免方案"
  - 需要多步骤推理、跨领域知识整合
```

### 按来源分类

| 来源 | 优点 | 缺点 | 适用阶段 |
|------|------|------|---------|
| 人工标注 | 质量最高、可控 | 贵、慢、规模有限 | 种子数据、关键任务 |
| 强模型合成 | 规模大、成本低 | 多样性可能不足、有偏 | 主体数据 |
| 拒绝采样 | 质量有保证 | 依赖好的评分函数 | 质量提升 |
| 人机协作 | 平衡质量和效率 | 流程复杂 | 最佳实践 |

## 11.4 SFT 数据量的决策

### 你到底需要多少 SFT 数据？

这取决于你的目标和基座模型的质量。

```
经验性指导：

极简方案（LIMA 风格）：
  - 数据量：1K-5K 条
  - 适用：强基座模型 + 简单对话场景
  - 特点：每条数据都经过人工精选

标准方案：
  - 数据量：10K-100K 条
  - 适用：通用助手模型
  - 特点：人工精选 + 合成数据混合

工业方案（Llama 3 风格）：
  - 数据量：100K-1M+ 条
  - 适用：全能力覆盖的商业模型
  - 特点：多阶段 SFT，覆盖数十种任务类型

关键原则：
  不是"越多越好"，而是"覆盖越全面越好"。
  重复加入同类型的简单数据，边际效用快速递减。
```

---

## 本章要点回顾

1. **表面对齐假说**：SFT 教的是"如何表达"，不是"知道什么"
2. **质量远大于数量**：1000 条精选 > 100K 条噪声
3. **SFT 有能力边界**：补知识应该回到中训练/预训练
4. **数据需要分层设计**：任务类型、难度、来源三个维度
5. **数据量不是越多越好**：覆盖全面比总量大更重要
