如何提升 LLM 的 Agentic Coding 能力

面向大模型预训练 / 后训练团队的系统性技术综述

研究领域：代码 LLM · Agentic AI · 强化学习 截止时间：2025 年 12 月 文档版本：v1.0

执行摘要 Abstract

Agentic Coding 是指大语言模型（LLM）以自主 Agent 的方式完成真实软件工程任务的范式，区别于传统单步代码补全：模型需要在多轮交互中自主分解任务、导航代码库、调用工具（终端、文件系统、测试运行器）、检测与修复错误，最终完成从需求到可运行代码的完整链路。这一范式已成为 2024—2025 年 AI 编程领域最核心的研究方向。

核心发现（五条）：

  1. 数据是第一瓶颈：SWE-smith（2025）证明，通过自动化合成 5 万条真实仓库级任务实例，训练出的开源 32B 模型在 SWE-bench Verified 上达到 40.2%，远超此前最优开源模型——规模化的高质量 agentic 训练数据是突破性能上限的关键。
  2. RL with Verifiable Rewards（RLVR）是 agentic 后训练的核心技术路径：Agent-RLVR（2025）、DeepSWE 等工作证明，以单元测试通过率为可验证奖励，结合 agent 引导机制，可将基础模型性能从 9.4% 提升至 22.4%（Qwen-2.5-72B），甚至通过纯 RL 从零训练达到 SOTA（DeepSWE）。
  3. 扩展思考（Extended Thinking）对复杂多步任务效果显著：o3、Claude 3.7 Sonnet、DeepSeek-R1 的实践表明，long CoT + GRPO 训练可使代码推理能力大幅提升，尤其是在需要多步规划的 agentic 场景中。
  4. 自我改进（Self-Improvement）循环是可行且高效的：SWE-Gym 的实验表明，通过拒绝采样微调（RFT）让模型在自己生成的轨迹上迭代训练，7B 模型可在 SWE-bench Lite 上达到 10%，32B 达到 20%，且存在清晰的训练/推理时算力扩展规律。
  5. 评估体系亟需升级：SWE-bench 存在数据污染和任务难度分布问题，SWE-bench Verified、SWE-bench Pro、CodeClash 等新 benchmark 正在形成更完善的多维评估体系；内部评估体系的建立对团队迭代至关重要。

研究意义：本报告面向大模型预训练/后训练团队，系统梳理从数据工程到训练策略、从架构设计到评估体系的完整技术路径，重点给出可操作的工程建议，帮助团队在 6—12 个月内建立有竞争力的 Agentic Coding 能力。

第一章：Agentic Coding 概述

1.1 定义与范畴

传统代码生成（Code Generation）是单步、被动的：给定一个自然语言提示或代码上下文，模型一次性输出代码片段，不涉及后续执行、验证或修正。GitHub Copilot、早期 Codex 均属于此范式。

Agentic Coding 则是多步、主动的自主软件工程能力。根据 Wang et al.（2025, arXiv:2508.11126）的综述定义，Agentic Coding Agent 需要具备：

Agentic Coding 核心特征： 自主分解目标 → 制定执行计划 → 调用外部工具（编译器、调试器、测试运行器、版本控制） → 观察执行结果 → 基于反馈迭代修正 → 最终交付可运行的代码变更

具体的核心能力维度如下：

1.1.1 任务理解与分解（Task Understanding & Decomposition）

真实软件工程任务往往描述模糊（如 GitHub issue）、需求隐含约束（向后兼容、性能要求）。模型必须将高层目标分解为可执行子任务序列，并在执行过程中动态调整计划。SWE-agent 的实验表明，良好的任务分解是提高解决率的首要因素。

1.1.2 代码库导航与上下文检索（Codebase Navigation）

真实仓库动辄数十万行代码，模型无法将全部内容纳入上下文窗口。需要通过文件搜索（grep/ast-grep）、定义跳转（LSP）、相关文件识别等手段，检索出与当前任务最相关的上下文。Agentless 的实验证明，精准的文件定位（file localization）是决定最终成功率的关键子任务之一。

1.1.3 工具使用（Tool Use）

Agentic Coding 系统需要调用多种工具：

  • 终端（Terminal）：执行 bash 命令、git 操作、包管理
  • 文件系统：读取、创建、修改文件，导航目录结构
  • 测试运行器：pytest/unittest/cargo test 等，获取失败信息
  • 搜索工具：grep、ripgrep、代码搜索引擎
  • Web 搜索：查阅文档、API 参考
  • 代码执行沙箱：Python REPL、Node.js 解释器

1.1.4 多步规划与执行（Multi-step Planning & Execution）

一个典型的 SWE-bench 任务可能需要 20—80 步工具调用：首先复现 bug，然后定位问题文件，分析根因，设计修复方案，实现代码变更，运行测试验证，再根据测试结果进一步迭代。模型需要在整个过程中维护全局计划和局部执行状态。

1.1.5 错误检测与自我修正（Error Detection & Self-Correction）

Agentic 系统区别于单步生成的核心优势之一是执行反馈驱动的修正循环。模型需要能从编译错误、测试失败、运行时异常中提取有效信号，诊断根因，并生成修正代码。这要求模型具备对错误信息的深度语义理解，而非简单的表面匹配。

1.1.6 长上下文处理（Long Context）

Agentic 任务中的上下文包括：仓库文件内容、执行历史轨迹（tool calls + observations）、任务描述，累积可达 32K—200K+ tokens。如何在超长上下文中保持关键信息的注意力是核心挑战。

第二章：预训练数据层面的提升

2.1 代码预训练数据的构成

2.1.1 公开代码语料

当前主流代码预训练数据集包括：

  • The Stack v2（BigCode, 2024）：覆盖 600+ 编程语言，约 67B 代码文件，经过去许可证过滤，是目前规模最大的开源代码语料。
  • StarCoder2 训练数据：在 The Stack v2 基础上加入代码相关网页文本（Stack Overflow、文档）、Jupyter notebooks，共 约 4T tokens。
  • GitHub 全量爬取：DeepSeek-Coder 系列使用了覆盖 87% 代码、13% 自然语言的混合比例，代码来自 GitHub 全量（含协议过滤）。
  • 代码相关网页文本：技术博客、官方文档、Stack Overflow、Reddit 技术讨论等，为模型提供代码使用的自然语言上下文。

2.1.2 数据规模与代码 token 占比

从已公开的技术报告来看，代码 token 占比与模型 coding 能力存在显著正相关，但存在边际收益递减：

模型 总预训练 tokens 代码占比 HumanEval 特点
Llama 2 (2023) 2T ~4% 29.9% 通用基础模型
DeepSeek-Coder-Base (2024) 2T 87% 78.6% 专用代码模型
Qwen2.5-Coder (2024) 5.5T ~70% 92.7% 数据+规模双提升
StarCoder2-15B (2024) 4T ~95% 46.8% 极高代码占比
DeepSeek-V3 (2024) 14.8T ~30% 86.6% 通用+代码平衡
关键发现：纯代码模型在 HumanEval 上表现优异，但在需要自然语言理解的 agentic 任务（如解读 issue 描述、与用户交互）上表现较弱。最优代码预训练比例通常在 30%—70% 之间，具体取决于是否有后续的专项微调。对于 Agentic Coding 任务，建议在大规模通用预训练基础上，使用代码密集的 Continual Pre-training（继续预训练），而非从零开始的纯代码预训练。

2.1.3 多语言分布设计

编程语言分布设计直接影响模型的跨语言泛化能力。核心原则：

  • 主流语言保证覆盖深度：Python、JavaScript/TypeScript、Java、C/C++、Rust、Go 等应有足够的高质量数据量
  • 长尾语言保证覆盖广度：通过上采样（oversampling）确保 SQL、Bash、Dockerfile、YAML 等配置/脚本语言的表示
  • 跨语言对齐数据：相同算法的不同语言实现对（Python↔Java, Python↔C++）有助于迁移学习
  • 配套自然语言：代码注释、文档字符串、README、变更日志等是不可忽视的"代码自然语言"

2.1.4 代码质量过滤

原始 GitHub 数据质量参差不齐，需要多层过滤：

# 代码质量过滤流水线（示意）
filters = [
    "remove_auto_generated",    # 去除自动生成代码（如 protobuf 生成的桩代码）
    "remove_minified",          # 去除压缩/混淆代码
    "exact_dedup_hash",         # 基于 hash 的精确去重
    "near_dedup_minhash",       # MinHash LSH 近似去重
    "syntax_check",             # 语法合法性检查（可解析 AST）

    "license_filter",           # 许可证合规过滤
    "pii_removal",              # 个人信息脱敏
]

2.2 Agentic 相关数据类型

这是本章最关键的部分。与传统代码补全数据不同，Agentic Coding 需要包含工具交互和执行反馈的轨迹数据。

2.2.1 执行轨迹数据（Execution Trajectories）

最直接、最有效的 agentic 预训练数据是完整的问题解决轨迹，格式为：

# ReAct 格式轨迹示例
Thought: 我需要先理解这个 issue：用户报告 `sort()` 在空列表时崩溃。
         让我先找到相关代码。

Action: bash
```
grep -r "def sort" /repo/src/ --include="*.py"
```

Observation:
src/utils/sorting.py:45:def sort(items):

Thought: 找到了，让我查看完整实现。

Action: read_file
```
/repo/src/utils/sorting.py
```

Observation:
def sort(items):
    pivot = items[0]  # BUG: 空列表会 IndexError
    ...

Thought: 找到根因。需要处理空列表边界情况。

Action: edit_file
```
if not items:
    return []
pivot = items[0]
```

Action: run_tests
```
pytest tests/test_sorting.py -v
```

Observation: PASSED (5/5 tests)

这类数据的主要来源：

  • SWE-smith（2025）：自动从 128 个 GitHub 仓库构造 5 万条任务实例，用强模型生成解决轨迹，经测试验证过滤
  • SWE-Gym（2024）：2,438 个真实 Python 任务，用 GPT-4o 和 Claude 3.5 Sonnet 采样轨迹
  • 真实 git commit 历史：从 GitHub commit 中提取"问题报告 → 代码修改 → 测试"的自然链路，规模可达亿级
  • Issue → PR → Merge 链路：包含完整的需求描述、讨论、实现、审查过程

2.2.2 工具调用数据（Tool Use Data）

工具调用能力需要特殊的数据来训练模型正确格式化和执行工具调用：

  • Function Calling 格式数据：OpenAI、Anthropic 格式的工具调用 JSON 数据，包含工具定义、调用参数、返回结果三元组
  • Shell 命令执行序列：从真实开发者的终端日志、CI/CD 脚本中提取的命令序列（带错误处理）
  • API 调用序列：REST API、SDK 调用的完整序列，包含认证、错误处理、分页等真实模式
  • 文件系统操作日志：真实的文件读写操作序列，训练模型理解目录结构和文件依赖

2.2.3 多文件上下文数据

Agentic Coding 的核心难点之一是跨文件的依赖理解。关键数据类型：

  • 代码库级别 QA 数据：给定一个代码库的多个文件，回答关于代码行为、潜在 bug、依赖关系的问题
  • 跨文件重构任务：如重命名函数并更新所有调用点、抽取公共接口等，需要理解全局依赖
  • 大型代码库导航数据：训练模型从模糊任务描述出发，系统性地定位相关文件的能力

2.2.4 测试驱动数据（Test-Driven Data）

  • 测试用例 + 实现代码对：先给测试，要求生成通过测试的实现（TDD 范式）
  • 报错信息 → 修复代码对：从 Stack Overflow、GitHub issue 中提取的 "error → fix" 对，规模庞大且质量多样
  • TDD 完整执行轨迹：包含多轮测试失败→修改→重测的完整迭代过程

2.3 数据质量提升策略

2.3.1 基于执行结果的数据过滤

这是 SWE-smith 等工作的核心创新：只保留能通过单元测试的轨迹数据。

  • 构建可执行的沙箱环境（Docker 容器，每个仓库独立）
  • 运行模型生成的解决方案对应的测试套件
  • 只有"所有相关测试通过"的轨迹才进入训练集
  • 这保证了训练数据的正确性，而非仅仅看起来合理
实证效果： SWE-smith 使用这种过滤策略，从约 2M 生成轨迹中筛选出约 5 万条高质量数据（约 2.5% 通过率），用于训练的 SWE-agent-LM-32B 达到了 40.2% 的开源 SOTA。相比未过滤数据，质量过滤带来了约 8—15% 的绝对性能提升。

2.3.2 LLM 辅助质量打分

使用强模型（如 GPT-4o、Claude）对代码数据进行多维度打分：

  • 可读性：变量命名、代码组织、注释质量
  • 最佳实践：是否遵循语言惯例、错误处理是否完善
  • 教育价值：代码是否清晰展示了某种算法或设计模式
  • 解决方案质量：相对于"hack"式修复，是否是真正的根因解决

2.3.3 教育价值评估

Qwen2.5-Coder 的技术报告中提出了"教育价值（Educational Value）"指标：参考 Stack Overflow 高赞回答、权威教程、知名开源库的核心代码，这些内容在预训练中应获得更高权重。具体实施方式：

  • Stack Overflow 答案按点赞数加权采样
  • 官方文档代码示例优先级高于随机爬取的博客代码
  • 知名开源项目（如 CPython、Linux kernel、NumPy）的核心代码给予 2—3× 权重

2.4 数据配比与 Curriculum 学习

2.4.1 代码 vs 自然语言比例

经验表明，最优的代码 LLM 需要充足的自然语言能力以理解需求、与用户交互、生成注释和文档。推荐的配比参考：

训练阶段 代码占比 自然语言占比 代码相关文本占比 适用场景
通用预训练 20-35% 55-70% 5-10% 通用+代码能力均衡
代码继续预训练 60-80% 10-20% 10-20% 代码能力专项强化
Agentic 继续预训练 40-60% 10-15% 10-20% 引入轨迹数据

2.4.2 难度课程设计（Curriculum Learning）

课程学习（先易后难）对代码 LLM 效果存在争议，但以下策略已被多个工作验证有效：

  • 阶段 1（基础代码）：单文件函数级代码，有完整的输入/输出类型和注释
  • 阶段 2（复杂单任务）：多函数、有依赖的模块级代码；带测试用例的实现任务
  • 阶段 3（仓库级任务）：跨文件修改、轨迹格式数据；短 trajectory（5—10步）
  • 阶段 4（长轨迹数据）：完整的 issue → fix 轨迹（20—80步）；多文件重构
重要洞察：课程的关键不仅是代码复杂度，还有工具交互深度。建议从"无工具调用"到"单工具调用"再到"多轮多工具调用"逐步递进，让模型先掌握单步工具使用，再学习工具组合和错误恢复。

2.4.3 领域覆盖设计

确保训练数据覆盖主要软件工程领域，防止在特定领域的能力盲点：

  • Web 开发：前端（React/Vue）、后端（Django/FastAPI）、全栈
  • 系统编程：操作系统、网络编程、并发、内存管理
  • 数据科学/ML：NumPy/Pandas 数据处理、PyTorch/TensorFlow 模型代码
  • 数据库：SQL 编写、ORM、数据库迁移脚本
  • DevOps：CI/CD 脚本、Docker/K8s 配置、自动化运维
  • 安全：常见漏洞模式、安全编码实践

第三章：后训练层面的提升

3.1 SFT（监督微调）

3.1.1 SFT 数据构建策略

Agentic Coding 的 SFT 数据与普通代码 SFT 存在本质区别：需要多步轨迹格式，而非单轮问答。

数据来源优先级排序：

  1. 可验证的自动生成轨迹（最高优先级）：利用 SWE-smith 等框架，从 GitHub 仓库自动生成并通过测试验证的轨迹。成本低（约 $0.01—0.1/条），质量可靠，规模可达数十万条。
  2. 强模型蒸馏数据：用 Claude Opus / GPT-4o 在 SWE-bench 类任务上生成解决轨迹，过滤出成功的 trajectories。质量高但成本较高（$0.5—5/条）。
  3. 人工标注的黄金轨迹：由专业软件工程师完成的标注轨迹，质量最高但成本极高（$50—500/条）。用于困难样本和多样化覆盖。
  4. 从 benchmark 衍生的训练数据：利用 SWE-bench Verified 以外的任务（训练集）构造 SFT 数据，但需注意不得使用 Verified 测试集的任务。

3.1.2 Agentic SFT 的格式设计

当前主流的 agentic SFT 格式有三种，各有优劣：

格式 设计理念 优点 缺点 代表系统
ReAct Reason + Act 交替 直觉清晰，易理解 思考与行动分离，效率较低 早期 SWE-agent
CodeAct 用代码作为统一行动空间 可组合、可并行、减少格式错误 需要可执行环境 OpenDevin
Tool-integrated Reasoning 思考 token 内嵌工具调用 推理更深入，错误分析更充分 训练难度更高 Claude 3.7, o3

业界最佳实践：倾向于 Tool-integrated Reasoning 格式，其中 <thinking> 块内进行自由推理，工具调用以结构化 JSON 或 XML 格式输出。这使模型能在"行动前深思熟虑"，减少鲁莽行动导致的错误传播。

3.1.3 中间步骤正确性处理

多步轨迹的一个核心挑战是：整体成功≠每步最优。处理策略：

  • 仅监督最终结果正确的轨迹：最保守但最可靠，避免学习错误子步骤
  • 过程监督（Process Supervision）：对每步的"合理性"单独打分（ORM vs PRM），但需要额外标注成本
  • 容错轨迹（Fault-tolerant Trajectories）：保留"走了弯路但最终成功"的轨迹，这类数据教会模型错误恢复能力，尤其有价值

3.2 强化学习（RLHF/RLVR）

RL 是提升 Agentic Coding 能力最具潜力的后训练方法。其核心优势在于：模型可以超越专家轨迹的质量上限，通过自主探索发现更好的解决策略。

3.2.1 奖励信号设计

① 可验证奖励（Verifiable Rewards）— 首选

代码任务天然适合可验证奖励，因为代码的"正确性"可以通过执行来客观验证。这是当前学术和工业界最主流的路径：

奖励类型 具体实现 优点 局限性
单元测试通过率 pass@1 by test suite 客观、精确、直接对齐目标 测试覆盖率不足时可能 reward hacking
编译/语法正确 AST parse success 零成本、快速信号 仅保证语法，不保证语义
Lint/静态分析 pylint/eslint scores 代码质量导向 可能与任务完成度不一致
diff 应用成功率 git apply && tests pass 端到端验证完整流程 需要完整执行环境

② LLM-as-Judge 奖励

对于难以用程序验证的方面，使用强模型作为评判者：

  • 代码质量 Rubric：按可读性（1-5）、效率（1-5）、安全性（1-5）等维度打分
  • 任务完成度评估：Judge 模型判断代码是否在语义上解决了 issue 中描述的问题（即使测试套件不完整）
  • 参考实现相似度：对于有参考答案的任务，评估语义等价性（非字面匹配）
警告： LLM-as-Judge 方法存在奖励模型 hacking 风险：模型可能学会生成"看起来好但实际错误"的代码来欺骗 Judge。建议将可验证奖励作为主信号，LLM-as-Judge 作为辅助信号或用于 reward shaping。

③ 过程奖励（Process Reward Models, PRM）

Process Reward 对轨迹中每个步骤单独打分，而非仅对最终结果：

  • 工具调用的合理性：是否选择了正确的工具、参数是否合理
  • 中间思考步骤质量：推理是否逻辑连贯、是否正确识别了问题所在
  • 挑战：需要大量人工标注中间步骤，成本高昂。SWE-Gym 通过训练轨迹验证器（trajectory verifier）来近似 PRM，取得了一定效果。

3.2.2 RL 训练策略对比

算法 核心机制 Agentic Coding 表现 计算开销 推荐场景
PPO 策略梯度+价值函数+clip 稳定，适合长轨迹 高（需要 critic 模型） 长轨迹 agentic 任务
GRPO 组相对策略优化（无 critic） DeepSeek-R1 验证有效 中（组内对比） 代码推理、竞赛题
DPO 偏好对比学习（离线） 对齐任务，coding 效果次于 PPO 低（无需 online rollout） 风格对齐、代码质量
RFT 拒绝采样微调 简单有效，SWE-Gym 验证 中（需大量采样） 自我改进初始阶段
Agent-RLVR RLVR+引导机制 解决稀疏奖励问题 中高 难 agentic 任务

3.2.3 解决 Agentic RL 的稀疏奖励问题

Agentic 任务的核心 RL 难点是奖励稀疏性：对于复杂的 20—50 步任务，大多数尝试都以失败告终，模型几乎得不到正向信号，无法有效学习。

Agent-RLVR（2025）提出了Agent Guidance 机制来解决这个问题：

  1. 模型先自主尝试任务，生成初始轨迹
  2. 单元测试验证结果（大多失败）
  3. 对失败轨迹，提供导师引导信息：高层策略提示（"你需要先找到问题文件"）、错误分析（"测试失败原因是..."）、环境反馈（"你遗漏了边界情况"）
  4. 模型在引导下重新尝试，成功率显著提升
  5. 用引导后的轨迹进行 RLVR 更新
实证结果：Agent-RLVR 将 Qwen-2.5-72B-Instruct 在 SWE-bench Verified 上从 9.4% 提升至 22.4%，进一步结合测试时奖励模型达到 27.8%。这比简单 RLVR（无引导）提升了约 10 个百分点。

3.2.4 On-policy 数据采集策略

RL 训练需要持续的 on-policy 数据：

  • 任务分布：在难度梯度上采样任务，避免只用简单/困难任务
  • 温度采样：使用 temperature=0.7—1.0 增加轨迹多样性，防止 mode collapse
  • 执行环境隔离：每个 rollout 使用独立 Docker 容器，保证环境干净
  • 并行采样：Agentic 任务耗时长（每条轨迹 1—10 分钟），需要大规模并行推理基础设施

3.3 扩展思考与推理（Extended Thinking / CoT）

3.3.1 Long CoT 在 Agentic Coding 中的作用

o1/o3 风格的扩展思考在复杂 coding 任务中展现出显著优势。OpenAI 的实验（arXiv:2502.06807）显示，o3 在国际信息学奥林匹克（IOI 2024）中可在不依赖任何手工启发式策略的情况下获得金牌。

Long CoT 的作用机制：

  • 深度规划：在输出任何代码前，先在思考 token 中进行充分的方案探索和比较
  • 错误预防：在"行动"前预先考虑边界情况、潜在 bug 和测试覆盖
  • 系统性调试：通过"思考"对执行失败进行深度根因分析，而非直觉性修改
  • 多方案比较：同时思考多种实现路径，选择最合适的一种

3.3.2 DeepSeek-R1 式 GRPO 训练

DeepSeek-R1 的训练范式为代码推理提供了重要参考：

# GRPO 训练范式简述
for each_task in coding_tasks:
    # 1. 对同一任务采样 G 个轨迹
    trajectories = [model.rollout(task) for _ in range(G)]
    
    # 2. 用可验证奖励（单元测试）评分
    rewards = [run_tests(traj.output) for traj in trajectories]
    
    # 3. 组内归一化（无需 critic 模型）
    baseline = mean(rewards)
    advantages = [(r - baseline) / std(rewards) for r in rewards]
    
    # 4. 策略梯度更新
    loss = -sum(advantages[i] * log_prob(trajectories[i]) 
               for i in range(G))
    loss.backward()

3.3.3 思考 Token 预算设计

思考 token 的数量与任务复杂度应该动态匹配：

  • 简单任务（单文件 bug fix）：100—500 thinking tokens 通常足够
  • 中等任务（多步骤功能实现）：500—2000 thinking tokens
  • 复杂任务（架构重构、跨多文件）：2000—10000 thinking tokens
  • 训练建议：不对思考 token 长度强制约束，让模型学会自适应分配思考预算（类似 o3 的 adaptive compute）

3.4 Self-Improvement 与 Iterative Training

3.4.1 拒绝采样微调（Rejection Sampling Fine-tuning, RFT）

RFT 是 Self-Improvement 的核心基础技术，也是进入完整 RL 训练前的重要步骤：

  1. 用当前模型对训练任务采样 K 个解（K=16—64 是常见范围）
  2. 通过单元测试验证每个解的正确性
  3. 只保留正确的解，用于 SFT 更新模型
  4. 重复上述过程（多轮迭代）
SWE-Gym 实验数据：通过 RFT 自我改进，Qwen-2.5-32B 在 SWE-bench Lite 上从 ~12% 基础性能提升至 20%，Qwen-2.5-7B 提升至 10%。关键发现是性能与训练 / 推理时计算量存在清晰的 Scaling Law，暗示这一方向有很大的提升空间。

3.4.2 AlphaCode 风格大规模采样

AlphaCode 2（DeepMind）和类似方法使用"宽"而非"深"的策略：

  • 对每道竞赛题生成 1000—100万 个候选解
  • 使用聚类 + 过滤：将候选解按语义聚类，每类选一个代表，通过初步测试的提交
  • 最终选择得分最高的候选提交
  • 局限性：适合竞赛编程（有官方测试用例），在 agentic 场景中执行环境成本较高

3.4.3 多轮 Self-Improvement 循环

完整的 self-improvement 循环（参考 STaR、V-STaR 等工作）：

  • 轮次 1：用 SFT 模型生成轨迹，RFT 过滤，更新模型
  • 轮次 2：更新后的模型能解决更多之前失败的任务，用这些新数据再次 SFT
  • 轮次 3+：引入 RL 信号（RLVR）进一步提升，防止纯 SFT 的 distribution shift 问题
  • 关键挑战：随着模型变强，能自主解决的任务中"简单任务"比例上升，需要持续引入更难的任务来维持训练信号质量

第四章：架构与推理层面

4.1 长上下文处理

4.1.1 Agentic Coding 的上下文需求分析

Agentic Coding 任务中的上下文来源和规模：

上下文来源 典型规模 重要性
任务描述（issue / PR） 200—2,000 tokens 必须完整保留
相关代码文件 1,000—50,000 tokens 核心上下文
历史工具调用轨迹 5,000—100,000 tokens（长任务） 关键状态信息
测试失败信息 500—5,000 tokens 错误诊断依据
代码库概览/README 1,000—10,000 tokens 架构理解
总计（复杂任务） 50,000—200,000+ tokens —

4.1.2 长上下文训练技术

  • RoPE 扩展（YaRN / LongRoPE）：通过缩放 RoPE 旋转位置编码，将模型从 8K context 扩展到 128K—200K，训练成本相对较低
  • 长文档持续预训练：在长文本数据（代码仓库、书籍、法律文档）上继续预训练，帮助模型适应长距离依赖
  • LongFilter（2025）：通过量化"长程信息增益"筛选长上下文训练数据，保留真正需要长距离依赖理解的样本
  • Sliding Window / GQA：减少注意力计算开销，在保持长上下文能力的同时降低推理成本

4.1.3 代码的特殊 Attention 模式

代码与自然语言的 attention 模式有显著差异：

  • 局部性：函数内部的 token 相互关注度高（局部语法结构）
  • 跨文件引用：跨越数千个 token 的函数调用、类继承关系（长程依赖）
  • 标识符一致性：同一变量/函数名在整个上下文中需要保持一致的语义绑定
  • 缩进感知：Python 等语言中缩进层级直接影响语义

4.2 工具调用架构

4.2.1 Function Calling 训练方式

有效的工具调用训练需要覆盖：

  • 工具选择：从工具集中选择最合适的工具（多分类问题）
  • 参数生成：正确填充工具参数（结构化输出）
  • 调用时机：判断何时需要调用工具（vs. 直接生成答案）
  • 结果解析：正确理解工具返回结果并更新内部状态
  • 错误处理：工具调用失败时的恢复策略（重试、使用替代工具、报告无法完成）

4.2.2 工具错误处理训练

一个常被忽视但极其重要的能力：在工具失败时优雅降级。训练数据应覆盖：

  • 文件不存在 → 搜索正确路径
  • 命令执行失败 → 检查权限/语法，尝试替代命令
  • 测试失败 → 分析错误信息，定位 bug 并修复
  • 网络超时 → 本地缓存查询或跳过
  • 权限不足 → 请求用户授权或选择其他方案

4.2.3 并行工具调用

对于无依赖关系的工具调用，并行执行可显著减少总耗时。训练模型识别可并行的工具调用序列，例如：同时读取多个文件、并行运行多个独立测试模块。这需要模型输出特殊的并行调用格式，并能正确处理并发结果。

4.3 多智能体架构

4.3.1 Orchestrator + Worker 模式

将复杂任务分解给多个专业化 Agent 协同完成：

  • Orchestrator：接收高层任务，分解子任务，分配给 Worker，整合结果
  • Worker 类型：代码搜索 Agent、代码修改 Agent、测试执行 Agent、文档查询 Agent
  • 通信机制：结构化的任务/结果消息格式，清晰的上下文传递规范
  • 训练挑战：多 Agent 系统难以端到端训练，通常需要分别训练各角色，再通过系统级评估调整

4.3.2 代码 Review 循环

Generator → Critic → Reviser 三角结构可显著提升代码质量：

  • Generator：生成初始解决方案
  • Critic：指出代码中的问题（逻辑错误、边界情况、代码质量）
  • Reviser：根据 Critic 的反馈修改代码
  • 多轮迭代直至 Critic 认可或达到最大轮次
  • 关键发现：Critic 和 Generator 可以是同一个模型，通过不同 prompt 驱动不同行为

4.3.3 测试驱动的 Agentic Loop

将单元测试结果作为 agentic loop 的核心反馈信号：

  1. 理解需求，编写测试（TDD）
  2. 实现代码，运行测试
  3. 失败 → 分析错误信息 → 修改代码 → 重新测试
  4. 成功 → 检查边界情况、添加更多测试
  5. 全部通过 → 提交

这一 loop 的优势是自然地将可验证奖励嵌入推理过程，使 RL 训练的目标与实际推理行为完全对齐。

第五章：关键挑战与开放问题

挑战 1：长轨迹泛化（Long Trajectory Generalization）
训练时的 trajectory 长度 vs 推理时的真实长度
高优先级

训练时，由于计算成本，多数工作使用 10—30 步的轨迹数据（约 4K—16K tokens 的 context）。但真实的复杂软件工程任务可能需要 50—100+ 步，context 累积到 64K—200K tokens。这导致模型在推理时面临分布外（Out-of-Distribution） 的轨迹长度，容易出现注意力退化、重复步骤、忘记之前的发现等问题。

当前解决方向：① 轨迹压缩（summarize 历史轨迹）；② 长上下文专项训练（参见 4.1）；③ 基于检索的状态管理（将历史写入外部记忆）；④ 子任务分解（将长任务拆分为独立短任务）。

挑战 2：错误传播（Error Propagation）
早期错误如何在多步推理中级联放大
高优先级

在多步 agentic 任务中，第 3 步的错误判断可能导致后续 20 步的努力全部基于错误前提，最终以失败告终。这个问题在需要精确理解代码语义的任务中尤为突出（如错误地定位到了非相关文件）。

缓解策略：① 训练模型频繁进行自我校验（"我的理解是否与测试结果一致？"）；② 过程奖励模型（PRM）提供中间步骤反馈；③ 训练包含"纠错恢复"的轨迹数据；④ 多路径投票（beam search over trajectories）在推理时选择最可靠路径。

挑战 3：探索 vs 利用平衡（Exploration vs. Exploitation）
RL 阶段如何有效探索解空间
中优先级

代码 RL 训练中，模型倾向于快速收敛到少数已知有效策略（如"先用 grep 搜索"），而忽略可能更优但需要更多探索才能发现的策略（如"先阅读测试文件来理解期望行为"）。过于保守的探索会导致模式单一、在新类型任务上泛化差。

解决方向：① 温度/采样多样性调度；② KL 散度正则化防止策略崩塌；③ 课程任务调度保持任务多样性；④ Intrinsic curiosity reward 鼓励探索新策略；⑤ 多 Agent 并行探索不同策略路径。

挑战 4：评估对齐（Benchmark-Reality Gap）
Benchmark 分数 vs 真实工程价值
中优先级

SWE-bench 存在多个已知问题：① 数据集相对老旧（多数 issue 来自 2017—2020 年的 Python 库），与最新工程实践存在分布差异；② 测试套件可能不完整，允许"取巧"通过（如删除测试）；③ 重度使用者可能发生训练集/测试集污染；④ 仅覆盖 Python，代表性有限；⑤ 典型任务比真实生产代码简单。

SWE-bench Pro、SWE-bench Multilingual、LiveCodeBench、CodeClash 等新 benchmark 正在尝试解决这些问题。团队应建立自己的内部评估集（真实业务代码库上的任务），防止 benchmark 过拟合。

挑战 5：安全与可控（Safety & Controllability）
Agentic 系统的权限控制与沙箱
不可忽视

赋予 Agent 真实的代码执行能力（写文件、运行命令）带来了显著的安全风险：模型可能意外删除重要文件、执行不安全的网络请求、写入恶意代码（prompt injection 攻击）。在企业内网部署时，风险更为突出。

主要缓解措施：① 沙箱隔离（Docker 容器，只读挂载关键目录）；② 工具权限白名单（只暴露必要的工具）；③ 危险操作确认机制；④ 操作日志审计；⑤ 在训练数据中包含"拒绝危险请求"的样本。

挑战 6：成本效率（Cost & Latency）
长轨迹推理的计算开销与延迟问题
工程关键

一个典型的 SWE-bench 任务平均消耗 20,000—100,000 tokens（Claude 3.5 约 $0.1—$1.0/任务）。在企业级应用中，成本和延迟是产品化的主要障碍。扩展思考（thinking tokens）进一步增加 2—5× 的 token 消耗。

优化方向：① Speculative decoding 加速 token 生成；② KV Cache 跨轮次复用；③ 小模型做初步过滤，大模型做复杂决策；④ 训练更高效的轨迹（减少冗余步骤）；⑤ 并行工具调用减少等待时间；⑥ 结果缓存（相似任务复用历史轨迹片段）。

第六章：代表性工作深度解析

SWE-agent
Princeton NLP Group | 设计了首个系统性的 Agent-Computer Interface
2024

SWE-agent 由普林斯顿大学 NLP 团队于 2024 年发布，是将 LLM 应用于真实软件工程任务的奠基性工作之一。其核心贡献是Agent-Computer Interface（ACI）的系统性设计。

ACI 的核心理念是：就像 GUI 是为人类设计的界面，ACI 是专门为 LLM Agent 设计的计算机交互界面。具体设计包括：① 专用文件查看工具：与 cat 不同，SWE-agent 的 open 命令会显示行号并支持滚动，更适合 LLM 处理；② 搜索工具：专为 LLM 优化的 search_file 和 find_file，输出格式经过裁剪，只显示相关上下文；③ 编辑工具：edit 命令接受起止行号和新内容，相比生成完整文件更高效；④ 内置错误处理：工具层面自动捕获常见错误并给出友好提示。

实验表明，ACI 的精心设计对 Agent 性能有显著影响——使用相同的 GPT-4，优化 ACI 后性能从约 7% 提升至 12%。这一发现强调了"接口设计"对 Agentic 系统的重要性，不亚于底层模型能力。

SWE-agent 在 2025 年 3 月推出了 1.0 版本，成为 SWE-bench Lite 的开源 SOTA；mini-SWE-agent 则以仅 100 行 Python 代码实现了 65% 的 SWE-bench Verified 性能，证明了简洁设计的竞争力。

CodeAct
University of Illinois Urbana-Champaign | 统一代码执行作为行动空间
2024

CodeAct（Wang et al., UIUC, 2024）提出了一个优雅而强大的观点：用可执行代码作为 LLM Agent 的统一行动空间，而非设计专用的工具调用格式或自然语言指令。

传统 Agent 框架为每种操作定义专门的 action 格式（读文件={"action":"read","path":"..."}，执行命令={"action":"bash","cmd":"..."}），这带来了 schema 设计负担和格式错误风险。CodeAct 的核心洞察是：Python 代码天然地是一种图灵完备的行动语言，可以表达任意操作，且语法已经被 LLM 深度学习。

在 CodeAct 框架中，每次 Agent "行动"都输出一段 Python 代码：读文件用 open()，执行 shell 用 subprocess，调用 API 用 requests——所有操作统一用 Python 表达。这带来了几个关键优势：① 可组合性：复杂操作可以用 Python 控制流自然表达；② 并行化：使用 ThreadPoolExecutor 自然并行执行无依赖操作；③ 错误处理：Python 的 try/except 天然支持异常处理；④ 减少格式错误：LLM 生成 Python 代码比生成自定义 JSON schema 更可靠。

实验显示，CodeAct 在 28 个任务上比 JSON-based 方法平均提高 20+%。这一思想已被 OpenDevin/All-Hands 等平台采纳为核心设计原则。

OpenDevin / All-Hands
All-Hands AI | 开源 Agentic Coding 平台
2024—2025

OpenDevin（现更名为 OpenHands，由 All-Hands AI 维护）是目前最活跃的开源 Agentic Coding 平台，累计 GitHub Stars 超过 40K（截至 2025）。它不仅是一个推理框架，也提供了完整的训练基础设施。

核心技术特点：① CodeAct 2.0 框架：改进的代码执行行动空间，支持状态保持的 Jupyter 内核和 bash 会话；② 沙箱化执行环境：每个 Agent 会话运行在独立 Docker 容器中，提供安全隔离；③ 事件流架构：所有 Agent 行动和环境观察以统一的事件流格式记录，天然适合轨迹数据收集和回放；④ 多 Agent 框架：支持 Orchestrator-Worker 模式，不同 Agent 可以专业化分工。

在训练方面，All-Hands 与学术团队合作（如 SWE-Gym 论文），将 OpenHands 作为 Agent 框架，在 SWE-Gym 环境中采样训练轨迹，实现了显著的性能提升（基于 32B 模型达到 SWE-bench Verified 32%）。这为"以 OpenHands 为框架构建训练数据流水线"提供了完整的开源参考实现，对资源有限的团队极具价值。

Agentless
UIUC | 无 Agent 框架的简单有效方法
2024

Agentless 来自 UIUC（Liu et al.，2024），以其"反直觉"的设计理念引发了广泛关注：在 SWE-bench 上取得具有竞争力成绩，而完全不使用 Agent 框架（无自主工具调用，无多步推理循环）。

Agentless 的核心流程分三个固定阶段：① 文件定位（Localization）：利用代码库结构（文件路径、函数名）和 issue 关键词，通过多轮 LLM 调用确定需要修改的文件和代码位置；② 修复生成（Repair）：针对定位到的代码段，生成多个（如 10 个）候选补丁；③ 补丁过滤（Filtering）：运行现有测试套件，筛选出通过测试的候选补丁，必要时用 LLM-as-Judge 排序。

Agentless 的价值在于揭示了两个关键洞察：一是文件定位准确性（localization）对最终成功率的影响比 Agent 框架本身更大；二是在有足够计算预算的情况下，"生成多个候选+过滤"的简单策略可以与复杂 Agent 竞争。这对资源有限时的工程选型和对 Agentic 训练数据中"候选生成+验证"组件的重视都有指导意义。

Claude 3.7 Sonnet
Anthropic | 扩展思考 + 专项 Coding 能力提升
2025

Claude 3.7 Sonnet 是 Anthropic 在 2025 年初发布的旗舰模型，在 Agentic Coding 上取得了当时的 SOTA 成绩（SWE-bench Verified ~62%）。其最重要的技术特性是扩展思考（Extended Thinking）。

扩展思考允许模型在输出前进行长达数千 token 的内部推理过程。与 o1 的隐藏 thinking 不同，Claude 3.7 可以选择性地向用户展示部分思考过程，增加了可解释性。在 coding 任务中，扩展思考的价值体现在：① 系统性分析 issue 的根本原因，而非急于跳到代码修改；② 在复杂多文件任务中维护全局状态和计划；③ 在提交修复前进行"思维实验"（预演修改后的代码行为）。

Anthropic 同期推出了 Claude Code（命令行 agentic 工具），并发布了专项的 agentic coding 训练技术，包括：专门的工具调用格式训练、长上下文 coding 对话数据、以及针对 agentic loop 的 RLHF 数据。后续的 Claude Opus 4 系列（2025 下半年）在这一方向上进一步演进，SWE-bench Verified 突破 70%，成为业界领先水平。

DeepSeek-Coder V2 / V3
DeepSeek AI | 开源 SOTA 代码模型系列
2024—2025

DeepSeek 代码模型系列是开源社区在 Agentic Coding 能力上最具竞争力的成果之一。

DeepSeek-Coder V2（2024）基于 MoE 架构（16B active / 236B total），使用 87% 代码 + 13% 自然语言的预训练比例，在 HumanEval 达到 90.2%，在多项代码 benchmark 上超越 GPT-4 Turbo（当时）。关键技术：FIM（Fill-in-the-Middle）训练、代码特定的 tokenizer 优化、多语言对齐数据。

DeepSeek-V3（2024 末）是通用大模型，但代码能力极强（SWE-bench Verified ~42%），技术亮点：① 从 DeepSeek-R1 的推理能力蒸馏；② Multi-Token Prediction（MTP）辅助目标；③ 使用 FP8 混合精度训练，仅消耗约 278 万 H800 GPU 小时完成 14.8T token 的训练，成本效率极高。

DeepSeek-V3.2（2025）进一步引入Agentic 能力专项优化：通过 Agentic RL 训练提升工具使用和规划能力，SWE-bench Verified 接近 78%，成为当时开源最优水平。其 Sparse Attention 和对 agentic 任务的专门后训练是核心技术差异点。DeepSeek 系列证明了以较低训练成本实现 SOTA 的可能性，对资源受限团队极具参考价值。

Qwen2.5-Coder
Alibaba DAMO Academy | 数据为中心的代码 LLM
2024

Qwen2.5-Coder 是阿里巴巴 DAMO 院于 2024 年 11 月发布的代码 LLM 系列，覆盖 0.5B—72B 多个规模，其核心贡献是系统性的数据工程方法论，而非新颖的架构设计。

Qwen2.5-Coder 的数据构建包括四个层次：① 基础代码语料：在 The Stack v2 基础上，对 GitHub 代码进行额外爬取，总量超过 5.5T tokens；② 质量过滤：针对代码库特性设计的多维质量过滤，包括教育价值（educational value）、代码清晰度、注释质量等，使用专门训练的分类器打分；③ 合成数据：使用 Qwen2.5-72B-Instruct 生成大量高质量代码问答、代码解释、Bug 修复对等合成数据；④ Code Evolution 数据：模拟代码的演化过程，包含多版本代码和变更说明。

Qwen2.5-Coder-72B-Instruct 在 HumanEval 上达到 92.7%，在 MBPP 上达到 90.9%，在多个代码 benchmark 上超越 GPT-4o。其7B 模型表现尤其亮眼，在小模型中具有显著竞争力，对计算资源有限的应用场景非常友好。团队的数据工程方法论已成为代码 LLM 领域的重要参考。

SWE-smith + SWE-agent-LM-32B
MIT CSAIL / Princeton | 规模化 SWE 训练数据生成
2025

SWE-smith（Yang et al., 2025, arXiv:2504.21798）是 2025 年 Agentic Coding 训练数据领域最重要的工作之一，直接解决了训练数据稀缺的核心痛点。

核心贡献：设计了一套自动化的软件工程训练数据生成流水线。给定任意 Python 代码库，SWE-smith 能：① 自动构建对应的可执行 Docker 环境；② 分析代码库结构，找到现有测试套件；③ 自动在代码库中"注入"bug（修改代码使某些测试失败），生成任务实例；④ 用多个强 LLM（GPT-4o, Claude 3.5）尝试解决每个任务，采样多条候选轨迹；⑤ 只保留通过所有测试的成功轨迹作为训练数据。

利用 SWE-smith，团队从 128 个 GitHub 仓库生成了约 5 万条高质量训练实例，比此前工作大一个数量级（此前最大约 5,000 条）。基于此数据训练的 SWE-agent-LM-32B 在 SWE-bench Verified 上达到 40.2%，成为当时开源 SOTA。这证明了数据规模对 Agentic Coding 能力的决定性作用，也为后续扩展提供了方向：通过扩展到更多仓库（目前 128，可扩展至数千）和更多语言（当前主要是 Python），进一步提升训练数据量。

第七章：实践建议（面向训练团队）

7.1 数据工程优先级

短期（1—3 个月）

ROI 最高的投入

  • 建立 SWE-bench Verified 评估基线
  • 收集并清洗现有开源代码语料（The Stack v2）
  • 从 GitHub 爬取高质量 issue→PR→merge 链路
  • 构建代码质量过滤器（语法检查+LLM打分）
  • 建立简单的 SFT pipeline（500—2000 条人工/蒸馏轨迹）
中期（3—6 个月）

Agentic 数据工程

  • 部署 SWE-smith 类流水线，自动生成 1—5 万条训练实例
  • 建立 Docker 沙箱执行基础设施
  • 收集工具调用数据（shell 序列、function calling）
  • 构建测试驱动数据集（test→impl 对）
  • 开始 RFT 自我改进实验
长期（6 个月+）

完整 Agentic RL Pipeline

  • 搭建完整 RLVR 训练框架（PPO or GRPO）
  • 扩展训练数据至 10—50 万条
  • 覆盖更多编程语言（Go, Rust, Java）
  • 引入 extended thinking 训练
  • Multi-agent 训练框架
ROI 分析：在短期投入中，优先级建议为：① 高质量 SFT 数据（500条蒸馏轨迹）> ② 代码质量过滤 > ③ 数据规模扩展。500 条高质量 agentic 轨迹（SWE-Gym 论文验证）可带来 +14% 绝对性能提升，超过单纯增加数据量 10× 的效果。质量优先于数量是核心原则。

7.2 评估体系建立

7.2.1 内部 Benchmark 构建建议

外部 benchmark 存在数据污染风险和业务不对齐问题，建议同步建立内部评估集：

  • 真实业务仓库任务：从团队实际的代码库中，选取有意义的历史 bug fix 作为评估任务（10—50 个，代表不同难度和类型）
  • 合成边界任务：针对已知薄弱点（如多文件重构、特定语言、DevOps 脚本）手工设计评估任务
  • 污染检测机制：定期用新生成的任务（确保未在训练集中出现）进行评估，验证真实泛化能力

7.2.2 推荐评估指标体系

指标维度 具体指标 权重（建议） 评估工具
核心能力 SWE-bench Verified 解决率 40% SWE-bench 官方
代码推理 LiveCodeBench Pass@1 20% LiveCodeBench
终端交互 Terminal-Bench 通过率 15% Terminal-Bench
业务对齐 内部 benchmark 解决率 15% 团队自建
效率 平均轨迹步骤数、Token 消耗 10% 自定义监控

7.2.3 持续监控体系

  • 评估频率：每次模型更新后触发自动评估（SWE-bench Lite 约 10 美元/轮，可作为频繁评估标准；Verified 约 100 美元/轮，用于重要里程碑）
  • 回归检测：建立基准线，自动报警任何 >2% 的性能下降
  • 能力分析：按任务类型（bug fix / feature / refactor）、代码库大小、修改文件数等维度分解性能，定位薄弱点
  • 错误分析：定期人工分析失败案例，分类常见失败模式

7.3 训练技术路线图

阶段 名称 核心技术 预期提升 关键风险
阶段一 代码预训练增强 Continual Pre-training on code + trajectory data 基础 coding +10—20% 灾难性遗忘通用能力
阶段二 Agentic SFT 500—5000 条高质量 agentic 轨迹 SFT SWE-bench +15—25% 数据质量参差，格式过拟合
阶段三 RLVR 训练 单元测试可验证奖励 + PPO/GRPO SWE-bench +10—20% 奖励 hacking，训练不稳定
阶段四 Self-Improvement Loop RFT → RLVR 多轮迭代 + 难度课程 持续提升，无明确上限 模式坍塌，需持续补充新任务

具体行动项

阶段一（代码预训练增强）实施细节：

  • 从 The Stack v2 + GitHub 收集 500B—2T code tokens
  • 应用多层质量过滤（语法 + LLM 打分 + 去重）
  • 包含代码执行轨迹数据（git commit history, issue-PR pairs）
  • 混合比例：代码 60% + 代码文档 20% + 通用文本 20%
  • 使用 LoRA 或 Full Fine-tuning，学习率 ≤ 预训练的 1/10（防灾难性遗忘）

阶段二（Agentic SFT）实施细节：

  • 目标数据量：500—2000 条（短期），5000—20000 条（中期）
  • 数据来源优先级：SWE-smith 类自动生成 > 强模型蒸馏 > 人工标注
  • 格式：Tool-integrated Reasoning（<thinking>...</thinking> + JSON 工具调用）
  • 包含 15—20% 的"错误恢复"轨迹（走弯路但最终成功）
  • 多轮格式：每条数据包含完整任务，平均 15—30 步工具调用

阶段三（RLVR 训练）实施细节：

  • 首选算法：GRPO（无需 critic，实现简单，DeepSeek 验证有效）
  • 奖励函数：单元测试通过率（主信号，权重 0.8）+ 代码质量 LLM-Judge（辅助，权重 0.2）
  • 任务难度：先从 SWE-bench Lite 中等难度任务开始（基础成功率 10—30%），逐步提升难度
  • 关键超参：组大小 G=8—16，KL 系数 β=0.05—0.1，学习率 1e-6
  • 针对稀疏奖励：引入 Agent Guidance（Agent-RLVR 方案）或课程调度

阶段四（Self-Improvement）实施细节：

  • 每月执行一轮：在最新任务池（包括新增任务）上采样 K=16 个轨迹/任务
  • 过滤：只保留通过测试的轨迹（预计 5—15% 通过率）
  • 用新数据进行增量 SFT（混入旧数据防止遗忘）
  • 持续扩充任务池：每月添加新 GitHub 仓库，防止任务池饱和
  • 监控多样性：如果模型在大多数任务上已能成功，立即切换到更难的任务池
关键成功因素总结：
  1. 可执行验证是核心：所有训练数据（SFT + RL）都应经过单元测试验证，宁缺毋滥。
  2. 数据质量 > 数据量：500 条高质量轨迹比 10,000 条低质量轨迹效果好。
  3. 从 SFT 开始，再 RL：不要从零开始用 RL 训练，先用 SFT 建立基础能力再 RL 精调。
  4. 评估先行：在任何训练前建立完善的评估体系，确保能量化每一步投入的价值。
  5. 避免 benchmark 过拟合：使用多个 benchmark 和内部评估集，定期轮换测试任务。
