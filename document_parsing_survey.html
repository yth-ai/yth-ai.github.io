文档解析技术综述

从规则匹配到多模态大模型的全面演进

文档解析是 RAG、知识库构建、信息抽取等一切下游任务的"第一公里"。本报告系统梳理规则方法、Pipeline 流水线、 端到端 OCR 模型、VLM 直接解析四大技术路线的核心优劣，深度介绍 MinerU、Docling、Marker、olmOCR、GOT-OCR、 Nougat、Mistral OCR 等主流方案，并梳理 2025–2026 年前沿进展与技术展望。

🌍文档解析的挑战

文档解析不只是"提取文字"——它需要理解版式结构、语义关系、多模态内容，并以机器可消费的格式输出。真实文档的多样性让这个任务远比想象中困难。

📐 版式多样性

  • 单/多列混排，期刊、PPT、报表各不相同
  • 复杂表格：跨行/列合并、嵌套表格、无边框表
  • 公式、代码块、图注、脚注的阅读顺序还原
  • 页眉页脚、水印、批注的干扰噪声去除

🖼️ 内容多模态

  • 扫描件（图片 PDF）vs 数字原生 PDF 处理差异巨大
  • 图表（柱状图/折线图）需理解而非仅截图
  • LaTeX 公式、化学式的结构化还原
  • 手写内容、低分辨率、偏斜、污损的鲁棒性

⚙️ 工程现实

  • 速度 vs 精度的 trade-off：工业场景对吞吐量要求极高
  • 部署成本：GPU 依赖、模型大小、推理延迟
  • 多语言（中/日/韩/阿拉伯语等）支持难度
  • 下游适配：Markdown / JSON / HTML 等格式要求不同

🗂️方案分类体系

按技术路线可分为四大类，各有其适用场景和能力边界。

① 规则 / 启发式方法

PyMuPDF · pdfminer · pdfplumber
  • 直接解析 PDF 内部数字结构（字符位置、字体信息）
  • 无需模型，速度极快（毫秒级），内存占用极低
  • 对数字原生 PDF 效果好；扫描件/复杂版式束手无策

② Pipeline 流水线方法

MinerU · Marker · Docling · Surya
  • 版式检测（YOLO/LayoutLM）+ OCR + 后处理多阶段串联
  • 各模块可替换，工程灵活性高；对扫描件/复杂文档强
  • 模块错误累积，推理速度受限于最慢模块

③ 端到端 OCR 模型

GOT-OCR 2.0 · Nougat · olmOCR · TrOCR
  • 图像→文本的统一端到端模型，无模块划分
  • 对特定领域（学术论文/公式）精度高
  • 输出结构相对简单；推理速度受模型大小影响

④ VLM 直接解析

GPT-4o · Mistral OCR · Gemini · Claude
  • 多模态大模型直接理解文档图像，输出结构化文本
  • 理解能力最强，表格/图表语义解析能力突出
  • API 成本高、速度慢、难以本地部署

⚖️核心 Trade-off 深度分析

文档解析没有"最好的方案"，只有"最合适的方案"。以下是四组最关键的取舍。

⚡ 速度 vs 精度

规则方法处理一页 PDF <1ms；Pipeline 方法（如 MinerU）约 0.5–3s/页（GPU）；VLM 方案每页调用可达 5–20s。精度方向则完全相反。

💡 实践建议：两阶段策略——先用规则方法快速分类文档类型（数字/扫描/混合），再分流到对应的精度优先方法。
🏠 本地部署 vs API 服务

MinerU/Marker/Docling 支持纯本地 GPU 部署，适合隐私敏感场景（医疗/法律/金融）。Mistral OCR / GPT-4o 等仅提供 API，数据上传风险不可忽视。

💡 数据隐私优先选 Pipeline 本地方案；精度优先且数据非敏感选 VLM API。
📊 结构提取 vs 语义理解

Pipeline 方法善于还原文档结构（阅读顺序、标题层级、表格边界），但不理解"这个表格说的是什么"。VLM 善于理解语义，但结构还原不一定准（如复杂嵌套表格的 Markdown 格式）。

💡 RAG 场景用 Pipeline 还原结构；需要摘要/问答理解场景可叠加 VLM 二次处理。
🎯 通用性 vs 领域专精

通用方案（MinerU/Docling）覆盖场景广，但在专业领域（医学影像报告、工程图纸、法律合同）不如针对性微调的模型。Nougat 专攻学术论文效果远超通用方案，但对其他文档类型效果差。

💡 数据源单一且专业，优先考虑专项微调；来源多样的通用场景用 MinerU/Docling。

📐规则 / 启发式方法详解

直接解析 PDF 的数字结构，速度最快、成本最低，是数字原生文档处理的首选。

PyMuPDF（fitz）

MuPDF 引擎 · Python binding · 工业级
  • 核心能力：极速提取文字、图像、元数据、超链接；支持输出结构化 JSON（含位置坐标）
  • 版本亮点：PyMuPDF 1.24+（2024）加入 RAG/LLM 专用模式（page.get_text("markdown")），自动处理列排版
  • 局限：扫描件完全无效；复杂跨列表格识别错误率高；公式仅提取纯文本
  • 适用：数字原生 PDF 批量处理、速度 > 精度场景、无 GPU 环境

pdfplumber / pdfminer.six

Python · 细粒度坐标控制
  • 核心能力：暴露每个字符的位置、字体、颜色信息；pdfplumber 在表格提取方面比 PyMuPDF 更细腻（可基于线段识别表格边界）
  • 表格提取：通过检测 PDF 内的线段自动识别表格，对有边框表格效果好
  • 局限：仅限数字 PDF；大文件性能较差；无列排版还原能力
  • 适用：需要精确坐标的文档结构分析、有边框表格提取
✅ 优势
  • 处理速度极快（<10ms/页），支持百万级批量处理
  • 零 GPU 依赖，部署成本几乎为零
  • 输出确定性高，可复现，调试方便
  • 对有序数字 PDF 精度极高（>95%）
  • 工程控制性强，每一步都可调试和干预
❌ 劣势
  • 扫描件/图像 PDF 完全无法处理（需先 OCR）
  • 多列排版阅读顺序容易混乱
  • 公式仅能提取纯文本符号，无法还原 LaTeX 结构
  • 复杂嵌套表格识别效果差
  • 依赖 PDF 编码质量，"坏 PDF"会产生乱码
🔧Pipeline 流水线方法详解

模块化设计，版式检测 + OCR + 结构理解串联，是当前开源文档解析的最主流方案。

⛏️
MinerU（PDF-Extract-Kit）
opendatalab · OpenDataLab/MinerU · 2024–2025 · 最活跃的开源 PDF 解析项目

架构

  • 版式检测：DocLayout-YOLO（opendatalab 自研，97.5 mAP，YOLOv10 based）
  • OCR：PaddleOCR（默认）或 PP-OCRv4 server 模型；支持 surya OCR 替换
  • 公式识别：UniMERNet 端到端公式识别，支持行内/块级公式 LaTeX 输出
  • 表格识别：TableMaster / RapidTable，输出 HTML 格式
  • 阅读顺序：LayoutReader 模型还原多列顺序
  • 输出：Markdown + JSON（含坐标）

2025 年更新亮点

  • 1.3.x 系列（2025 Q1-Q2）：公式解析优化，公式渲染成功率大幅提升
  • 支持 VLM 模式：将版式检测换成 VLM 整页理解，精度进一步提升
  • MinerU CLI 大幅优化，支持批量并行处理
  • ModelScope 一键部署，国内用户友好
  • OmniDocBench 评测中综合排名第一（开源方案）
📖
Docling
IBM Research · 2024.09 开源 · arXiv:2408.09869 · IBM watsonx 生产系统来源

架构与设计理念

  • 版式检测：DocLayNet 数据训练的 RT-DETR 模型，对多种文档类型覆盖广
  • 表格结构：TableFormer（IBM 自研），在有/无边框表格上均表现稳定
  • OCR：EasyOCR / Tesseract 可选；扫描件走 OCR 流程，数字 PDF 直接解析
  • 设计重点：统一的 DoclingDocument 数据结构，所有格式输出一致
  • 输出：Markdown / HTML / JSON，以及 LlamaIndex/LangChain 集成

适用场景与优势

  • CPU 友好：无需 GPU 也可运行（速度会慢），对轻量部署更友好
  • 表格识别稳定：TableFormer 对无边框表格效果优于大多数开源方案
  • 生态完善：官方提供 LlamaIndex 和 LangChain 集成，RAG 场景开箱即用
  • Docling v2（2025.01）：加速 3–5× ，加入 VLM 增强模式（docling-ibm-models）
  • 局限：公式识别不如 MinerU；中文支持相对弱
🖊️
Marker
Vik Paruchuri · 2023.12 开源 · 轻量 Pipeline，社区广泛使用

架构

  • 版式检测：Surya layout（自研小模型）
  • OCR：Surya OCR（自研，多语言支持好，90+ 语言）
  • 公式：Texify 模型（数学公式 → LaTeX）
  • 输出：Markdown，简洁可读性强
  • 特点：代码简洁，易于定制，社区活跃

Marker v1 更新（2025）

  • 加入 VLM 增强路径（可用 Claude/GPT-4V 做 fallback），大幅提升复杂页面精度
  • 原生支持 --use_llm 参数，一行命令启用 VLM 增强
  • 速度最优：纯 Surya 模式下处理速度是三大开源方案中最快的
  • 局限：表格识别精度稍低于 Docling/MinerU；公式密集文档效果不如 MinerU
✅ Pipeline 整体优势
  • 模块可替换，工程灵活性高，便于针对特定文档类型优化
  • 完全本地化部署，无数据隐私风险
  • 扫描件与数字 PDF 都能处理，通用性强
  • 开源生态完善，社区活跃，持续迭代
  • 输出格式标准化（Markdown/JSON），下游集成方便
❌ Pipeline 整体劣势
  • 模块串联误差累积：版式检测错 → OCR 范围错 → 结构错
  • 需要 GPU 才能达到工业级速度（1–3s/页）
  • 模型组合复杂，调参门槛较高
  • 手写内容、严重倾斜/污损文档效果较差
  • 复杂嵌套表格、跨页表格识别仍是难题
🔤端到端 OCR 模型详解

统一的图像→文本模型，无需手动组装 pipeline，特定领域精度可超过流水线方案。

🎓
Nougat
Meta AI · EMNLP 2023 · arXiv:2308.13418 · 学术论文专用

设计与原理

  • 架构：Donut 风格（Vision Encoder + AR Decoder），图像→LaTeX/Markdown 端到端
  • 训练数据：arXiv 论文 PDF 与对应 LaTeX 源码对齐，学术文档分布
  • 公式：行内/块级 LaTeX 公式还原精度业界最高（学术文档场景）
  • 输出：结构化 Markdown，含标题层级、公式、引用

局限与适用

  • 强烈领域偏置：在非学术文档（报表、合同）上效果急剧下降
  • 速度较慢：AR 解码逐 token 生成，长页面耗时明显
  • 幻觉问题：AR 模型会在低置信区域产生不存在的内容
  • 最佳场景：arXiv 风格学术论文批量转 Markdown，公式密集文档
🌐
GOT-OCR 2.0
阶跃星辰（StepFun）· arXiv:2409.01704 · 2024.09 · 580M 参数统一端到端模型

设计理念：OCR-2.0

  • 统一架构：高压缩 ViT 编码器 + 长上下文 Qwen 解码器，580M 参数
  • 多任务覆盖：纯文本 OCR、场景文字、公式（LaTeX）、乐谱、图表（纯文字输出）、手写体
  • Local Attention：支持高分辨率输入（1024×1024），减少计算量
  • 交互式解析：支持框选区域 OCR、颜色区域 OCR

公式识别特长

  • 在 UniMER-Test 等公式识别基准上超越 Nougat 和专用公式识别模型
  • 支持行内公式、块级公式、化学式的统一识别
  • 局限：表格识别为纯文本提取，无结构（行/列）理解；多列文档阅读顺序不稳定
  • 最佳场景：公式识别、单页混合内容文档、不需要表格结构的场景
🌿
olmOCR / olmOCR 2
Allen Institute for AI (AI2) · arXiv:2502.18443 · 2025.02 / v2 arXiv:2510.19817 · 2025.10

设计目标：为预训练数据生产而生

  • 核心目标：高吞吐量地从 PDF 中提取高质量文本，用于 LLM 预训练数据集构建
  • 架构：基于 Qwen2-VL-7B 微调，在 26 万页 PDF（10 万文档）上训练
  • 保留阅读顺序：特别优化阅读顺序恢复，保证提取文本语义连贯性
  • 多类型支持：含表格、公式、手写内容

olmOCR 2 升级（2025.10）

  • Unit Test Rewards：用 RLVR 风格训练（可验证奖励），显著提升结构化区域准确率
  • 规模：可横向扩展到 100+ GPU 并行，适合万亿 token 规模文本提取
  • 完全开源：模型权重 + 训练数据 + 代码全部开源
  • 局限：主要为文本质量优化，不输出 Markdown 结构标注（表格输出较简单）
✅ 端到端 OCR 整体优势
  • 无模块拼接问题，错误不累积
  • 对特定领域可做高度针对性微调（如 Nougat 的学术论文）
  • GOT-OCR 等小模型（580M）部署成本相对低
  • 公式识别精度是四类方案中最强的（GOT/Nougat）
❌ 端到端 OCR 整体劣势
  • AR 解码速度慢，长文档吞吐量低
  • 幻觉问题：低置信区域容易凭空生成内容
  • 表格结构理解弱于专用 Pipeline 方案
  • 通用性不如 Pipeline，跨领域表现不稳定
🤖VLM 直接解析详解

多模态大模型直接"看"文档图像并输出结构化文本，理解能力最强，成本最高。

Mistral OCR / Mistral OCR 3

Mistral AI · API · 2025.03 / OCR3 2025.12
  • 定位：专为文档解析优化的多模态 API（非通用对话模型），结构感知架构
  • OCR3 性能（2025.12）：对 OCR2 综合胜率 74%，在表单/扫描件/复杂表格/手写体上大幅提升
  • 优势：输出稳定的 Markdown/JSON，多语言支持极强，价格相对 GPT-4o 便宜
  • 定价（2025）：约 $1/1000 页，比 GPT-4o 便宜 3–5×
  • 局限：API 调用，数据需外传；速度较慢；中文特殊版式偶有问题

GPT-4o / Claude 3.5+ Vision

OpenAI / Anthropic · API
  • 语义理解最强：表格内容语义、图表含义、专业术语理解远超 Pipeline 方案
  • 复杂版式：对各类奇特版式有最强的鲁棒性（营销材料、创意排版等）
  • 成本：GPT-4o 约 $3–5/1000 页（取决于图像分辨率）；Claude 3.5 Sonnet 相近
  • 工程集成：原生支持批处理（Batch API），成本降低 50%
  • 局限：逐页处理吞吐量有限；严格结构输出（如精确 Markdown 表格）有时需要 prompt 工程

dots.ocr

arXiv:2512.02498 · 2025.12 · 统一 VLM 架构
  • 设计：单一 VLM 架构同时做版式检测 + OCR + 结构理解，无需多模型串联
  • 创新：证明 VLM 架构在文档版式解析上的可行性和 SOTA 性能
  • 多语言：单一模型支持多语言，不需要语言特定的 OCR 模型
  • 目前仍为研究状态，无生产可用版本

DocVLM（CVPR 2025）

arXiv · CVPR 2025 · 高效文档 VLM
  • 设计：让通用 VLM 成为"高效文档阅读器"，专注于文本密集文档的高效处理
  • 贡献：解决通用 VLM 在文本密集图像上压缩视觉特征导致的性能下降问题
  • 文档专用特征提取，比直接用通用 VLM 速度快 2–3×，精度相当
✅ VLM 解析优势
  • 语义理解能力最强：图表含义、表格上下文、跨页关系
  • 对复杂/异常版式鲁棒性最好
  • 多语言、手写体、低质量扫描件处理能力极强
  • 可在同一次调用中完成解析+理解+摘要
❌ VLM 解析劣势
  • API 成本高，大规模批处理经济性差
  • 推理速度慢，吞吐量受限
  • 数据需外传，隐私敏感场景不可用
  • 输出不完全稳定（需 prompt 工程控制格式）
  • 本地部署需要极大 GPU 显存（70B+ 模型）

🚀2025–2026 前沿进展

2025 年文档解析领域发生了范式级别的转变：VLM 从"辅助增强"走向"核心路线"，端到端能力快速逼近 Pipeline 方案的结构精度。

📌 2025 年五大趋势

趋势① VLM 统一架构
dots.ocr（2025.12）首次证明单一 VLM 可同时完成版式检测+OCR+结构理解，性能达到 SOTA。Pipeline 架构的模块化优势正在被 VLM 统一性优势侵蚀。
趋势② RLVR 训练 OCR
olmOCR 2（2025.10）将可验证奖励强化学习（RLVR）引入 OCR 训练，通过程序化验证文字识别正确性做 RL 反馈，在结构化区域精度上大幅提升。
趋势③ 混合 Pipeline+VLM
Marker v1 / MinerU VLM 模式：用规则/YOLO 做粗版式检测，VLM 做精细内容理解。两者结合在速度和精度上取得更好的 Pareto 最优点。
趋势④ 评测体系完善
OmniDocBench（CVPR 2025）建立了统一的文档解析基准，覆盖多种文档类型和内容类型，解决了此前各家自测不可比的问题。
趋势⑤ 预训练数据工程化
olmOCR 的定位是"为 LLM 预训练解锁万亿 token"，文档解析开始成为数据工程的标准基础设施，而非独立研究方向。
趋势⑥ 多模态文档理解
从"文字提取"扩展到"图表理解"：图表→数字/描述的自动解析（图表 QA）、科学图像的结构化描述正在成为新的评测方向（CVPR 2025 相关工作）。

🔭 2026 年重点方向

端到端 VLM 文档解析的工程化

  • dots.ocr 类方案从研究走向生产，单 VLM 完成全流程
  • 推理加速（投机解码、量化、批处理优化）使 VLM 方案的吞吐量接近 Pipeline
  • 小模型专项蒸馏（从 GPT-4o 蒸馏文档解析能力到 7B 以下模型）

跨页/长文档结构理解

  • 跨页表格、跨页公式的上下文还原
  • 章节层级自动识别（不依赖字体大小启发式）
  • 文档知识图谱构建：从文字提取到关系提取的一体化

图表语义理解（Chart Understanding）

  • 从"截图保存"到"数字/描述提取"：柱图/折线图的数值自动读取
  • 科学论文图（architecture diagram、结果曲线）的自动解读
  • 图表 QA 基准的完善（ChartQA、FigureQA 的升级版本）

手写与低质量文档

  • 手写识别（HTR）与印刷识别统一：olmOCR、GOT-OCR 已有初步探索
  • 历史文献、老旧扫描件（噪声、褪色、装订折痕）的鲁棒性
  • 多语言混合文档（中英混排、注释语言不同）的处理

🔭技术展望

文档解析正处于范式转变期：从"结构提取工具"演进为"文档智能理解系统"。

📈 未来 1–2 年（2026–2027）
  • VLM 统一架构普及：dots.ocr 类端到端 VLM 方案工程化，逐步取代多模块 Pipeline 作为新的默认方案
  • 推理加速缩小速度差距：量化/投机解码让 7B VLM 的吞吐量接近当前 Pipeline 方案
  • 图表语义理解成熟：图表→结构化数字/描述的自动解析成为文档解析标配
  • 评测标准化：OmniDocBench 类统一基准成为业界标准，终结各家自测乱象
🔬 中长期挑战（3–5 年）
  • 文档知识图谱：从结构提取到跨文档知识关系构建，文档解析与知识图谱的融合
  • 主动理解：模型不仅提取文字，还能主动识别文档中的不一致/错误（审阅式解析）
  • 实时流式解析：文档摄像头/扫描流的实时解析，用于智能会议记录、工业质检等
  • 可信度量化：解析结果附带置信度，下游 RAG 系统可据此决策是否需要人工核查
核心判断： 文档解析将在 2026–2027 年经历一次类似"OCR → 深度学习"的范式切换——从"规则+模型 Pipeline"到"端到端 VLM"。 但这不意味着 Pipeline 方案消亡：在隐私敏感、成本极端敏感、特定领域专精的场景，轻量级 Pipeline 方案仍将长期存在。 实践建议：现在就应该在核心业务文档类型上测试 VLM 方案，为迁移做准备，同时保留 Pipeline 作为低成本/离线场景 fallback。
