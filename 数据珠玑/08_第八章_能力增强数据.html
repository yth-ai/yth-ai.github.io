Chapter 8 · Capability Enhancement Data
第8章 能力增强数据
面向数学推理、长上下文、代码和多语言的专项能力数据构建——让模型在特定维度上突破通用预训练的能力天花板。

8.1 数学与推理能力的专项训练数据

数学推理被广泛视为衡量 LLM 通用推理能力的"金标准"。从 GPT-3 在 GSM8K 上接近随机的表现，到 GPT-4/Claude 3/Gemini Ultra 接近完美的解题能力，数学能力的飞跃背后，有着数据策略的根本性变革。

数学数据的类型层次

数学训练数据可以按复杂度分为四个层次：

层次数据类型代表数据集能力培养目标
L1数学文本（教材、百科）ArXiv、Wikipedia 数学页面数学概念与术语理解
L2数学题目+答案GSM8K、MATH、AMC/AIME 题库基础解题能力
L3题目+完整推理链（CoT）NuminaMath、MetaMath分步推理能力
L4题目+验证过程+错误分析DeepSeek-R1 生成数据、OpenAI o1（推测）自我验证与纠错

合成数学数据的主要范式

MetaMath（Yu et al., 2023）：对 GSM8K 和 MATH 中的题目进行多维度改写（问题重述、答案增强、逆向问题生成），将原始 ~15K 题扩展到 395K 条训练数据。在 7B 模型上，MetaMath 数据使 GSM8K 准确率从约 50% 提升至 66.5%，证明了通过数据扩充进行数学能力增强的有效性。

NuminaMath（2024）：收集和整理来自 AMC、AIME、奥林匹克数学竞赛、中国高考等高质量竞赛题，配合完整的 CoT 解题过程，总量约 860K 道题。作为 2024 年 AIMO Progress Prize 竞赛中多个冠军方案的训练数据，是目前公开的高质量数学数据集中最具竞争力的之一。

过程奖励模型（PRM）数据：不仅对最终答案打标，而是对每一步推理过程给出正确/错误的标注。代表工作是 OpenAI 发布的 PRM800K（800K 步级标注），以及 Math-Shepherd（基于自动验证的大规模步级标注）。过程监督被认为是突破复杂数学推理能力天花板的关键数据策略。

过程奖励 vs 结果奖励（训练信号对比）：
结果奖励（ORM）：仅对最终答案正确与否给 +1/-1
过程奖励（PRM）：对每个推理步骤给 {+1, -1, 0} 的评分

PRM 优势：稀疏奖励问题得到缓解，模型能从中间步骤错误中学习
PRM 难点：需要人工或强模型对每个步骤进行正确性判断，成本高

8.2 长上下文能力的数据构建

标准预训练数据通常将文本截断为固定长度（如 2K 或 4K Token），导致模型缺乏处理长文档的能力。长上下文中训练通过专门的数据构建策略，将模型的有效上下文窗口从数千 Token 扩展至十万乃至百万 Token。

长上下文预训练的数据要求

  • 天然长文档：书籍全章、长篇学术论文、法律合同、代码仓库等，避免简单拼接造成的语义不连贯
  • 多跳推理数据：需要跨越多个文档段落、整合多个信息片段才能回答的问题，如长文档问答、多文档摘要
  • 位置编码训练数据：为确保模型在长距离位置上的注意力机制有效，训练数据应包含各种长度分布的样本，不能仅有短文本

RoPE 扩展与数据配合

长上下文训练通常伴随着位置编码（Position Embedding）的修改。LongRoPE、YaRN、Positional Interpolation（PI）等方法通过修改 RoPE 的频率参数，将位置编码外推到更长的序列。但仅修改位置编码还不够——必须用足够长的上下文数据进行中训练，才能激活模型在长距离上的注意力能力。

LongLLaMA 与 LongLoRA 的数据策略

LongLoRA（2023）：仅需少量长文本数据（约 1B Token）+LoRA 参数高效微调，将 LLaMA-2 的上下文从 4K 扩展到 100K。关键发现：在长上下文中训练中，不需要所有序列都是目标最大长度——混合不同长度（4K、8K、32K、100K）的数据，训练效率更高。

数据来源建议：书籍数据（GutenbergProject、Books3 残余）、长篇新闻文章、代码仓库拼接、维基百科全文，这些来源能提供天然连贯的长文档，避免"拼接文档带来的语义噪声"。

8.3 代码能力中训练：Fill-in-the-Middle 与仓库级代码

代码中训练除了第2章介绍的标准代码预训练外，还有两种高价值的专项数据构建策略：Fill-in-the-Middle（FIM）和仓库级代码（Repository-level Code）。

Fill-in-the-Middle（FIM）

标准的自回归语言建模只训练模型从左到右生成文本，无法处理"代码补全"（光标中间填写代码）的场景。FIM 通过重新排列训练序列解决这一问题：

# 原始代码： def calculate_area(radius): pi = 3.14159 area = pi * radius * radius return area # FIM 变换后的训练格式（PSM：Prefix-Suffix-Middle）： <PRE> def calculate_area(radius): pi = 3.14159 <SUF> return area <MID> area = pi * radius * radius

Bavarian et al.（2022）的研究表明，用约 50% FIM 格式数据训练，可以在不损失自回归生成能力（Prefix completion）的同时，大幅提升代码补全能力。CodeLlama、StarCoder2 等主流代码模型均采用了 FIM 训练策略。

仓库级代码（Repository-level Code）

单文件代码训练让模型看到的是代码的"碎片"，缺乏跨文件的依赖关系理解。仓库级代码数据将整个代码仓库的文件按依赖关系（import 顺序）拼接，使模型学习完整的项目结构和跨文件调用关系。

RepoCoder 与 DeepSeek-Coder 的实验（2023）：使用仓库级代码训练的模型，在"跨文件代码补全"任务（CrossCodeEval）上比单文件训练的同规模模型提升 15-25%，在实际工程场景（修改已有代码库中的 bug）中表现更接近真实开发者的行为模式。

代码执行反馈数据

最新趋势是引入"代码执行反馈"（Execution Feedback）数据——将模型生成的代码实际运行，根据测试用例通过情况构建训练信号。代表工作包括 CodeRL、PPOCoder、RLTF 等。这一策略将在后训练章节详细讨论，但在中训练阶段，基于执行反馈的拒绝采样（Rejection Sampling）已被证明是提升代码正确率的有效手段。

8.4 多语言能力的持续增强

即使预训练阶段已包含多语言数据，许多低资源语言的能力仍远未达到英语水平。中训练阶段可以针对特定语言进行专项能力增强，同时借助跨语言迁移加速低资源语言的学习。

跨语言迁移的工程利用

多项研究表明，在预训练基础上，即使以较少量（数十亿 Token）的目标语言数据进行中训练，也能显著提升该语言的能力——因为模型已经通过英语数据学习了大量通用知识，中训练只需"激活"这些知识与目标语言的映射关系。

平行语料的价值：高质量的平行语料（同一内容的多语言版本）在多语言中训练中具有极高价值，帮助模型建立跨语言语义对齐。来源包括：OPUS 平行语料库、维基百科多语言版本、联合国文件等。

Chinese-LLaMA 系列的工程实践（Cui et al., 2023）：在 LLaMA（英语为主）基础上，仅使用约 20B Token 的中文数据（中文维基、C4-zh、百度百科等）进行中训练，并扩展词表（加入中文字符）以提升 tokenization 效率，最终在中文理解基准上从接近随机提升至接近 BELLE、ChatGLM 等专项中文模型的水平。这证明了跨语言迁移的巨大潜力——英语预训练的通用知识可以高效地迁移到中文能力上。

多语言能力增强的数据优先级

在有限资源下，多语言中训练数据应优先覆盖：

  1. 高质量百科数据：维基百科、Baidu Baike 等，知识密度高
  2. 新闻与正式文本：语言规范，减少口语化噪声
  3. 与英语的平行语料：最大化跨语言知识迁移
  4. 目标语言的指令数据前驱：包含少量问答、摘要等任务型数据，为后续 SFT 预热
