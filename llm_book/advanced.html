第一章 · Tokenization

文字是怎么变成数字的？

计算机只认识 0 和 1，但人类的文字千变万化。从文字到数字，第一步是分词——把文字切成模型能处理的"原子单位"。

Token BPE Vocabulary UTF-8 词表大小 Token 计费

1.1 为什么不能直接用文字？

你可能会问：为什么不能把汉字"你"直接喂给 AI，而要先把它变成数字？

原因很简单：计算机的 CPU 和 GPU 做的事情，本质上都是数字运算——加减乘除，矩阵乘法。它不会"理解"字符串，只会处理数字。所以，第一步就是把所有文字转成数字编号。

最简单的想法：给每个汉字编一个号。"你"=1，"好"=2，"世"=3，"界"=4……但汉字有五万多个，常用的也有七千个左右，英文还有大小写、标点、特殊符号——如果每个字符一个编号，词表就会非常庞大，而且很多罕见字只出现一两次，训练数据太少，模型根本学不好。

另一个极端：把每个字节（byte）当一个 Token。这样词表只有 256 个，但一个汉字用 UTF-8 编码需要 3 个字节，一个句子会变成一串很长的数字序列，模型要处理的长度大大增加，效率很低。

两个极端都不行，所以人们发明了一种"折中方案"——子词分词（Subword Tokenization），其中最流行的算法叫 BPE（Byte Pair Encoding，字节对编码）。

1.2 BPE 算法：合并高频对

BPE 的核心思想非常优雅，可以用一句话概括：反复找出出现最频繁的相邻字符对，把它们合并成一个新 Token，直到词表达到目标大小为止。

用一个简单的例子来看。假设我们的训练文本里有以下单词（括号里是出现次数）：

# 初始词表（每个字母都是独立 Token） l o w (5次) l o w e r (2次) n e w e s t (6次) w i d e s t (3次) # 第1步：找最高频相邻对 → "e s" 出现 9次（6+3） # 合并 "e" + "s" → "es" l o w (5) l o w e r (2) n e w es t (6) w i d es t (3) # 第2步：最高频对 → "es t" 出现 9次 # 合并 "es" + "t" → "est" n e w est (6) w i d est (3) # 最终：词表里就有了 "est" 这个子词 Token

通过反复合并，BPE 自动学会了英文中的常见词缀（如 -est、-ing、-tion）、常见词根，甚至汉字的常见组合。最终得到一个词表，里面有约 3 万～10 万个 Token。

🔍 类比

BPE 就像用砖头砌墙。最开始每个字母都是"沙子"，太细了。通过不断合并高频对，沙子变成砖块（子词），砖块有的变成预制板（常用词）。最终建筑用的是不同大小的构件，既不浪费（罕见词用小块拼），又高效（常见词一块搞定）。

1.3 实际分词示例

1.4 词表（Vocabulary）

经过 BPE 训练，我们得到一张词表（Vocabulary）：一个从 Token 到整数 ID 的映射表。GPT-4 的词表有 100,277 个 Token（tiktoken cl100k），涵盖了英文、中文、代码、数学符号等。

💡 技术小贴士：为什么 GPT 处理中文更贵？

GPT-4 的词表里，英文大量常用词是一个 Token，但中文每个汉字有时需要 2-3 个 Token（因为中文字符的 UTF-8 编码是 3 字节，而 BPE 初始从字节出发）。这意味着：同等内容，中文消耗的 Token 数量是英文的 1.5-3 倍，API 费用更高。国产模型（Qwen、DeepSeek）专门优化了中文词表，每个汉字通常只需 1 个 Token。

1.5 特殊 Token

除了普通文字 Token，词表里还有一些特殊 Token，它们承担着控制模型行为的特殊任务：

  • <|endoftext|> — 文本结束标记，告诉模型"这个文档到这里就结束了"
  • <|im_start|> / <|im_end|> — 对话轮次的开始和结束
  • [PAD] — 填充 Token，把不同长度的序列对齐到相同长度
  • [MASK] — 掩码 Token（BERT 类模型用，GPT 类不用）

正弦位置编码：PE(pos,2i) = sin(pos/10000^(2i/d))，颜色越蓝表示值接近-1，越红表示接近+1。不同维度对应不同频率的波，高维度变化缓慢（低频），低维度变化快（高频），让模型能区分任意位置。

✅ 本章小结
  • 计算机不认识文字，所有输入都要先转成数字
  • Token 是语言的"原子单位"——比词小，比字节大
  • BPE 通过反复合并高频字符对来构建词表
  • 词表大小通常是 3万-15万，覆盖常见子词和完整词
  • 中文词表优化影响 Token 消耗效率和 API 成本
  • 特殊 Token 用于控制模型行为（对话格式、结束信号等）

第二章 · Embedding

Token 怎么变成向量？

把"猫"编号为 1234，把"狗"编号为 1235，这两个数字挨在一起并不意味着猫和狗的意思相近。数字 ID 不携带语义信息，我们需要一种更聪明的表示方式——向量。

Embedding 向量空间 语义相似度 Positional Encoding 维度

2.1 从 ID 到向量：为什么要这一步？

假设词表里"猫"的 ID 是 100，"狗"是 101，"飞机"是 102。如果直接用这些整数 ID，模型看到的是：

  • 猫 (100) 和 狗 (101) 相差 1
  • 狗 (101) 和 飞机 (102) 相差 1

这意味着：根据数字大小，模型会认为"狗和飞机的关系"跟"猫和狗的关系"一样近——这显然是错的。数字 ID 不含语义，不能直接用于计算。

解决方案是：给每个 Token 分配一个向量（vector）——一串浮点数，比如 768 个数字组成的列表。这个向量不是手动设计的，而是在训练过程中自动学习出来的。语义相似的词，它们的向量在高维空间中距离也近。

🔍 类比：用坐标描述"词的位置"

想象一个二维坐标系，横轴代表"是否是动物"，纵轴代表"体型大小"。那么：猫在左上（动物、小），狗在中上（动物、中），大象在右上（动物、大），飞机在右下（非动物、大）。坐标越接近的词，语义越相关。真实的 Embedding 不是 2 维，而是 768、1024 甚至 4096 维——但原理一样，都是用坐标来表达语义关系。

2.2 King - Man + Woman ≈ Queen

2013 年，谷歌研究员 Tomas Mikolov 发现了一个让所有人震惊的现象：学好的 Word2Vec 向量满足：

vector("国王") - vector("男人") + vector("女人") ≈ vector("女王") vector("巴黎") - vector("法国") + vector("德国") ≈ vector("柏林") vector("走路") - vector("脚") + vector("翅膀") ≈ vector("飞翔")

这意味着：Embedding 向量不仅仅是一串随机数字，它实际上编码了词语之间的语义关系。"国王"和"女王"的差异，可以用"男性"和"女性"的向量差来表达。这是 AI 真正开始"理解"语义关系的开端。

图2-1：二维语义空间示意（实际是 768+ 维）

语义相近的词在向量空间中距离更近。向量运算可以捕捉到"性别"这类抽象语义关系。

2.3 Embedding 矩阵：一张大查找表

实现上，Embedding 就是一个巨大的矩阵：

  • 行数 = 词表大小（例如 100,277）
  • 列数 = 向量维度（例如 768 或 4096）

当模型接收到 Token ID = 2392（"我"），它就在这张矩阵的第 2392 行找到对应的 768 维向量。这个查找操作称为 Embedding lookup，相当于查字典。

图2-2：Embedding 矩阵（词表大小 × 向量维度）
dim0 dim1 dim2 … dim767
我 [2392] 0.32 -0.18 0.91 … -0.04
喜欢 [15940] -0.11 0.73 0.25 … 0.66
… … … … … …

每行是一个 Token 的向量表示，颜色深浅代表数值大小。这张表在训练中不断更新，最终使语义相近的词向量距离更近。GPT-3 的 Embedding 矩阵有 50,257 × 12,288 = 6.2 亿个参数。

2.4 位置编码（Positional Encoding）

现在有个问题：Transformer 处理序列时，把所有 Token 的向量放进去，但它本质上不知道顺序。"我爱你"和"你爱我"，如果不加位置信息，模型看到的是同样的三个 Token 向量，无法区分。

解决方案是给每个 Token 的向量加上位置编码（Positional Encoding）——根据这个 Token 在句子中的位置（第1个、第2个……），给它的向量加上一个独特的"位置标记"。

💡 两种主流位置编码方式

正弦/余弦编码（原始 Transformer）：用不同频率的正弦波来生成位置向量，数学上优雅，但长度固定。

RoPE（旋转位置编码，LLaMA/Qwen/DeepSeek 等使用）：把位置信息编码成向量的"旋转"，更好地泛化到训练时没见过的长序列，是目前主流选择。GPT-4 的超长上下文（128K token）就依赖 RoPE 的变体。

✅ 本章小结
  • Token ID 是整数，不含语义；Embedding 向量才承载语义信息
  • Embedding 矩阵是一张查找表，每个 Token 对应一行向量
  • 语义相似的词，向量在空间中距离更近（可以做向量运算）
  • 向量维度越高，能表达的语义越丰富（GPT-4 约 12,288 维）
  • 位置编码解决 Transformer 不感知顺序的问题
  • RoPE 是目前主流位置编码，支持超长上下文扩展

第三章 · Attention · 核心

注意力机制——AI 的"找重点"能力

注意力机制（Attention）是整个大语言模型最核心的创新。它解决了一个根本问题：在理解一个词时，应该参考句子里的哪些其他词？

Query / Key / Value Attention Score Softmax Multi-Head Attention Scaled Dot-Product Causal Mask

3.1 为什么需要注意力？

考虑这两个句子：

  • "我把鱼放进冰箱后，它坏掉了。"
  • "我把鱼放进冰箱后，它很冷。"

第一句中的"它"指的是"鱼"，第二句中的"它"指的是"冰箱"。人类一眼就能判断，但计算机怎么做到这一点？

早期的 RNN 模型用一个"记忆槽"从左到右传递信息，但长距离依赖很难维持。Transformer 的创新在于：对于每个词，直接让它和序列中所有其他词"交流"，通过注意力分数来决定参考谁、参考多少。

3.2 Q、K、V 是什么？

注意力机制用三个向量来工作：Query（查询）、Key（键）、Value（值）。每个 Token 的 Embedding 向量，会通过三个不同的线性变换矩阵，生成它自己的 Q、K、V 向量。

注意力计算过程：Q 和所有 K 计算相似度 → 得到匹配分数 → Softmax 归一化成概率 → 按概率加权聚合所有 V → 得到这个词的最终表示。

3.3 注意力分数计算（无公式版）

3.4 Softmax：把分数变成概率

原始的注意力分数可能是任意实数（正负都有）。Softmax 函数把这些分数转化成概率分布：所有值都变成正数，且加起来等于 1。

比喻：班级里投票选班长，原来的"支持度"是 [-2, 1, 5, 3]，Softmax 把它变成得票率 [0.002, 0.019, 0.878, 0.101]——绝对领先的候选人得票率接近 1，其他人接近 0。

💡 为什么要除以 √d（缩放点积）？

注意力公式是：Attention = Softmax(QKT / √d) · V。除以 √d（维度的平方根，比如 √64 = 8）是为了防止点积结果太大——如果分数太大，Softmax 会接近 0 或 1，梯度会消失，训练困难。这就是"Scaled Dot-Product Attention"中"Scaled"的含义。

3.5 Multi-Head Attention：多角度看问题

单个注意力头只能从一种"角度"关注词之间的关系。实际上，一个句子里的词存在多种关系：语法关系、语义关系、指代关系、情感关系……

Multi-Head Attention（多头注意力） 的做法是：把向量分成若干份（比如 12 份），每份独立做一套 Q/K/V 计算，每个"头"学习不同角度的关系，最后把所有头的输出拼接起来。

✅ 本章小结
  • 注意力机制让每个词可以"看到"并参考句中任意其他词
  • Q=我要查什么，K=我的标签，V=我的内容（图书馆类比）
  • 注意力分数 = Q·K 点积，经 Softmax 变成概率权重
  • 输出 = 用注意力权重对所有 V 做加权平均
  • Multi-Head：多个头并行，各学习不同语言关系，最后拼接
  • GPT 类模型用因果掩码（Causal Mask），只能看左边，不能"偷看"右边

第四章 · Transformer

Transformer 架构——把注意力组装起来

注意力机制只是 Transformer 的核心组件之一。真正让模型变得强大的，是把注意力和其他模块以特定方式组合起来，形成能处理复杂语言任务的深层网络。

Transformer Block Feed-Forward Network Residual Connection Layer Normalization Decoder-only 参数量

4.1 一个 Transformer 层里有什么？

每个 Transformer 层（Block）包含两个子模块，外加两个"维护稳定"的技术：

图4-1：一个 Transformer Block 的内部结构
输入向量（来自上一层）
Multi-Head Attention
每个词融合上下文信息
Add & LayerNorm
残差连接 + 归一化
Feed-Forward Network
两层全连接，扩展-压缩
Add & LayerNorm
再次残差连接 + 归一化
输出向量（传给下一层）

GPT-3 把这个结构堆叠了 96 层。输入向量的维度从头到尾保持不变（12,288 维），但每层的向量表示越来越"深刻"。

4.2 前馈网络（FFN）：信息的加工厂

Multi-Head Attention 负责"收集信息"——把上下文信息聚合到当前词的向量里。而前馈网络（Feed-Forward Network，FFN） 负责"加工信息"——对每个词的向量独立做非线性变换。

FFN 是两层全连接网络：先把维度扩大 4 倍（768→3072），经过激活函数（如 GeLU），再压缩回来（3072→768）。这个"先膨胀后压缩"的结构让模型有足够空间做复杂的特征提取。

有趣的是：研究发现 FFN 层像是"知识存储库"——模型记住的大量世界知识（"法国首都是巴黎"）就存在 FFN 的权重里。

4.3 残差连接：防止"忘了自己是谁"

想象你在玩"传话游戏"：第一个人说"今天天气不错"，经过 96 个人传递后，最后一个人说出来可能完全变样了。深度神经网络也有这个问题——梯度在反向传播时经过太多层，信号会变得非常微弱（梯度消失），导致前面的层几乎学不到东西。

残差连接（Residual Connection） 的解决方案很巧妙：每个子模块的输出，加上它的输入（跳跃连接）。数学上：

# 没有残差连接： output = Attention(x) # 信息全靠 Attention 变换 # 有残差连接： output = x + Attention(x) # 加上原始输入 x # 这意味着：即使 Attention 什么都不做（输出全0）， # 原始信息 x 仍然完整传递下去——最坏情况是"什么都没变"

这个小技巧让 Transformer 可以堆叠上百层而不出现梯度消失，是深度学习的重大突破之一（来自 2015 年的 ResNet 论文）。

4.4 参数量怎么算？

GPT-3 宣称有 1750 亿参数，这意味着什么？让我们粗略估算：

# GPT-3 规格：96层，12288维，96头 每层参数量： Attention (Q,K,V,O 矩阵): 4 × 12288 × 12288 ≈ 6亿 FFN (两层，4倍扩展): 2 × 12288 × 49152 ≈ 12亿 LayerNorm 等小参数: 可忽略 单层参数 ≈ 18亿 96层合计 ≈ 1728亿 加上 Embedding 矩阵: 50257 × 12288 ≈ 6亿 总计 ≈ 1750亿 参数 ✓ # 存储：每个参数用 float16（2字节） 1750亿 × 2字节 = 350GB ← 光加载就要 5块 A100（80GB显存）
图 5-3　自回归生成步骤动画
每次只生成 1 个 Token，循环直到遇到结束符

自回归生成（Autoregressive Generation）：模型每次接收当前所有 Token，预测下一个 Token，然后将新 Token 加入序列，如此反复。高亮（蓝色）表示本轮新生成的 Token。

✅ 本章小结
  • Transformer Block = Attention + FFN + 两次 Add&Norm
  • Attention 收集上下文信息；FFN 独立加工每个位置的信息
  • 残差连接解决梯度消失，让模型可以堆叠很多层
  • LayerNorm 让每层输入的数值分布稳定，加快训练
  • GPT 类模型只用 Decoder（无 Encoder），用因果掩码自回归生成
  • 参数量 = 层数 × 每层矩阵大小；存储量 = 参数数 × 每参数字节数

第五章 · Generation

下一个 Token 预测——AI 生成文字的秘密

前四章讲了文字如何进入模型、如何被处理。现在来看最核心的问题：模型是怎么"输出"文字的？

Autoregressive Logits Softmax Temperature Top-K / Top-P Greedy / Sampling

5.1 预测下一个 Token

GPT 类模型的训练目标只有一个：给定前面所有词，预测下一个词。这个看起来很简单的目标，却逼迫模型学会了语法、语义、逻辑、常识……因为要准确预测下一个词，就必须深刻理解所有之前的词。

推理（生成）时，模型每次前向计算会输出词表里每个 Token 的得分（叫 logits），经过 Softmax 变成概率分布，然后从中采样一个 Token 作为输出，再把这个 Token 拼到输入末尾，重复这个过程——这就是自回归生成（Autoregressive Generation）。

# 自回归生成示意 输入: ["今", "天", "天", "气"] ↓ 模型计算 ↓ 全词表概率: {"真":0.28, "不":0.18, "很":0.15, "好":0.12, ...} 采样: 选 "真" 输入: ["今", "天", "天", "气", "真"] ↓ 模型计算 ↓ 全词表概率: {"好":0.41, "棒":0.22, "差":0.08, ...} 采样: 选 "好" 输入: ["今", "天", "天", "气", "真", "好"] ↓ 模型计算 ↓ ...继续生成...

5.2 Temperature：创意度旋钮

Temperature（温度） 参数控制采样的"随机程度"，是影响 AI 输出风格的最重要参数之一。

5.3 Top-K 和 Top-P：只在好的选项里挑

即使 Temperature=1，词表里还有大量概率极低的奇怪词。为了避免偶尔采样到"垃圾词"，实际使用中会配合截断策略：

  • Top-K：只保留概率最高的 K 个 Token（比如 K=40），其余设为 0，重新归一化后采样
  • Top-P（Nucleus Sampling）：保留累计概率超过 P 的最小 Token 集合（比如 P=0.9）——动态调整候选数量，概率集中时候选少，概率分散时候选多
💡 实际 API 推荐参数

精确任务（代码、数学）：Temperature=0.1, Top-P=0.9 → 稳定可重复

创意写作：Temperature=0.8-1.0, Top-P=0.95 → 多样有创意

对话聊天：Temperature=0.7, Top-P=0.9 → 自然流畅

注意：Temperature 和 Top-P 不要同时设极端值，否则互相冲突。

  • 自回归：把生成的 Token 拼到输入末尾，再预测下一个
  • Logits → Softmax → 概率 → 采样 → Token
  • Temperature 控制随机程度：低=保守稳定，高=创意发散

  • 同一问题每次答案可能不同，因为有随机采样

第六章 · Training

训练是怎么回事？

模型不是手工编程的——没有人写下"法国首都是巴黎"这条规则。所有知识都来自训练：用海量文本反复"纠错"，让模型的预测越来越准确。

预训练 损失函数 梯度下降 反向传播 GPU 并行 过拟合

6.1 预训练：海量文本的完形填空

GPT 的预训练任务非常简单：给模型看互联网上的文本，然后遮住最后一个词，让模型猜。猜对了，什么都不用改；猜错了，调整参数让下次猜得更准。

这个过程在 GPT-3 的训练中，处理了 3000 亿个 Token（约相当于 45 万本书）。每处理一个 Token，都有可能微调模型的 1750 亿个参数。这就是为什么训练需要数千块 GPU 并行运算数周时间。

6.2 损失函数：答错了有多严重？

损失函数（Loss Function） 量化模型的"答错程度"。LLM 常用的是交叉熵损失（Cross-Entropy Loss）：如果正确答案"好"的概率是 0.41，损失 = -log(0.41) ≈ 0.87；如果概率只有 0.01，损失 = -log(0.01) ≈ 4.6。概率越低，损失越大，说明答得越差。

6.3 梯度下降：蒙眼下山

训练的目标是最小化损失。梯度下降就是实现这个目标的算法：

图6-1：梯度下降——在损失曲面上找最低点

梯度告诉我们"现在所在坡面的方向和陡度"。沿着梯度反方向移动（下坡），就能降低损失。学习率（Learning Rate）控制每步走多远。

6.4 反向传播：错误从输出端"倒流"回去

反向传播（Backpropagation） 是计算梯度的算法。它利用微积分的链式法则，从输出端的损失值出发，逐层计算每个参数对损失的影响（偏导数）。想象信号从前往后（正向传播），然后"错误信号"从后往前流（反向传播），沿途更新每个参数。

💡 为什么训练要用 GPU？

梯度下降的每一步都要对海量文本做矩阵乘法。一个 batch（一批训练样本）可能包含上千个序列，每个序列的每个 Token 都要做几十次 12288×12288 的矩阵乘法。GPU 有数千个小核心，天生适合这种高度并行的运算。A100 GPU 做矩阵乘法的速度是 CPU 的几百倍。GPT-3 训练使用了 10,000 块 V100，连续运行约 30 天。

第七章 · Alignment

从预训练到对话——SFT、RLHF 与对齐

预训练完成后，模型只会"续写文本"，不会"回答问题"。把它变成一个有用的助手，还需要两个关键步骤。

SFT RLHF 奖励模型 PPO 对齐 DPO

7.1 预训练模型的局限

假设你问预训练完的 GPT："法国首都是哪里？"它可能回答："……全球主要城市有巴黎、伦敦、纽约……"——因为它学到的是"这类问题后面通常跟着什么文字"，而不是"如何回答问题"。

要变成一个真正有用的对话助手，需要额外的训练：

📚
Step 1：预训练（Pre-training）
3000亿 Token 互联网文本，学会预测下一个 Token。获得世界知识，但不会对话。

Step 2：有监督微调（SFT）

InstructGPT（ChatGPT 的前身）在 2022 年首次公开了这个三步流程，被认为是 LLM 从"续写工具"变成"有用助手"的关键突破。

7.2 RLHF 的核心思路

RLHF（Reinforcement Learning from Human Feedback）分三步：

  1. 收集人类偏好数据：对同一个问题，让模型生成 2-4 个不同的回答，让人类标注者排序（哪个更好、更安全、更准确）
  2. 训练奖励模型（Reward Model）：用这些排序数据训练一个模型，让它学会预测"人类会给这个回答打多少分"
  3. 强化学习优化 LLM：用 PPO（一种强化学习算法）让 LLM 生成的回答尽可能得高分
🔍 类比：学生作文与老师打分

SFT 像是：老师给你看优秀作文范本，你照着写。RLHF 像是：你写了好几篇作文，老师给它们排名打分（不用给具体分数，只要说哪篇更好）。然后你根据老师的审美反复改写，直到老师满意为止。奖励模型就是学习了老师审美的"AI 老师"。

7.3 对齐（Alignment）是什么？

对齐是指：让 AI 的行为目标和人类的价值观相符。一个未对齐的模型可能会：拒绝说"我不知道"（因为训练时"不知道"很少出现）、帮助用户做有害的事情（因为训练目标只是"让用户满意"而非"对人类有益"）。

RLHF 中标注者会被要求拒绝给危险内容打高分，这样奖励模型就学会了"有帮助但无害"的评判标准。这是当前最主流的对齐方法。

💡 DPO：更简单的 RLHF 替代方案

2023 年提出的 DPO（Direct Preference Optimization）发现：可以绕过奖励模型，直接用 (好回答, 坏回答) 对来优化 LLM，数学上等价于 RLHF，但实现更简单、训练更稳定。LLaMA 3、Qwen2.5 等都使用了 DPO 或其变体。

✅ 本章小结
  • 预训练只学会续写；SFT 教会模型用对话格式回答问题
  • RLHF 通过人类偏好排序让回答更有帮助、更安全
  • 奖励模型学习人类的评判标准，替代人类给 LLM 打分
  • 对齐问题核心：如何确保 AI 追求的目标和人类价值观一致
  • DPO 是 RLHF 的简化替代，目前广泛使用
08
第八章 · Inference

推理与部署——AI 怎么"跑起来"？

训练好的模型需要部署才能被用户使用。推理阶段的效率优化，直接决定 API 响应速度和运营成本。

推理 vs 训练 KV Cache 量化 Prefill / Decode Batch 显存

8.1 推理和训练有什么区别？

训练需要：正向传播 + 反向传播 + 参数更新。推理只需要：正向传播（把输入变成输出）。所以推理比训练快很多，显存需求也少一半（不需要存梯度）。

但推理有自己的挑战：用户一个接一个地发请求，模型要实时响应，延迟不能太高。而且每次生成都要做大量矩阵乘法——生成 1000 个 Token 就要做 1000 次前向传播。

8.2 KV Cache：不要重复计算

自回归生成有一个低效的地方：每次生成新 Token 时，都需要计算整个序列的 Attention。但之前 Token 的 K（键）和 V（值）向量已经算过了——可以缓存起来，下次直接用！

KV Cache 使推理速度提升 10 倍以上。代价是显存占用增加——上下文越长，缓存越大。128K Token 的 KV Cache 可能占用数十 GB 显存。

8.3 量化：压缩模型体积

训练时参数用 float32（4字节）或 float16（2字节）存储，推理时可以压缩到 int8（1字节）甚至 int4（0.5字节）。这叫量化（Quantization）。

量化的代价是精度略有损失，但通常对最终输出质量影响不大（误差在 1-2% 以内）。好处是：模型体积缩小一半到四分之一，推理速度大幅提升，可以在更小的 GPU 上运行。

GGUF 格式（llama.cpp 使用）和 GPTQ、AWQ 等都是量化方案，让普通消费级显卡（甚至 CPU）可以运行 7B-13B 的模型。

💡 Prefill vs Decode 两个阶段

Prefill（预填充）：处理用户输入的 Prompt，一次性计算所有 Token 的 KV Cache。这个阶段是高度并行的，速度很快。

Decode（解码）：逐个生成输出 Token，每次只处理一个新 Token。这个阶段是串行的，受限于 GPU 显存带宽（而非计算能力）——这是生成速度的主要瓶颈。

你感受到的 ChatGPT "逐字输出"就是 Decode 阶段的实时流式输出（Streaming）。

✅ 本章小结
  • 推理只需正向传播，不需要梯度，比训练更快
  • KV Cache 缓存历史 Token 的 K/V 向量，避免重复计算
  • 量化把 float16 参数压缩到 int8/int4，体积缩半，速度更快
  • Prefill 阶段快（并行），Decode 阶段慢（串行，受显存带宽限制）
  • 本地运行模型（Ollama/llama.cpp）依赖量化和 KV Cache 优化

第九章 · Future

大模型的边界与未来

LLM 已经很强了，但还差什么？未来会走向哪里？本章讨论前沿方向和未解问题。

涌现能力 上下文窗口 RAG Agent 多模态 推理模型

9.1 涌现能力：大到一定程度，突然"开窍"

模型规模越大，能力越强——但不是线性的。当模型参数量超过某个临界点，某些能力会突然出现，叫做涌现（Emergence）。比如：GPT-2（15亿参数）做不了数学推理；GPT-3（1750亿参数）可以做简单数学；GPT-4（估计超过 1 万亿参数）可以解大学数学题。

图9-1：涌现能力——规模超过临界点后能力突然提升

涌现能力不是"更好一点点"，而是从"几乎不能"到"基本会了"的质变。这种不可预测性让大模型的能力边界难以预先评估。

9.2 RAG：给 AI 配一个"外脑"

LLM 的知识来自训练数据，训练结束后就固定了（有知识截止日期）。想让模型知道最新信息怎么办？RAG（检索增强生成，Retrieval-Augmented Generation）的思路是：回答问题前，先从外部知识库（向量数据库、搜索引擎）检索相关文档，把文档塞进 Prompt，再让模型基于这些文档回答。

用户问: "DeepSeek-V3 的参数量是多少？" # RAG 流程： 1. 向量检索: 用问题在知识库里搜索 → 找到相关文档: "DeepSeek-V3 技术报告: 671B 参数..." 2. 拼接 Prompt: 系统: 根据以下资料回答问题。 资料: [DeepSeek-V3 技术报告内容] 问题: DeepSeek-V3 的参数量是多少？ 3. 模型生成: "DeepSeek-V3 共有 671B 参数，采用 MoE 架构..."

9.3 Agent：能自己使用工具的 AI

普通 LLM 只能"说"，Agent 能"做"。Agent 框架让模型可以：调用工具（搜索、代码执行、API）、分解任务、多步规划、根据执行结果调整策略。OpenAI 的 GPT-4o、Anthropic 的 Claude Computer Use、Manus 都是 Agent 能力的具体体现。

9.4 推理模型：慢思考比快思考更聪明

GPT-4 类模型是"快思考"——直接生成答案。2024-2025 年出现了"慢思考"模型：在回答前先花时间"思考"（生成思维链 Chain-of-Thought），这让复杂推理能力大幅提升。OpenAI o1/o3、DeepSeek-R1、Kimi k1.5 都是这类模型。思考的本质是：用更多 Token（计算量）换取更高质量的答案。

💡 LLM 还差什么？

世界模型：目前的 LLM 是"语言模型"不是"世界模型"——它知道"苹果会掉落"，但不理解引力是什么。

持续学习：LLM 不能像人类一样边用边学——每次更新知识都要重新训练。

真正的推理：LLM 的"推理"本质上是统计模式匹配，容易被新颖题型击败（如 ZeroBench 中得分接近 0%）。

长期记忆：上下文窗口（即使 128K）仍然有限，跨会话记忆是难题。
