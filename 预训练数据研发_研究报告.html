预训练数据研发概述

1.1 数据规模演变趋势

预训练数据规模已经历了数个数量级的跨越式增长。从 GPT-2（2019年，约 40B tokens）到如今的万亿级 Token 训练集，数据规模本身已成为模型能力的核心驱动力之一。

关键趋势

  1. 主流旗舰模型已突破 10T tokens 大关
  2. 仅增加数量已无法保证质量提升，数据质量工程地位日益突出
  3. 专有领域数据（数学、代码）的比例策略成为差异化竞争核心

1.2 数据质量 vs 数据规模的权衡

Hoffmann et al. 的 Chinchilla Scaling Law（2022）指出，在固定算力预算下，模型参数量和训练 Token 数量应保持约 1:20 的比例。

Phi-1 的启示：使用 1.3B 参数模型，仅训练 7B tokens 的高质量"教材级"数据，在 HumanEval 上达到 50.6% 的 pass@1，超越了使用数百亿 token 训练的同参数级别模型。[Gunasekar et al., 2023]
FineWeb 实验（2024）：在相同 Token 数量下，经过精细过滤的 Common Crawl（FineWeb: 15T tokens）显著优于未过滤版本，且 FineWeb-Edu（1.3T，教育类过滤）在 MMLU 等知识密集型基准上更优。[Penedo et al., 2024]
DCLM 竞赛（2024）：系统评估了去重、过滤、数据混合等策略，确认基于模型的过滤（model-based filtering）是构建高质量训练集的核心手段。[Li et al., 2024]

1.3 预训练数据研发的核心挑战

  • 规模 vs 质量：海量低质数据 vs 少量高质数据的效益边界
  • 数据污染（Contamination）：基准泄露导致评估失效
  • 合规与版权：互联网数据的法律风险持续上升
  • 多语言平衡：英语主导数据集对非英语语言的不公平
  • 动态数据分布：互联网内容随时间演变，模型时效性问题
  • Model Collapse：合成数据循环污染问题
  • 可复现性：主流闭源模型数据集无法公开，研究壁垒高

2.1 Web 数据：Common Crawl 处理流程

Common Crawl（CC）是当前最主要的互联网语料来源，每月约产生 250TB 的 WARC 格式原始爬取数据。

    ↓

    ↓

    ↓

    ↓

    ↓

    ↓

2.3 主流模型数据来源构成

LLaMA 3（Meta, 2024） — 总计 15T tokens。采用 FastText + 基于 LLaMA 2 输出的质量分类器打分；多语言数据占比约 5%（支持 8 种语言）；知识型数据显著上调。
Qwen 2.5（阿里巴巴, 2024） — 总计 18T tokens，较 Qwen 2 的 7T 大幅提升。数据清洗引入 "recall" 和 "precision" 双维度优化；数学和代码数据比例显著提高；多语言覆盖约 30 种语言。
DeepSeek-V3（DeepSeek, 2024） — 总计 14.8T tokens。三阶段训练：主阶段 → 长上下文扩展 → 退火阶段（高质量短文档 + 推理增强数据）。退火阶段约 500B tokens，大幅提升数学、代码、推理能力。

数据质量过滤

3.1 规则过滤（启发式规则）

# 典型规则过滤示例（伪代码）
def rule_filter(doc):
    # 语言过滤
    if langdetect(doc.text) != "zh" and langdetect(doc.text) != "en":
        return False
    # 长度过滤
    words = doc.text.split()
    if len(words) < 50 or len(words) > 100000:
        return False
    # 特殊字符比例
    special_char_ratio = count_special_chars(doc.text) / len(doc.text)
    if special_char_ratio > 0.1:
        return False
    return True

经典规则集参考（Gopher 规则，DeepMind 2021）：

  • 文档词数 50–100,000
  • 行平均字数 ≥ 3
  • ... 行比例 < 30%
  • # 开头行比例 < 90%（针对 markdown 垃圾）
  • 字母字符比例 ≥ 70%

3.2 模型打分（质量分类器）

  • FastText 分类器：正样本为 Wikipedia/书籍，负样本为未过滤 CC；速度快，用于粗筛
  • CCNet 管道：使用 n-gram KenLM 语言模型计算困惑度（Perplexity），仅保留 head 部分
  • LLaMA 3 质量分类器：使用 LLaMA 2 为 "Would this be useful for learning English?" 打分，约过滤掉 50% 以上的原始 CC 数据
DCLM 的发现：基于模型的过滤（如训练 fastText 分类器）比纯规则过滤在 MMLU 上高出约 6.6 个百分点。[Li et al., 2024]

3.3 去重技术

精确去重

基于 URL / MD5 / SHA256 哈希进行文档级去重，识别完全重复的文档，代价低廉；局限：无法识别近似重复。

模糊去重：MinHash LSH

1. 对文档进行 n-gram（通常 n=5）分词
2. 用 MinHash 签名压缩为固定长度向量
3. Locality Sensitive Hashing（LSH）将相似文档分桶
4. 计算 Jaccard 相似度，阈值通常 0.7–0.8
5. 对同一聚类保留一份，丢弃其余重复文档

LLaMA 3 使用 MinHash LSH（128 hash functions，5-gram），在 15T token 的规模下仍有效去重。

语义去重（Semantic Deduplication）

SemDeDup [Abbas et al., 2023] 使用文本 embedding 计算语义相似度。在 LAION 数据集上验证，移除 50% 语义重复数据仅损失很小精度。计算代价高，一般仅用于精选高价值数据集。

子字符串去重

Lee et al. (2022) 提出后缀数组方法，直接检测跨文档的长重复字符串。在 Pile 数据集上去除约 19% 的数据后模型质量提升明显。

3.4 有害内容过滤

  • NSFW 过滤：使用专门的 NSFW 分类器（基于 BERT/DistilBERT 微调）
  • CSAM 检测：哈希匹配（PhotoDNA）+ 内容分类器
  • 毒性过滤：Perspective API 或本地 Detoxify 模型
  • PII 过滤：正则表达式识别邮箱、电话、身份证号等
  • 政治敏感内容：基于关键词列表 + 分类器联合判断

数据配比策略（Data Mixture）

4.1 Domain Mixing：领域比例设计

数据类型LLaMA 3（近似）Qwen 2.5（近似）DeepSeek-V3（近似）
网页文本65%50%50%
代码10%15%20%
数学3–5%10%15%+
书籍5%5%5%
学术/百科5%10%5%
多语言5%10%5%

关键设计原则

  1. 代码数据的溢出效应：代码预训练显著提升推理、结构化输出和指令遵循能力，即使任务不涉及代码 [Madaan et al., 2022]
  2. 数学数据的推理增益：数学推理数据的增加与 ARC、MMLU 等推理基准强相关
  3. 高质量数据的重复采样（Upsampling）：Wiki、书籍等高质量数据通常会被采样 2–4 遍

4.2 语言配比（多语言设计）

  • Temperature Sampling：pi ∝ ni1/T，T 通常取 0.7–0.8，可平滑语言间差异
  • mC4/CC-100 专门支持 100+ 语言
  • Qwen 2.5 支持约 30 种语言，亚洲语言（日语、韩语、阿拉伯语）均有专项数据增强

4.3 动态配比：DoReMi 与 RegMix

DoReMi [Xie et al., 2023, NeurIPS]：使用 Group DRO（分布鲁棒优化）训练小代理模型（280M），通过最大化最坏域的性能来优化跨域配比。用优化后的配比训练大模型（8B），获得 6.5% 的下游任务提升。
MiniCPM 的 WSD 学习率调度器：Warmup-Stable-Decay 三阶段 LRS。Decay 阶段允许快速切换数据域，解耦了学习率调度与数据混合，提升灵活性。[Hu et al., 2024]

4.4 数据配比对下游任务的影响

调整方向主要受益任务代价
提升代码比例HumanEval, MBPP, SWE-bench轻微降低通用语言理解
提升数学比例GSM8K, MATH, AIME影响不大，推荐增加
提升书籍/新闻比例知识性 QA, 长文本生成可能降低代码能力
提升多语言比例多语言基准英语性能轻微下降
提升学术论文比例MMLU, GPQA增加训练成本
05

合成数据（Synthetic Data）

5.1 合成数据的动机与挑战

动机
  • 数据稀缺性：特定领域天然缺乏
  • 质量可控：按需生成满足特定格式/难度的数据
  • 数据飞轮：强模型生成数据 → 弱模型训练 → 能力提升
挑战
  • Model Collapse：错误和偏差被放大传递
  • 多样性不足：单一生成模型缺乏变化
  • 真实性缺失：缺乏真实世界的边缘情况
  • 计算成本：高质量合成数据生成代价高昂

5.2 代码合成

  • StarCoder：基于 The Stack v1 的真实代码 + FIM 训练策略，合成代码文档和单元测试
  • DeepSeek-Coder：合成大量代码推理路径（"Scratchpad"），为算法题目生成 Think step-by-step 格式解题过程
  • LLaMA 3 代码增强：使用 LLaMA 2 注释代码库，生成单元测试 + 可执行代码对

5.3 数学合成

  • MetaMath：通过改写 GSM8K/MATH 数据集问题，多角度重新表述答案，使用 ChatGPT 多轮自我完善
  • NuminaMath（2024）：专门为 AIME/AMC 竞赛数学设计，将专业解题过程转化为 Chain-of-Thought 格式
rStar-Math（Microsoft, 2025）：通过 MCTS + 代码验证生成自我进化数学推理数据。4 轮自我进化处理 747k 数学题，Qwen2.5-Math-7B 从 58.8% 提升到 90.0%（MATH benchmark）。[Guo et al., 2025]

5.4 通用合成：Phi 系列的 Textbook-quality 数据

  • Phi-1（2023）：6B tokens "textbook quality" 网页数据 + 1B tokens GPT-3.5 生成 Python 教材；1.3B 参数 HumanEval 达到 50.6%
  • Phi-1.5 / Phi-2 / Phi-3：逐步扩大合成数据范围，形成"合成教材 + 合成练习"双轨

5.5 合成数据质量保证与多样性控制

质量保证策略

  1. Rejection Sampling：只保留可验证（代码可执行/数学答案正确）的答案
  2. 自一致性过滤：多次生成，只保留主流共识答案
  3. 人工审核抽样：对 1–5% 的合成数据进行人工质量评估
  4. 多模型验证：使用独立模型验证另一模型的输出

多样性控制策略

  1. 多样化 Prompt 模板，避免固定格式导致的同质化
  2. Topic Seeding：从真实语料提取话题词引导多样话题
  3. 难度分级：控制简单/中等/困难题目的比例

5.6 Model Collapse 问题及解决方案

Model Collapse（Shumailov et al., 2023, Nature）指当模型在合成数据上反复训练时，数据分布逐渐退化，尾部罕见信息丢失，导致模型能力下降。

  • 保留原始人工数据：始终混合一定比例的真实人类生成数据
  • 数据多样性增强：人为控制合成数据的话题分布
  • 迭代验证：每轮合成数据训练后，出现退化立刻终止
  • 独立验证集：使用从未见过的人工数据作为锚点，检测分布漂移
06

课程学习与训练策略

6.1 数据课程设计（先易后难）

困难度的定义方式：语言模型困惑度（PPL）、教育价值打分（Phi 系列）、词汇复杂度、知识密度（实体密度、概念多样性）。

6.2 多阶段训练

阶段 1：主预训练（Main Pretraining）
    - 大规模混合数据（≥10T tokens）
    - 固定 Context Length（通常 4K–8K tokens）
    - 目标：学习语言基础能力、世界知识

阶段 2：长上下文扩展（Long Context Training）
    - 减少训练数据量（0.5T–1T tokens）
    - 扩展 Context Length（32K→128K）
    - 使用 RoPE 扩展/YaRN/LongRoPE 等位置编码调整
    - 数据组成：长文档（书籍/代码库/长对话）比例上调

阶段 3：退火阶段（Annealing / Cooldown）
    - 极小数据量（50B–500B tokens）
    - 高质量精选数据
    - 学习率线性衰减至 0
    - 数学/代码/推理数据比例大幅提升
DeepSeek-V3 的退火设计：退火阶段使用约 500B tokens 精选语料，重点包含：高质量推理（数学、物理、CS 竞赛题）、高质量代码、优质 Wikipedia。退火期间 loss 下降异常平滑，无 spike。

6.3 OLMo 2 的 Dolmino Mix 退火策略

OLMo 2 在退火阶段引入了专门设计的 Dolmino Mix 1124，包含 DCLM-baseline 精华子集、Numina Math、Code 高质量子集、StackExchange 和学术论文精华。实验证明在多个基准上超过了 LLaMA 3.1（使用更少的 FLOPs）。[OLMo Team, 2025]

6.4 持续预训练（Domain Adaptation）

场景代表模型继续训练数据量
代码专化Code Llama（from LLaMA 2）500B tokens 代码数据
医疗专化BioMedLM、MedPaLM医学文献
法律专化LexLLM法律语料
中文增强多个中文增强模型中文语料补充

技术要点：使用 Replay Buffer 混合少量原领域数据防止灾难性遗忘；降低学习率（通常为主训练的 10–30%）。

数据基础设施与工程实践

7.1 数据处理流水线设计

[数据源] → [Executor（Slurm/Ray/Spark）]
                ↓
    ┌─────────────────────────┐
    │ Pipeline Stage 1: 读取  │  WARC → JSON
    │ Pipeline Stage 2: 过滤  │  语言/长度/规则
    │ Pipeline Stage 3: 去重  │  MinHash LSH
    │ Pipeline Stage 4: 质量  │  分类器打分
    │ Pipeline Stage 5: 写出  │  分片 JSONL
    └─────────────────────────┘
                ↓
    [分布式存储：S3/HDFS/GCS]
  • 流式处理：避免中间结果全量落盘，节省存储
  • 并行去重：跨节点的分布式 MinHash，需 All-to-All 通信
  • 增量处理：新 CC 快照只处理增量，复用历史签名库

7.2 大规模去重工程

1. 签名生成：每文档生成 128–256 个 MinHash 值
2. 分桶（Banding）：将 128 个签名分为 b=16 个 band，每 band 8 个值
3. 候选对生成：只比较同 band 落入同 bucket 的文档对
4. 最终确认：计算 Jaccard 相似度，阈值 0.7 过滤
5. Union-Find：合并去重聚类，每聚类保留最长/质量最高文档

7.3 数据版本管理

  • 数据卡（Data Cards）：记录每个数据集的来源、处理步骤、质量指标
  • Schema 版本控制：使用 Git LFS 或 DVC
  • Snapshot 机制：保留关键时间点的数据快照，支持消融实验
  • 元数据追踪：每个文档保留来源 URL、处理时间戳、质量分数

7.4 在线数据抽样与动态更新

Online Data Mixing：在训练过程中动态调整各数据源的采样率，根据各域的 validation loss 变化自适应调整权重，避免了静态数据集固化带来的灵活性损失。

数据飞轮（Data Flywheel）：强模型生成标注 → 数据质量提升 → 新一代模型 → 能力更强的标注器。这是大型科技公司（OpenAI、Google、Meta、阿里）构建数据竞争壁垒的核心机制。

面向 TerminalBench 的预训练数据设计建议

8.1 终端操作相关数据的识别与采集

一级优先数据来源

  • 🔴 Shell 脚本仓库：GitHub 上的 .sh、.bash、.zsh 文件，覆盖自动化/运维/CI-CD 脚本
  • 🔴 Stack Overflow / Unix StackExchange：包含大量 "如何用命令行完成 X" 的 QA 对
  • 🔴 man page 库：Linux man pages，包含准确的命令语法和示例
  • 🔴 Ansible/Terraform/Kubernetes YAML：基础设施即代码
  • 🔴 运维博客/技术教程：DigitalOcean Tutorials、Linux Journey、Arch Wiki

二级优先数据来源

  • 🟡 GitHub Issues/PR（含 bug 描述和修复命令）
  • 🟡 Docker Hub 文档
  • 🟡 crontab/systemd 配置示例
  • 🟡 CTF Writeup（命令行安全操作的详细步骤）

8.2 Shell/Bash 代码数据的处理特点

# 需要特别注意的Shell代码质量标准：

# 1. 可执行性验证 - shellcheck 静态检查
shellcheck --severity=warning script.sh

# 2. 安全性过滤 - 避免危险命令模式
DANGEROUS_PATTERNS=(
    "rm -rf /"
    "dd if=/dev/zero"
    ":(){ :|: & };:"  # fork bomb
)

# 3. 上下文保留 - 多行命令不应被截断
if [ condition ]; then
    # 应完整保留整个 if-then-fi 块
    command
fi

8.3 推荐的数据配比调整方向

标准配比基础上的增量调整：

+5%  Shell/Bash/命令行脚本代码
+3%  Unix/Linux 系统管理文档（man pages, 官方文档）
+2%  Stack Overflow（Unix & Linux, Server Fault 子站点）
+2%  DevOps 相关内容（GitHub Actions, CI/CD, Kubernetes）
+1%  安全与渗透测试相关（CTF writeup, 安全博客）
-5%  普通网页文本（适当减少）
-3%  商业内容/广告类文本

具体建议比例（Total Pretraining Data）：
- Shell 脚本：2–3%
- 系统文档：2%
- 技术 QA（偏运维/SysAdmin）：3%
- 通用代码（含 Python/Go/C 等）：15%（保持较高）
- 数学/推理：5%（保持推理能力）

9.1 数据污染（Contamination）检测

方法原理局限
n-gram 重叠检测计算 13-gram 重叠率对改写的污染无效

9.3 数据飞轮（Data Flywheel）

用户使用模型 → 产生交互数据（偏好/反馈/使用模式）
    ↓
数据用于改进下一代模型（RLHF/DPO/持续预训练）
    ↓
更好的模型吸引更多用户
    ↓
形成数据壁垒，后来者难以追赶
DeepSeek 的低成本破局：通过技术创新（MLA、MoE专家路由优化）降低训练成本，用 2.788M H800 GPU hours（约 557万美元）训练了 DeepSeek-V3，打破了数据飞轮的算力壁垒。[DeepSeek-AI, 2024]

9.4 Scaling Law 对数据的新启示

  1. 数据重复的 Scaling Law：适当重复（2–4次）高质量数据比使用等量低质量数据更优 [Muennighoff et al., 2023]
  2. 数据质量的非线性影响：高质量数据的边际收益显著高于低质量数据
  3. MiniCPM 的发现：最优数据-模型比例远高于 Chinchilla 预测，某些设置下超过 100:1 [Hu et al., 2024]
  4. Inference-Time Compute Scaling（2025）：DeepSeek-R1、Kimi k1.5 开辟了新方向，推理时扩展在某些任务上可补偿预训练数据不足
  5. 数据 Pruning vs. Scaling：SemDeDup 等工作显示删除 50% 的语义冗余数据几乎不损失性能
