数据工程的未来

数据墙、多模态、Agent 数据与开放问题

发布于：2026年3月21日 未来展望 数据墙 多模态 Agent

“预测未来最好的方法是创造它。” 从 GPT-3 到 DeepSeek-R1，短短五年间数据工程经历了多次范式转变。本章试图描绘接下来几年可能发生的变化——不是作为预言，而是作为一张值得探索的地图。

22.1 即将到来的变化

22.1.1 数据墙（Data Wall）：互联网文本的天花板

问题的本质：

人类文明积累的高质量互联网文本总量大约在 10-15T tokens 的量级。当前最大的模型（Llama 3 训练了 15T tokens）已经逼近甚至触及了这个天花板。去重后的高质量文本就更少了——FineWeb 2 的全量去重数据也不过约 15T tokens。

互联网高质量文本的供给曲线（估算）

    可用量
    (T tokens)
    15 |            ┌──────── 天花板 ───────
       |          ╱
    10 |        ╱
       |      ╱
     5 |    ╱
       |  ╱
     0 |╱
       +──────────────────────────────
       2020   2022   2024   2026   2028

问题：2026 年之后，高质量互联网文本的增量已经
     无法满足模型规模的指数增长需求。

应对策略矩阵：

策略成熟度效果风险
合成数据较成熟可扩展 3-5 倍Model Collapse
多模态数据快速发展中开辟新数据源标注成本高
数据效率优化研究前沿同量数据训练效果更好理论不完善
数据重复使用有争议简单直接收益递减，过拟合
主动数据收集早期探索针对性强成本和伦理问题
Agent 交互数据刚刚起步全新数据类型质量参差不齐

🔮 我的判断：数据墙不是一堵不可逾越的墙，而更像是一个需要换道超车的弯道。从”被动利用互联网存量数据”到”主动构造高信息密度的训练信号”，这个范式转变已经开始了。

22.1.2 合成数据的未来

合成数据是应对数据墙最直接的方案，但它的演进方向值得深思。

当前：第一代合成数据

  • 方法：用 GPT-4/Claude 直接生成
  • 特点：质量受限于教师模型能力
  • 问题：多样性不足、存在系统性偏差、有 Model Collapse 风险

未来：第二代合成数据

第二代合成数据的四大特征：

1. 验证驱动（Verification-driven）
   ┌─────────────────────────────────────┐
   │ 生成 → 验证 → 反馈 → 再生成        │
   │ 不是"生成即可用"，而是"验证后可用"  │
   └─────────────────────────────────────┘
   
2. 环境交互（Environment-grounded）
   ┌─────────────────────────────────────┐
   │ 模型在环境中行动 → 收集轨迹数据     │
   │ 代码执行、工具调用、网页浏览        │
   └─────────────────────────────────────┘
   
3. 自我博弈（Self-play）
   ┌─────────────────────────────────────┐
   │ 多个模型相互博弈 → 产生新的训练信号 │
   │ 类似 AlphaGo 的自我进化             │
   └─────────────────────────────────────┘
   
4. 课程化（Curricular）
   ┌─────────────────────────────────────┐
   │ 根据模型当前能力动态调整合成难度     │
   │ 始终保持在"最近发展区"               │
   └─────────────────────────────────────┘

微软 SynthLLM 框架给出了一个有价值的方向：不是让模型直接生成答案式数据，而是让模型设计”知识图谱 → 教材 → 习题”这样的结构化教学内容。这种方法产生的合成数据信息密度更高，多样性也更好。

22.1.3 多模态数据：下一个数据洪流

互联网上的文本数据在逼近天花板，但视觉数据、音频数据、视频数据的可用量还远未被充分挖掘。

多模态数据的规模估算：

模态估算可用量当前利用率潜力
文本10-15T tokens~80%低（接近饱和）
图文对100B+ 对~20%高
视频数十亿小时<5%极高
音频/播客数十亿小时<5%极高
代码+执行轨迹无限（可生成）<10%极高
3D/空间数据快速增长<1%极高

多模态数据工程面临的新挑战：

  1. 对齐问题：图片和文字的语义对齐、视频和旁白的时序对齐
  2. 质量定义：多模态数据的”质量”比纯文本更难定义和度量
  3. 处理成本：视频数据的存储和处理成本比文本高 100-1000 倍
  4. 标注复杂度：多模态标注需要更复杂的标注协议和更专业的标注员
  5. 去重难度：跨模态的语义去重是尚未解决的问题

22.1.4 Agent 数据：从对话到行动

2025-2026 年，AI Agent 从概念走向产品。Agent 需要一种全新类型的训练数据——行动轨迹数据（Action Trajectory Data）。

传统对话数据:
  User: "帮我查一下明天的天气"
  Assistant: "明天北京多云，最高温度 25°C..."

Agent 轨迹数据:
  User: "帮我查一下明天的天气，如果会下雨就帮我在日历上标注带伞"
  
  [Thought] 需要先查天气，然后根据结果决定是否操作日历
  [Action] call_api("weather", location="北京", date="tomorrow")
  [Observation] {"condition": "rain", "temperature": "18°C", ...}
  [Thought] 会下雨，需要在日历上添加提醒
  [Action] call_api("calendar", action="add_event", ...)
  [Observation] {"status": "success", "event_id": "..."}
  [Response] 明天北京有雨，18°C。我已经在你的日历上标注了"带伞"提醒。

Agent 数据构造的核心挑战：

  1. 环境多样性：Agent 需要在各种环境中行动——网页、API、文件系统、数据库。每种环境需要不同的数据
  2. 错误恢复：好的 Agent 数据不仅包含成功轨迹，还要包含错误-恢复的轨迹
  3. 规划与反思：需要包含显式的思考过程（CoT），而不仅是 action 序列
  4. 安全边界：Agent 可以”真正”操作系统，因此安全约束的数据尤为重要

Agent 数据的未来架构：

层次数据类型构造方法
基础能力层工具调用格式数据模板合成 + 人工验证
环境交互层成功/失败轨迹沙盒环境自动采集
规划推理层多步骤推理链强模型蒸馏 + RL
安全约束层安全边界案例红队测试 + 规则生成
自我改进层反思-修正轨迹自我博弈 + 人工审核

22.1.5 持续学习：模型如何不断从新数据中学习

当前的大模型训练范式是”训练一次，部署很久”。但这带来一个根本问题——模型的知识会过时。

持续学习的数据挑战：

挑战 1: 知识更新 vs 遗忘
  - 注入新知识时如何不破坏旧知识？
  - 中训练的回放策略是初步方案，但不够优雅
  
挑战 2: 增量训练的数据策略
  - 每周/每月有新数据到来
  - 哪些旧数据需要保留？哪些可以丢弃？
  - 新旧数据的配比如何动态调整？
  
挑战 3: 灾难性遗忘的长期管理
  - 经过 10 次增量训练后，最初的能力还剩多少？
  - 有没有"数据保鲜期"的概念？

挑战 4: 数据流的质量保证
  - 持续到来的数据如何保持一致的质量标准？
  - 自动化质量监控如何适应数据分布的漂移？

🔮 我的判断：未来的大模型数据工程不再是”一次性项目”，而将成为一个持续运转的数据系统——类似于搜索引擎的持续爬取和索引系统。数据工程师的角色将从”建设者”转变为”运营者”。

22.2 数据工程师的进化

22.2.1 从”数据清洗工”到”数据科学家”

大模型数据工程这个领域还很年轻——2020 年之前几乎不存在”专门做大模型数据”的岗位。它的演变大致如下：

2020: "数据清洗工"
  └─ 会写 Python 脚本处理文本就行
  └─ 主要工作：去重、过滤、格式化

2022: "数据工程师"
  └─ 需要理解 NLP 基础知识
  └─ 主要工作：建设完整的数据管线

2024: "数据科学家"
  └─ 需要理解模型训练原理
  └─ 主要工作：数据策略设计、配比优化、质量模型

2026: "数据架构师"
  └─ 需要跨越数据和模型的完整认知
  └─ 主要工作：端到端数据系统设计、数据-模型协同优化

2028+: "数据-模型协同工程师"（我的预测）
  └─ 数据工程和模型训练不再分离
  └─ 模型在训练过程中自主决定需要什么数据
  └─ 工程师的角色是设计和维护这个自动化系统

22.2.2 数据直觉的培养

好的数据工程师有一种”数据直觉”——看一眼数据就能判断它对训练有没有价值。这种直觉不是天生的，而是通过大量实践培养的。

培养数据直觉的建议：

  1. 看数据：每个项目至少花 10 个小时亲自阅读原始数据。不是采样统计，而是逐条阅读。你会发现很多统计无法捕捉的问题。

  2. 做消融：每次修改管线后，都做一个小规模消融实验。不需要训练大模型——400M 参数的小模型在 5B tokens 上训练，就能反映大多数数据质量问题。

  3. 建基准：建立你自己的”数据质量直觉基准”——收集 100 条你认为是”好数据”和 100 条”坏数据”，用它来校准你的判断。

  4. 跨领域：不要只看文本数据。理解代码数据、数学数据、多模态数据的不同特征，可以让你对”质量”有更全面的认识。

  5. 读论文：系统阅读 FineWeb、DCLM、RedPajama、Dolma 等项目的技术报告。这些报告中有大量的数据工程经验和反直觉发现。

22.2.3 与模型训练团队的协作模式

数据工程师和训练工程师的协作模式也在演变：

传统模式（2020-2023）：

数据团队                       训练团队
  │                              │
  │ "这是处理好的数据"           │
  │ ────────────────────────→   │
  │                              │ 训练...
  │                              │
  │        "效果不好"            │
  │ ←────────────────────────   │
  │                              │
  │ 调整管线...                  │
  │                              │
  │ "新版数据"                   │
  │ ────────────────────────→   │
  │                              │
  反馈循环很慢，通常需要数周

现代模式（2024-2026）：

数据团队                       训练团队
  │                              │
  │ ←── 共享评估 Dashboard ──→   │
  │                              │
  │  实时监控训练 loss 和指标    │
  │  发现异常后立即调查数据      │
  │                              │
  │  数据版本与 checkpoint 对应  │
  │  可以精确定位哪个数据版本    │
  │  导致了哪个能力变化          │
  │                              │
  反馈循环缩短到天级甚至小时级

未来模式（2027+）：

  ┌─────────── 自动化数据-训练循环 ───────────┐
  │                                            │
  │  模型训练中                                 │
  │    ↓                                       │
  │  在线评估发现薄弱能力                       │
  │    ↓                                       │
  │  自动调整数据配比 / 合成针对性数据          │
  │    ↓                                       │
  │  数据注入训练流                             │
  │    ↓                                       │
  │  再评估...                                 │
  │                                            │
  └────────────── 人工监督+干预 ──────────────┘

22.3 开放问题

22.3.1 我们还需要多少数据？

这是一个看似简单实际极深刻的问题。不同的回答反映了对智能本质的不同理解。

乐观派：数据需求有上限。当模型足够大、数据足够好时，模型可以从有限的数据中”涌现”出超越数据的能力。证据包括：

  • 少量高质量合成数据可以训练出强大模型（Phi 系列）
  • LIMA 表明 1000 条数据就能实现表面对齐
  • DeepSeek-R1-Zero 仅用 RL 就能涌现推理能力

悲观派：数据需求会持续增长。模型本质上是数据的压缩表示，没有足够多样的数据就不可能获得足够多样的能力。证据包括：

  • 更大的数据一致带来更好的效果（Scaling Laws）
  • Model Collapse 表明合成数据不能替代真实数据
  • 现实世界的长尾分布意味着永远有未覆盖的场景

我的观点：数据的总量需求可能会趋于平缓，但数据的类型和质量需求会持续演化。未来的瓶颈不是”有多少数据”，而是”有多少正确类型的、高信息密度的数据”。

22.3.2 合成数据能否完全替代真实数据？

基于 Nature 发表的 Model Collapse 研究和后续的大量实验，短期内的答案是否。

但这个问题的更深层含义值得思考：

如果合成数据不能完全替代真实数据，
那么"真实数据"中到底有什么是合成数据缺乏的？

可能的答案：
1. 信息的多样性——真实世界的长尾分布
2. 噪声的价值——适度的噪声帮助模型泛化
3. 文化和常识——隐含在自然文本中的社会知识
4. 创造性——人类创作的非规律性和意外性
5. 时效性——真实世界不断产生新的信息

如果我们能精确定义这些因素，
或许就能设计出不会导致 collapse 的合成方法。

一个有趣的类比：这就像营养学中的”合成维生素 vs 天然维生素”之争。合成维生素化学结构相同，但天然食物中还包含微量元素、纤维、酶等协同因素。训练数据也类似——真实文本中包含大量我们尚未识别但对模型有益的”微量元素”。

22.3.3 数据质量有没有统一的度量标准？

至今没有。这是数据工程领域最大的理论缺口之一。

当前的困境：

  • “质量”是一个依赖于下游任务的概念——对预训练好的数据对 SFT 可能不好
  • 不同的质量代理指标（PPL、教育价值分类器、LLM 评分）有各自的偏差
  • 没有类似”BLEU for translation”或”FID for images”这样被广泛接受的数据质量度量

可能的方向：

  1. 基于信息论的度量：数据对模型参数的信息增益
  2. 基于学习曲线的度量：加入数据后的学习效率变化
  3. 多维度组合度量：流畅性 × 信息密度 × 多样性 × 安全性的加权组合
  4. 任务无关的内在质量度量：不依赖下游任务的数据本身质量分数

我的期待：未来 3-5 年内，社区需要建立一个像 ImageNet 对计算机视觉那样的数据质量 Benchmark——不是评估模型，而是评估数据的好坏。这将是数据工程从手艺走向科学的关键一步。

22.3.4 最优的数据策略是否可以自动发现？

这是最令人兴奋的开放问题。当前的数据策略（配比、过滤阈值、课程顺序）主要依靠人类经验和大量实验。有没有可能让模型自己发现最优的数据策略？

已有探索：

  • DoReMi：用参考模型引导数据配比优化
  • Data Mixing Laws：用小实验预测大模型最优配比
  • AutoML for Data：将数据处理参数作为超参数搜索
  • 在线配比调整：根据训练中的 loss 信号动态调整数据比例

arXiv 2603.14712 综述提出了一个引人注目的愿景：Agent-based Data Preparation Systems——用 AI Agent 自动完成数据准备的全流程，包括采集、清洗、过滤、配比。

未来的自动化数据工程系统（概念架构）：

┌─────────────────────────────────────────┐
│           数据策略 Agent                 │
│  ┌──────────┐  ┌──────────┐            │
│  │ 配比优化  │  │ 质量评估  │            │
│  │ Agent    │  │ Agent    │            │
│  └────┬─────┘  └────┬─────┘            │
│       │              │                  │
│  ┌────▼──────────────▼─────┐           │
│  │    数据处理管线 Agent     │           │
│  │  ┌──────┐ ┌──────┐     │           │
│  │  │ 清洗  │ │ 过滤  │     │           │

22.4 写在最后

22.4.1 数据工程的黄金时代

我们正处于大模型数据工程的黄金时代。这个领域足够年轻——核心方法论在过去 3-5 年内才逐步成型；又足够重要——“Data is all you need” 不是口号，而是现实。

回顾过去五年的发展脉络：

年份核心认知
2020数据越多越好
2021Chinchilla: 数据和参数要匹配
2022质量比数量更重要
2023合成数据可以弥补真实数据不足
2024系统化的数据工程是可复现的科学
2025数据工程进入精细化、多阶段时代
2026数据系统走向自动化和持续化

每一年的认知都在演变，而且经常推翻之前的”共识”。这意味着——如果你在读这本书时发现某些观点已经过时了，那是一件好事，说明这个领域还在快速进步。

22.4.2 给数据工程师的建议

如果你刚入行：

  • 从动手做一个完整的小规模数据管线开始（比如处理 10GB 的 Common Crawl 数据）
  • 训练一个 400M 的小模型来验证你的数据质量——这是最快的反馈循环
  • 精读 FineWeb、DCLM 的技术报告——它们是最好的数据工程教材

如果你已有经验：

  • 关注合成数据和 Agent 数据这两个快速演进的方向
  • 建立你的数据质量直觉——这是很难被自动化替代的核心能力
  • 思考如何让你的数据管线从”一次性项目”变成”持续运转的系统”

如果你是技术负责人：

  • 数据工程的投入应占整个大模型项目的 30-50%——很多团队严重低估了这个比例
  • 建立数据和训练团队之间的紧密反馈循环
  • 投资数据基础设施——它的回报周期比你想象的短

22.4.3 最后的话

写这本书的过程中，我反复感受到一件事：数据工程是一门手艺，更是一门科学。

说它是手艺，因为很多关键决策依赖于经验和直觉——某个过滤阈值设 0.5 还是 0.6，某两类数据的配比是 7:3 还是 8:2，这些看似微小的选择可能对最终模型有巨大影响。好的数据工程师就像好的厨师，配方重要，但火候和手感同样关键。

说它是科学，因为这个领域正在快速建立自己的理论基础——Scaling Laws、Data Mixing Laws、合成数据的信息论分析、遗忘的定量度量。这些理论让数据决策从”拍脑袋”逐渐走向”可推导”。

未来几年，数据工程将成为大模型技术栈中最值得投入的方向之一。不是因为模型架构不重要，而是因为架构的优化空间正在收窄，数据的优化空间却依然广阔。

在这个”数据决定一切”的时代，愿这本书能帮你在数据的海洋中找到方向。
