第 13 章 第三部分：SFT 数据

SFT 数据质量的精细控制

好样本标准、清洗过滤、多样性保证与多轮对话

最后更新：2026年5月1日 SFT质量 数据清洗 多样性 多轮对话

“SFT 数据的质量控制不是一个检查清单，而是一种持续精进的工艺。“

13.1 什么是好的 SFT 样本

好样本的四个维度

维度 1: 指令清晰
  ✅ "用 Python 实现一个 LRU 缓存，要求线程安全，支持过期时间"
  ❌ "写个缓存"

维度 2: 回复准确
  ✅ 代码能正确运行，事实准确无误
  ❌ 代码有 bug，或引用不存在的 API

维度 3: 信息密度
  ✅ 每句话都传递有价值的信息
  ❌ 大量填充语（"好的，让我来帮你..."）和无关内容

维度 4: 格式规范
  ✅ 代码在代码块中，步骤用列表，关键词加粗
  ❌ 代码和文字混在一起，没有结构

案例分析：好样本 vs 差样本

指令：“解释什么是梯度消失问题，以及如何解决”

❌ 差样本：

好的！梯度消失问题是深度学习中很常见的一个问题哦~简单来说就是
当神经网络很深的时候梯度会变得很小很小然后就没法训练了。解决方
法有很多比如用ReLU激活函数或者用残差连接等等。希望这个回答对
你有帮助！

问题：口语化严重、信息密度低、没有结构、技术细节不足

✅ 好样本：

## 梯度消失问题

### 现象
在深层神经网络的反向传播中，梯度通过链式法则逐层相乘。当每层
的梯度值 < 1 时，经过 N 层后梯度指数级衰减到接近零，导致浅层
参数几乎无法更新。

### 数学本质
对于 L 层网络：∂Loss/∂W₁ = ∏ᵢ₌₁ᴸ (∂hᵢ/∂hᵢ₋₁) · ∂Loss/∂hₗ

当 |∂hᵢ/∂hᵢ₋₁| < 1 时，乘积 → 0

### 主要解决方案

| 方法 | 原理 | 典型应用 |
|------|------|---------|
| ReLU 激活函数 | 正区间梯度恒为 1 | CNN |
| 残差连接 | 梯度可以跳过层直接回传 | ResNet, Transformer |
| BatchNorm | 标准化中间表示，控制梯度范围 | 多数深度网络 |
| LSTM 门控 | 选择性保留/遗忘梯度 | RNN 序列模型 |
| 参数初始化 | Xavier/He 初始化保持方差稳定 | 所有网络 |

推理 SFT 数据的特殊质量要求

上述四维度是通用 SFT 数据的基线框架。随着 Long-CoT 和推理增强成为主流训练范式，推理 SFT 数据需要额外的质量维度。MCSQ（Multi-dimensional quantitative framework for evaluating distilled CoT SFT Data Quality, EMNLP 2025 Findings）提出了一个关键实证结论：结构胜于规模——结构化推理链的质量比数据量更重要。

推理 SFT 数据需要在通用四维度之上额外评估三个维度：

维度 5: 推理链完整性
  ✅ 每一步都有明确的前提和推导过程，无逻辑跳步
  ❌ 从问题直接跳到答案，或关键推导步骤缺失

维度 6: 步骤粒度合理性
  ✅ 每步推理恰好包含一个可验证的逻辑单元
  ❌ 一步内堆叠多个推理跳跃（过粗），或拆分为过于琐碎的子步骤（过细增加噪声）

维度 7: 中间结论可验证性
  ✅ 中间步骤的结论可以被独立验证（如数学表达式可以代入验证）
  ❌ 中间结论模糊不清，无法判断对错

这些维度的核心洞见在于：结构胜于规模——结构化推理链的质量比数据量更重要。2026年最新实证研究证实了这一观点：100 条结构完整、步骤粒度合理的推理链训练 128 个 epoch，在 AIME’24/‘25 和 GPQA 上显著优于 51,200 条数据训练 1 个 epoch。

推理 SFT 泛化的三个条件

Long-CoT SFT 泛化研究（2026.04）揭示了推理 SFT 泛化需要满足三个条件：

  1. 充分训练：多 epoch 训练（如 128 epoch）显著优于单 epoch，模型需要反复”消化”推理模式才能内化为泛化能力
  2. 高质量推理数据：推理链结构完整、步骤粒度合理、中间结论可验证
  3. 强基座模型：弱模型即使获得最好的推理数据也难以泛化，基座能力构成推理 SFT 的”地板”

这三个条件解释了为什么同样的推理数据在不同模型上效果差距巨大，也指导了推理 SFT 数据的质量评估标准。

13.2 数据清洗与过滤

自动化质量检查管线

"""
SFT 数据质量检查管线
"""

from dataclasses import dataclass
from typing import List, Optional
import re

@dataclass
class QualityCheckResult:
    passed: bool
    score: float  # 0-1
    issues: List[str]

def check_sft_sample(
    instruction: str,
    response: str,
    language: str = "zh",
) -> QualityCheckResult:
    """
    SFT 样本的多维度质量检查
    """
    issues = []
    score = 1.0

    # === 格式检查 ===

    # 1. 代码块完整性
    code_block_opens = response.count("```")
    if code_block_opens % 2 != 0:
        issues.append("代码块未正确闭合")
        score -= 0.3

    # 2. Markdown 结构检查
    if len(response) > 500 and "##" not in response and "\n-" not in response:
        issues.append("长回复缺少结构化格式")
        score -= 0.1

    # === 内容检查 ===

    # 3. 空回复或过短回复
    if len(response.strip()) < 20:
        issues.append("回复过短")
        score -= 0.5

    # 4. 过长的前缀废话
    filler_patterns = [
        r"^(好的|当然|没问题|好呢)[!！,，。.]*\s*(我来|让我)",
        r"^(Sure|Of course|Absolutely)[!,.]",
    ]
    for pattern in filler_patterns:
        if re.match(pattern, response):
            issues.append("包含不必要的前缀填充语")
            score -= 0.05

    # 5. 重复内容检测
    sentences = [s.strip() for s in response.split("。") if len(s.strip()) > 10]
    if len(sentences) > 3:
        unique_ratio = len(set(sentences)) / len(sentences)
        if unique_ratio < 0.7:
            issues.append(f"内容重复度高 (unique ratio: {unique_ratio:.2f})")
            score -= 0.2

    # 6. 截断检测
    if response.rstrip()[-1:] not in ["。", ".", "!", "！", "?", "？", "`", "）", ")", "】", "]", "\n"]:
        if not response.rstrip().endswith("```"):
            issues.append("回复可能被截断")
            score -= 0.15

    # === 一致性检查 ===

    # 7. 指令-回复相关性（简单关键词检查）
    instruction_keywords = set(instruction.lower().split())
    response_keywords = set(response.lower().split()[:100])
    overlap = len(instruction_keywords & response_keywords)
    if overlap < 1 and len(instruction_keywords) > 3:
        issues.append("回复可能与指令不相关")
        score -= 0.2

    # 8. 语言一致性
    if language == "zh":
        zh_chars = len(re.findall(r'[\u4e00-\u9fff]', response))
        total_chars = max(len(response), 1)
        if zh_chars / total_chars < 0.1 and "```" not in response:
            issues.append("中文指令但回复可能不是中文")
            score -= 0.3

    return QualityCheckResult(
        passed=score >= 0.6,
        score=max(0, score),
        issues=issues,
    )

IFD 方法：Instruction-Following Difficulty

IFD（Instruction-Following Difficulty）是一种数据筛选方法：用模型自身来评估每条数据的”价值”。

核心思想：

  • 对每条 (instruction, response) 对，计算模型生成 response 的困惑度（perplexity）
  • PPL 过低 → 模型”已经会了”，这条数据的训练价值有限
  • PPL 过高 → 回复可能与模型的能力差距太大，训练效果也不好
  • PPL 适中 → 这条数据在模型的”学习区”（zone of proximal development），价值最高
数据价值的 PPL 分布：

  PPL 很低（< 5）     → "已经会了" → 训练价值低
  PPL 适中（5-50）    → "学习区"  → 训练价值最高
  PPL 很高（> 50）    → "太难了"  → 训练效果差

PPL 的更深层含义：模型-数据兼容性

上面的三档分类是直觉上的入门框架。大规模 SFT 实验（EMNLP 2025 Main）揭示了 PPL 更深层的含义：PPL 实际衡量的是模型-数据兼容性，而非绝对难度。

这一区分至关重要。同一条 SFT 数据对不同基座模型会产生不同的 PPL 值——因此”好数据”是相对于模型而言的，不存在脱离模型的绝对质量。实验发现：

  1. PPL 是 SFT 有效性的最佳预测指标——优于训练数据与 benchmark 的表面相似度。高相似度的数据如果与模型内部表示不兼容，SFT 效果可能很差。
  2. 中层权重变化与性能提升高度相关。PPL 能预测 SFT 效果的原因在于，它间接反映了数据能否有效修改模型的中层表示——真正的”学习”发生在中间层，而非表层。
  3. 实践含义：IFD 筛选不应该只做一次——当基座模型更新后，同一批数据的 PPL 分布会改变，需要重新评估。

Token 级推理质量评估：TI-DPO 方法

传统的序列级质量评估主要关注整体回复质量，但推理链的质量信号在 token 级高度不均匀。TI-DPO（Token-level Inference DPO, ICLR 2026）研究发现，推理链中少量关键 token 贡献了大部分判别信号。

关键发现：

  • 推理链中约 5-10% 的 token（通常是推理步骤的开头、转折点、结论点）决定了整个推理链的质量
  • 序列级评估可能掩盖关键 token 的质量问题
  • token 级评估能更精准地识别推理链中的薄弱环节

实践方法：

def identify_critical_tokens(reasoning_chain: str) -> List[int]:
    """识别推理链中的关键 token 位置"""
    # 1. 步骤开头 token（如"首先"、"第一步"、"因此"）
    step_starters = ["首先", "第一步", "因此", "所以", "因为", "由于"]
    # 2. 转折点 token（如"但是"、"然而"、"相反"）
    transition_words = ["但是", "然而", "相反", "尽管如此"]
    # 3. 结论点 token（如"综上"、"因此"、"结论是"）
    conclusion_words = ["综上", "因此", "结论是", "最终"]
    
    tokens = reasoning_chain.split()
    critical_positions = []
    
    for i, token in enumerate(tokens):
        if token in step_starters + transition_words + conclusion_words:
            critical_positions.append(i)
    
    return critical_positions

应用 TI-DPO 方法时，应重点关注这些关键 token 的质量，而非均匀评估整个序列。

IFD 的升级用法：

  传统：    PPL 三档 → 留中间、去两端
  升级：    PPL × 模型版本 → 每次换基座重新筛选
            PPL × 任务类型 → 不同任务的"学习区"阈值不同

13.3 数据多样性的保证

多样性的多个维度

维度 1: 任务多样性
  - 覆盖尽可能多的任务类型
  - 避免某一类任务过度主导

维度 2: 领域多样性
  - 科技、医学、法律、教育、金融、日常...
  - 每个领域都有代表性样本

维度 3: 长度多样性
  - 短回复（一两句话）到长回复（数千字）
  - 不要让模型形成固定的长度偏好

维度 4: 难度多样性
  - 从入门到专家级的全谱系覆盖

维度 5: 风格多样性
  - 正式/非正式、技术/通俗、详细/简洁

使用 Embedding 进行多样性分析

"""
基于 Embedding 的 SFT 数据多样性分析
"""

import numpy as np
from typing import List

def analyze_diversity(
    instructions: List[str],
    embeddings: np.ndarray,  # (N, dim) 的 embedding 矩阵
    n_clusters: int = 20,
) -> dict:
    """
    分析 SFT 数据集的多样性
    """
    from sklearn.cluster import KMeans
    from sklearn.metrics import silhouette_score

    N = len(instructions)

    # 1. 聚类分析
    kmeans = KMeans(n_clusters=n_clusters, random_state=42, n_init=10)
    labels = kmeans.fit_predict(embeddings)

    # 各类别大小
    cluster_sizes = np.bincount(labels)
    size_std = np.std(cluster_sizes)

    # 2. 覆盖度分析
    # 理想情况下，类别大小应该相对均匀
    coverage_score = 1 - (size_std / np.mean(cluster_sizes))

    # 3. 平均余弦距离（全局多样性）
    # 采样计算避免 O(N²)
    sample_size = min(5000, N)
    indices = np.random.choice(N, sample_size, replace=False)
    sample_emb = embeddings[indices]
    # 归一化
    norms = np.linalg.norm(sample_emb, axis=1, keepdims=True)
    normalized = sample_emb / np.maximum(norms, 1e-8)
    # 平均余弦距离
    cos_sim_matrix = normalized @ normalized.T
    avg_similarity = (cos_sim_matrix.sum() - sample_size) / (sample_size * (sample_size - 1))
    avg_diversity = 1 - avg_similarity

    # 4. 识别稀疏区域（可能缺少样本的类别）
    sparse_clusters = [i for i, size in enumerate(cluster_sizes)
                       if size < N / n_clusters * 0.3]

    return {
        "total_samples": N,
        "n_clusters": n_clusters,
        "cluster_sizes": cluster_sizes.tolist(),
        "coverage_score": round(float(coverage_score), 3),
        "avg_diversity": round(float(avg_diversity), 3),
        "sparse_cluster_count": len(sparse_clusters),
        "silhouette_score": round(float(silhouette_score(embeddings, labels)), 3),
        "recommendation": (
            "多样性良好" if coverage_score > 0.7 and avg_diversity > 0.5
            else "建议增加稀疏领域的样本"
        ),
    }

避免合成数据中的多样性不足

合成数据最常见的问题是模式坍缩：模型倾向于生成相似模式的回复。

检测方法：

def detect_pattern_collapse(responses: List[str], n_samples: int = 1000) -> dict:
    """
    检测合成回复中的模式坍缩
    """
    from collections import Counter

    # 1. 开头句式检测
    first_sentences = [r.split("。")[0][:50] if "。" in r else r[:50]
                       for r in responses[:n_samples]]
    opening_counter = Counter(first_sentences)
    top_openings = opening_counter.most_common(10)
    repetitive_openings = sum(c for _, c in top_openings if c > n_samples * 0.05)

    # 2. 结构模式检测
    structures = []
    for r in responses[:n_samples]:
        # 简单的结构指纹：标题数量、列表项数量、代码块数量
        n_headers = r.count("##")
        n_list_items = r.count("\n-") + r.count("\n1.")
        n_code_blocks = r.count("```") // 2
        structures.append((n_headers, n_list_items, n_code_blocks))

    structure_counter = Counter(structures)
    top_structures = structure_counter.most_common(5)

    # 3. 汇总
    collapse_risk = "low"
    if repetitive_openings > n_samples * 0.3:
        collapse_risk = "high"
    elif repetitive_openings > n_samples * 0.15:
        collapse_risk = "medium"

    return {
        "collapse_risk": collapse_risk,
        "top_repeated_openings": top_openings[:5],
        "repetitive_opening_ratio": repetitive_openings / n_samples,
        "top_structural_patterns": top_structures,
        "unique_structures": len(structure_counter),
    }

多样性的训练目标维度

上面的检测方法都是从数据层面入手。但模式坍缩的根源可能不仅在数据，还在训练目标本身。

SED-SFT（Selectively Encouraging Diversity in SFT, 2026）揭示了一个被忽视的机制：标准交叉熵（CE）损失天然鼓励模式坍缩。CE 优化的是 argmax 概率——它把所有概率质量集中到”最可能的下一个 token”上，系统性地挤压那些不是最优但完全合理的表达。换句话说，即使训练数据足够多样，CE 损失也在训练过程中把多样性”抹平”。

SED-SFT 的应对策略是在 token 级别自适应鼓励多样性：对探索空间大的位置（如句子开头、风格选择点）加入熵正则项，保留模型在这些位置的多路径能力；对确定性高的位置（如事实表述、代码语法）仍然正常优化。

另一个视角来自 DFT（Dynamic Fine-Tuning, ICLR 2026），它从强化学习视角分析 SFT 的有限泛化能力，发现仅在 NLL 损失基础上加入奖励信号校正——本质上是一行代码的修改——就能显著提升 SFT 的泛化性。

这些发现的实践意义是：真正的多样性保证需要数据层面和目标函数层面双管齐下。

层面方法效果
数据层面多样化采集 + embedding 聚类 + 模式坍缩检测保证输入多样性
目标函数层面SED-SFT 熵正则 / DFT 奖励修正保证训练过程不丢失多样性

仅靠其中一个层面是不够的——多样化的数据配合 CE 损失仍会坍缩，正则化的目标函数配合单一数据也无法凭空创造多样性。

13.4 多轮对话数据的特殊考量

多轮依赖关系的构造

多轮对话数据需要有真实的轮次间依赖——不能是”把多个单轮拼在一起”。

❌ 假多轮（没有真正的依赖）：
  User: "什么是 TCP？"
  Assistant: "TCP 是传输控制协议..."
  User: "Python 怎么读文件？"  ← 和第一轮完全无关
  Assistant: "可以使用 open() 函数..."

✅ 真多轮（有上下文依赖）：
  User: "什么是 TCP？"
  Assistant: "TCP 是传输控制协议..."
  User: "它和 UDP 有什么区别？"  ← 依赖第一轮的上下文
  Assistant: "TCP 和 UDP 的主要区别在于..."
  User: "什么场景下应该用 UDP？"  ← 依赖前两轮
  Assistant: "UDP 适合实时性要求高..."

上下文一致性检查

def check_multi_turn_consistency(
    conversation: list,  # [{"role": "user"|"assistant", "content": str}, ...]
) -> dict:
    """
    多轮对话一致性检查
    """
    issues = []

    for i in range(1, len(conversation)):
        current = conversation[i]
        if current["role"] == "assistant":
            # 检查是否与自己之前的回复矛盾
            for j in range(i - 1, -1, -1):
                if conversation[j]["role"] == "assistant":
                    # 简化检查：关键实体是否前后一致
                    # 实际中应该用 NLI 模型检查
                    pass

            # 检查是否遵循了最近的用户指令
            prev_user = None
            for j in range(i - 1, -1, -1):
                if conversation[j]["role"] == "user":
                    prev_user = conversation[j]["content"]
                    break

            if prev_user and len(current["content"]) < 10:
                issues.append(f"Turn {i}: 回复过短，可能未充分回应用户")

    # 检查对话长度合理性
    assistant_turns = [c for c in conversation if c["role"] == "assistant"]
    lengths = [len(c["content"]) for c in assistant_turns]
    if len(lengths) > 2:
        if max(lengths) / max(min(lengths), 1) > 10:
            issues.append("各轮回复长度差异过大")

    return {
        "turn_count": len(conversation),
        "issues": issues,
        "is_consistent": len(issues) == 0,
    }

从线性到分支：多轮数据的结构化升级

上面讨论的真多轮 vs 假多轮区分，以及上下文一致性检查，解决的是多轮对话数据的基础质量问题。更进一步，Conversation Forests（2025）提出了一种超越线性序列的数据组织方式——分支对话树。

传统的多轮对话数据是线性的：User → Assistant → User → Assistant → …。但在真实交互中，同一个对话上下文下可能有多种合理的后续发展方向。分支对话树允许在每个节点探索不同的回复路径：

           User: "解释 Transformer"
                 │
          Assistant: "Transformer 是..."
                 │
    ┌────────────┼────────────┐
    │            │            │
 "注意力机制    "和RNN比     "代码实现
  怎么工作？"   有什么优势？"  能看看吗？"
    │            │            │
  (分支A)      (分支B)      (分支C)

分支结构天然编码了多样性——模型从中学到的不是”给定上下文只有一种正确回应”，而是”同一上下文下有多种合理的发展方向”。在医疗对话等需要探索多种诊断路径的场景中，分支对话树已被验证能显著提升对话质量。

不过需要注意，分支对话树目前的验证主要集中在垂直领域（如医疗对话），推广到通用 SFT 的效果还需要更多实证。这一范式更适合作为高价值对话数据的构造策略，而非通用的大规模数据生产方法。

13.5 SFT 的评估与迭代

如何评估 SFT 数据的效果

不要把”模型表现”和”数据质量”混为一谈。以下是专门针对 SFT 数据质量的评估方法：

1. 数据影响力分析

找出哪些训练样本对模型表现贡献最大：

方法 A: Leave-One-Out（太贵，不实用）
  → 每次移除一条数据，重新训练，观察影响

方法 B: 影响函数（Influence Functions）
  → 近似计算每条数据的影响
  → 计算成本可控

方法 C: 数据 Shapley 值
  → 用采样方法估计每条数据的边际贡献
  → 最公平但也最贵

方法 D: RL-readiness 评估
  → 不看 SFT 后的表现，而是看经 RL 后的表现
  → PEAR 的发现：SFT 得分最高 ≠ RL 后得分最高

上述前三种方法都从”SFT 后模型表现”角度评估数据质量。PEAR（Policy Evaluation-inspired Algorithm for Offline Learning Loss Re-weighting, 2026）提出了一个颠覆性的发现：SFT 性能更强的检查点，经过相同 RL 训练后反而可能表现更差。

这意味着评估 SFT 数据质量时，仅看 SFT 阶段的指标可能产生误导。好的 SFT 数据应该为下游 RL 保留足够的探索空间，而非过度拟合到一种”最优回复”。PEAR 通过重要性采样对损失进行重加权，使 SFT 检查点更适合后续 RL 优化。

RL-readiness 评估的实践方法：

快速 RL 探测（RL Probing）：

  1. 在候选 SFT 数据集上训练得到检查点
  2. 用少量 RL step（如 100-200 step）测试检查点的可优化性
  3. 对比不同数据集训练出的检查点在 RL 探测后的提升幅度
  4. 选择 RL 提升幅度最大的数据集——它保留了最多的优化空间

  关键指标：improvability = RL后得分 - SFT得分
  → improvability 越高，说明 SFT 数据为 RL 留出了越好的起点

2. 类别覆盖度评估

def evaluate_data_coverage(

认知变化：从”一次性标完”到”持续数据飞轮”

SFT 数据迭代的认知演变

  • 2022：“标注 10K 条，训练，发布”
  • 2023：“标注 → 训练 → 评估 → 补充标注 → 再训练”
  • 2024：“建立数据飞轮——模型的失败案例成为新的标注重点”
  • 2025-2026：“全自动化的数据飞轮——模型评估 → 自动识别弱点 → 自动合成补充数据 → 人工验证 → 训练”
  • 2026（新认知）：“不是所有场景都需要更多数据”——推理 SFT 中精品重复优于规模扩展，数据×基座×训练策略三位一体

数据飞轮的流程：

  模型部署
      ↓
  收集真实用户交互 + 失败案例
      ↓
  分析失败模式（按类型、领域、难度分类）
      ↓
  针对性地合成/标注补充数据
      ↓
  质量过滤 + 人工抽检
      ↓
  混入新数据 + 重新训练
      ↓
  评估 → 如有改善，部署新版本
      ↓
  循环 ↑

精品重复 vs 规模扩展：场景决定策略

数据飞轮隐含一个假设：更多的数据总是更好。最新实证（2026, Nuremberg/Mistral/NVIDIA）挑战了这一假设：在推理 SFT 场景下，对 400 条精品数据训练 128 个 epoch，在 AIME 和 GPQA 上显著优于对 51,200 条数据训练 1 个 epoch。泛化条件研究（2026.04）进一步确认，Long-CoT SFT 在充分训练（多 epoch）+ 高质量数据 + 强基座模型三条件下可以泛化。

这意味着数据飞轮的”补充数据”步骤需要根据场景做不同决策：

场景策略核心驱动力
通用指令跟随多样性驱动的规模扩展覆盖更多任务类型和领域
推理/CoT质量驱动的精品重复推理链结构完整性 > 数据量
领域专精深度驱动的垂直扩展领域知识的准确性和深度

更深层的认知转变是：数据飞轮不仅是补充数据的循环，而是数据×基座×训练策略的联合优化。同一批数据在弱模型上可能无效，在强模型上效果显著；同一个模型用不同训练策略（单 epoch vs 多 epoch）处理同一批数据，结果也可能天壤之别。

推理 SFT 数据的三个泛化条件

“SFT 只是记忆，RL 才是泛化”——这是 2025 年以来的流行叙事。最新研究（2026.04）对此提出了有力挑战：Long-CoT SFT 在满足以下三个条件时，同样可以实现泛化。

条件 1：充分训练。多 epoch 训练显著优于单 epoch。这与预训练中”数据只看一次”的最佳实践相反——在推理 SFT 中，模型需要反复”消化”推理模式才能内化为泛化能力。

条件 2：高质量推理数据。不是更多的推理数据，而是更好的推理数据。质量标准包括：验证通过率（数学题有正确答案、代码能通过测试）、推理链步骤完整性（参见 13.1 节的 MCSQ 框架）、以及解题路径的多样性。

条件 3：强基座模型。弱模型即使获得最好的推理数据也难以泛化——基座能力构成了推理 SFT 的”地板”。这解释了为什么同样的推理数据在 7B 和 70B 模型上效果差距巨大。

三条件的实践意义是：推理 SFT 数据质量的”及格线”比通用 SFT 高得多，但所需数据量可能远低于预期。投入应从”标注更多推理数据”转向”确保每条推理数据都满足三条件”。

推理 SFT 数据质量决策树：

  基座模型是否足够强？
    ├─ 否 → 先提升基座能力（预训练/中训练），数据再好也难泛化
    └─ 是 → 推理数据质量是否达标？（MCSQ 七维度）
               ├─ 否 → 优先提升数据质量，不要扩展规模
               └─ 是 → 训练策略是否充分？（多 epoch + 合适学习率）
                          ├─ 否 → 调整训练配方
                          └─ 是 → 三条件满足，预期可泛化

推理 SFT 数据决策树：实践指导

基于泛化三条件，推理 SFT 数据决策应遵循以下流程：

决策流程：

1. 评估基座模型强度
   ├─ 弱模型（< 7B）→ 优先提升基座能力
   ├─ 中等模型（7B-30B）→ 可进行推理 SFT，但预期泛化有限
   └─ 强模型（> 30B）→ 适合高质量推理 SFT

2. 数据质量评估
   ├─ 低质量（MCSQ 评分 < 0.7）→ 优先提升数据质量
   ├─ 中等质量（0.7-0.85）→ 可训练，但需多 epoch 补偿
   └─ 高质量（> 0.85）→ 适合标准训练配方

3. 训练策略选择
   ├─ 数据质量高 + 强基座 → 标准多 epoch 训练（64-128）
   ├─ 数据质量中 + 强基座 → 增加 epoch（128-256）
   └─ 数据质量低 → 不建议训练，优先改进数据

4. 规模扩展决策
   ├─ 数据质量高 → 精品重复优于规模扩展
   ├─ 数据质量中 → 可适度扩展规模（2-4倍）
   └─ 数据质量低 → 扩展规模无效，优先提升质量

关键实践指导：

  • 当基座模型强度和数据质量都高时，对 100-500 条精品数据训练 128 个 epoch 通常优于对 10,000 条普通数据训练 1 个 epoch
  • 数据质量不足时，盲目扩展规模反而会降低训练效率
  • 强基座模型是推理 SFT 成功的必要条件，但不是充分条件——高质量数据同样重要

"""
SFT 数据集质量审计工具
综合本章的多个质量检查维度
"""

import re
from collections import Counter
from typing import List, Dict

def audit_sft_dataset(samples: List[Dict]) -> dict:
    """对 SFT 数据集进行全面质量审计"""

    issues = Counter()
    scores = []
    length_distribution = []

    for sample in samples:
        inst = sample.get("instruction", "")
        resp = sample.get("response", "")
        sample_issues = []

        # --- 格式检查 ---
        # 代码块闭合
        if resp.count("```") % 2 != 0:
            sample_issues.append("代码块未闭合")
        # 长回复无结构
        if len(resp) > 500 and "##" not in resp and "\n-" not in resp and "\n1." not in resp:
            sample_issues.append("长回复缺结构")

        # --- 内容检查 ---
        # 过短
        if len(resp.strip()) < 30:
            sample_issues.append("回复过短")
        # 填充语
        if re.match(r'^(好的|当然|没问题)[，！]', resp):
            sample_issues.append("不必要的前缀")
        # 截断
        if resp and resp.rstrip()[-1:] not in "。.！!？?`）)】]\n":
            if not resp.rstrip().endswith("```"):
                sample_issues.append("可能被截断")

        # --- 重复检测 ---
        sentences = [s.strip() for s in resp.split("。") if len(s.strip()) > 10]
        if len(sentences) > 3:
            unique = len(set(sentences))
            if unique / len(sentences) < 0.7:
                sample_issues.append("内容重复度高")

        # 记录
        for issue in sample_issues:
            issues[issue] += 1

        score = max(0, 1.0 - len(sample_issues) * 0.2)
        scores.append(score)
        length_distribution.append(len(resp))

    # 生成报告
    n = len(samples)
    avg_score = sum(scores) / max(n, 1)
    pass_rate = sum(1 for s in scores if s >= 0.6) / max(n, 1)

    return {
        "总样本数": n,
        "平均质量分": round(avg_score, 3),
        "通过率": f"{pass_rate*100:.1f}%",
        "平均回复长度": f"{sum(length_distribution)/max(n,1):.0f} 字符",
        "长度范围": f"{min(length_distribution)}-{max(length_distribution)}",
        "问题分布": dict(issues.most_common(10)),
        "建议": (
            "质量良好，可以使用" if pass_rate > 0.85
            else "需要清洗，建议修复上述问题后重新检查" if pass_rate > 0.6
            else "质量较差，建议重新构造数据"
        ),
    }

"""
分析 SFT 数据集的多样性
用简单的词袋模型代替 embedding
"""

from collections import Counter

def analyze_diversity(instructions: List[str]) -> dict:
    """分析指令集的多样性"""

    # 词频分析
    all_words = []
    for inst in instructions:
        words = re.findall(r'[\u4e00-\u9fff]+|[a-zA-Z]+', inst)
        all_words.extend(words)

    word_freq = Counter(all_words)

    # 指令开头模式
    openings = [inst[:10] for inst in instructions]
    opening_freq = Counter(openings)
    repetitive_openings = sum(1 for _, c in opening_freq.items() if c > 1)

    # 长度多样性
    lengths = [len(inst) for inst in instructions]
    length_cv = (max(lengths) - min(lengths)) / (sum(lengths) / len(lengths))

    # 关键词覆盖度
    topic_keywords = {"代码", "数学", "写作", "翻译", "分析",
                      "解释", "设计", "优化", "对比", "总结"}
    covered = sum(1 for kw in topic_keywords
                  if any(kw in inst for inst in instructions))

    return {
        "指令数": len(instructions),
        "独特词数": len(word_freq),
        "重复开头数": repetitive_openings,
        "长度变异系数": round(length_cv, 2),
        "主题覆盖度": f"{covered}/{len(topic_keywords)}",
        "多样性评估": (
            "良好" if covered >= 7 and length_cv > 1.0
            else "需要改善" if covered >= 4
            else "严重不足"
        ),
    }

本章要点回顾

  1. 好的 SFT 样本四要素：指令清晰、回复准确、信息密度高、格式规范
  2. 推理 SFT 数据有额外的质量维度：推理链完整性、步骤粒度合理性、中间结论可验证性——结构胜于规模
  3. 自动化质量检查应覆盖格式、内容、一致性三个层面
  4. IFD 方法用模型自身的困惑度筛选最有价值的训练样本——PPL 衡量的是模型-数据兼容性，而非绝对难度
  5. 多样性需要数据和目标函数双管齐下：CE 损失天然鼓励模式坍缩，需要 SED-SFT 熵正则或 DFT 奖励修正配合
  6. 多轮对话的核心是真实的轮次间依赖；分支对话树是更高级的数据组织范式
  7. 评估 SFT 数据不能只看 SFT 得分：RL-readiness 评估揭示 SFT 最优 ≠ RL 后最优
  8. 数据飞轮需要区分场景：通用指令跟随走规模扩展，推理/CoT 走精品重复
  9. 推理 SFT 泛化的三条件：充分训练（多 epoch）+ 高质量推理数据 + 强基座模型
  10. 质量控制的视野应扩展到全流程：数据×基座×训练策略的联合优化，而非仅关注数据本身
