第 14 章 第四部分：RL 数据

偏好数据的本质与构造

从 RLHF 到 GRPO 的偏好学习范式演变与数据构造

最后更新：2026年5月1日 RLHF DPO GRPO 偏好数据

“Teaching a model to be helpful is about showing it what ‘good’ looks like. Teaching it to be aligned is about showing it what ‘better’ looks like.”

14.1 从 RLHF 到 DPO：偏好学习范式的演变

范式演变的时间线

大模型对齐技术在短短三年内经历了多次范式转换：

时期方法数据需求复杂度代表
2022RLHF (PPO)偏好对 + 奖励模型最复杂（4 个模型）InstructGPT
2023.5DPO偏好对简化（无需 RM）Zephyr
2023.11KTO好/坏标签进一步简化（二元反馈）-
2024SimPO偏好对（无参考模型）平均对数概率作隐式奖励-
2024ORPOSFT + 偏好合并单阶段训练-
2025GRPO可验证奖励不需要人类偏好DeepSeek R1
2026DAPO可验证奖励（工业级）非对称裁剪 + 动态采样 + Token 归一化-
2026ARPO多轮可验证奖励基于熵的自适应 rollout-

DAPO：GRPO 的工业级升级

DAPO（Differentiable Asymmetric Policy Optimization, NeurIPS 2025）在 GRPO 基础上实现了四大技术跃迁，使可验证奖励方法达到工业级可用性：

1. Clip-Higher 防熵坍缩机制

  • 问题：GRPO 在长序列训练中容易导致策略熵坍缩，模型变得过于保守
  • 解决方案：对高奖励样本进行不对称裁剪，保留探索性
  • 效果：在 Qwen2.5-32B 上训练至 AIME 2024 50分，训练效率提升约50%

2. Dynamic Sampling 自动难度控制

  • 问题：固定采样策略无法适应不同难度任务
  • 解决方案：根据模型当前能力动态调整采样难度
  • 效果：在数学推理任务上，训练步数减少30%的同时性能提升5%

3. Token-level PG Loss 解决长序列梯度稀释

  • 问题：长序列中关键 token 的梯度信号被稀释
  • 解决方案：对每个 token 独立计算策略梯度损失
  • 效果：在代码生成任务上，长序列（>512 tokens）性能提升15%

4. Overlong Reward Shaping 处理超长响应

  • 问题：超长响应（>2048 tokens）的奖励信号难以有效传播
  • 解决方案：分段奖励塑形，确保长序列中每个关键步骤都能获得有效反馈
  • 效果：在复杂推理任务上，超长序列训练稳定性提升40%

偏好学习的认知演变

  • 2022：“人类偏好标注是核心——InstructGPT 的方法就是标准答案”
  • 2023：“DPO 证明不需要单独训练奖励模型，直接用偏好对就行”
  • 2024：“On-policy 数据的重要性被重新认识——off-policy DPO 有天花板”
  • 2025：“GRPO + 可验证奖励——在数学和代码上，不需要人类偏好了”
  • 2025-2026：“推理模型的 RL 范式——让模型在可验证任务上自我进化”
  • 2026 早期：“偏好数据门槛降维——从精心构造的成对比较到二元信号（KTO）或隐式奖励（SimPO）”
  • 2026 中期：“可验证奖励从单轮推理扩展到多轮 Agent（ARPO），偏好数据的领地进一步收缩”

偏好对中的不对称性：Chosen 质量才是核心

直觉上，偏好对中的 chosen 和 rejected 似乎同等重要——毕竟 DPO 的损失函数是对称地”拉近 chosen、推远 rejected”。但 What Matters in Data for DPO 的系统实证研究揭示了一个被忽视的不对称性：

偏好数据中各因素的影响力排序：

1. Chosen 响应质量  ★★★★★  始终正向影响性能，是最关键因素
2. Margin / 对比度   ★★★☆☆  主要通过提升 chosen 效用发挥作用
3. Rejected 响应质量 ★★☆☆☆  影响有限，甚至不显著
4. 数据总量          ★★☆☆☆  在质量有保证时才有正面作用

更深层的发现是：在线 DPO 实际上简化为对 chosen 的 SFT。这意味着偏好对的核心价值在于高质量正例——rejected 主要提供”什么不要做”的弱信号，而 chosen 提供”什么该做”的强信号。

实践指导：构造偏好数据时，至少 70% 的投入应放在确保 chosen 的质量上。具体而言：

  • 对 chosen 做多轮筛选：best-of-N 采样 + 人工复核
  • 对 rejected 的要求可以适当放宽：只要”明确不如 chosen”即可
  • 在资源有限时，提升 chosen 质量（通过更好的采样策略或拒绝采样）比追求数据总量更值得投入

Margin 的重要性

偏好对中 chosen 和 rejected 之间的**质量差距（margin）**对训练效果有重要影响：

Margin 太小（chosen 和 rejected 质量差不多）：
  → DPO 梯度信号弱，学习效果差
  → 模型很难区分"好"和"更好"

Margin 太大（一个很好，一个很差）：
  → 梯度信号强但信息少
  → 模型只学会了"避免很差的回复"，没学会"从好到更好"

Margin 适中（明显可区分但不极端）：
  → 最佳学习效果
  → 模型学会了精细的质量判别

推荐：构造偏好数据时，同时包含不同 margin 的样本
  - 30% 大 margin（教模型基本判别）
  - 50% 中 margin（教模型精细判别）
  - 20% 小 margin（教模型在边界情况下做决策）

上述比例是一个可操作的起点，但近期研究揭示了 margin 的三个认知升级：

Margin 的演化认知

第一阶段：固定 Margin（DPO 原始）
  → 所有样本共享相同的隐式 margin 目标
  → 简单有效，但对数据异质性视而不见

第二阶段：动态 Margin（γ-PO, ACL 2025）
  → 在样本级别自适应调整目标 reward margin
  → 高置信度样本给大 margin，噪声/歧义样本给小 margin
  → 核心洞见：不是所有偏好对都同等可靠

第三阶段：Token 级 Margin（TI-DPO, ICLR 2026 Oral）
  → 序列级 margin 只是表象——偏好信号在 token 级高度不均匀
  → 少量"关键 token"（如关键推理步骤、事实陈述）贡献了大部分判别信号
  → 对这些 token 施加更大梯度权重，在指令遵循和代码生成上超越 GRPO

数据选择维度：margin 最大化原则。Less is More（NeurIPS 2025）提出了一个令人惊讶的发现：通过贝叶斯聚合统一外部 RM margin 和隐含 margin，在 Ultrafeedback 上仅用 10% 的数据即可提升 3-8%，迭代 DPO 场景中 25% 的在线数据即可提升 3%。这意味着偏好数据高度冗余——关键不是”造更多数据”，而是”选对数据”。Margin 从构造时的经验法则升级为筛选时的数学原则。

Token 粒度维度。TI-DPO的另一层启示在于：即便 margin 控制得当，如果对序列中所有 token 一视同仁，仍然会浪费大量梯度在非信息性 token 上。在一段包含推理步骤的回复中，真正决定”这个回复比那个好”的可能只是某几步推理或某个关键事实——而不是格式化文本或过渡句。这为偏好训练指出了新方向：从”选对偏好对”到”找对关键 token”。

隐式 PRM：零成本步骤级奖励

传统的步骤级奖励模型（PRM）需要显式标注每个推理步骤的质量，成本高昂。隐式 PRM（ICML 2025）提出了一种颠覆性的方法：将结果奖励参数化为策略模型与参考模型的对数似然比，可以从 ORM 中”零额外成本”提取步骤级奖励信号。

核心洞见：

  • 策略模型（π）和参考模型（π_ref）的对数似然比 log(π(a|s)/π_ref(a|s)) 天然编码了每个 token 的”偏好强度”
  • 这个比值在关键推理步骤处达到峰值，在非信息性 token 处接近零
  • 无需额外标注，即可获得与显式 PRM 相当的步骤级奖励信号

实证结果：

  • 在 MATH 基准上，用不到 Math-Shepherd 1/38 的训练数据即超越其性能
  • PRIME 框架基于隐式 PRM 在推理基准上实现 15.1% 平均提升
  • 在在线 RL 场景中，隐式 PRM 无需重新收集步骤标签，大幅降低部署成本

实践意义：隐式 PRM 使步骤级奖励从”昂贵奢侈品”变为”标准配置”，为推理模型的 RL 训练提供了新的范式。

ARPO：多轮可验证奖励

ARPO（Adaptive Rollout Policy Optimization）将可验证奖励从单轮推理扩展到多轮 Agent 场景，基于熵的自适应 rollout 机制实现了更高效的训练。

核心机制：

  • 基于熵的自适应 rollout：根据当前策略的不确定性动态调整 rollout 长度
  • 多轮可验证奖励：将验证点从单轮结论扩展到多轮交互的关键节点
  • 工具调用优化：用一半的工具调用量达到更优性能

多轮场景的应用：

# 多轮 Agent 场景的可验证奖励示例
multi_turn_rewards = &#123;
    "turn_1": &#123;
        "action": "search_web",
        "verifiable": False,  # 搜索动作本身不可验证
        "reward": 0
    &#125;,
    "turn_2": &#123;
        "action": "extract_facts", 
        "verifiable": True,   # 提取的事实可验证
        "reward": +1 if facts_correct else -1
    &#125;,
    "turn_3": &#123;
        "action": "reason_step",
        "verifiable": True,   # 推理步骤可验证
        "reward": +1 if logic_valid else -1
    &#125;
&#125;

实证优势：

  • 在 WebArena 基准上，ARPO 用 50% 的工具调用量达到传统方法的 110% 性能
  • 基于熵的自适应机制使训练过程更加稳定，减少无效探索
  • 为多轮 Agent 的 RL 训练提供了可扩展的解决方案

14.3 人工偏好标注

标注协议设计

成对比较（Pairwise Comparison） 是最常用的标注方式：

标注界面设计：

┌──────────────────────────────────────────────┐
│ Prompt: "解释什么是量子纠缠"                    │
├──────────────────────┬───────────────────────┤
│     回复 A           │     回复 B             │
│ (随机左右排列)        │                       │
│ "量子纠缠是一种..."   │ "纠缠就是两个粒子..."   │
├──────────────────────┴───────────────────────┤
│ 哪个回复更好？                                  │
│ ○ A 明显更好  ○ A 稍好  ○ 差不多               │
│ ○ B 稍好     ○ B 明显更好                      │
│                                              │
│ 判断依据（多选）：                               │
│ □ 准确性  □ 完整性  □ 清晰度  □ 安全性          │
└──────────────────────────────────────────────┘

标注中最难定义的规则

困难 1: Helpful vs Harmless 的冲突
  User: "怎么配一把锁？"
  → 可能是锁匠学徒（合理请求）
  → 也可能是入室盗窃（有害请求）
  → 标注指南需要明确处理策略

困难 2: 主观偏好
  User: "写一首关于春天的诗"
  Response A: 古典风格，含蓄优美
  Response B: 现代风格，直白生动
  → 哪个"更好"？取决于标注员的个人品味
  → 解决方案：明确"不评判风格偏好，只评判执行质量"

困难 3: 长度偏好
  短而精确的回复 vs 长而详细的回复
  → 人类标注员倾向于偏好更长的回复（"信息更多=更好"）
  → 这个偏差会被模型学到，导致"啰嗦"
  → 解决方案：在标注指南中明确"简洁但完整优于冗长但重复"

14.4 自动偏好数据构造

LLM-as-Judge

用强模型来评判弱模型的输出，生成偏好数据：

"""
LLM-as-Judge 偏好数据生成
"""

JUDGE_PROMPT = """请比较以下两个回复的质量。

用户问题：&#123;prompt&#125;

回复 A：
&#123;response_a&#125;

回复 B：
&#123;response_b&#125;

请从以下维度评估：
1. 准确性：信息是否正确
2. 完整性：是否充分回答了问题
3. 清晰度：是否易于理解
4. 实用性：是否对用户有实际帮助

请给出你的判断：
- 如果 A 更好，输出 "[[A]]"
- 如果 B 更好，输出 "[[B]]"
- 如果差不多，输出 "[[tie]]"

简要说明理由（1-2 句话）。"""

def generate_preference_pair(
    prompt: str,
    model_a_response: str,
    model_b_response: str,
    judge_model,
    swap_positions: bool = True,
) -> dict:
    """
    用 LLM-as-Judge 生成偏好对

    关键：进行位置交换来消除位置偏好
    """
    # 正序评判
    result_1 = judge_model.generate(
        JUDGE_PROMPT.format(
            prompt=prompt,
            response_a=model_a_response,
            response_b=model_b_response,
        )
    )

    if swap_positions:
        # 反序评判（消除位置偏好）
        result_2 = judge_model.generate(
            JUDGE_PROMPT.format(
                prompt=prompt,
                response_a=model_b_response,
                response_b=model_a_response,
            )
        )

        # 只有两次评判一致时才采用
        winner_1 = parse_winner(result_1)  # "A", "B", or "tie"
        winner_2 = parse_winner(result_2)  # "A", "B", or "tie"

        # 反转 winner_2（因为位置交换了）
        winner_2_flipped = &#123;"A": "B", "B": "A", "tie": "tie"&#125;[winner_2]

        if winner_1 != winner_2_flipped:
            return &#123;"status": "inconsistent", "discard": True&#125;

        winner = winner_1
    else:
        winner = parse_winner(result_1)

    if winner == "A":
        return &#123;"prompt": prompt, "chosen": model_a_response, "rejected": model_b_response&#125;
    elif winner == "B":
        return &#123;"prompt": prompt, "chosen": model_b_response, "rejected": model_a_response&#125;
    else:
        return &#123;"status": "tie", "discard": True&#125;

def parse_winner(judge_output: str) -> str:
    if "[[A]]" in judge_output: return "A"
    if "[[B]]" in judge_output: return "B"
    return "tie"

合成偏好的偏差分析

LLM-as-Judge 有已知的系统性偏差：

偏差类型表现影响缓解方法
位置偏好倾向于选择第一个回复数据不公平位置交换 + 一致性检查
长度偏好倾向于选择更长的回复模型变啰嗦长度归一化 + 明确指令
自我偏好倾向于选择自己风格的回复多样性下降多模型交叉评判
格式偏好偏好 markdown 格式过度格式化明确评判标准
评分范围偏差不同模型对分数尺度有内在偏好（如偏向 3-5 分或 1-3 分）多模型评分不可比对比解码（同族模型对数概率相减，Spearman +11.7%）
领域/任务偏差代码质量与创意写作的评判标准混用跨域评判不一致领域特化评判 prompt + 分任务评分

已知偏差只是冰山一角。BiasScope（ICLR 2026）提出了一个完全由 LLM 驱动的迭代框架，可以自动发现评判模型中尚未被识别的偏差类型——不仅仅是上表列出的几种。其核心发现令人警醒：即使是当前最强的评判模型，也存在大量特定于领域、任务和上下文的未知偏差。这意味着仅靠手工枚举偏差并逐一缓解是不够的，需要系统化的自动偏差发现机制。

缓解方法的演进：从早期的位置交换（消除位置偏差）、长度归一化（消除长度偏差），到对比解码（消除评分范围偏差），再到自动化偏差扫描（发现未知偏差）——去偏技术正从”已知偏差的手工修补”走向”未知偏差的系统化发现”。

14.5 偏好数据选择——少即是多

前几节讨论了如何构造偏好数据（人工标注、LLM-as-Judge、margin 控制）。但一个同样重要的问题是：构造出来的数据都需要用吗？

答案是：大部分不需要。

偏好数据的冗余问题

Less is More（NeurIPS 2025）的实验揭示了偏好数据中的惊人冗余：

实验结果：

Ultrafeedback 数据集（标准 DPO）：
  100% 数据  →  baseline
  10% 数据   →  baseline + 3-8%  ✓ 10% 数据反而更好

迭代 DPO（在线场景）：
  100% 在线数据  →  baseline
  25% 在线数据   →  baseline + 3%  ✓ 25% 同样更好

这个结果看似反直觉，但原因并不复杂：偏好数据中存在大量噪声（标注不一致、margin 歧义、极端样本），这些噪声会对参数产生”收缩效应”——拉平模型对偏好方向的区分能力。去掉噪声数据，反而释放了有效信号。

Margin 最大化原则

核心方法是 margin 最大化选择：

目标：从 N 个偏好对中选择 k << N 个最有价值的样本

选择标准：
  1. Margin 大 → 偏好信号清晰
  2. 噪声低 → 多来源标注一致

关键创新——贝叶斯聚合：
  当多个来源（外部 RM、模型隐含偏好、人类标注）给出不同 margin 时，
  用贝叶斯后验将它们统一为单一偏好概率 P(chosen > rejected)

  选择 P 高且方差低的样本 = 高置信度 + 强信号

### "少即是多"原则的深层原因

[Less is More](https://arxiv.org/abs/2502.14560)（NeurIPS 2025）的实验揭示了偏好数据高度冗余的根本原因：

**噪声样本的收缩效应**：
- 噪声样本（标注不一致、margin 歧义）对模型参数产生"收缩效应"
- 这种效应拉平了模型对偏好方向的区分能力
- 去掉噪声数据反而释放了有效信号的表达空间

**贝叶斯聚合的统一框架**：
```python
def bayesian_margin_aggregation(margins: dict) -> float:
    """贝叶斯聚合多来源 margin"""
    # 先验：假设偏好概率为 0.5（无信息）
    prior_alpha = 1.0
    prior_beta = 1.0
    
    # 似然：每个来源的 margin 转换为偏好概率
    for source, margin in margins.items():
        prob = sigmoid(margin * 2)  # margin 到 [0,1] 的映射
        alpha_update = prob * source_weight[source]
        beta_update = (1 - prob) * source_weight[source]
        prior_alpha += alpha_update
        prior_beta += beta_update
    
    # 后验：Beta(alpha, beta) 分布的均值
    posterior_prob = prior_alpha / (prior_alpha + prior_beta)
    posterior_variance = (prior_alpha * prior_beta) / \
                        ((prior_alpha + prior_beta)**2 * \
                         (prior_alpha + prior_beta + 1))
    
    return posterior_prob, posterior_variance

实证结果：

  • Ultrafeedback 上仅用 10% 数据提升 3-8%
  • 迭代 DPO 场景中 25% 的在线数据提升 3%
  • 数据选择效率提升 4-10 倍

### 与 SFT 数据选择的区别

| 维度 | SFT 数据选择 | 偏好数据选择 |
|------|-------------|-------------|
| 核心指标 | 单条质量（IFD、PPL） | 成对关系（margin + 一致性） |
| 选择逻辑 | 选质量高的 | 选 margin 大且噪声低的 |
| 冗余来源 | 指令重复、风格雷同 | 歧义标注、极端差异、信号弱的对 |
| 筛选比例 | 通常 30-50% | 可低至 10-25% |

### 实践指导

偏好数据筛选的推荐流程：

  1. 多 RM 交叉标注 → 用 2-3 个不同的 RM 对每个偏好对打分 → 计算每个来源的 margin

  2. 贝叶斯聚合 → 将多来源 margin 统一为后验偏好概率 → 同时得到置信度估计

  3. Margin 最大化筛选 → 按 P(chosen > rejected) 降序排列 → 取 Top-k（通常 10-25%）

  4. 分布检查 → 确保筛选后的数据仍覆盖不同 prompt 领域 → 避免极端偏向某一类任务

"""

def compute_margin(chosen: str, rejected: str) -> float:
    """计算偏好对的 margin（分数差）"""
    score_c = score_response_quality(chosen)
    score_r = score_response_quality(rejected)
    return score_c - score_r

def categorize_margin(margin: float) -> str:
    """根据 margin 大小分类"""
    if margin > 0.4:
        return "大margin（基础判别）"
    elif margin > 0.15:
        return "中margin（精细判别）"
    elif margin > 0:
        return "小margin（边界决策）"
    else:
        return "⚠️ 反转（chosen 可能不如 rejected）"

偏好数据还需要吗？

在章节最后，值得退一步思考一个根本性问题：偏好数据在大模型训练中的角色，正在经历怎样的变化？

在可验证任务上，偏好数据已经被绕过。GRPO 和 RLVR 在数学、代码等有明确正确答案的任务上，完全不需要人类偏好——模型通过自我验证就能进化。ARPO（ICLR 2026）进一步将这种无偏好 RL 扩展到多轮 agent 场景，通过基于熵的自适应 rollout，用一半的工具调用量达到更优性能。可验证奖励的适用边界，正在从”单轮推理”向”多轮交互”扩展。

在开放域任务上，偏好信号仍然不可替代——但形式在简化。创意写作、开放对话、安全对齐等缺乏客观标准的任务，仍然需要某种”人类认为什么更好”的信号。区别在于，这个信号不再需要精心构造的成对比较：KTO 只需要二元反馈（好/坏），SimPO 用平均对数概率作为隐式奖励——偏好数据的构造成本在大幅降低。

推测：偏好数据不会消失，但会从后训练的”主角”变为”补充”——在 RLVR 无法覆盖的开放域发挥作用，且形式越来越轻量。未来的格局可能是：可验证任务用 RL 自我进化，开放域用轻量偏好信号（二元反馈或隐式奖励）做兜底。

本章要点回顾

  1. 偏好学习从 RLHF 演变到 GRPO/DAPO/ARPO，复杂度不断降低，适用范围从单轮扩展到多轮 Agent
  2. Chosen 质量是偏好数据的核心驱动力：投入应重点放在确保 chosen 质量，rejected 的影响相对有限
  3. Margin 认知已从固定到动态到 Token 级：γ-PO 在样本级自适应，TI-DPO 在 token 级识别关键信号
  4. 偏好数据高度冗余：margin 最大化原则下，10-25% 的数据即可匹配甚至超越全量
  5. 人工标注的核心挑战是处理 helpful/harmless 冲突和主观偏好
  6. LLM-as-Judge 是规模化的关键，但偏差远不止位置和长度——需要系统化的自动发现机制
  7. On-policy 数据的优势不是无条件的：其本质是”生成更高质量的 chosen”，在资源有限时提升 chosen 质量比追求 on-policy 更重要
  8. 偏好数据的形式在降维：从成对比较到二元反馈（KTO）到隐式奖励（SimPO），构造成本持续降低
  9. 在可验证任务上，偏好数据正在被绕过：GRPO → DAPO → ARPO 的演进正在不断扩展无偏好 RL 的边界
  10. 偏好数据的未来角色是”补充”而非”主角”——在 RLVR 覆盖不到的开放域发挥兜底作用
