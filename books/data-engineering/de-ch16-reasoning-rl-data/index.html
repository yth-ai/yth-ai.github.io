第 16 章 第四部分：RL 数据

推理模型的 RL 数据——新范式

DeepSeek-R1 启示、可验证奖励与推理数据构造

最后更新：2026年5月4日 推理模型 R1 可验证奖励 GRPO

“The most exciting phrase to hear in science, the one that heralds new discoveries, is not ‘Eureka!’ but ‘That’s funny…’” ——Isaac Asimov

DeepSeek-R1 的”aha moment”——模型自发地学会了反思和自我纠正——正是这样一个”That’s funny”时刻。

16.1 DeepSeek-R1 的启示

纯 RL 训练出推理能力的里程碑

2025 年 1 月，DeepSeek 发布了 R1 系列模型，其中 DeepSeek-R1-Zero 是一个里程碑：它仅通过强化学习（无 SFT warm-start）就获得了强大的推理能力。

DeepSeek-R1-Zero 的训练过程：

  基座模型（DeepSeek-V3-Base）
          ↓
  直接进入 RL 训练
  ├── 算法：GRPO（Group Relative Policy Optimization）
  ├── 数据：数学+代码推理题（含可验证答案）
  ├── 奖励：基于答案正确性的规则奖励
  └── 无 SFT 数据、无人类偏好数据

  训练过程中出现的"涌现"行为：
  1. 模型自发地学会了"chain of thought"推理
  2. 出现了自我反思：生成 "wait, let me reconsider..."
  3. 出现了"aha moment"：模型发现自己之前的错误并修正

GRPO 算法对数据的要求

GRPO 的核心是不需要独立的奖励模型，而是用”组内相对排名”来提供学习信号：

"""
GRPO 的数据流程（简化）
"""

def grpo_data_pipeline(
    prompt: str,          # 推理问题
    model,                # 当前策略模型
    reward_fn,            # 奖励函数（如答案正确性）
    group_size: int = 16, # 每组采样数量
) -> dict:
    """
    GRPO 的一次数据生成循环
    """
    # Step 1: 对同一个 prompt，采样 group_size 个回复
    responses = []
    for _ in range(group_size):
        response = model.generate(prompt, temperature=1.0)
        responses.append(response)

    # Step 2: 对每个回复计算奖励
    rewards = [reward_fn(prompt, r) for r in responses]

    # Step 3: 计算组内相对优势
    mean_reward = sum(rewards) / len(rewards)
    std_reward = max(np.std(rewards), 1e-8)
    advantages = [(r - mean_reward) / std_reward for r in rewards]

    # Step 4: 高于平均的回复被"鼓励"，低于平均的被"抑制"
    training_pairs = []
    for response, advantage in zip(responses, advantages):
        training_pairs.append({
            "prompt": prompt,
            "response": response,
            "advantage": advantage,
        })

    return {
        "prompt": prompt,
        "pairs": training_pairs,
        "group_stats": {
            "mean_reward": mean_reward,
            "max_reward": max(rewards),
            "min_reward": min(rewards),
            "solve_rate": sum(1 for r in rewards if r > 0) / group_size,
        },
    }

从 GRPO 到后续改进：四个工程方向

GRPO 奠定了“无需独立 Critic 模型”的基础，但在实际大规模训练中仍会暴露若干工程瓶颈——熵坍缩（策略过早收敛到少数模式）、长序列梯度稀释、以及对无效 batch 的计算浪费。2025 年之后，多篇公开论文和工程报告都在围绕这些问题做改进，可以归纳为四个方向：

技术解决的问题核心思路数据层面的意义
Clip-Higher熵坍缩对概率提升方向设置更高的裁剪上界，保持策略多样性允许模型在更多元的推理路径上探索，间接扩大了有效数据空间
Dynamic Sampling无信息量 batch自动过滤全对/全错的 prompt（solve rate 为 0% 或 100%），只保留有梯度信号的 batch隐式实现了 16.3 节所描述的”最佳 solve rate”课程调度——无需人工设定难度梯度
Token-level PG Loss长 CoT 梯度稀释将策略梯度从回复级改为 token 级，每个 token 都获得梯度信号长推理链（数百步）的每一步都参与学习，数据利用率大幅提升
Overlong Reward Shaping超长响应对超出长度限制的响应施加软惩罚（而非截断），引导模型学会在有限长度内完成推理减少因截断产生的无效样本，提高数据效率

这里最值得数据工程师关注的是 Dynamic Sampling——它让难度控制从一个需要预先设计的静态参数，变成训练过程中的自适应行为。后续 16.3 节会回到这个话题。

16.2 可验证奖励与 RLVR 范式

什么是可验证奖励

可验证奖励是指可以通过规则或程序自动判断正确性的奖励信号，不需要人类标注或奖励模型。

任务类型验证方法奖励信号可靠性
数学题比较最终答案答案正确 +1，错误 -1高
代码题运行测试用例全部通过 +1，部分 +0.5极高
逻辑推理形式化验证器证明有效 +1极高
翻译BLEU/参考答案连续分数中
问答字符串匹配匹配 +1中

关键认知：可验证奖励消除了人类偏好标注的瓶颈。在数学和代码领域，你可以生成几乎无限量的训练信号——只需要有足够的题目和正确答案。

RLVR：从”奖励类型”到”独立范式”

可验证奖励最初只是奖励信号的一种选择，但到 2025-2026 年，围绕它构建的训练范式——RLVR（Reinforcement Learning with Verifiable Rewards）——已经成为推理能力训练里最重要的技术路线之一。DeepSeek-R1 之后，数学、代码和形式化任务上的“可验证奖励”成为各家公开讨论的重点。

维度RLHFRLVR
奖励来源人类标注 → 奖励模型规则/程序自动验证
标注成本高（每轮迭代都需要新标注）零（验证器是确定性的）
信号质量受标注者噪声和偏差影响无噪声（对/错是确定的）
可扩展性受限于标注产能受限于算力
适用范围所有任务（含主观任务）有明确正确答案的任务
数据瓶颈偏好标注高质量题库
典型代表InstructGPT, ClaudeDeepSeek-R1, DAPO

RLVR 的端到端数据流程：题库 → 在线采样 → 规则验证 → 梯度更新。整个循环中没有任何人类参与，数据瓶颈从”标注产能”完全转化为”算力和题库规模”。

适用范围与局限

可验证奖励适用的场景：
  ✅ 数学推理（有标准答案）
  ✅ 代码生成（有测试用例）
  ✅ 逻辑推理（可形式化验证）
  ✅ 棋类/游戏（有明确胜负）

正在拓展的场景：
  🔄 通用推理（多任务可验证奖励设计仍在探索）
  🔄 多步 Agent 任务（环境反馈作为验证信号）

仍不适用的场景：
  ❌ 创意写作（什么是"好"的创意？）
  ❌ 对话质量（"有帮助"很难量化）
  ❌ 安全对齐（"无害"的边界模糊）

→ 实际的 RL 训练通常混合使用：
   - 可验证任务用 RLVR
   - 主观任务用 RLHF
   DeepSeek-R1 的完整训练正是这一混合策略

验证器本身的局限

RLVR 假设验证器是完美的，但现实中并非如此：

  • 数学边界情况：等价但形式不同的答案（如 2/2\sqrt{2}/22​/2 与 1/21/\sqrt{2}1/2​）可能被误判
  • 代码测试覆盖不完整：测试用例无法覆盖所有 edge case，模型可能学会”通过测试但逻辑有缺陷”的捷径
  • 回吐验证器漏洞：模型可能发现验证器的假阳性模式，产生 reward hacking

针对噪声验证器的去偏校正已成为一个活跃的研究方向。核心思路是：在 RL 训练中显式建模验证器的错误率，并在梯度更新中做相应补偿——类似于带噪声标签学习中的 loss correction 技术。

16.3 推理数据的构造

数学推理数据的难度梯度

难度级别设计：

Level 1: 基础运算和简单应用
  - GSM8K 级别（小学数学）
  - 只需 2-4 步推理
  - 用途：warm-up，建立基础推理习惯

Level 2: 中等数学
  - MATH 数据集中的中低难度
  - 需要 4-8 步推理
  - 涉及代数、几何基础
  - 用途：主体训练数据

Level 3: 竞赛预备
  - AMC/AIME 级别
  - 需要 8-15 步推理
  - 需要组合不同数学工具
  - 用途：提升上限

Level 4: 竞赛级
  - IMO/Putnam 级别
  - 需要深度证明和创造性思考
  - 用途：少量使用，推动极限

推荐配比：
  Level 1: 20%  |  Level 2: 45%  |  Level 3: 30%  |  Level 4: 5%

代码推理数据的粒度

函数级 → 文件级 → 项目级

函数级：
  "实现一个函数，判断一棵二叉树是否是 BST"
  验证：10-20 个测试用例
  适用：基础代码能力

文件级：
  "实现一个 LRU 缓存类，支持 get/put/delete，线程安全"
  验证：完整的单元测试套件
  适用：模块设计能力

项目级：
  "实现一个简单的 key-value 数据库，支持持久化和并发访问"
  验证：集成测试 + 性能测试
  适用：系统设计能力（当前最难）

难度控制

RL 训练对难度非常敏感：

太简单（模型 solve rate > 90%）：
  → 梯度信号太弱，学不到东西
  → 浪费计算资源

太难（模型 solve rate < 5%）：
  → 几乎所有采样都是失败的
  → 无法产生有效的正向信号
  → 可能导致训练不稳定

最佳难度（solve rate 20-50%）：
  → 足够的正向信号来学习
  → 足够的挑战来提升能力
  → 组内差异大，GRPO 梯度信号强

启动方案——静态配比：如果你刚开始搭建 RL 训练管线，Level 1-4 按 20/45/30/5% 配比是合理的起点。它足够简单，且在多数场景下表现不差。

进阶方案——自适应机制：静态配比的问题在于，随着模型能力提升，固定难度分布会逐渐偏离”最佳学习区”。DAPO 的 Dynamic Sampling 提供了一个优雅的解决方案：直接过滤掉 solve rate 为 0% 或 100% 的 prompt，让训练自动聚焦在有信息量的数据上。 实验表明，这种自适应机制比任何人工设定的固定配比都更高效——DAPO 的训练步数比 R1-Zero 减少约 50%。

2026 年的 ICLR 研究进一步量化了数据效率：通过课程策略和数据选择的组合，RLVR 训练可以在使用显著更少的题目的情况下达到相当甚至更好的性能。这意味着 RL 数据的瓶颈不在量，而在”信息密度”——每道题在当前模型能力边界上能提供多少梯度信号。

16.4 RL 数据的在线生成与迭代

On-policy vs Off-policy

Off-policy 数据：
  用旧版本的模型（或其他模型）生成的数据
  优点：可以提前准备，不阻塞训练
  缺点：与当前模型的分布不匹配，效果有折扣

On-policy 数据：
  用当前版本的模型实时生成的数据
  优点：分布完全匹配，学习效率最高
  缺点：需要实时推理，增加训练复杂度

当前的共识：On-policy 数据显著优于 off-policy，特别是在 RL 训练的后期。DeepSeek-R1 采用的就是完全 on-policy 的方式。

数据新鲜度的重要性

"""
RL 训练中的数据新鲜度管理
"""

def rl_data_freshness_check(
    model_version: int,
    data_generated_at_version: int,
    max_stale_versions: int = 3,
) -> dict:
    """
    检查 RL 数据的新鲜度
    """
    staleness = model_version - data_generated_at_version

    status = (
        "fresh" if staleness == 0 else
        "acceptable" if staleness <= max_stale_versions else
        "stale"
    )

    return {
        "model_version": model_version,
        "data_version": data_generated_at_version,
        "staleness": staleness,
        "status": status,
        "recommendation": {
            "fresh": "直接使用",
            "acceptable": "可以使用但建议混入新数据",
            "stale": "丢弃，重新生成",
        }[status],
    }

动态难度调整

一个高级的 RL 训练系统应该能够动态调整题目难度：

自适应难度调整：

  每 N 步 → 评估当前模型在各难度级别的 solve rate
      ↓
  Level 1 solve rate > 90%? → 降低 Level 1 比例
  Level 3 solve rate < 10%? → 降低 Level 3 比例
      ↓
  重新平衡难度分布，确保"最佳学习区"
      ↓
  继续训练

16.5 自验证与自博弈：走向完全自主的训练循环

RLVR 解决了”谁来评判答案”，但仍然依赖人类设计的题库。2025-2026 年的两个前沿方向正在消除最后的人类依赖。

自验证（RISE）

RLVR 训练的模型会”做对题”，但不一定会”检查自己的推理”——即使最终答案正确，中间步骤可能包含逻辑跳跃或隐含错误。RISE 在同一个 RL 过程中同时训练问题求解和自我验证能力：模型不仅要生成解答，还要学会在推理过程中捕捉自身的错误。

这对数据的要求是：训练题目需要同时提供答案级验证（最终结果对不对）和过程级验证（中间步骤有没有逻辑缺陷）的信号。数学和代码领域比较容易构造这种双层验证——数学可以检查每一步推导的合理性，代码可以在中间状态插入断言。

合成自博弈（SPICE）

SPICE 框架引入了一个”Challenger-Reasoner”双角色结构：

  • Challenger 从已有文档中挖掘生成推理任务（而非人工设计题目）
  • Reasoner 求解这些任务
  • 文档锚定防止幻觉放大——每道题都可以追溯到源文档

实验结果：数学推理 +8.9%，通用推理 +9.8%。

SPICE + RLVR 构成了一个完全无人参与的自适应训练循环：模型自己出题 → 自己解题 → 规则验证 → 梯度更新。16.4 节描述的 on-policy 数据生成是”用当前模型生成回答”，SPICE 更进一步，连题目本身也由模型生成。这是否会导致训练分布的退化（类似 Model Collapse）？目前的证据表明，文档锚定和可验证奖励的双重约束提供了足够的接地力，但长期效果仍需观察。

16.6 认知演变

推理模型 RL 数据的认知演变——五代去依赖

时间认知去掉了什么
2023”推理能力需要 CoT 数据 + SFT”—
2024.01”o1 证明 RL 可以产生强大的推理——但方法未公开”—
2025.01”DeepSeek-R1 开源证明纯 RL 也能训练出推理。关键：可验证奖励 + GRPO”去掉 SFT warm-start
2025.03”DAPO 四大技术超越 GRPO。Dynamic Sampling 让难度控制全自动化”去掉独立 Critic 模型 + 人工难度设定
2025-2026”RLVR 成为独立范式，与 RLHF 并列。数据瓶颈从标注转为算力”去掉人类偏好标注
2026”SPICE 自博弈 + RISE 自验证 → 完全自主训练循环。模型自己出题、解题、验证”去掉人工设计的静态题库

五代演进的主线清晰：每一代去掉一个外部依赖，从 SFT 数据、Critic 模型、人类标注、静态数据集，到最后连题目设计都交给模型自己。

本章要点回顾

  1. DeepSeek-R1 证明了纯 RL 可以训练出推理能力——这是范式级的转变
  2. GRPO 消除了单独奖励模型的需求，用组内相对排名提供信号
  3. **DAPO 四大技术（Clip-Higher / Dynamic Sampling / Token-level PG / Overlong Shaping）**全面超越 GRPO，训练效率提升约 50%
  4. RLVR 已从”奖励类型”升级为”独立范式”——与 RLHF 并列，适用于所有有明确正确答案的任务
  5. 可验证奖励是关键突破——在数学/代码上可以生成无限训练信号，瓶颈从标注转为算力
  6. 验证器并非完美——边界情况误判、测试覆盖不完整、reward hacking 都需要去偏校正
  7. 难度控制至关重要：solve rate 20-50% 是最佳学习区，Dynamic Sampling 让这一控制自动化
  8. On-policy 数据远优于 off-policy——RL 训练的数据必须”新鲜”
  9. RISE 自验证 + SPICE 自博弈正在构建完全无人参与的自适应训练循环
  10. 五代演进的主线：每一代去掉一个外部依赖——SFT → Critic → 人类标注 → 静态题库
