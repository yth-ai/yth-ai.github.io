为什么数据是大模型的灵魂

从 0.1% 脏数据的故事讲起——数据质量如何决定大模型的成败

最后更新：2026年5月4日 数据质量 预训练 认知演进

“We don’t have better algorithms. We just have more data.” ——Peter Norvig

这句话在大模型时代需要修正：我们不仅需要更多数据，更需要正确的数据。

1.1 一个 0.1% 的事故模式

先从一个大模型训练里很典型的事故模式讲起。

一个训练团队完成了一个大模型的预训练，英文 benchmark 表现正常，但中文问答里开始频繁夹杂无关的日语片段。模型结构、训练超参、显卡状态都查过，没有明显异常。

问题最后被追到数据侧：一批 Common Crawl 网页里混入了少量中日混杂的低质量页面。这类页面常见于自动翻译站、采集站和模板化垃圾站，HTML 的 lang 属性可能标成中文，但正文实际是机器翻译日文、中文片段、导航栏和广告文本的混合物。语言识别器如果只给一个整体置信度，很容易把它们放过过滤阈值。

比例可以很小，甚至只有千分之一量级。但在万亿 token 级训练集里，千分之一就已经是十亿到百亿 token。这个量级足以让模型在语言边界、词表使用和上下文模式上学到错误关联。

修复通常不是“删掉几条脏数据”这么简单。你需要回溯数据版本，定位污染来源，调整语言识别阈值，增加行级或段落级二次检测，再重新生成数据切片。最坏的情况下，已经消耗掉的训练计算也要部分作废。

这个例子是一个复合案例，不指向某一家具体团队。它想说明的是一个更普遍的规律：在大模型训练里，小比例、系统性、分布集中的脏数据，比随机噪声更危险。

“Garbage in, garbage out” 在大模型时代有了新的含义：你不需要太多垃圾就能毁掉一次训练。

数据质量 vs 模型规模：一个正在被重塑的认知

长期以来，大模型领域存在一种朴素的信念：模型够大，就能从噪声数据中学到有用的信号。这种信念有其来源——GPT-3 的论文（Brown et al., 2020）展示了 175B 参数的模型在仅做了最小化清洗的数据上就取得了惊人的效果。这给很多人留下了”数据粗糙一点没关系，模型够大就行”的印象。

但这个认知正在被系统性地推翻。

2024 年的多项研究表明，在相同的计算预算下，用更少但更干净的数据训练的模型，一致性地优于用更多但更脏的数据训练的模型。HuggingFace 的 FineWeb 项目是一个标志性的案例：他们通过精心设计的数据管线（从 Common Crawl 出发，经过文本提取、语言过滤、启发式清洗、模糊去重、质量过滤等多个步骤），产出了 15 万亿 token 的高质量英文数据集。用这个数据集训练的模型，在多个 benchmark 上超过了之前用更大但清洗不够彻底的数据集训练的模型。

更有趣的是 2025 年的一篇论文《When Bad Data Leads to Good Models》（arXiv: 2505.04741）从另一个角度审视了”质量”的概念。研究者发现，某些传统认为”低质量”的数据（比如论坛上的口语化讨论、社交媒体上的非正式文本），在特定的下游任务上反而是有价值的。问题不在于数据的”绝对质量”，而在于数据的”适配性”——什么样的数据适合你要训练的模型所面向的目标。

这引出了本书的一个核心观点：

数据质量不是一个固定的属性，而是一个依赖于目标的函数。“好数据”是在给定目标下能最大化模型效用的数据。

1.2 大模型训练的四个数据阶段

一个现代大语言模型的训练，从数据视角来看，可以分为四个泾渭分明的阶段。每个阶段对数据的需求截然不同——不仅数量级不同，质量要求、构造方式、评估标准都不同。

四阶段总览

阶段目标数据量级数据形式核心矛盾
预训练通识教育：学习语言、知识、推理1T~15T+ tokens原始文本（网页、书籍、论文、代码）量够大 vs 质够好
中训练能力强化：长上下文、代码、数学100B~1T tokens领域文本 + 合成数据学新知识 vs 不忘旧知识
SFT行为对齐：学会”怎么说话、怎么做事”10K~10M 条指令-回复对质量 vs 数量 vs 多样性
RL偏好对齐：学会”什么更好”10K~500K 条偏好对 / 奖励信号人类偏好 vs 可验证奖励

让我们用具体的数字感受一下这个量级差异。

预训练阶段：Meta 的 Llama 3（2024）使用了 15.6 万亿 token 进行预训练，相比 Llama 2 的 2 万亿 token 增长了近 8 倍。DeepSeek V3（2024）同样使用了 14.8 万亿 token。这些数据主要来自互联网爬取，经过多步清洗后使用。如果把这些 token 转换成纯文本文件，大约需要 40-60TB 的存储空间。

中训练阶段：数据量降到了百亿到千亿 token。比如 Llama 3.1 在预训练之后，额外用了大量的长文本数据来扩展上下文窗口能力，并用高质量代码和数学数据来增强推理能力。中训练的数据量通常是预训练的 1%-10%，但每条数据的”知识密度”要求高得多。

SFT 阶段：LIMA（2023）用 1000 条精选数据就达到了不错的效果，表明 SFT 阶段的数据量需求远小于预训练。但工业界的实践通常使用几十万到几百万条，以覆盖足够多的任务类型。关键是：SFT 阶段，一条质量差的数据对模型的伤害，可能比预训练阶段的一万条脏数据还大。

RL 阶段：偏好数据的规模更小——几万到几十万条。但这些数据的标注成本最高，因为需要人类标注员对模型的多个输出做比较判断。DeepSeek R1 的一个里程碑式发现是，对于数学和编程这类有明确正确答案的任务，可以用”可验证奖励”（verifiable rewards）替代人类偏好标注，大大降低了数据构造的成本和主观性。

一个关键认知：数据量递减，价值递增

从预训练到 RL，数据量以数量级递减（15T → 500B → 1M → 100K），但单条数据对模型行为的影响力却在递增。一条预训练数据对模型的影响微乎其微，但一条 SFT 数据可以改变模型的回复风格，一条 RL 偏好数据可以改变模型的价值取向。

这意味着，越到后面的阶段，数据工程越需要精细化管理。预训练阶段的数据管理是”大数据”问题——核心挑战是如何从海量数据中高效地筛选和清洗。而 SFT 和 RL 阶段的数据管理更像是”小数据”问题——核心挑战是如何精心设计每一条数据的质量和构成。

四个阶段之间的数据关系

四个阶段并不是孤立的。它们之间存在深刻的相互影响：

  • 预训练数据的质量决定了 SFT 的上限。如果预训练数据中缺乏某类知识（比如最新的科学发现），SFT 阶段无论怎么精心构造指令，模型都不可能在这个领域表现好。SFT 能做的是”激活”和”塑形”，不能”凭空创造”。

  • 中训练可以弥补预训练的不足。如果你的预训练数据中代码比例不够，可以在中训练阶段用高质量代码数据来补强。但中训练有一个核心风险——灾难性遗忘。注入新知识的同时可能丢失旧知识。

  • RL 数据的分布要和 SFT 后的模型匹配。如果 RL 阶段的偏好数据是基于旧版模型生成的（off-policy 数据），效果往往不如基于当前模型生成的（on-policy 数据）。这意味着 RL 数据需要持续更新。

理解这四个阶段的数据关系，是做好大模型数据工程的前提。本书的核心组织结构就是按这四个阶段展开，每个部分都会深入到具体的工程细节。

1.3 数据工程的三个层次

无论处于哪个训练阶段，数据工程都可以抽象为三个层次的工作。理解这三个层次有助于你在面对具体问题时，快速定位问题所在的层面。

第一层：采集层——从哪里获取数据？

这是最基础的问题。不同阶段的数据来源差异巨大：

  • 预训练数据：主要来自互联网爬取（Common Crawl 是最大的公共来源），辅以高质量语料（维基百科、书籍、学术论文、代码仓库）。采集层面临的核心问题是覆盖度和合规性——如何获取足够多样的数据，同时遵守版权和隐私法规。

  • 中训练数据：在预训练数据的基础上，重点收集特定领域的高质量数据。比如要增强代码能力，就需要大量高质量的开源代码仓库；要增强长上下文能力，就需要长文档（书籍、法律文本、学术论文）。采集层的核心问题是领域覆盖和知识密度。

  • SFT 数据：来源更加多元——人工标注、模型合成（Self-Instruct、Evol-Instruct）、从已有对话数据中提炼。采集层的核心问题是任务覆盖度和标注成本。

  • RL 数据：主要来自人类偏好标注或自动评估。DeepSeek R1 之后的一个重要趋势是使用可验证奖励（verifiable rewards），这改变了 RL 数据的采集方式——不再需要人类一一比较，而是通过执行验证（代码运行测试用例、数学答案校验）来自动获得奖励信号。

第二层：处理层——如何清洗、过滤、转化？

这是数据工程的”重体力劳动”，也是细节最密集、坑最多的部分。

  • 文本提取：把 HTML 变成干净的纯文本。听起来简单，但不同提取工具（trafilatura、Resiliparse、jusText）在同一个网页上的输出可以差异巨大。导航栏、侧边栏、广告、Cookie 弹窗、JavaScript 渲染内容——每一个都是陷阱。

  • 语言识别：判断一段文本是什么语言。对英文、中文、日文等主流语言来说准确率不错，但对混合语言文本（比如学术论文中的中英混合、代码注释中的多语言）和低资源语言就困难得多。

  • 清洗与过滤：去除噪声数据。包括启发式规则（行级过滤、文档级过滤）和基于模型的过滤（质量分类器、困惑度过滤）。这里的核心两难是：过滤太松，脏数据太多；过滤太严，好数据也被误杀。FineWeb 的研究显示，C4 数据集（GPT-3 的训练数据之一）的一些过滤规则实际上误杀了大量有价值的数据。

  • 去重：消除重复或近似重复的内容。精确去重（hash-based）和模糊去重（MinHash+LSH）各有适用场景。大规模去重（万亿 token 级别）是一个严肃的工程挑战。

  • PII 处理：检测和处理个人身份信息（邮箱、电话、身份证号等），这是合规要求。

第三层：策略层——怎么配比、怎么排课、怎么评估？

这是最体现”工程判断力”的层面，也是最容易被忽视的。

  • 数据配比：预训练阶段，网页文本、书籍、学术论文、代码、数学数据各占多少比例？不同配比可以导致模型在不同能力上的表现差异达到 10% 以上。Llama 3 大幅提升了代码数据的比例，结果不仅代码能力提升了，推理能力也跟着上来了——这个”跨领域溢出效应”是一个重要发现。

  • 课程学习（Curriculum Learning）：数据的排列顺序是否重要？先让模型学简单的再学难的，还是反过来？还是随机打乱最好？目前的研究结论并不一致，但在某些场景下，精心设计的课程确实能带来收益。

  • 数据评估：如何知道你的数据好不好？训练 loss 下降并不等于数据质量好——loss 下降可能只是因为数据有规律性（重复内容）。需要设计消融实验（data ablation），用小模型快速验证数据策略的有效性，再扩展到大模型。

三个层次之间是递进关系：采集层解决”有没有”，处理层解决”干不干净”，策略层解决”用得好不好”。很多团队在采集和处理层投入了大量精力，但在策略层的思考不够。本书的一个重要目标，就是帮助读者在策略层建立系统的思维框架。

1.4 认知演进的时间线

大模型数据工程不是一个静态的领域。在短短几年间，业界对数据的认知经历了多次深刻的转变。理解这些转变，是避免重复前人错误的最好方式。

2020：GPT-3 时代——“数据越多越好”

GPT-3 使用了约 300B token 的混合数据集，包括 Common Crawl（经过基本过滤后约 410B token，在训练中被采样使用）、WebText2、Books1&2、Wikipedia。数据处理相对粗糙——主要是基于 URL 的去重和简单的启发式过滤。

这个时期的核心认知是：规模是第一优先级。数据清洗虽然重要，但只要做到”差不多”就行。Kaplan et al.（2020）的 scaling law 研究更强化了这个认知：模型性能主要取决于模型参数量和训练数据量，数据质量的影响被隐式地忽略了。

2022：Chinchilla——“数据和参数要匹配”

Hoffmann et al.（2022）的 Chinchilla 论文是一个转折点。他们发现，在给定的计算预算下，GPT-3 这样的大模型其实是”欠训练”的——参数太多，数据太少。Chinchilla 法则建议，参数量和训练 token 数应该大致按 1:20 的比例匹配（即 1B 参数需要约 20B token）。

这篇论文的影响是深远的：它让业界意识到，盲目加大模型不是最优策略，还需要确保有足够的训练数据来”喂饱”模型。但后续的实践表明，Chinchilla 法则的系数估计可能有误差——Epoch AI 在 2024 年的复现研究指出，最优的 token-to-parameter 比例可能接近 1:26 而非 1:20。

更重要的是，工业界很快发现了”过度训练”（overtraining）的价值：即使超过了 Chinchilla 最优点继续训练，推理成本的节省（用更小的模型达到相近的性能）往往值得额外的训练计算开支。Llama 3 的 8B 模型使用了 15T token，token-to-parameter 比例高达 1:1875——远超 Chinchilla 建议的 1:20。这意味着：在实际部署中，最优的训练策略不是”计算最优”，而是”推理最优”。

这个转变对数据工程的启示是：我们需要的数据量可能比 Chinchilla 预测的要大得多。数据的获取和质量控制变得更加关键。

2023：LIMA / Phi——“质量远重要于数量”

2023 年出现了两个看似相反但实际互补的发现。

LIMA（Less Is More for Alignment, Zhou et al., 2023）用仅仅 1000 条精心挑选的指令-回复对进行 SFT，就训练出了表现不俗的模型。这提出了”表面对齐假说”（Superficial Alignment Hypothesis）：SFT 阶段需要的信息量远比想象的少，它主要是激活预训练已经学到的知识，并赋予模型合适的输出格式。

Phi-1 / Phi-1.5 / Phi-2（Microsoft，2023）则证明了另一个方向：用极高质量的合成数据（“教科书质量”）训练的小模型，可以在特定任务上打败大得多的模型。Phi-1 只有 1.3B 参数，但在代码生成上超过了很多 10B+ 的模型。

这两个发现的共同启示是：数据质量对模型效果的影响可能被严重低估了。不是所有 token 都平等——一个 token 的”教学价值”可以有数量级的差异。

2024：FineWeb / DCLM——“系统化的数据策略是可复现的科学”

2024 年是大模型数据工程走向科学化的关键一年。

FineWeb（HuggingFace，2024）不仅发布了一个高质量数据集，更重要的是公开了完整的数据管线和详细的消融实验。他们系统性地测试了每一个处理步骤的贡献：

  • 文本提取工具的选择（trafilatura vs resiliparse）——差异可以达到 2-3 个 benchmark 点
  • 去重策略（MinHash 参数选择）——过度去重反而降低了性能
  • 质量过滤策略（教育价值分类器 vs 通用质量分类器）——教育价值分类器带来了显著提升
  • “过度过滤”的代价：他们发现 C4 数据集的一些过滤规则（比如”含有坏词的句子整条删除”）误杀了大量优质内容

DCLM（DataComp for Language Models，Li et al., 2024）则把数据策略的比较做成了一个标准化的竞赛框架：固定模型结构和训练配置，只改变数据，比较不同数据策略的效果。这使得数据工程第一次有了”可控变量实验”的范式。

这一年的核心认知转变是：数据工程不再是”黑魔法”，而是可以系统化、可复现、可优化的工程科学。

2025-2026：合成数据 + 可验证奖励 + 精细化数据管理

进入 2025-2026 年，数据工程的认知继续快速演化：

合成数据从争议走向成熟。早期人们担心合成数据会导致”模型坍缩”（model collapse）——用模型生成的数据训练模型，会导致分布逐渐退化。Nature 上的一篇论文（Shumailov et al., 2024）给出了理论上的警告。但实践表明，混合使用真实数据和合成数据（通常合成数据比例不超过 30%）是安全且有效的。关键在于质量控制和多样性保证。

DeepSeek R1（2025）的出现改变了 RL 数据的范式。它表明，在数学和编程这类有明确正确性标准的领域，可以大量使用可验证奖励（验证答案是否正确、代码是否通过测试用例）来替代人类偏好标注的一部分工作。这意味着：对于有明确正确性标准的任务，RL 数据的构造正在从“主观偏好”转向“客观验证”。

数据管理的精细化成为新趋势。2026 年 3 月发表的综述论文《Towards Next-Generation LLM Training: A Data Perspective》（arXiv: 2603.14712）系统性地总结了数据工程的最新实践，标志着这个领域正在从经验驱动转向体系化。

  • 这不是一本”大模型入门”书。我们假设你已经了解 Transformer 架构、自注意力机制、基本的训练流程。

1.6 写在四个阶段之前

在进入具体的技术章节之前，我想分享一个观点。

大模型的训练数据，在本质上是对人类知识和人类偏好的一种数字化压缩。预训练数据是对人类集体知识的压缩，SFT 数据是对人类期望行为的压缩，RL 数据是对人类价值判断的压缩。数据工程师做的事情，本质上是在决定：模型应该看到人类世界的哪些面？以什么比例？以什么顺序？

这意味着数据工程不仅仅是一个技术问题，它还涉及深层的判断：什么知识更重要？什么行为更合适？什么价值更优先？这些判断会通过数据传递给模型，再通过模型影响数以亿计的用户。

所以，做好数据工程不仅需要技术能力，还需要对”好”与”坏”的深思熟虑。我希望本书不仅帮助你成为一个更好的数据工程师，也帮助你成为一个更审慎的数据决策者。

练习 1：构造”受污染”数据集

用 Python 模拟一个简单的数据污染实验。我们会在一个干净的中文语料中注入不同比例的噪声，然后观察语言模型困惑度（perplexity）的变化。

"""
数据污染敏感性实验
目标：验证 0.1%、1%、5% 的脏数据对 n-gram 语言模型困惑度的影响
"""

import random
import math
from collections import Counter, defaultdict

def create_clean_corpus(n_docs: int = 1000) -> list:
    """模拟干净的中文语料（用主题句模板生成）"""
    topics = [
        "机器学习是一种从数据中自动学习规律的技术方法",
        "深度学习使用多层神经网络来提取数据的层次化特征",
        "自然语言处理旨在让计算机理解和生成人类语言",
        "数据预处理是机器学习流水线中最关键的步骤之一",
        "模型评估需要在独立的测试集上进行以避免过拟合",
    ]
    corpus = []
    for _ in range(n_docs):
        doc = random.choice(topics) + "。" + random.choice(topics) + "。"
        corpus.append(doc)
    return corpus

def inject_noise(corpus: list, noise_ratio: float) -> list:
    """注入指定比例的噪声文档"""
    n_noise = int(len(corpus) * noise_ratio)
    noise_docs = [
        "asdfgh 乱码 #$%^ qwerty !@#$" * 3,
        "クリック ここ 下載 무료 ダウンロード click here",
        "买买买 加微信 免费领取 扫码关注 限时优惠 点击这里",
    ]
    noisy_corpus = corpus.copy()
    for i in random.sample(range(len(noisy_corpus)), n_noise):
        noisy_corpus[i] = random.choice(noise_docs)
    return noisy_corpus

def compute_bigram_ppl(corpus: list) -> float:
    """计算 bigram 语言模型的平均困惑度"""
    # 统计 bigram 频率
    bigram_counts = Counter()
    unigram_counts = Counter()
    for doc in corpus[:800]:  # 前 800 条做训练
        chars = list(doc)
        for c in chars:
            unigram_counts[c] += 1
        for i in range(len(chars) - 1):
            bigram_counts[(chars[i], chars[i+1])] += 1

    # 在剩余数据上计算困惑度
    total_log_prob = 0
    total_tokens = 0
    vocab_size = len(unigram_counts)
    for doc in corpus[800:]:  # 后 200 条做评估
        chars = list(doc)
        for i in range(len(chars) - 1):
            bg = (chars[i], chars[i+1])
            # 加一平滑
            prob = (bigram_counts[bg] + 1) / (unigram_counts[chars[i]] + vocab_size)
            total_log_prob += math.log2(prob)
            total_tokens += 1

    avg_log_prob = total_log_prob / max(total_tokens, 1)
    return 2 ** (-avg_log_prob)

# 实验
clean = create_clean_corpus(1000)
print("噪声比例 → 困惑度")
print("-" * 30)
for ratio in [0.0, 0.001, 0.01, 0.05, 0.10]:
    noisy = inject_noise(clean, ratio)
    ppl = compute_bigram_ppl(noisy)
    print(f"  {ratio*100:5.1f}%  →  PPL = {ppl:.1f}")

"""
四阶段数据量级对比可视化
"""

stages = {
    "预训练":   {"tokens": 15e12, "单条影响": 0.001, "单条成本": 0.00001},
    "中训练":   {"tokens": 500e9, "单条影响": 0.01,  "单条成本": 0.0001},
    "SFT":      {"tokens": 1e6,   "单条影响": 1.0,   "单条成本": 0.1},
    "RL":       {"tokens": 1e5,   "单条影响": 10.0,  "单条成本": 1.0},
}

print(f"{'阶段':<8} {'数据量':>15} {'单条影响力':>10} {'单条成本':>10}")
print("-" * 50)
for stage, info in stages.items():
    tokens = info["tokens"]
    if tokens >= 1e12:
        amount = f"{tokens/1e12:.1f}T tokens"
    elif tokens >= 1e9:
        amount = f"{tokens/1e9:.0f}B tokens"
    elif tokens >= 1e6:
        amount = f"{tokens/1e6:.0f}M 条"
    else:
        amount = f"{tokens/1e3:.0f}K 条"
    print(f"{stage:<8} {amount:>15} {info['单条影响']:>10.3f} {info['单条成本']:>10.5f}")

  1. 微小比例（~0.1%）的脏数据就能导致严重的训练问题——大模型对数据质量的敏感度超出预期
