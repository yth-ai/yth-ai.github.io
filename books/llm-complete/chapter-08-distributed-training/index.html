分布式训练工程

数据并行、模型并行、流水线并行——在数千张 GPU 上训练大模型的工程实践

最后更新：2026年4月17日 分布式训练 数据并行 模型并行 流水线并行 通信

为什么需要分布式训练

一个 70B 参数的模型，仅权重就需要 140GB（BF16 格式）。加上优化器状态（mmm 和 vvv，各占 280GB）、梯度（140GB）和激活值，单卡（即使 80GB H100）完全无法装下。

更重要的是训练时间。以 2T tokens 训练 70B 模型为例：

FLOPs≈6×N×D=6×70×109×2×1012=8.4×1023\text{FLOPs} \approx 6 \times N \times D = 6 \times 70 \times 10^9 \times 2 \times 10^{12} = 8.4 \times 10^{23}FLOPs≈6×N×D=6×70×109×2×1012=8.4×1023

一张 H100 的 BF16 算力约为 990 TFLOPS，但实际 MFU（Model FLOPs Utilization）通常在 40-50%：

时间=8.4×1023990×1012×0.45≈1.89×109 秒≈60 年\text{时间} = \frac{8.4 \times 10^{23}}{990 \times 10^{12} \times 0.45} \approx 1.89 \times 10^9 \text{ 秒} \approx 60 \text{ 年}时间=990×1012×0.458.4×1023​≈1.89×109 秒≈60 年

一张卡需要 60 年。用 2048 张卡，约 11 天。这就是为什么大模型训练必须是分布式的。

数据并行（Data Parallelism）

基本 DP

最简单的并行策略：每张卡都持有完整的模型副本，各自处理不同的数据 batch，然后同步梯度。

gglobal=1K∑k=1Kgkg_{global} = \frac{1}{K}\sum_{k=1}^{K} g_kgglobal​=K1​∑k=1K​gk​

使用 AllReduce 操作在所有 GPU 间同步梯度。

ZeRO（Zero Redundancy Optimizer）

DeepSpeed ZeRO（Rajbhandari et al., 2020）是数据并行的革命性优化。核心思想：DP 中每张卡都存了完整的优化器状态、梯度和参数——这是巨大的冗余。

ZeRO 分三个阶段消除冗余：

阶段分片内容每卡显存（70B 模型）
无 ZeRO全部~1120 GB
ZeRO-1优化器状态~560 GB
ZeRO-2+ 梯度~420 GB
ZeRO-3+ 参数~280/K GB（K 卡）

ZeRO-3 将所有内容均匀分片到 K 张卡上，需要时再通过通信获取。

FSDP（Fully Sharded Data Parallel）

PyTorch FSDP 是 ZeRO-3 在 PyTorch 中的原生实现。核心 API：

from torch.distributed.fsdp import FullyShardedDataParallel as FSDP

model = FSDP(
    model,
    sharding_strategy=ShardingStrategy.FULL_SHARD,
    mixed_precision=MixedPrecision(param_dtype=torch.bfloat16),
    auto_wrap_policy=transformer_auto_wrap_policy,
)

上面展示的是 FSDP1 的 API。PyTorch 2.5+ 引入了 FSDP2，基于 DTensor 重构，带来了几个关键改进：

  • Per-parameter sharding：每个参数独立分片，而非 FSDP1 的 flat_param（将多个参数拼接再分片）。这使得分片边界与参数边界对齐，简化了调试和混合精度配置。
  • 与 TP/CP 原生组合：DTensor 统一了数据并行和张量并行的分片逻辑，FSDP2 + TP 不再需要手动处理两者的交互。
  • 更灵活的内存管理：可以按层控制是否分片，对 MoE 等非均匀结构更友好。

TorchTitan（Liang et al., 2024）是 PyTorch 官方推荐的端到端分布式训练系统，原生集成了 FSDP2、Zero-Bubble PP 和 Context Parallelism，作为统一的 3D+ 并行训练平台。

张量并行（Tensor Parallelism）

基本原理

Megatron-LM（Shoeybi et al., 2020）提出的方法：将模型内部的权重矩阵沿特定维度切分到多张卡上。

MLP 层的切分：

对于 Y=XW1⋅W2Y = XW_1 \cdot W_2Y=XW1​⋅W2​，将 W1W_1W1​ 按列切分，W2W_2W2​ 按行切分：

W1=[W1(1),W1(2)],W2=[W2(1)W2(2)]W_1 = [W_1^{(1)}, W_1^{(2)}], \quad W_2 = \begin{bmatrix} W_2^{(1)} \\ W_2^{(2)} \end{bmatrix}W1​=[W1(1)​,W1(2)​],W2​=[W2(1)​W2(2)​​]

每张卡计算 Y(i)=XW1(i)W2(i)Y^{(i)} = X W_1^{(i)} W_2^{(i)}Y(i)=XW1(i)​W2(i)​，然后做 AllReduce 得到最终结果。

Attention 层的切分：

按 head 维度切分——每张卡计算一部分 attention heads。

通信开销

张量并行每一层都需要两次 AllReduce 通信（前向 + 反向各两次）。因此张量并行只适合在高带宽互连（如 NVLink）的卡之间使用，通常限制在同一节点内的 4-8 张卡。

流水线并行（Pipeline Parallelism）

基本思想

将模型的不同层分配到不同的 GPU 上，形成”流水线”：

GPU 0: Layer 0-7    → GPU 1: Layer 8-15  → GPU 2: Layer 16-23 → GPU 3: Layer 24-31

Bubble 问题

简单的流水线会产生大量的”气泡”（bubble）——某些 GPU 空闲等待上游/下游的数据：

GPU 0: [F1][F2][F3][F4]                    [B4][B3][B2][B1]
GPU 1:     [F1][F2][F3][F4]            [B4][B3][B2][B1]
GPU 2:         [F1][F2][F3][F4]    [B4][B3][B2][B1]
GPU 3:             [F1][F2][F3][F4][B4][B3][B2][B1]
                                ^^^^bubble^^^^

Bubble 比例约为 (P−1)/(P−1+M)(P-1)/(P-1+M)(P−1)/(P−1+M)，其中 PPP 是 pipeline 阶段数，MMM 是 micro-batch 数。

1F1B 调度

PipeDream 提出的 1F1B（One Forward One Backward）调度策略大幅减少 bubble：

GPU 0: [F1][F2][F3][F4][B1][F5][B2][F6][B3][B4][B5][B6]
GPU 1:     [F1][F2][F3][B1][F4][B2][F5][B3][F6][B4][B5][B6]
...

交替执行前向和反向，显著减少了空闲时间。

Zero-Bubble Pipeline

Qi et al. (2024) 提出了零气泡流水线并行，通过更精细的调度几乎消除了 bubble。核心思想是将反向传播拆分为输入梯度计算和权重梯度计算两个阶段，更灵活地调度。

序列并行（Sequence/Context Parallelism）

当序列很长时（如 128K tokens），单卡可能无法装下完整的激活值。序列并行将长序列切分到多张卡上。

Ring Attention

Ring Attention（Liu et al., 2023）是最优雅的序列并行方案：

  1. 将序列切分为 KKK 段，分配给 KKK 张卡
  2. 每张卡计算自己的 Q 与当前 K、V 的 Attention
  3. 以环形方式传递 K、V 给下一张卡
  4. 经过 KKK 轮环形传递后，每张卡都计算了完整的 Attention

通信与计算可以重叠（pipeline），实现近线性扩展。

DeepSpeed Ulysses

DeepSpeed Ulysses 使用 AllToAll 通信在序列维度上分片：

  1. 每张卡持有序列的一部分，但所有 attention heads
  2. AllToAll 交换 → 每张卡持有所有序列，但部分 heads
  3. 计算 attention
  4. AllToAll 交换回来

Expert 并行（MoE 专用）

MoE 模型引入了独特的并行维度——不同专家可以放在不同的 GPU 上。

All-to-All 通信

每个 token 需要路由到其被选中的专家所在的 GPU。这需要 All-to-All 通信：

Token 1 → Expert 3 (GPU 1)
Token 2 → Expert 7 (GPU 3)
Token 3 → Expert 1 (GPU 0)
Token 4 → Expert 5 (GPU 2)

All-to-All 的通信模式不规则，对网络带宽和拓扑敏感。

DeepSeek V3 的 EP 策略

DeepSeek V3 的 256 个专家分布在 8 个节点上，每个节点 32 个专家。为减少跨节点通信：

  1. 限制跨节点路由：每个 token 最多路由到 kkk 个不同节点的专家
  2. 节点内共享专家：1 个共享专家在所有 GPU 上都有副本
  3. 负载均衡 loss：训练时添加辅助 loss 促进均匀路由

通信拓扑与硬件

GPU 互连层次

现代 GPU 集群的互连通常有多个层次：

层次带宽示例
GPU 内存 ↔ GPU 计算~3.35 TB/sHBM3e
GPU ↔ GPU（节点内）900 GB/sNVLink 5.0 (H100)
GPU ↔ GPU（跨节点）400 GbpsInfiniBand NDR
节点 ↔ 节点400-800 GbpsRoCE / IB

并行策略映射：

  • 张量并行 → NVLink 内（节点内，高带宽）
  • 流水线并行 → 跨节点（对延迟敏感但带宽要求低）
  • 数据并行 → 跨节点（AllReduce，通信与计算可重叠）

NCCL

NVIDIA NCCL（NVIDIA Collective Communications Library）是 GPU 集群通信的事实标准。它自动处理 NVLink/PCIe/IB 之间的拓扑感知路由。

训练精度演进

分布式训练的”算力需求”不仅取决于模型大小和数据量，还取决于训练时使用的数值精度。精度越低，同样的硬件可以完成更多计算。

精度GPU 支持显存计算吞吐现状
FP32所有1x基线已基本弃用（仅 loss 计算/优化器状态）
BF16Ampere+0.5x~2x当前主流训练精度
FP8 (E4M3/E5M2)Hopper H1000.25x~4x已被 DeepSeek V3 等模型验证
FP4 (E2M1)Blackwell B2000.125x~7x vs Hopper FP8早期验证中（Quartet）

BF16 混合精度训练已是默认做法——权重和激活用 BF16 计算，优化器状态保持 FP32。

FP8 训练在 Hopper GPU 上成为可能。NVIDIA 的 MXFP8 最佳实践使用 per-tensor scaling factor 保证精度，DeepSeek V3 证明了 FP8 训练在万亿参数规模下的可行性。

FP4 训练是下一个前沿。Quartet（2025）在 Blackwell GPU 上证明 FP4 原生训练可以达到接近 BF16 的效果。但 FP4 训练尚未在工业规模验证——Blackwell 刚开始大规模铺开，真正的 FP4 大规模预训练数据还有待公开。

低通信训练范式

上述所有并行策略都假设一个前提：GPU 之间有高带宽互连（NVLink + InfiniBand）。但如果 GPU 分布在不同的数据中心呢？

DiLoCo 与 DiPaCo

DiLoCo（Distributed Low-Communication）（Google DeepMind，2023）和 DiPaCo（Distributed Path Composition）（Google DeepMind，2024）探索了一个激进的方向——将通信量降低 100-500 倍，使跨数据中心训练成为可能。

DiLoCo 的核心机制：

  1. 每个 worker 在本地用标准 SGD 训练若干步（如 500 步）
  2. 周期性地同步外部优化器梯度（使用 Nesterov momentum）
  3. 同步频率极低——每 500 步一次，通信量仅为每步 AllReduce 的 1/500

DiPaCo 在 DiLoCo 基础上引入模块化路径——模型被拆分为多条路径（path），不同路径可以在不同的数据中心训练。推理时通过路由选择路径，类似 MoE 的思路。

局限性：DiPaCo 目前仅在 1B 规模验证，百亿级以上的大规模实验尚未公开。这条路径的工业化还有距离，但它打开了”互联网级分布式训练”的想象空间。

故障恢复

在数千张 GPU 的集群上训练数天/数周，节点故障是常态而非意外。

检查点恢复：定期保存模型状态、优化器状态、数据位置等。关键是异步检查点保存——保存检查点不应该阻塞训练。

弹性训练：TorchElastic 和 Bamboo 支持节点动态加入/退出，无需重启整个训练任务。
