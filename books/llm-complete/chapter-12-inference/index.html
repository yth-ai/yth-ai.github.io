第 12 章 第四部分：推理与部署

推理优化

从 KV Cache 到推测解码——让大模型跑得又快又省

最后更新：2026年4月17日 推理优化 KV Cache 推测解码 注意力优化 批处理

推理成本的构成

LLM 推理（inference）的成本结构与训练截然不同。训练是一次性投入，而推理是持续运营成本——每一次用户请求都需要实实在在的 GPU 时间。

对于一个 70B 模型，生成一个 token 需要约 2×70×109=1.4×10112 \times 70 \times 10^9 = 1.4 \times 10^{11}2×70×109=1.4×1011 FLOPs。在 H100 上（990 TFLOPS BF16），理论上每秒生成约 7000 tokens——但实际上，由于内存带宽瓶颈，只能做到 30-100 tokens/s。

计算密集 vs. 内存密集

LLM 推理有两个截然不同的阶段：

阶段名称特点瓶颈
Prefill预填充并行处理整个输入序列计算密集
Decode解码逐 token 自回归生成内存带宽密集

Prefill 阶段：输入 nnn 个 token，一次性计算所有 token 的 KV，类似训练的 forward pass。GPU 利用率高。

Decode 阶段：每步只生成 1 个 token，但需要读取整个 KV Cache。计算量很小（O(d)O(d)O(d)），但内存访问量很大（O(n⋅d)O(n \cdot d)O(n⋅d)），GPU 大部分时间在等数据从显存搬到计算单元。

Arithmetic Intensity（算术强度）= FLOPs / Bytes。Decode 阶段的算术强度极低（~1-10），远低于 GPU 的 roofline 拐点（~100-300），因此是内存带宽受限的。

KV Cache：空间换时间

基本原理

自回归生成中，每个新 token 需要与之前所有 token 做 Attention。如果每次都重新计算所有 token 的 K,VK, VK,V，复杂度是 O(n2)O(n^2)O(n2)。

KV Cache 的思路很简单：缓存之前计算过的 K,VK, VK,V，新 token 只需计算自己的 K,VK, VK,V，然后与缓存拼接：

Step 1: K = [k_1], V = [v_1]
Step 2: K = [k_1, k_2], V = [v_1, v_2]        ← k_1, v_1 来自缓存
Step 3: K = [k_1, k_2, k_3], V = [v_1, v_2, v_3]  ← k_1, k_2, v_1, v_2 来自缓存

KV Cache 的显存占用

对于一个典型的 LLM，KV Cache 的大小为：

KV Cache=2×nlayers×nkv_heads×dhead×seq_len×batch_size×dtype_size\text{KV Cache} = 2 \times n_{\text{layers}} \times n_{\text{kv\_heads}} \times d_{\text{head}} \times \text{seq\_len} \times \text{batch\_size} \times \text{dtype\_size}KV Cache=2×nlayers​×nkv_heads​×dhead​×seq_len×batch_size×dtype_size

以 LLaMA 3 70B 为例（80 层，8 个 KV head，128 维，BF16）：

KV Cache 的显存占用在长上下文场景下会超过模型权重本身，成为主要的扩展瓶颈。

KV Cache 压缩

GQA/MQA（见第 2 章）是最直接的压缩手段——减少 KV head 数量。

其他方法：

  • StreamingLLM：只保留 attention sink token + 最近的窗口，丢弃中间的 KV
  • H2O（Heavy-Hitter Oracle）：根据 attention 分数，只保留”重要” token 的 KV
  • SnapKV：在 prefill 阶段识别重要 token，只缓存这些 token 的 KV
  • KV Cache 量化：将 KV 从 FP16 量化到 INT8 甚至 INT4，显存减半或减到四分之一
  • KVTC（KV Cache Transform Coding）（ICLR 2026）：借鉴经典媒体压缩方法，结合 PCA 特征去相关+自适应量化+熵编码。核心洞察：相邻 token 的 KV 向量高度相关，可以用变换编码利用这种空间冗余。实现 4-8x 压缩比且近无损。与 StreamingLLM/H2O 的”选择性丢弃”和量化的”降精度”不同，KVTC 是第三条路径——“利用冗余结构做变换编码”

这些方法可以分为三个流派：选择性保留（StreamingLLM/H2O/SnapKV——丢弃不重要的 token）、降精度（量化——用更少的位存储每个值）、冗余压缩（KVTC——利用 token 间的相关性）。实际部署中往往组合使用。

注意力优化

FlashAttention

FlashAttention（Dao et al., 2022）和 FlashAttention-2 是推理加速的基石。核心思想是IO-aware 算法——利用 GPU 的 SRAM（快但小）和 HBM（慢但大）的层级结构，通过 tiling 减少 HBM 访问：

标准 Attention:
  1. 计算 S = QK^T → 写 HBM (O(n²))
  2. 计算 P = softmax(S) → 读写 HBM (O(n²))
  3. 计算 O = PV → 读 HBM (O(n²))
  总 HBM 访问: O(n² + n·d)

FlashAttention:
  将 Q, K, V 切成 tiles，在 SRAM 中计算注意力
  总 HBM 访问: O(n·d)  ← 减少了 O(n²) 项！

FlashAttention 在推理中的加速效果：

序列长度标准 AttentionFlashAttention-2加速比

8K1.0x2.4x2.4x
32K1.0x3.8x3.8x
128KOOM可运行∞

PagedAttention

PagedAttention（Kwon et al., 2023，vLLM 团队）解决了 KV Cache 的内存碎片问题。

传统方式为每个请求预分配连续显存（按最大序列长度），浪费 60-80% 的 KV Cache 空间。PagedAttention 借鉴操作系统的虚拟内存分页机制：

  • 将 KV Cache 划分为固定大小的 page（如 16 tokens）
  • 使用 page table 管理逻辑→物理映射
  • 按需分配，不浪费

效果：KV Cache 内存利用率从 ~20-40% 提升到 >95%，吞吐量提升 2-4 倍。

MLA 与 FlashMLA

GQA/MQA 通过减少 KV head 数来压缩 KV Cache，但压缩比有上限——最激进的 MQA（只有 1 个 KV head）也只能压缩到 1/H（H 为 query head 数）。

MLA（Multi-Latent Attention）（DeepSeek，2024）走了一条完全不同的路径：将 KV 投影到低秩潜在空间。不减少 head 数量，而是将每个 head 的 KV 通过一个低秩投影压缩到很小的潜在向量。KV Cache 只需要存储这个潜在向量——大小降低约 93%（256 heads → 等效 1 个低维 latent head）。

工程问题在于：MLA 的 decode 阶段需要在线展开潜在向量再做注意力计算，如果实现不好反而会更慢。FlashMLA（DeepSeek 开源）解决了这个问题：

  • FP8 KV Cache + BF16 计算：潜在向量用 FP8 存储，注意力计算用 BF16，兼顾压缩和精度
  • Token 级稀疏 decode：只对活跃 token 计算，跳过 padding
  • 分页 KV Cache：与 PagedAttention 兼容
  • H100 优化：在 H100 上达到 410 GB/s 带宽利用率（decode 场景），接近硬件理论上限

目前 DeepSeek V2/V3/R1 均使用 MLA 架构。MLA 与 GQA 的定位关系：GQA 是”减少副本数”，MLA 是”降维压缩”——两种正交的 KV Cache 瘦身策略。

批处理策略

Continuous Batching（连续批处理）

传统 static batching 的问题：一个 batch 中最长的请求完成前，短请求只能空等。

Continuous Batching（也叫 iteration-level batching，Orca, Yu et al., 2022）：

  • 每个 decode step 检查是否有请求完成
  • 完成的请求立即释放资源，新请求立即填入
  • GPU 利用率从 <30% 提升到 >80%

Prefill-Decode 分离

Splitwise 和 DistServe 提出将 prefill 和 decode 分到不同的 GPU 上：

Prefill GPU (计算密集)          Decode GPU (内存密集)
├── 处理新请求的输入              ├── 逐 token 生成
├── 高 GPU 利用率                ├── 高内存带宽利用率
└── 计算完 KV 后传给 Decode       └── 低延迟响应

这样每种 GPU 可以针对自己的工作负载进行优化。

KVCache 中心架构：Mooncake

Splitwise/DistServe 是”第一代分离”——prefill 和 decode 用不同的 GPU，但 KV Cache 仍然是 GPU 上的临时数据。Mooncake（Moonshot AI，FAST 2025 / CACM 2025）将分离推进到了一个新层次：KV Cache 不再是附属于 GPU 的缓存，而是独立的分布式存储服务。

Mooncake 的三层架构：

Prefill 节点                KVCache Store              Decode 节点
├── 处理输入序列          ← RDMA 传输 →         ├── 逐 token 生成
├── 生成 KV Cache          ├── 多副本存储          ├── 按需拉取 KV
└── 写入 Store             ├── 自动淘汰策略        └── 低延迟响应
                           └── SSD 卸载扩展

关键设计决策：

  • KVCache Store 是独立的分布式存储层，支持 RDMA 跨节点传输，不依赖特定 GPU
  • 多副本：热门前缀（如系统提示）的 KV Cache 自动创建多副本，避免成为瓶颈
  • SSD 卸载：冷 KV Cache 可以卸载到 SSD，显存只保留热数据
  • 解耦扩缩容：Prefill 节点、Store 节点、Decode 节点可以独立扩缩

在 Kimi 的长上下文场景（128K+）下，Mooncake 相比传统架构有效请求容量提升 59%~498%（取决于负载特征）。这代表了推理架构从”GPU 中心”到”KV Cache 中心”的范式转变——当上下文越来越长，KV Cache 的管理效率比 GPU 算力更决定系统性能。

推测解码（Speculative Decoding）

基本思想

推测解码（Leviathan et al., 2022; Chen et al., 2023）是近年最巧妙的推理加速技术之一：

  1. 用一个小的 draft 模型（如 1B）快速生成 γ\gammaγ 个候选 token
  2. 将这 γ\gammaγ 个 token 送入大的 target 模型一次性验证
  3. 接受匹配的 token，拒绝不匹配的，从拒绝位置重新采样

关键性质：推测解码在数学上保证与直接用大模型生成的分布完全一致——不损失质量，只加速。

接受率分析

设 draft 模型在位置 ttt 生成 token xxx 的概率为 q(x)q(x)q(x)，target 模型为 p(x)p(x)p(x)，则接受概率为：

P(accept)=min⁡(1,p(x)q(x))P(\text{accept}) = \min\left(1, \frac{p(x)}{q(x)}\right)P(accept)=min(1,q(x)p(x)​)

平均每次验证接受的 token 数为：

E[accepted]=∑i=1γ∏j=1iαj\mathbb{E}[\text{accepted}] = \sum_{i=1}^{\gamma} \prod_{j=1}^{i} \alpha_jE[accepted]=∑i=1γ​∏j=1i​αj​

其中 αj\alpha_jαj​ 是第 jjj 个位置的平均接受率。draft 模型越好（与 target 分布越接近），接受率越高。

内核级优化

Operator Fusion

将多个顺序操作融合为一个 GPU kernel，减少内存读写：

Before: LayerNorm → Linear → SiLU → Linear
  (4次 kernel launch, 4次 HBM 读写)

After: FusedFFN(LayerNorm + Linear + SiLU + Linear)
  (1次 kernel launch, 1次 HBM 读写)

CUDA Graph

通过 CUDA Graph 预录制一系列 GPU 操作，避免每步的 CPU→GPU 调度开销。对 decode 阶段（计算量小、launch overhead 占比高）特别有效。

量化推理

在推理时使用低精度计算：

精度类型显存速度质量
FP16/BF16标准1x1x基线
INT8 (W8A8)全量化0.5x~1.5x几乎无损
INT4 (W4A16)仅权重量化0.25x~1.8x轻微损失
FP8 (E4M3)H100 原生支持0.5x~1.8x几乎无损
NVFP4 (E2M1)Blackwell 原生支持0.25x~2.5xper-block scaling，几乎无损

NVFP4 的出现改变了大模型部署的经济学——结合 FP4 权重 + FP4 KV Cache + MLA，单张 B200（192GB）可以部署 70B 模型的全精度推理，而之前需要 2 张 H100。

推理框架

vLLM

vLLM 是目前最流行的开源推理框架：

  • PagedAttention：高效 KV Cache 管理
  • Continuous Batching：动态批处理
  • Tensor Parallelism：多卡推理
  • Quantization：支持 GPTQ、AWQ、FP8
  • OpenAI-compatible API：兼容 OpenAI API 格式

TensorRT-LLM

TensorRT-LLM（NVIDIA）通过编译优化实现极致性能：

  • 图优化：算子融合、常量折叠
  • FP8 支持：在 H100 上充分利用 FP8 Tensor Core
  • In-flight Batching：NVIDIA 版连续批处理
  • 量化：SmoothQuant、INT8/FP8 KV Cache

SGLang

SGLang（LMSYS 团队）在 2025-2026 年迅速崛起，在多个场景下吞吐量超过 vLLM：

  • RadixAttention：自动 KV Cache 复用（前缀共享），前缀密集型负载下比 vLLM 快 6.4x
  • Constrained Decoding：JSON schema、正则表达式约束输出
  • Fork/Join 并行：多个生成分支并行执行
  • Disaggregated Serving：Prefill 和 Decode 阶段分离到不同的 GPU 集群

2026 格局：SGLang 在 H100 上实测 16,200 tok/s vs vLLM 的 12,500 tok/s（Llama 3.1 70B），特别在结构化输出和前缀缓存场景下优势明显。但 vLLM 的硬件支持更广（TPU、Trainium 等）。

llama.cpp

llama.cpp 是纯 C/C++ 实现的 CPU/GPU 推理引擎：

  • GGUF 量化格式：支持 Q2 到 Q8 多种精度
  • 跨平台：macOS (Metal), Linux (CUDA/ROCm), Windows
  • 消费级硬件友好：MacBook 就能跑 70B 模型（量化后）

章节小结

  1. LLM 推理的核心瓶颈是 decode 阶段的内存带宽，不是计算
  2. KV Cache 是空间换时间的核心机制，管理好它（PagedAttention、压缩、量化）是关键
  3. 推测解码 是唯一”免费午餐”——加速但不损失质量
  4. Continuous Batching 将 GPU 利用率从 <30% 提升到 >80%
  5. 实际部署中，上述技术组合使用，往往能实现 5-10 倍的成本降低
